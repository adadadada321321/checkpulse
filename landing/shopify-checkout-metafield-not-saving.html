Shopify Checkout Metafield Not Saving — Line Item Properties, Custom Fields & Cart Attributes

Checkout custom data · Data persistence · Shopify Admin API · Checkout Extensibility · Updated June 2026
Shopify has three parallel custom-data systems at checkout: metafields (structured, namespace/key, API-managed), line item properties (cart-level key-value pairs attached to a specific product), and cart attributes (order-level note fields). Each travels through checkout differently, requires a different API to write and read, and fails silently when you use the wrong one. The most common mistake: treating metafields as if they can be written at checkout from client-side JavaScript. Order metafields don't exist until after the order is created — they can only be written server-side via the Admin API.
Architecture

Shopify's Three Custom-Data Systems at Checkout

Before diagnosing why data isn't persisting, you need to know which system you're using and where it lives in the order lifecycle. All three look similar in code but differ fundamentally in scope, timing, and API access.

System 1

Line Item Properties

Key-value string pairs attached to a single cart line item. Set via properties[key]=value in product forms or the AJAX Cart API. Travels with the line item through checkout. Visible in admin under Orders → line item. Cannot be queried via metafields API.

System 2

Cart Attributes

Order-level custom fields. Two types: note (single string, "Note from customer" in admin) and attributes (key-value pairs on the cart object). Set via /cart/update.js or Storefront API. Persists as order.note and order.note_attributes on the order. Not metafields.

System 3

Metafields

Structured typed data attached to a Shopify resource (order, product, variant, customer, collection). Defined with namespace + key + type. Written via Admin API (POST /metafields.json) or GraphQL. Order metafields can only be written server-side after the order exists. Not writable at checkout from the browser.

System 3a (Plus)

Checkout Extensibility Metafields

Shopify Plus only. Checkout UI Extensions can write CartLine metafields (attached to a line item within the checkout session) via the applyCartLinesChange API. These become order line item properties post-purchase. Customer metafields can be written via UI extensions with write_customers scope.

The fundamental timing issue

Metafields live on the order. Orders don't exist until checkout completes. This means you cannot write an order metafield during checkout — only after it. The correct pattern: orders/create webhook fires → your app writes metafields via Admin API. Any solution that tries to write order metafields "at checkout" from client-side code will fail silently.

Root Causes

Five Reasons Custom Fields Don't Persist Through Shopify Checkout

RC1

Using metafield APIs for cart-level data

The most common mistake: a developer writes to POST /admin/api/metafields.json with resource_type: "order" expecting the data to appear on the checkout, but the order doesn't exist yet. The API call either fails (no order ID to attach to) or writes to a draft order that never gets linked to the live order. The fix: use line item properties for per-product custom data, or cart attributes for order-level notes. Reserve metafields for structured data written by a webhook after the order is placed.

RC2

Buy Now button bypasses line item properties

Line item properties are set on the cart. The Shopify Buy Now button creates a direct checkout session, bypassing the cart entirely. Any JavaScript that adds properties to a cart line item after the product is added (e.g., via /cart/change.js or a cart-drawer script) will not run for Buy Now purchases. Only properties embedded in the product form's name="properties[key]" inputs are included when the Buy Now button submits the form — and only if the button is inside that same form.

RC3

Metafield type mismatch causes silent validation failure

Shopify's metafield type system enforces types at write time. Writing a string like "15" to a metafield defined as type number_integer returns HTTP 422 with a validation error. Writing a multi-line string to single_line_text_field fails if the value contains newlines. Writing a JSON object to a non-json type fails. These errors are often swallowed by apps that check only for HTTP 200 from the outer request, not the inner validation message. The metafield appears to save but nothing is persisted. Fix: always validate the metafield type definition before writing, and check the errors object in the response body.

RC4

Draft order metafields not copied to live order

When an agency or app creates a draft order (B2B quote flow, custom product configurator, invoice tool) and writes metafields to the draft, those metafields live on the DraftOrder resource — not on the Order resource that gets created when the draft is paid. Shopify does not automatically copy draft order metafields to the live order. The app must re-write the metafields via POST /admin/api/orders/{order_id}/metafields.json after the draft_orders/completed webhook fires.

RC5

Namespace collision between apps

Two apps using the same namespace.key pair silently overwrite each other. App A writes custom.delivery_date = "2026-07-01". App B writes custom.delivery_date = "standard". Last write wins — no conflict error, no notification. The data from App A disappears. Fix: apps must use unique namespaces (typically their app name or a UUID prefix) and document all namespace.key pairs used. If you manage multiple apps writing to the same order, audit GET /admin/api/orders/{id}/metafields.json to see all current values and who wrote last.

Silent Failures

Seven Silent Failure Modes

These failures produce no error at checkout. Data simply disappears or arrives wrong.

Engraving text visible in cart, absent from order

Line item property set via JavaScript after add-to-cart. Buy Now button creates fresh checkout session, JavaScript never runs, property not attached.

Gift message missing from order notification

Cart attribute set correctly, arrives on order as note_attributes. But order confirmation Liquid template doesn't include the note_attributes loop — visible in admin, invisible to merchant.

Metafield write returns 200 but value missing

Type mismatch triggers a validation error inside the response body. App only checks HTTP status code. Value silently rejected, metafield not updated.

Draft order data lost at payment

B2B configurator writes metafields to draft. Customer pays, draft becomes order. Metafields not re-written to live order — they exist on the draft only.

Confirmation email sends before webhook fires

Webhook handler writes order metafields asynchronously. Email sends within seconds of order creation — metafields don't exist yet. Merchant sees no custom data in the notification.

Underscore-prefixed property hidden from customer

Properties with keys starting with _ (e.g., _tracking_id) are hidden from the customer-facing order status page and some email templates. Still exist on the order — just not visible to customers. Often confused with "missing" data.

Storefront metafield not accessible — storefront visibility off

Product/variant metafields must have storefront_access: true (or be listed in the Headless channel's accessible metafields) to be readable via Storefront API or in themes. A product metafield defined via Admin API without storefront visibility set returns empty at checkout even though the value exists.

Diagnostics

How to Diagnose Where Your Data Is Being Lost

If you're a merchant or developer

Identify which system you're using. Is the data per-product (→ line item properties), order-level unstructured (→ cart attributes / note), or structured typed data needed post-order (→ metafields)? Mixing them up is the root cause in most cases.

Check if line item properties exist on the cart. Open your browser console and run fetch('/cart.js').then(r=>r.json()).then(d=>console.log(d.items.map(i=>i.properties))). If properties are present in the cart but absent on the order, the issue is Buy Now bypass or a cart-clearing operation at checkout redirect.

Verify the order in Shopify admin. Go to Orders → open the order → scroll to each line item — properties appear as indented key-value pairs under the product name. Also check the note_attributes section near the bottom of the order page for cart attributes.

Check metafields via the API. Run GET /admin/api/2024-01/orders/{order_id}/metafields.json with your Admin API token. If the metafield exists but has the wrong value, you have a namespace collision. If it doesn't exist at all, the write failed — check your webhook handler logs for the orders/create event.

Test the Buy Now path explicitly. Add the product to cart via the normal form, then also test using the Buy Now button. Compare line item properties on both resulting orders. If properties are present in the first but absent in the second, the issue is Buy Now bypassing your property-injection script.

Check metafield type definitions. In Shopify admin → Settings → Custom data → view your metafield definitions. Compare the defined type against what your app is writing. A common mismatch: app writes JSON string to a single_line_text_field, or writes a float to a number_integer. Fix the type definition or the value format — not both at once.

Deep Dive B

Shopify Metafield Type System — Common Type Mismatch Errors

Shopify enforces metafield types at write time. The full list of available types includes:

Type Accepts Common Mistake
single_line_text_field String, no newlines Writing multiline text or JSON string — fails if value contains \n
multi_line_text_field String with newlines Correct for long notes; don't use for structured data
number_integer Integer only, no decimals Writing "15.5" or "15" (string) — 422 validation error
number_decimal Decimal number as string Writing a raw JS number instead of a string — type coercion issues
boolean "true" or "false" as string Writing JavaScript true (boolean, not string) — fails
date ISO 8601 date string YYYY-MM-DD Writing "June 19, 2026" or Unix timestamp — fails validation
date_time ISO 8601 with time Missing timezone offset — Shopify requires full ISO string
json Any valid JSON string Passing a JavaScript object instead of JSON.stringify(obj) — writes [object Object]
color Hex color string #RRGGBB Writing "red" or rgb(255,0,0) — fails
url Full URL with protocol Writing relative path /products/handle — fails, needs https://
How to catch type validation errors

When Shopify rejects a metafield write due to type mismatch, the HTTP status is 422 (Unprocessable Entity). The response body contains an errors object with the field and message. Example: {"errors":{"value":["is invalid"]}}. Most app frameworks log the status code but not the response body — add explicit body logging to your metafield write path to catch these silently-swallowed errors.

Deep Dive C

Checkout Extensibility and Metafields (Shopify Plus)

For Shopify Plus merchants, Checkout Extensibility opens additional options for reading and writing custom data during the checkout flow. But the scope is more limited than it appears.

What Checkout UI Extensions can do with custom data

Checkout UI Extensions (Plus) can: (1) Read product and variant metafields if they have storefront access enabled; (2) Write CartLine attributes (equivalent to line item properties, visible on the resulting order's line items); (3) Display customer metafields if the app has write_customers scope; (4) Capture custom fields (text inputs, dropdowns) and store values as CartLine attributes or note_attributes. They cannot write order metafields directly — those still require a post-order webhook handler.

CartLine attributes vs CartLine metafields (Shopify Functions)

CartLine attributes (set via applyCartLinesChange with type: "updateCartLine", attributes: [{key, value}]) become line item properties on the order. CartLine metafields (a different concept, set via Shopify Functions at the cart/checkout level) are more structured but require a Cart Transform Function with appropriate permissions. For most use cases, CartLine attributes are sufficient and simpler to implement.

Storefront access requirement for reading metafields at checkout

Product, variant, and collection metafields must have storefront access enabled to be readable by Checkout UI Extensions or the Storefront API. To enable: Shopify Admin → Settings → Custom data → [your metafield definition] → toggle "Storefront access" on. Alternatively, via API: include "access": "PUBLIC_READ" in the metafield definition. Without this, the metafield value returns null at checkout even though the data exists — a silent failure that looks like a missing value.

Order note_attributes from Checkout Extensibility

Checkout UI Extensions can write to the order's note_attributes by appending to the checkout's attributes via applyAttributeChange. These appear on the order as note_attributes (same as cart attributes written via /cart/update.js). This is the recommended approach for capturing order-level custom data during Plus checkout — it doesn't require a post-order webhook, is visible immediately in Shopify admin, and accessible in Liquid notification templates via order.note_attributes.
