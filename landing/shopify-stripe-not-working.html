Shopify Stripe Not Working — Payments Failing, Not Processing, Charges Missing (2026)

Stripe payments declining at checkout, charges not appearing in your Stripe dashboard, or the Stripe connection silently broken — here are all 8 failure modes for Stripe breaking on Shopify, with exact diagnostic steps and fixes for each.

Updated June 2026 8 failure modes API Keys · Test Mode · Account Restricted · Radar Rules · SCA/3DS · Webhooks · Currency Mismatch · Transaction Fees

Stripe is the most popular third-party payment gateway on Shopify, but its integration is more complex than native Shopify Payments. Where Shopify Payments is a first-party integration that Shopify manages end-to-end, Stripe on Shopify is a third-party gateway connected via API keys — which means more configuration points, more places to silently break, and a harder path to diagnosing what went wrong when payments fail.

Stripe failures on Shopify fall into two categories: connection failures (the gateway can't process any payments) and transaction failures (the connection works but specific payments are declined). Both fail without clear error messages — the customer sees a generic "payment failed" or simply gets no confirmation, and the merchant sees an incomplete order with no explanation of why Stripe rejected it.

Quick Check: Test Mode vs Live Mode
The single fastest diagnostic: go to your Stripe dashboard → top-right toggle → check if "Test mode" is active. If you see "Test mode" highlighted in orange, your Stripe account is in test mode. Charges processed in test mode appear only in Stripe's test data view and collect no real money — even though they show as successful in Shopify's order system. This is the most common reason "Stripe is working but I'm not getting paid."

Failure Mode 1: Wrong or Expired API Keys (Test Keys in Production)

Shopify connects to Stripe using two API keys: a publishable key (used in the browser to tokenize card data) and a secret key (used server-side to create charges). Both keys exist in two versions — test keys (prefixed pk_test_ and sk_test_) and live keys (prefixed pk_live_ and sk_live_). Using test keys in a live Shopify store is the most common Stripe configuration error.

Keys can also become invalid if you rotate them in Stripe (which Stripe recommends periodically for security). If you generate new API keys in Stripe without updating them in Shopify, the connection breaks immediately and all payments fail with a generic authentication error.

How to verify and fix your API keys

  1. Go to Shopify Admin → Settings → Payments → Third-party payment providers
  2. Find Stripe in your active payment providers and click Edit
  3. Note whether the publishable key shown starts with pk_test_ (wrong) or pk_live_ (correct)
  4. Go to your Stripe Dashboard → Developers → API keys
  5. Copy the live publishable key and live secret key (click "Reveal test key" → switch to live first if needed)
  6. Paste both live keys back into Shopify's Stripe payment provider settings and save
  7. Test with a real card for a small amount to confirm live charges process correctly
Never Share Your Secret Key
Your Stripe secret key (sk_live_...) gives full API access to your Stripe account including the ability to issue refunds, create charges, and modify your account. It belongs only in Shopify's payment settings server-side — never in your theme code, never in a JavaScript file, never in a public GitHub repo. If you accidentally expose your secret key, go to Stripe Dashboard → Developers → API keys and roll (rotate) it immediately.

Failure Mode 2: Stripe Account Still in Test Mode

Stripe accounts start in test mode when created. In test mode, every Stripe API call processes successfully from a technical standpoint — Shopify creates orders, sends confirmation emails, and marks payments as paid. But no actual money moves. Test charges are fake. If you set up Stripe in test mode and forgot to activate your live account, you may have processed dozens of "orders" that collected zero revenue.

This is the most financially damaging failure mode because it produces no error signals — everything looks normal. Customers receive order confirmations. Shopify shows paid orders. The only indication is that your Stripe dashboard shows $0 in live revenue, or that your bank account receives no deposits.

How to activate your Stripe account for live payments

  1. Log into your Stripe Dashboard
  2. Look for the banner "Activate your account" or the orange "Test mode" toggle in the top right
  3. Click Activate account and complete Stripe's verification process
  4. Provide: business type, business address, business description, bank account details, and identity verification (SSN last 4 for US individuals, or company registration details for businesses)
  5. Once activated, Stripe generates your live API keys
  6. Replace your test keys in Shopify with the live keys (see Failure Mode 1 above)
What to Do About Past Test-Mode Orders
If you've been in test mode and collected "orders" that weren't actually charged, you have two options: (1) manually contact those customers to collect payment through another method, or (2) fulfill orders for existing customers as a goodwill gesture and ensure your store is properly set up going forward. Shopify will not automatically re-charge customers for orders processed with test API keys — each affected customer would need to re-purchase or be invoiced separately.

Failure Mode 3: Stripe Account Restricted or Paused

Stripe monitors accounts for compliance, fraud signals, and regulatory requirements. If Stripe flags your account — due to an unusually high dispute rate, suspicious transaction patterns, an identity verification request you haven't completed, or operating in a restricted business category — your account enters a restricted state where new charges are declined or payouts are paused.

Unlike test mode, restricted mode usually produces actual error messages at checkout ("Your payment could not be processed"). But the error looks identical to a card decline, making it hard to distinguish from a customer card issue. The definitive indicator is that every payment fails, not just some.

Stripe account restriction patterns and symptoms

Restriction Type Symptom Where to Check Resolution
Identity verification required Charges process but payouts blocked Stripe Dashboard → Home → Action required banner Complete verification — submit ID and business docs
High dispute rate (>1%) New charges may be declined; account under review Stripe Dashboard → Radar → Disputes Respond to open disputes; implement fraud prevention

Unusual transaction patterns Individual charges flagged; manual review triggered Stripe Dashboard → Radar → Reviews Resolve reviews; adjust Radar rules; contact support
Stripe vs Shopify Payments for High-Risk Categories
If your business is in a category Stripe considers elevated-risk (supplements, CBD, adult content, firearms accessories, high-ticket items), Stripe may restrict or close your account without warning. Consider whether Shopify Payments (which has its own similar restrictions but a different enforcement threshold) or a specialized high-risk payment processor is a better fit for your business before building on Stripe.

Failure Mode 4: Stripe Radar Blocking Legitimate Cards

Stripe Radar is Stripe's machine-learning fraud detection system that evaluates every transaction before it's authorized. Radar uses signals from across the Stripe network — not just your store — to identify potentially fraudulent transactions. A card that seems valid and belongs to a real customer can be blocked by Radar if it matches patterns associated with fraud on other Stripe merchants.

Radar blocks are invisible to merchants at first glance. The customer sees "Your card was declined" — the same message they'd get for an insufficient funds error. But the decline code in your Stripe dashboard tells a different story: codes like card_declined with a reason of fraudulent, or transactions appearing in your Radar → Reviews queue, indicate Radar intervention rather than a bank-level decline.

Common Radar block triggers

  • Card previously used in fraud: Cards that have been used fraudulently on other Stripe merchants are flagged network-wide. A legitimate customer using a card that was previously compromised (without their knowledge) may be blocked.
  • Billing/shipping address mismatch: When billing address doesn't match the card's registered address, Radar elevates the risk score. Customers entering a work address or a new address they haven't registered with their bank are common victims.
  • VPN or proxy usage: Customers using VPNs (common for privacy-conscious shoppers) may be blocked if their apparent IP country doesn't match their card's issuing country.
  • Velocity rules: Multiple purchases in a short time from the same IP, same card, or same email — even by a legitimate customer making bulk purchases — can trigger velocity-based blocks.
  • High-risk country IP: IP addresses from countries Radar considers elevated-risk may be blocked even for legitimate transactions originating there.
  • New card or card bin: Cards from newly issued bins or prepaid card bins trigger higher risk scores by default.

Diagnosing Radar blocks

  1. Go to Stripe Dashboard → Radar → Reviews — transactions that Radar flagged for manual review appear here
  2. Go to Stripe Dashboard → Payments and filter for "Failed" — click any failed payment to see the decline code and decline reason
  3. Look for decline reason fraudulent or card_velocity_exceeded — these are Radar, not bank, blocks
  4. If a legitimate customer is being blocked, you can allowlist their email or card in Radar → Rules → Allowlist
  5. For systematic issues, review your Radar rules in Radar → Rules and consider relaxing rules that produce false positives — but do this carefully, as loosening rules increases fraud exposure
Do Not Disable Radar to Fix Decline Rates
Disabling or significantly loosening Stripe Radar to improve authorization rates is a risky tradeoff. Radar blocking 2–3% of legitimate transactions may seem costly, but removing those blocks typically results in a proportionally larger increase in fraudulent transactions, chargebacks, and the elevated dispute rate that triggers Stripe account restrictions. Fix specific false-positive patterns with targeted allowlisting, not blanket rule disabling.

Failure Mode 5: SCA / 3D Secure Authentication Failures for EU Customers

The EU's Payment Services Directive 2 (PSD2) and the UK's equivalent regulation require Strong Customer Authentication (SCA) for most online card payments from EU/EEA and UK cardholders. SCA is implemented via 3D Secure (3DS) — an additional authentication step where the customer approves the payment through their bank's app or via SMS OTP before the charge is authorized.

Stripe handles 3DS automatically for most SCA-required transactions. However, if your Shopify store's checkout integration is using an outdated Stripe.js version, or if your checkout theme has JavaScript that conflicts with Stripe's 3DS redirect flow, European customers will hit authentication errors and fail to complete payment. From the customer's side, this appears as the payment being declined after they've entered their card details.

SCA failure patterns by symptom

Symptom Likely Cause Fix
EU customers declined, US customers fine SCA/3DS flow not completing correctly Ensure Stripe integration uses latest Stripe.js; check for JS conflicts in theme
"Authentication required" error at checkout 3DS triggered but redirect not initiated Update Stripe app or reconnect; contact Shopify support for Stripe integration version
Payment shows as pending then fails 3DS popup blocked by browser or popup blocker Advise customers to disable popup blockers; Stripe's hosted 3DS page is a redirect, not a popup, but some integrations use iframes
Works in Stripe test mode (test 3DS cards) but fails live Bank not enrolled in 3DS or using older 3DS 1.0 Stripe handles 3DS version negotiation — contact Stripe support if specific bank's cards consistently fail

SCA exemptions Stripe can apply

Not every EU transaction requires 3DS — Stripe can request SCA exemptions for low-value transactions (under €30), merchant-initiated transactions (subscriptions), and transactions where the risk score is very low. These exemptions reduce checkout friction for EU customers without compromising compliance. Review your Stripe Dashboard → Settings → Radar to ensure exemption settings are configured appropriately for your transaction mix.

Failure Mode 6: Stripe Webhook Endpoint Misconfigured

Webhooks are HTTP callbacks that Stripe sends to your Shopify store to notify it about payment events: successful charges, refund completions, dispute notifications, and subscription updates. If the Stripe webhook endpoint in your Shopify store is misconfigured, outdated, or returning errors, these events never reach Shopify — which means refunds don't process correctly, disputes create orphaned charges, and order status updates break.

Webhook failures don't prevent initial payment processing — the charge succeeds at Stripe's end. The failure appears later: refunds initiated in Shopify don't reach Stripe, or vice versa; Stripe dispute alerts never arrive in Shopify admin; or subscription renewal payments don't update Shopify order status.

How to verify and fix Stripe webhooks

  1. Go to Stripe Dashboard → Developers → Webhooks
  2. Find the endpoint registered for your Shopify store (it will be a Shopify URL like https://your-store.myshopify.com/stripe/webhooks or similar)
  3. Click on the endpoint and review the Recent deliveries section — look for failed deliveries (red) and their HTTP response codes
  4. Common failure codes: 401 (authentication failed — webhook secret mismatch), 404 (endpoint not found — URL changed), 500 (Shopify processing error)
  5. If using Shopify's native Stripe integration (through Shopify's app store), remove and reinstall the Stripe payment provider to reset the webhook registration
  6. If using a custom integration, update the webhook secret in both Stripe and your integration code
Stripe Webhook Signing Secrets vs API Keys
Stripe webhooks use a separate signing secret (prefixed whsec_) that is different from your API secret key. The signing secret lets your integration verify that incoming webhook requests genuinely come from Stripe, not from a malicious third party. If you rotate your API keys, the webhook signing secret remains valid unless you explicitly rotate it. Conversely, rotating the webhook signing secret doesn't invalidate your API keys. Check both independently when debugging webhook failures.

Failure Mode 7: Store Currency Not Supported by Stripe Account

Stripe supports over 135 currencies for payment processing, but your specific Stripe account has a settlement currency based on your country and banking setup. When your Shopify store's currency doesn't align with what Stripe can process for your account, charges fail or are automatically converted — sometimes at unfavorable rates that erode margins without explanation.

The most common scenario: a merchant in the UK sets up a Shopify store with USD as the display currency to target US customers, but their Stripe account settles in GBP. Currency conversion happens at Stripe's exchange rate (which includes a 2% conversion fee on top of Stripe's standard processing rate), and the merchant is surprised that their effective processing cost is much higher than expected.

Stripe currency support by account type

Stripe Account Country Default Settlement Currency Multi-Currency Payouts Available Notes
United States USD Yes Can add bank accounts in other currencies for multi-currency settlement
United Kingdom GBP Yes EUR, USD payouts available with additional bank accounts
European Union EUR Yes SEPA bank accounts supported; non-EUR currencies converted
Australia AUD Limited USD and EUR charges processed but converted to AUD for settlement
Canada CAD Limited USD charges settled in USD with USD bank account; others converted

When currency causes payment failures (not just conversion)

Some currencies that Stripe lists as "supported" are only available for payment acceptance, not settlement. If your Shopify store presents prices in a currency that Stripe cannot process for your specific account and country combination, charges will fail at the gateway level — not just incur a conversion fee. To verify: go to Stripe Dashboard → Settings → Bank accounts and currencies and confirm that the currencies your Shopify store uses are listed as processable.

Failure Mode 8: Shopify Transaction Fees Making Revenue Disappear

This isn't a payment failure in the traditional sense — payments process correctly. But many Shopify merchants using Stripe report that their revenue is "lower than expected" or that charges on the Stripe dashboard don't match payouts to their bank. The culprit is almost always Shopify's transaction fee, which is charged on top of Stripe's processing fee when Stripe is used as a third-party gateway instead of Shopify Payments.

Shopify deducts its transaction fee from your payout before the money reaches your bank. This deduction doesn't appear on your Stripe dashboard (which shows the full charge amount minus Stripe's fee), but it does reduce your actual payout. The effective total cost per transaction is Stripe's fee plus Shopify's fee — which can reach 4.9% + 30¢ on the Basic Shopify plan.

When to Use Stripe vs Shopify Payments
Use Shopify Payments if it's available in your country and your business type qualifies — it eliminates Shopify's transaction fee and integrates more deeply (instant payouts, Shop Pay, Afterpay, Buy now pay later). Use Stripe when Shopify Payments isn't available in your country, when you need Stripe-specific features (Connect for marketplace payouts, Billing for subscriptions, advanced Radar rules), or when you have existing Stripe infrastructure you're integrating with Shopify.

When a Stripe charge fails, the API returns a decline code that identifies the specific reason. These codes appear in your Stripe dashboard under the failed payment and are more informative than the generic error message shown to the customer. Here are the most common codes Shopify merchants encounter:

insufficient_funds Card has insufficient balance Customer needs to add funds or use different card

Stripe vs Shopify Payments: Which Should You Use?

Many of the failure modes above (API key confusion, test/live mode, webhook maintenance) are consequences of using Stripe as a third-party gateway instead of Shopify's native payment solution. Whether Stripe is the right choice depends on your specific needs:

Use Shopify Payments if:

  • Your country supports Shopify Payments (US, UK, Canada, Australia, many EU countries)
  • Your business type is eligible (Shopify Payments excludes certain high-risk categories)
  • You want BNPL options (Afterpay, Klarna, Shop Pay Installments) natively
  • You want to avoid Shopify transaction fees
  • You don't need Stripe-specific features like Connect or complex subscription billing

Use Stripe directly if:

  • Shopify Payments isn't available in your country
  • Your business category is rejected by Shopify Payments but accepted by Stripe
  • You're building a marketplace and need Stripe Connect for split payouts
  • You need Stripe's advanced subscription billing (Stripe Billing)
  • You have existing Stripe infrastructure (e.g., a SaaS product) that needs to integrate with your Shopify store
  • You need Stripe's more granular Radar rule customization for high-volume fraud management

Note that Shopify Payments uses Stripe as its underlying processor in the US — so the fraud protection and card network relationships are the same. The difference is integration depth, fee structure, and feature access.
