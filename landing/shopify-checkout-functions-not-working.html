Shopify Functions Not Working — 8 Failure Modes Explained

Discount function not applying? Shipping function not showing rates? Validation function throwing a generic checkout error? Function deploys fine but does nothing on the store? Here are all 8 root causes for Shopify Functions failures — for merchants and developers.

Updated June 2026 8 failure modes Discount API · Shipping Functions · Validation API · Scripts Migration · WebAssembly · run.graphql

Shopify Functions are the replacement for Shopify Scripts — WebAssembly-based logic that runs inside Shopify's checkout to customize discounts, shipping rates, payment methods, and order validation. Unlike Scripts, Functions require a completely separate development and deployment workflow: Shopify CLI, a Partners account, an app that contains the function, and explicit installation of that app on the target store. The gap between "function is deployed" and "function is active on the store" is where most failures live.

A deployed function that is not linked to a discount, shipping method, or validation rule does absolutely nothing at checkout — it exists in the Partners dashboard but has no effect on any store. A validation function that encounters an unhandled Rust error shows shoppers a generic "Something went wrong" message with no actionable information. A discount function that evaluates correctly but returns an empty discounts array applies no discount and logs no error. All eight failure modes are silent from the merchant's perspective.

Four Shopify Functions API Categories
Discount API: Order discounts, product discounts, and shipping discounts. Replaces Script Editor discount scripts.

Delivery Customization API: Show, hide, rename, and reorder shipping rates. Replaces carrier-calculated shipping scripts.

Cart and Checkout Validation API: Block checkout or show errors based on cart contents, buyer identity, or custom conditions. No Script equivalent — this is new.

Payment Customization API: Show, hide, or rename payment methods at checkout. Replaces payment gateway configuration logic.

All four categories require the same deployment workflow: Shopify CLI → Partners dashboard → app install → explicit rule creation on the store.

Failure Mode 1: Shopify Scripts vs Functions — Migration Required for Checkout Extensibility

The most common source of confusion for stores migrating to Shopify's new checkout (Checkout Extensibility): Shopify Scripts stop working when you upgrade to the new checkout. Scripts are a legacy feature tied to the old checkout.liquid system. Checkout Extensibility uses a completely different architecture — checkout.liquid is removed and all customization must be done through Checkout UI Extensions and Functions.

What breaks during migration

Shopify Scripts (Legacy) Shopify Functions (Extensibility) Migration Required?
Script Editor discount scripts Discount API function Yes — rewrite in Rust/JS
Script Editor shipping scripts Delivery Customization API function Yes — rewrite in Rust/JS
Script Editor line item scripts Discount API (product discounts) Yes — rewrite in Rust/JS
checkout.liquid customizations Checkout UI Extensions Yes — rewrite as extensions
checkout.liquid payment hiding Payment Customization API function Yes — rewrite in Rust/JS
Scripts Do Not Migrate Automatically
When a store upgrades to Checkout Extensibility, existing Script Editor scripts are not converted to Functions automatically. They stop executing silently. Merchants who had automatic discounts, free shipping thresholds, or payment method hiding via Scripts discover these features stopped working only after customers report missing discounts or payment options. Audit all active Scripts before migrating: Admin → Apps → Script Editor → see all active scripts.

How to check if Scripts are still active

  1. Admin → Apps → find "Script Editor" in your installed apps
  2. Open Script Editor → review all active scripts (discount, shipping, payment)
  3. Check your checkout version: Admin → Settings → Checkout → Checkout customization
  4. If "Upgrade to Checkout Extensibility" is available or already done, Scripts are inactive
  5. For each Script, build an equivalent Function using Shopify CLI before upgrading

Failure Mode 2: Deploy Does Not Mean Active — Install and Link Required

The most common developer error with Shopify Functions: deploying a function via shopify app deploy does not activate it on any store. The deploy command uploads the compiled WebAssembly binary to the Shopify Partners dashboard and creates a new version of your app — but this version has no effect on the store until two additional steps are completed.

The full activation chain

  1. Build and deploy: shopify app deploy — uploads function to Partners. Does nothing on the store yet.
  2. Install the app: The app containing the function must be installed on the target store. If the app was already installed, the new deployment auto-updates it. If this is a new app, you must install it via the Partners dashboard install link or a custom install URL.
  3. Create a rule from the function: This is the step most developers miss. For discount functions: Admin → Discounts → Create discount → select your function name from the list. For shipping functions: Admin → Settings → Shipping → Manage rates → create a carrier service linked to the function. For validation functions: the app must register the function at install time via the Admin API.
  4. Verify the rule is active: The discount, shipping rate, or validation rule must be enabled and not have expired dates or usage limits already hit.
Development Store vs Production Store
Functions deployed from development mode (shopify app dev) are available only on the development store linked to the Partners app. They do not automatically transfer to production stores. Production stores require a formal app deployment and installation. Additionally, some function APIs have different behavior in development stores — the Payment Customization API, for example, always shows all payment methods in development mode regardless of function output.

Failure Mode 3: Discount Function Not Applying — Target Type and Cart Conditions

A discount function that runs without error but applies no discount at checkout is caused by one of four issues: the cart conditions in your function logic are not met by the current cart, the function returns an empty discounts array, the discount target type (ORDER vs PRODUCT) is wrong, or a stacking conflict prevents the discount from applying alongside another active promotion.

Target type mismatch

Shopify discount functions have three distinct target types, each requiring a different function API and producing a different discount type in the Admin:

Target Type What It Discounts Admin Discount Category Function API
ORDER_DISCOUNTS The entire order total Amount off order order-discounts
PRODUCT_DISCOUNTS Specific line items Amount off products product-discounts
SHIPPING_DISCOUNTS Shipping rates Free shipping shipping-discounts

If you built a PRODUCT_DISCOUNTS function but created an "Amount off order" discount using it in the Admin, the function type and the discount type don't match — the discount applies to the wrong target and may appear to apply zero discount or not show at all.

Debugging discount function logic locally

# Create a local input JSON fixture matching your cart
# Then run the function locally to verify it returns the expected discount
shopify function run

# The CLI will prompt you to provide the input JSON.
# Paste a cart input that mirrors your production cart.
# The output shows exactly what discounts array the function returns.
# If the output shows: { "discounts": [] } — the cart doesn't meet your conditions.

Common cart condition mismatches

  • Product tag or collection not matching: Function checks line.merchandise.product.hasAnyTag(["sale"]) but the product in your test cart doesn't have that tag in the function's input data
  • Minimum quantity not met: Function requires 3+ units of a specific variant but the test cart has 2
  • Customer metafield check on guest cart: Function reads cart.buyerIdentity.customer.metafield but the customer is not logged in — the value is null and your function returns no discount
  • Currency comparison failing: Function compares subtotal_price > 5000 but subtotal is in minor units (cents) — $50.00 is 5000 minor units, so the condition is met only for orders over $50 exactly
Stacking Strategy Check
If another automatic discount is active on your store, check the stacking strategy. Go to Admin → Discounts → [the other discount] → see if it has "Combine with" options enabled. Your function's discount must also have compatible stacking. If one discount uses FIRST strategy (applies only if no other discount has been applied) and another discount already applied, your function's discount is skipped silently.

Failure Mode 4: Shipping Function Not Showing Rates — Empty Output, Missing Carrier Service

Shopify shipping functions (Delivery Customization API) that produce no rates at checkout are caused by the function returning an operations array that hides all rates and adds no new ones, a missing Carrier Service configuration, or a function runtime error that results in empty output.

Delivery Customization function must return at least one visible rate

The Delivery Customization API works by providing an array of operations that modify the checkout's available delivery options. If your function hides all existing carrier rates (hide operations on every rate) and does not add replacement rates, the checkout displays "No shipping methods available for your location." This is not an error — it is the correct behavior of the function returning no visible options.

// Common mistake: hiding rates without preserving at least one
FunctionRunResult {
  operations: vec![
    Operation::Hide(HideOperation {
      delivery_option_handle: "standard-shipping".to_string(),
    }),
    Operation::Hide(HideOperation {
      delivery_option_handle: "express-shipping".to_string(),
    }),
    // No Rename or Add operation — result is zero visible rates
  ]
}
Silent Rate Suppression Has No Merchant Alert
When a Delivery Customization function returns zero visible rates, Shopify does not alert the merchant or log an error in the Admin. From the merchant's view, the store appears operational — only customers trying to checkout discover there are no shipping options. This makes it impossible to detect without checkout monitoring or a test order.

Carrier Service must be explicitly created and linked

Unlike discount functions which become available in the Admin Discounts UI automatically after app installation, Delivery Customization functions require a Carrier Service to be created via the Admin API and associated with the function. The typical setup flow:

  1. Deploy the function and install the app on the store
  2. Register a Carrier Service via Admin API: POST /admin/api/2024-10/carrier_services.json with callback_url pointing to the function's endpoint
  3. The Carrier Service must be active (Admin → Settings → Shipping and delivery → Manage rates)
  4. The function must be linked to the Carrier Service handle in your app's configuration

Failure Mode 5: Validation Function Shows Generic Error — Rust Panic

Shopify Cart and Checkout Validation functions that block checkout with a generic "Something went wrong" message — instead of the structured error message you configured — are caused by an unhandled error (panic) inside the function's Rust code. When a function panics, Shopify cannot render the structured FunctionRunResult and falls back to a generic error that gives shoppers no actionable information.

The structured error response contract

A validation function must return a FunctionRunResult struct. An empty errors array means checkout is allowed. A non-empty errors array blocks checkout and displays each error's localizedMessage to the shopper. A panic bypasses this entirely:

// Correct: returns structured errors array
FunctionRunResult {
  errors: vec![
    FunctionError {
      localizedMessage: "You need at least 3 items to checkout".to_string(),
      target: "cart".to_string(),
    }
  ]
}

// Correct: allows checkout
FunctionRunResult {
  errors: vec![]
}

// Wrong: panics if cart.buyer_identity is None (guest checkout)
let customer = input.cart.buyer_identity.unwrap().customer.unwrap();
// .unwrap() on None = panic = generic "Something went wrong" error

Most common panic triggers in validation functions

  • .unwrap() on buyer identity for guest checkout: cart.buyer_identity is None for guest checkouts — any .unwrap() panics. Use .unwrap_or_default() or check with if let Some(...)
  • Metafield access without null check: If the metafield was not added to run.graphql or the product doesn't have the metafield set, the value is None — .unwrap() panics
  • Integer division by zero: If quantity can be 0 and your function divides by it
  • Array index out of bounds: line_items[0] panics on empty cart
Defensive Pattern for Validation Functions
Use ? operator or unwrap_or_default() everywhere, and wrap the entire run logic in a match that returns an empty errors array on any unexpected input shape. This ensures shoppers always see your structured messages, never a generic error — and an unexpected input shape allows checkout rather than blocking it.

Failure Mode 6: Input Data Not Available — run.graphql Missing Required Fields

Shopify Functions receive their input data via a run.graphql file that declares exactly what fields are available to the function at execution time. If a field is not in run.graphql, it is not available in the function's input — it is not null, it is simply absent from the JSON input entirely. This is the most subtle and confusing failure mode: the function compiles and deploys successfully, the field exists in Shopify's data model, but the function always receives null for that field because the query didn't request it.

Common missing fields in run.graphql

What You Want What Must Be in run.graphql Common Mistake
Customer tag-based discount cart { buyerIdentity { customer { tags } } } Only querying buyerIdentity { email } — tags always null
Product metafield in discount merchandise { ... on ProductVariant { product { metafield(namespace: "x", key: "y") { value } } } } Querying product { title } only — metafield always null
Cart custom attributes cart { attribute(key: "promo_code") { value } } Not querying cart attributes at all — attribute always null
Shipping address for validation cart { deliveryGroups { deliveryAddress { countryCode } } } Not including deliveryGroups in query — address always null
Discount codes already applied cart { discountCodes { code applicable } } Not querying discount codes — stacking logic always wrong
Larger Queries Slow Down Function Execution
Every field requested in run.graphql adds to the input data Shopify must serialize and pass to your function. Functions have a 5-second execution limit and a 10MB input limit. Do not request fields you don't use. If you need complex data joins (e.g., metafields on all line items), benchmark the execution time with a realistic cart before deploying to production.

How to verify what data your function receives

# Use shopify function run to see the exact input JSON
shopify function run

# Paste a test cart JSON input
# Inspect the output — if the field you need shows null in the input,
# it's not in your run.graphql query.
# Add it to run.graphql and redeploy.

Failure Mode 7: WebAssembly Build or Deploy Error

Shopify Functions must compile to WebAssembly (WASM) targeting the wasm32-wasip1 architecture. Build failures are usually caused by a missing Rust toolchain target, incorrect Cargo.toml configuration, or a mismatch between the @shopify/shopify_function package version and the Shopify API version your function targets.

Common WASM build errors

Error Cause Fix
error: no target named 'wasm32-wasi' WASM target not installed in Rust toolchain rustup target add wasm32-wasip1
Build failed: function exceeds size limit Compiled WASM binary exceeds 256KB limit Remove unused dependencies, enable LTO in Cargo.toml: lto = true
Deploy succeeds but Partners shows "Build failed" API version mismatch between shopify.extension.toml and package Align api_version in extension.toml with installed package version
linking with rust-lld failed Wrong WASM ABI — wasm32-wasi vs wasm32-wasip1 Ensure Rust toolchain is 1.78+ and use wasm32-wasip1 target
Function shows "inactive" in Partners after deploy Deploy succeeded but function has no linked app version Run shopify app release to publish the version as active 
# Verify your Cargo.toml has the correct WASM optimization settings
[profile.release]
lto = true        # Link-time optimization reduces binary size
opt-level = "s"   # Optimize for size
strip = "symbols" # Remove debug symbols

# Verify the extension.toml targets the correct API version
[extensions.[[extensions]]]
name = "my-discount-function"
type = "function"
api_version = "2024-10"  # Must match your installed shopify-functions package

Failure Mode 8: Function Passes Tests But Fails in Production

A function that produces correct output during local testing with shopify function run or unit tests but behaves differently in production is caused by a mismatch between the test input fixture and the actual data shape Shopify provides at runtime. Production checkout data is more complex than most test fixtures: prices are in minor units with currency codes, product data includes metafields only if they exist, guest checkouts have null buyer identity, and multi-currency stores send prices in the buyer's currency rather than the store's base currency.

Production data shape differences that break test-passing functions

  • Price minor units: subtotalPrice.amount in Functions is a string like "49.99" not an integer. Parsing it as integer and comparing to 5000 (expecting cents) fails — parse as float and multiply, or use Shopify's built-in money comparison utilities
  • Missing metafields on products without the metafield set: A product metafield is present in the input only for products that have that metafield. Your function must handle the case where some line items have it and some don't
  • Multi-currency price comparison: In a multi-currency store, subtotalPrice.amount is in the buyer's currency, not the store's base currency. A free shipping threshold of $50 USD does not match a cart total of £40 GBP even if both are roughly equivalent
  • POS checkout input shape: Point-of-sale checkouts have a different input shape than online checkouts — some fields present in online checkout are absent or have different values in POS context
  • Draft order context: Functions triggered during draft order creation receive a slightly different context than customer-initiated checkouts — buyer identity may differ
Capture a Real Production Input for Testing
The most reliable way to reproduce production failures is to capture the actual JSON input your function receives. You can do this by temporarily adding a log! macro call in your function that prints the serialized input to stderr — Shopify Functions logs stderr output to the Partners dashboard under the function's "Logs" tab. Copy that output as your local test fixture. Test-fixture-first development without a real production input is the root cause of most production-only failures.
