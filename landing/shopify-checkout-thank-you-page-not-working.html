Shopify Thank You Page Not Working — Order Status Page Blank, Post-Purchase Extension Missing, Custom Code Stopped, Conversion Pixel Gone

Shopify deprecated checkout.liquid and additional_scripts in June 2025, silently killing every custom script on the thank-you page. Post-purchase app extensions must now be registered in Checkout Customizations. Here are all 8 failure modes.

📅 Updated June 2026 ⏱ 10 min read 💡 Applies to: All Shopify plans
checkout.liquid Deprecated June 2025 — All Custom Thank-You Page Code Silently Stopped
If you had any custom JavaScript, tracking pixels, or HTML injected into Shopify's thank-you page via the additional_scripts Liquid variable or checkout.liquid, it stopped working silently on June 30, 2025. There is no error. The order status page loads normally — your code simply no longer runs. This is the root cause for most "thank you page not working" reports in 2025–2026.

The Shopify "Thank You Page" — What It Is and Why It's Confusing

Shopify has used several names for the post-checkout confirmation page across different product eras, and the terminology confusion causes merchants to look in the wrong place for settings:

Term Shopify's Current Name Where to Find It
"Thank you page" Order status page / Post-purchase page Settings → Checkout → Customize → Post-purchase page
"Thank you page URL" /thank_you redirect → order status URL Automatically generated per order
"Thank you page customization" Post-purchase Checkout UI Extension (Plus) or post-purchase app Settings → Checkout → Customizations
"Custom scripts on thank you page" Deprecated — checkout.liquid removed June 2025 No direct replacement; use Web Pixels API

The order status page URL follows this pattern after checkout: https://yourstore.myshopify.com/orders/{order-id}/authenticate?key={token}. Immediately after checkout, Shopify redirects the customer through /thank_you to this URL. Both terms refer to the same destination.

Failure Mode 1 — checkout.liquid / additional_scripts Deprecated, Custom Code Gone

Prior to Shopify's Checkout Extensibility, the only way to add custom code to the thank-you page was via the additional_scripts Liquid variable inside checkout.liquid. This was used for:

  • Google Analytics purchase events (UA and then GA4 via gtag.js)
  • Meta Pixel Purchase events
  • Pinterest Tag checkout completion
  • Affiliate network conversion pixels (Impact, ShareASale, Awin, CJ)
  • TikTok Pixel PlaceAnOrder events
  • Custom "thank you" welcome messages or product recommendations
  • Chat widget initialization with order context
All of This Stopped Working on June 30, 2025
Shopify deprecated checkout.liquid on June 30, 2025. Any code in additional_scripts — including every tracking pixel and custom widget installed this way — silently stopped executing. The order status page continued loading normally, so there was no visible error to alert merchants.

What to do instead

The replacement for additional_scripts tracking depends on the platform:

Platform Replacement How
Google Analytics 4 Google & YouTube app (official) Install from Shopify App Store → connects via Web Pixels API; fires purchase event in checkout sandbox
Meta Pixel Meta Pixel app (official) Install from Shopify App Store → creates web pixel extension; fires Purchase inside checkout
TikTok Pixel TikTok app (official) Install TikTok for Business app → web pixel handles PlaceAnOrder
Affiliate pixels Custom web pixel extension or tag manager app Build a Shopify web pixel extension; or use Elevar or Triple Whale which support Web Pixels API
Custom HTML content Post-purchase app (non-Plus) or Checkout UI Extension (Plus) See Failure Mode 2 and 4 below

Failure Mode 2 — Post-Purchase App Not Registered in Checkout Customizations

Apps like ReConvert, Zipify Pages, Carthook, and others that display a custom post-purchase offer page (upsell, cross-sell, survey, or custom thank-you content) all use Shopify's post-purchase extension system. Installation of the app is not enough — the app's extension must also be selected as the active post-purchase page in your checkout settings.

  1. Go to Shopify Admin → Settings → Checkout
  2. Scroll to "Post-purchase page" section
  3. Click "Add post-purchase page" (or check which app is currently set)
  4. Select your app's post-purchase extension from the list
  5. Save — the extension will now appear after every successful checkout
Only One Post-Purchase Extension Can Be Active at a Time
Shopify allows only one post-purchase page extension to be active simultaneously. If you install a second post-purchase app without deregistering the first, only the previously registered one shows. The new app's extension won't display until you explicitly select it in Checkout Settings → Post-purchase page.

App Embed also required for some apps

Some post-purchase apps also require their theme App Embed to be enabled, even though the post-purchase page runs inside the checkout sandbox, not on the storefront. If the app documentation mentions enabling it in the theme editor, check Online Store → Themes → Customize → App Embeds and confirm the app's embed is toggled on.

Failure Mode 3 — Conversion Tracking Purchase Event Not Firing

After checkout.liquid deprecation, purchase events must come from Shopify's Web Pixels API. The most common misconfigurations:

Google & YouTube app connected but purchase event missing

The Google & YouTube app creates a web pixel extension inside Shopify's checkout sandbox. If the app shows "Connected" in the app dashboard but GA4 still shows zero purchase conversions, check:

  • Admin → Settings → Customer events: Is the Google web pixel extension listed and active? If the extension was removed or disconnected here, it won't fire even if the app shows connected.
  • GA4 property ID mismatch: If your store was connected to one GA4 property and you created a new one, the web pixel may still be sending to the old property.
  • Google account permission change: If the Google account used to connect the app lost access to the GA4 property, the pixel continues to fire but GA4 rejects the hits.

Meta Pixel fires PageView but not Purchase

This is the most common Meta Pixel complaint after checkout.liquid deprecation. The Meta Pixel installed in your theme via theme.liquid fires storefront events (PageView, ViewContent, AddToCart) — but it cannot fire inside Shopify's checkout sandbox, which runs on a separate domain. The Purchase event can only fire from inside the checkout via a web pixel extension. If you don't have the Meta Pixel app installed, your Purchase event is missing entirely, even though other events appear in Events Manager.

  1. Go to Admin → Settings → Customer events
  2. Look for "Meta Pixel" in the list of active web pixel extensions
  3. If missing: install the Meta Pixel app from the Shopify App Store and connect your pixel
  4. If present but Purchase events are still missing: check Meta Events Manager → your Pixel → Test Events → simulate a purchase and verify the event fires
  5. Check for duplicate pixel IDs — if you have both the theme.liquid pixel and the web pixel extension with the same Pixel ID, deduplication should be occurring; compare event counts in Meta's Reports vs checkout sessions in Shopify Analytics

GDPR consent blocking EU conversion events

If your store has a consent management platform (OneTrust, Cookiebot, CookieYes, Axeptio), EU visitors who decline analytics consent will have all web pixels suppressed. This is correct behavior under GDPR, but it means 20–40% of EU purchase events may be permanently absent from your GA4 and Meta datasets. Server-side event forwarding (Conversions API for Meta, Measurement Protocol for GA4) with consent mode signals is required to recover this signal.

Failure Mode 4 — Post-Purchase UI Extension Blank on Plus

Shopify Plus merchants building custom post-purchase experiences via Checkout UI Extensions targeting the purchase.thank-you.block.render or purchase.thank-you.header.render extension point may find the extension renders blank after deployment.

Common causes of blank post-purchase UI extensions

  • Extension not deployed: Running shopify app dev shows the extension in development mode, but production deployments require shopify app deploy. A dev-only extension won't show on the live store.
  • Extension not added to checkout customization: After deploying, go to Admin → Settings → Checkout → Customize → find "Post-purchase" blocks and explicitly add your extension block to the template.
  • JavaScript runtime error: The extension renders but immediately crashes due to a runtime error. Check the browser console on the order status page for errors originating from the extension sandbox.
  • API call timing out: Post-purchase extensions have strict time limits. If your extension makes a slow external API call before rendering, Shopify may time it out and render nothing.
  • ShouldRender returning false: If your extension uses the ShouldRender export to conditionally show the extension, verify the conditions are met for the test orders you're using.
Test Post-Purchase Extensions in a Development Store First
Deploy the extension to a development store and use Shopify's checkout extension preview URL (available in the Partner Dashboard under your extension) to preview the post-purchase page without completing a real transaction. This lets you verify the rendering before pushing to production.

Failure Mode 5 — Order Status Page URL in Email Not Loading

The order status page URL included in Shopify's order confirmation email contains a customer-specific authentication token. Several conditions can cause this URL to fail:

Token broken by email client URL rewriting

Some email clients (notably Outlook on Windows and some corporate email gateways) rewrite URLs for click-tracking or safety scanning. If the URL rewriting mangles the key query parameter, the authentication check fails and the customer sees an error or an empty page. This is most common when the token contains special characters that get URL-encoded twice.

Guest checkout — wrong browser or device

For guest checkouts, the order status page is authenticated via the token in the URL — the customer doesn't need to be logged in. However, if the customer's browser has aggressive privacy settings that clear URL parameters, or if they forward the email to someone else and that person clicks the link while logged into a different Shopify account, they may see a restricted view or an error.

Order cancelled after email was sent

If the order was cancelled (by the merchant or automatically via fraud protection) after the confirmation email was sent, the order status page now shows the cancelled state — which can look like a "not working" page if the customer expects to see a confirmation.

  1. Check Admin → Orders → find the order → verify its status (Open, Cancelled, Refunded)
  2. Click "View order status page" to load the same URL the customer received — if it loads for you, the URL is valid
  3. If the customer reports a broken link, ask them to copy the raw URL from the email (not click it) and paste it in a browser
  4. Check if the order has a fraud risk flag — high-risk orders are sometimes auto-cancelled, which changes the order status page content

Failure Mode 6 — Subscription Renewal Orders Have No Thank-You Page

When a subscription app (Recharge, Bold Subscriptions, Appstle, Loop Subscriptions) processes a recurring billing cycle, there is no active checkout session — the charge is processed server-side without customer involvement. No checkout means no order status page is generated for that billing event.

This creates a specific support scenario: the customer receives a charge notification from their bank but searches for an order confirmation page and finds nothing. They contact support asking "where's my thank-you page?" — not understanding that rebill orders work differently.

What customers receive for subscription renewals

  • An order confirmation email (if the subscription app is configured to send one)
  • A bank/card charge notification
  • Access to the order in their customer account order history (if they have an account)
  • No real-time thank-you page — this is a platform limitation, not a configuration error

If customers report missing confirmation emails for renewals, check the subscription app's notification settings — Recharge has a separate "Charge Success Email" that must be enabled independently from Shopify's native order emails.

Failure Mode 7 — Headless Storefront Redirects to Wrong Page After Checkout

In headless or Hydrogen storefronts, the post-checkout experience is entirely custom — Shopify's hosted order status page is not automatically shown. The redirect after checkout completion must be configured explicitly.

Common headless thank-you page failures

After the customer completes checkout in Shopify's hosted checkout (which all headless storefronts still use for payment), Shopify redirects to the order status URL on checkout.shopify.com or the store's .myshopify.com domain by default — not back to the headless frontend. If the merchant expected customers to land on a custom yourstore.com/order-confirmed page, that redirect doesn't happen automatically.

  • Standard plan: Use a post-purchase app or configure a redirect via the cart's note attribute before checkout is initiated.
  • Shopify Plus: Use a Checkout UI Extension with a purchase.thank-you.header.render block, or configure checkoutCompleteBehavior in your Storefront API cart mutations to redirect back to the headless frontend.
  • Hydrogen / Remix: Use the redirects configuration in hydrogen.config.ts and ensure your post-checkout URL is handled by the headless router.
The Order Status URL Is on Shopify's Domain, Not Yours
Even with a custom domain for your headless storefront, the order status page URL after checkout completion will be on checkout.shopify.com or yourstore.myshopify.com — not your custom domain. The only way to show a fully custom post-checkout page on your own domain is to redirect the customer from Shopify's order status page to your custom URL using a post-purchase extension or the completionBehavior Storefront API setting.

Failure Mode 8 — Tracking Pixels Moved to theme.liquid Instead of Web Pixels API

A common mistake after the checkout.liquid deprecation: merchants move their tracking pixels from checkout.liquid to theme.liquid (the storefront theme) thinking this will restore purchase tracking. This does not work.

theme.liquid controls the storefront — product pages, collection pages, the cart page, and the homepage. It does not run inside the checkout flow. Shopify's checkout runs in a sandboxed environment on a separate domain, isolated from the storefront theme.

What this means in practice

  • A Meta Pixel in theme.liquid fires PageView, ViewContent, and AddToCart — but not Purchase
  • A GA4 gtag.js in theme.liquid tracks all storefront pages — but the purchase event is missing because the order status page is not part of the storefront theme
  • Moving the additional_scripts content to theme.liquid does not restore thank-you page tracking — it fires those events on the wrong pages
Verify Your Pixel Events Are Actually Coming From Checkout
In Meta Events Manager → Test Events, simulate browsing your storefront and completing a test checkout. You should see separate events for storefront actions (from your theme.liquid pixel) and a Purchase event (from the web pixel extension in checkout). If you only see one source of events, you're missing either the storefront pixel or the checkout web pixel.
