Shopify Gift Wrap Not Working — Gift Message Not Saving, Gift Option Not Showing at Checkout

Shopify has no native gift wrap feature on non-Plus plans. Gift messages via order notes are bypassed by every express checkout button. Line item properties are lost at checkout if set incorrectly. Gift wrap apps can be silently blocked. Here are all 8 failure modes.

📅 Updated June 2026 ⏱ 9 min read 💡 Applies to: All Shopify plans
Shopify Has No Built-In Gift Wrap Feature on Non-Plus Plans
There is no setting in Shopify Admin to enable a gift wrap option at checkout. Merchants who expect to find a toggle in Settings → Checkout will not find one. Gift wrapping must be implemented via order notes, line item properties, a third-party app, or (on Plus) a Checkout UI Extension. Each method has significant limitations, especially with express checkout flows.

How Gift Options Work in Shopify (And What Breaks)

Shopify has four implementation paths for gift wrapping and gift messages, each with different limitations:

Method Where It Appears Works with Buy Now / Shop Pay? Plan Required
Order notes (cart.note) Cart page (/cart) No — bypassed All plans
Line item properties Product page form No — Buy Now skips form All plans
Gift wrap app (cart page injection) Cart page No — cart page bypassed All plans
Checkout UI Extension (gift message TextField) Inside checkout Yes — survives express checkout Shopify Plus only

The practical consequence: on non-Plus Shopify stores, any customer who uses the Buy Now, Shop Pay, Apple Pay, or Google Pay button will never see a gift message field regardless of how it is implemented — because all these flows bypass the cart page and the product form.

Failure Mode 1 — Shopify Has No Native Gift Wrap Feature (Non-Plus)

When merchants look for a gift wrap setting in Shopify Admin, they search in Settings → Checkout, Online Store → Preferences, or the theme customizer. None of these locations contains a gift wrap toggle. Shopify's native checkout is sealed on non-Plus plans — no custom UI elements can be added inside the payment flow.

What Shopify does provide natively:

  • Order notes — an optional text area on the cart page (/cart) that lets customers leave a message. This is the closest native equivalent to a "gift message" field, but it is a generic order-level note, not a gift-specific field.
  • Line item properties — a mechanism to attach custom data to individual cart items via the product page form. Commonly used for personalization, but requires custom Liquid code to add the input field.
What Merchants Expect vs What Exists
Most merchants coming from other e-commerce platforms expect a "Gift Options" section inside checkout with checkboxes for gift wrap and a gift message textarea. Shopify does not have this except on Plus (via Checkout UI Extensions). Non-Plus merchants must work around this with product-page or cart-page solutions, each of which has the Buy Now bypass problem.

Failure Mode 2 — Gift Message Via Order Notes Bypassed by Express Checkout

The most common gift message implementation on non-Plus Shopify: enable the order note field in the cart template. The merchant's instruction to the customer is "add your gift message in the 'Order Notes' or 'Special Instructions' box before checkout." This works for customers who click the standard "Check Out" button. It fails silently for customers who use:

  • Buy Now button (dynamic checkout button on product pages)
  • Shop Pay (accelerated checkout)
  • Apple Pay
  • Google Pay
  • PayPal Express
  • Amazon Pay

All of these flows take the customer directly from the product page (or a one-click buy prompt) into the payment step, bypassing the cart page entirely. The customer never sees the note field. The order arrives with no gift message attached, and the merchant has no way to know a message was intended.

Mitigation for the express checkout bypass

  • Disable the Buy Now button on gift-eligible products — in the product template, wrap the {% render 'dynamic-checkout-buttons' %} tag in a conditional that hides it for tagged products (e.g., tag: gift-eligible). Trade-off: reduces one-click conversion on those products.
  • Add messaging near all express checkout buttons — a line of text: "To add a gift message, use the 'Add to Cart' button and add your message before checkout." Not ideal UX, but sets customer expectations.
  • Follow up via email — after every order, send an automated email asking if the order is a gift and offering to include a message with fulfillment. Requires a post-purchase email flow in Klaviyo, Omnisend, or similar.
  • Use a gift wrap app that injects into the checkout confirmation page — some apps use a post-purchase UI Extension (Plus) or order status page injection (non-Plus via additional_scripts) to collect gift message after purchase. Not ideal, but solves the bypass problem.

Failure Mode 3 — Line Item Properties Lost at Buy Now

Line item properties are the correct way to attach gift-specific data (gift wrap selection, gift message) to individual items rather than the whole order. They are set by adding input fields to the product form on the product page:

<!-- In product-form.liquid, inside the <form> tag -->
<label for="gift-message">Gift message (optional)</label>
<textarea
  id="gift-message"
  name="properties[Gift Message]"
  rows="3"
  placeholder="Write your gift message here..."
></textarea>

<label>
  <input type="checkbox" name="properties[Gift Wrap]" value="Yes" />
  Add gift wrap (+$5)
</label>

When a customer fills these fields and clicks "Add to Cart," the properties travel with the line item through checkout and appear on the packing slip and order line items in Admin. However:

Buy Now Button Skips the Product Form Entirely
The dynamic checkout button (Buy Now / Shop Pay / Apple Pay) does NOT submit the product form. It initiates checkout with the variant ID and quantity only — none of the custom property fields are included. A customer who checks the gift wrap checkbox and then clicks "Buy Now" instead of "Add to Cart" will complete an order with no gift wrap and no message, and there is no error shown.

Hiding dynamic checkout buttons on products with gift options

{% comment %} In your product.liquid template {% endcomment %}
{% unless product.tags contains 'gift-option' %}
  {{ form | payment_button }}
{% endunless %}

Tag any product that has a gift option with gift-option in Admin → Products → Tags. This suppresses the Buy Now button for those products, forcing customers through the standard Add to Cart → Cart → Checkout flow where the line item properties are captured.

Failure Mode 4 — Gift Wrap App JS Blocked by Cookie Consent Manager

Third-party gift wrap apps (Gift Wrap & Message by Cirkle Studio, Wrapr, Gift Options & Gift Wrap by Zestard, Personalize by Seguno, and others) work by injecting JavaScript into the cart page that renders a gift wrap UI and sets a cart attribute or line item property. This approach works well in most browsers but fails when:

  • Cookie consent manager blocks uncategorized scripts — apps like Cookiebot, OneTrust, Axeptio, Complianz, and Consentmo can block third-party app scripts if they are not explicitly categorized. The gift wrap UI never renders, but there is no visible error — the customer just sees no gift option.
  • Ad blockers — uBlock Origin, AdBlock Plus, and Privacy Badger may block scripts loaded from app CDNs. Estimated 20–35% of desktop users have ad blockers active.
  • App embed not toggled on in the theme — apps that use Shopify's App Embed Block system require the merchant to go to Online Store → Themes → Customize → App Embeds and toggle the app embed on. If this toggle is off (common after theme reinstall or theme switching), the app script never loads.
  • Theme update removing the app's Liquid snippet — older apps that inject directly into theme files (rather than using App Blocks) may have their code removed when the merchant updates the theme.

Diagnosing gift wrap app failures

  1. Open your store's cart page in an Incognito / Private window (no extensions, logged out). Check if the gift wrap option appears. If it does, the issue is browser extension or cookie consent related.
  2. Open Browser DevTools → Console. Look for JavaScript errors referencing the app name or any 403/blocked network requests.
  3. Open Online Store → Themes → Customize → App Embeds (left sidebar). Confirm the gift wrap app's embed is toggled ON.
  4. In the app's settings page in Admin, look for "Theme Setup" or "Installation" — confirm it shows "Installed" or "Active."
  5. Check if a recent theme update is the culprit: Admin → Online Store → Themes → click the three dots next to your current theme → Edit code → search for the app's snippet name (e.g., gift-wrap). If missing, re-install the app or add the snippet manually.

Failure Mode 5 — Checkout UI Extension Gift Message Not Saving (Plus Only)

On Shopify Plus, Checkout UI Extensions can render a gift message TextField directly inside the checkout flow. This is the only approach that survives express checkout. However, it requires explicit code to commit the value to the checkout object — simply rendering a TextField is not sufficient.

applyAttributeChange() Must Be Called Explicitly
In a Checkout UI Extension, a TextField component is a controlled input. The merchant's extension code must call applyAttributeChange({ key: 'gift_message', type: 'updateAttribute', value }) on the onChange or onBlur event. If this call is missing, the customer can type in the field but the value is never written to the checkout object — it disappears when they move to the next step. 
// In your Checkout UI Extension React component
import { TextField, useApplyAttributeChange } from '@shopify/ui-extensions-react/checkout';

export default function GiftMessageExtension() {
  const applyAttributeChange = useApplyAttributeChange();
  const [message, setMessage] = useState('');

  const handleChange = async (value) => {
    setMessage(value);
    // This line is REQUIRED — without it, the value is never saved
    await applyAttributeChange({
      key: 'gift_message',
      type: 'updateAttribute',
      value,
    });
  };

  return (
    <TextField
      label="Gift message (optional)"
      value={message}
      onChange={handleChange}
      multiline={3}
    />
  );
}

The saved attribute lands in order.note_attributes (as a key/value pair, not in order.note). To display it in packing slips and order confirmation emails, update the relevant notification templates to include:

{% for attribute in attributes %}
  {% if attribute.key == 'gift_message' %}
    Gift Message: {{ attribute.value }}
  {% endif %}
{% endfor %}

Failure Mode 6 — Cart-Modifying App Erasing Gift Attributes

Even when a gift message or gift wrap selection is correctly saved as a cart attribute or note, it can be silently erased by a subsequent cart modification. Apps that add upsells, apply discounts, modify quantities, or update the cart call the Shopify cart API without re-sending existing attributes:

  • A discount app calls /cart/update.js with only {"discount": "SUMMER10"} — omitting the existing {"attributes": {"gift_message": "Happy Birthday!"}} causes the attribute to be cleared
  • A upsell popup app calls cartAttributesUpdate in the Storefront API with a partial attributes object, overwriting the gift_message with blank
  • A cart drawer app re-fetches and re-renders the cart after a quantity change using a template that does not preserve the note — the note field shows empty and if the customer doesn't re-enter it, it submits blank

How to diagnose

  1. Add a product to cart, fill in the gift message field, then trigger the suspected app (apply a discount code, interact with the upsell popup, change the quantity in the cart drawer)
  2. After each interaction, check if the gift message field still shows the value you typed
  3. Also check: visit /cart.json in your browser after each interaction — the note field and attributes object in the response show what Shopify currently has stored
  4. Identify which app interaction clears the value, then check that app's settings for a "preserve cart attributes" or "maintain cart notes" option

Failure Mode 7 — Gift Order Flagged as High-Risk / Fraud

Gift purchases are structurally different from self-purchases in ways Shopify's fraud detection system penalizes. Understanding this pattern prevents legitimate orders from being held or cancelled:

Gift Order Signal Why Shopify's Fraud Filter Flags It Risk Level
Billing address ≠ shipping address Most fraudulent orders ship to an address different from the card's billing address (card-not-present fraud pattern) High weight
First-time customer No purchase history, higher statistical fraud risk Medium
Shipping to a different city/country than billing Amplifies the address mismatch signal High weight
High-value item, new card Classic card testing pattern High weight
Email does not match billing name May suggest the card belongs to someone else Medium
Order placed from IP far from both addresses VPN or location inconsistency Medium

Reducing false positives for gift orders

  • Add "This is a gift" checkbox to your checkout (Plus: UI Extension; non-Plus: cart page attribute). When checked, this signals intent to your fraud review team even if Shopify auto-flags the order.
  • Review high-risk orders manually before cancelling — Shopify's fraud score is a signal, not a verdict. Orders marked "High risk" for address mismatch alone are frequently legitimate gift purchases. Check the full order risk details (Admin → Orders → order → Risk assessment) before acting.
  • Lower the auto-cancel threshold — if using Shopify's auto-cancel for high-risk orders, consider disabling it for orders where the only high-risk signal is billing/shipping address mismatch without other fraud indicators.
  • Use a third-party fraud tool (Signifyd, NoFraud, Kount) that understands gift purchase patterns and provides chargeback guarantees — they apply more nuanced risk scoring than Shopify's built-in system.

Failure Mode 8 — No Native Gift Receipt Email in Shopify

Shopify's standard notification system does not include a gift receipt email template. There is no built-in option to suppress the order total from an order confirmation email when the item is being sent as a gift. By default:

  • The standard "Order confirmation" email goes to the billing email (the buyer), not the recipient
  • There is no "gift receipt" email that goes to the shipping address recipient
  • The packing slip included in the shipment by default shows the full order total and the buyer's email

Implementing gift receipt functionality

  • Suppress price on packing slip when gift attribute is set — in Admin → Settings → Notifications → Packing slip, add a conditional:
{% comment %} In packing-slip.liquid {% endcomment %}
{% assign is_gift = false %}
{% for attr in order.note_attributes %}
  {% if attr.name == 'Gift Order' and attr.value == 'Yes' %}
    {% assign is_gift = true %}
  {% endif %}
{% endfor %}

{% unless is_gift %}
  <!-- Show price section here -->
{% endunless %}
  • Send gift confirmation to recipient — use Klaviyo or Omnisend with an order trigger. Filter for orders where the gift_message attribute is not empty, then send a separate email to the shipping address contact email (if collected) with a friendly "You've received a gift!" message.
  • Use a gift wrap app with receipt functionality — apps like Gift Wrap & Message or Giftkart include recipient notification flows that send a personalized email to the gift recipient's email address (collected via the gift message form) when the order is placed or fulfilled.
