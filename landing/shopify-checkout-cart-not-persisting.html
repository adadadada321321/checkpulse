Shopify Checkout Troubleshooting

Shopify Cart Not Persisting — Items Disappearing, Cart Clearing After Checkout

Shopify carts live in a browser cookie. When that cookie is blocked, cleared, or expired — or when login merges go wrong — items vanish silently. Here's what's actually happening and how to fix each case.

Five Root Causes — Different Fixes

Most Common

Browser Cookie Blocked or Cleared

The cart token lives in a first-party cookie named cart. Safari ITP, private browsing, and cookie-clearing tools delete it. No cookie = no cart on next visit.

Login Flow

Anonymous-to-Logged-In Merge Failure

When a customer logs in, Shopify merges the anonymous cart into their account. Items can be silently dropped if the variant is OOS, already in the saved cart, or if the app stack interrupts the merge.

Inventory Change

Sold-Out Item Silently Removed

If a cart item goes out of stock between sessions, Shopify removes it from the cart on the next page load — no warning, no email. The customer just sees fewer items.

Multi-Device

Cart Doesn't Sync Across Devices

The cart cookie is device-specific. Items added on mobile don't appear on desktop unless the customer uses Shop Pay or a third-party saved-cart app.

App Conflicts

Cart Drawer or Upsell App Resets Cart

Some cart drawer apps, upsell apps, or custom checkout scripts replace the cart object on initialization, wiping items added by another channel or page reload.

How Shopify Carts Actually Work

Understanding the persistence model prevents most debugging dead ends.

Cart Storage

Server-Side Object, Browser Token

The cart itself lives on Shopify's servers as a cart object. The browser holds a cart cookie containing the cart token — a long random string like abc123def456.... Every cart operation (add, update, remove) uses that token to identify which server-side cart to modify.

Lifetime

30 Days from Last Activity

A cart persists for 30 days from the last item modification. The cart cookie is set to expire in 30 days and resets with each cart action. After 30 days of inactivity the cart is gone from both the server and the cookie. After checkout completes, the cart is deliberately cleared for the next session.

Key distinction: The cart is not stored in the browser's localStorage or sessionStorage in standard Shopify themes. It's a server-side object referenced by a cookie. This means clearing localStorage has no effect on the cart, but clearing cookies (or blocking them) breaks cart persistence entirely.

Inspecting the Cart Cookie

In Chrome DevTools: open Application → Cookies → your store domain. Look for a cookie named cart. If it's absent, the cart has no token and will appear empty. You can also fetch the cart directly:

/* Fetch current cart state */ GET /cart.js /* Response includes all cart items */ { "token": "abc123def456...", "item_count": 3, "items": [ { "variant_id": 12345678, "title": "Blue T-Shirt / Medium", "quantity": 2 } ] } /* If items is [] and item_count is 0, the cart is empty */ /* If the request returns a new empty cart token, the old cookie was lost */

Headless / Hydrogen Carts Are Different

Shopify's Hydrogen framework and custom headless storefronts use the Storefront API Cart object, not the classic Ajax Cart API. The cart is stored by a cartId (a global ID string) that the storefront must persist in localStorage, sessionStorage, or a cookie manually. If the headless app doesn't explicitly persist the cartId, cart contents are lost on every page reload. This is a framework responsibility — Shopify does not automatically handle persistence in headless implementations.

Failure Mode 1: Browser Cookie Blocking Loses the Cart Token

This is the most common cause of "cart was full, now it's empty" reports — especially from mobile Safari users.

Safari ITP (Intelligent Tracking Prevention): Apple's ITP restricts first-party cookies in some cross-site navigation scenarios. The cart cookie is a first-party cookie, so ITP doesn't normally delete it — but when a customer arrives at your store via a cross-site redirect (ad click, affiliate link, email link), ITP may classify the cookie as tracking-related and cap its lifetime to 24 hours instead of 30 days. This means a customer who added items via an ad click and returned the next day may find their cart empty. This affects iOS Safari in particular.

Common Cookie Blocking Scenarios

Scenario Effect on Cart Affected Browsers

Mitigations

Cart recovery email: Shopify's abandoned checkout emails capture the customer email at checkout entry. But if a customer adds items and never starts checkout, there's no email to trigger recovery. A third-party cart recovery app (like Klaviyo's Browse Abandonment or CartHook) can capture cart state before checkout begins.

Login incentive: If a customer is logged in, their cart can be associated with their account. While the cookie-based cart still applies, logged-in customers have the option for Shopify to prefill their cart from their last session. Encouraging guest customers to create accounts (or using Shop Pay) creates a fallback persistence layer.

Save cart URL: Shopify's Storefront API can generate a permalink for the cart: /cart/{variant_id}:{quantity}. Sending this URL to the customer via email lets them restore a specific cart without relying on cookie persistence.

Failure Mode 2: Anonymous-to-Logged-In Cart Merge Failure

When a customer who has been shopping as a guest logs into their account, Shopify attempts to merge the anonymous cart into the customer's account. This merge process has several failure points.

How cart merge works: Shopify associates each customer account with a saved cart — the last cart object the customer had when they were logged in. When a customer logs in, Shopify compares the anonymous cart items against the saved cart and attempts to combine them. The merge is additive (not replace), but items are silently dropped if: the variant is now out of stock with "Continue selling" disabled, the item has since been deleted from the store, or the merge would exceed Shopify's cart line item limit (currently 500 items, practically irrelevant for most stores). There is no error shown to the customer — the cart just has fewer items after login.

Merge Failure Scenarios

Most Confusing

Saved Cart Overwrites Anonymous Cart

If the customer has an older saved cart from a previous logged-in session that still has valid items, that saved cart can take precedence over the new anonymous items. The customer sees the old cart, not what they just added as a guest.

Silent Drop

Variant OOS Between Sessions

The customer adds Size M of a shirt while browsing as a guest. Size M sells out before they log in. During merge, Shopify drops that line item without notification. The customer sees the cart shrink after login.

App Interference

Upsell or Loyalty App Re-Initializes Cart

Some upsell apps or loyalty/reward apps reinitialize the cart object on the customer login event, replacing the merged cart with a fresh one based on their own logic. This is an app configuration issue, not native Shopify behavior.

How to Test the Merge Flow

Clear all cookies, open the store in normal mode

Start fresh. Add 2–3 specific items as a guest — note the exact variants and quantities. Do not start checkout.

Log in with a test customer account

Use a test account that has either (a) no previous saved cart, or (b) a known different saved cart. Note which scenario you're testing.

Check the cart immediately after login

Without refreshing or navigating, check the cart count. Fetch /cart.js in the browser console to see the exact current state. Compare to what you added as a guest.

Disable apps one by one if merge fails

If the cart isn't merged correctly, check apps that have "cart" or "checkout" permissions. Disable them temporarily in Shopify admin → Apps and re-run the test to isolate the culprit.

Failure Mode 3: Out-of-Stock Items Removed Silently

Shopify doesn't hold inventory for items in the cart — only for items in an active checkout. This means a product in a customer's cart can sell out at any time, and Shopify will silently remove it from the cart on the next cart page load.

How It Happens

No Inventory Hold at Cart Stage

Shopify only reserves inventory when a checkout is created (the customer clicks "Proceed to Checkout"). Items sitting in the cart have no inventory hold. Another customer can purchase the last unit, and your customer's cart will silently lose the item on their next visit.

No Notification

Customer Sees Fewer Items, No Explanation

There is no built-in notification when a cart item is removed due to OOS. The cart simply loads with fewer items. This is a significant source of silent abandoned cart — the customer assumes they removed the item themselves or that the site has a bug.

When "Continue Selling When Out of Stock" Changes Everything

In Shopify admin → Products → select product → Variants → Inventory → check "Continue selling when out of stock." If this is enabled, items stay in the cart even when inventory hits zero. They'll still be purchasable. If it's disabled (the default), inventory-tracked variants at zero stock are removed from the cart silently.

Practical recommendation: For popular products that frequently sell out, consider enabling "Continue selling when out of stock" to prevent silent cart drops — then manage backorder expectations via email after purchase. Alternatively, use a "Back in Stock" app (like Klaviyo, Restock Rocket, or Back in Stock by Appikon) that can email waiting customers and notify them before items actually sell out.

Product Deleted or Archived — Immediate Cart Drop

If a merchant archives or deletes a product that a customer has in their cart, that item is removed from the cart immediately on the next cart load. There is no grace period and no customer notification. This commonly happens during end-of-season catalog cleanup or when a product is replaced by a newer version. Always check active carts before archiving products — in Shopify admin, look at Analytics → Reports → Sales by product to see if there are pending carts before deleting.

Failure Mode 4: Cart Doesn't Sync Across Devices

This isn't a bug — it's the expected behavior of Shopify's cookie-based cart system. But it's one of the top sources of customer confusion.

How it works: The cart cookie is stored in a specific browser on a specific device. A customer who adds items on their iPhone has a cart token in Safari on that iPhone. When they switch to their laptop browser, there's no cart cookie there — so they see an empty cart. The two carts are completely independent server-side objects. Unless the customer is signed into Shop Pay, there's no automatic sync.

When Cart DOES Sync Across Devices

Shop Pay

Shop Pay Account Persistence

If a customer is signed into their Shop Pay account on multiple devices, Shop Pay can restore their last cart state. This is not native Shopify cart syncing — it's Shop Pay infrastructure. The customer must actively sign into Shop Pay on each device.

Logged-In + Same Browser

Account Saved Cart

Logged-in customers on the same browser have their cart associated with their account. If they log into their account on a second device using the same browser, Shopify may restore their saved cart. This is not guaranteed and depends on the merge behavior described above.

Third-Party Apps

Saved Cart Apps

Apps like Cart Saver, Save My Cart, or custom implementations using the Storefront API can persist cart state server-side and restore it across devices via login. These require the customer to have an account.

Shareable Cart URL Workaround

Shopify supports cart permalinks that can recreate a specific cart from a URL. The format is:

/* Single item cart permalink */ https://yourstore.com/cart/VARIANT_ID:QUANTITY /* Multiple items */ https://yourstore.com/cart/VARIANT_ID_1:QTY_1,VARIANT_ID_2:QTY_2 /* With discount pre-applied */ https://yourstore.com/cart/VARIANT_ID:QUANTITY?discount=SUMMER10 /* Example use: email the customer their cart link */ https://yourstore.com/cart/39484732:1,49283748:2?discount=COMEBACK

These permalinks recreate the cart from scratch using variant IDs — they don't restore a specific cart token, but they let the customer pick up where they left off on any device. Useful for cart abandonment emails, customer service recovery, or marketing campaigns.

Failure Mode 5: App Conflicts Reset or Replace the Cart

Third-party apps that interact with the cart object — particularly cart drawer apps, upsell/cross-sell apps, and loyalty reward apps — can inadvertently reset or replace the cart.

How App Cart Conflicts Happen

Shopify's Ajax Cart API uses the cart token cookie to identify which cart to modify. Most apps that interact with the cart use JavaScript on the storefront to call /cart/update.js, /cart/add.js, or /cart/change.js. Conflicts happen when:

  • Two cart apps both listen to the cart:updated event and each re-renders the cart, causing a race condition where one app's version of the cart overwrites the other's
  • An app uses /cart/clear.js to reset the cart before adding its own items (common in bundle builders and subscription apps)
  • A checkout extension or post-purchase app modifies cart attributes in a way that causes the storefront cart to re-initialize
  • An upsell app that adds "free gift" items removes them on re-initialization if conditions aren't re-met, and the re-initialization inadvertently re-fetches a stale cart state
/* Detect which app is modifying the cart — intercept fetch calls */ const origFetch = window.fetch; window.fetch = function(...args) { if (args[0] && args[0].includes('/cart/')) { console.log('Cart API call:', args[0], new Error().stack); } return origFetch.apply(this, args); }; /* Paste into DevTools Console, then interact with the page */ /* The stack trace shows which script is making cart API calls */

Isolating the Problem App

Reproduce in a clean theme

In Shopify admin → Online Store → Themes, duplicate your current theme and temporarily switch to Shopify's Dawn theme. Test the cart persistence issue. If it disappears, the problem is in your theme's cart code or an injected script.

Disable apps with cart permissions

In Shopify admin → Apps, look for apps that have requested "read/write cart" or "checkout" permissions. Disable them one at a time, testing after each, until the cart persistence issue stops.

Use the DevTools intercept above

Paste the fetch interceptor into your browser console. Trigger the cart loss scenario (add items, navigate away, come back). Look for unexpected calls to /cart/clear.js or /cart.js with empty payloads in the stack trace.

8-Step Cart Persistence Diagnostic

Run through this in order. Most issues resolve by step 4.

Reproduce in Chrome (non-incognito, no extensions)

If the cart loss only happens in Safari or incognito, it's a cookie issue (Failure Mode 1). If it happens in Chrome normal mode too, continue to step 2.

Check the cart cookie after adding items

Open DevTools → Application → Cookies → your store domain. Look for a cookie named cart with an expiry ~30 days out. If it's absent after adding items, there's a cookie-setting issue (possibly a CSP or SameSite attribute problem).

Navigate away and back — does the cookie survive?

Add items, navigate to Google, come back to your store. Check if the cart cookie is still present. If it disappeared, a script on your page may be clearing cookies (check your theme's JavaScript for document.cookie = 'cart=' or expires=Thu, 01 Jan 1970).

Test the login merge with a known cart

Add 2 specific items as a guest. Log in with a test account. Check if both items survive. If items are dropped during login, see Failure Mode 2 diagnostics above.

Check inventory status of the disappearing items

In Shopify admin → Products → Inventory, look at the items that disappear. Are they at zero stock? Is "Continue selling when out of stock" disabled? If yes, that's Failure Mode 3.

Test cross-device by adding items on mobile, checking on desktop

If the complaint is "cart empty on desktop, had items on phone," that's expected behavior unless Shop Pay or a saved-cart app is configured. This is Failure Mode 4.

Disable apps and re-test

Temporarily disable all apps with cart/checkout permissions. Test cart persistence. Re-enable one by one to find the culprit. See Failure Mode 5.

Check for headless / Hydrogen implementation

If you're using a headless storefront, the cart persistence is your framework's responsibility. Confirm the cartId is stored in localStorage or cookies and retrieved on every page load. If the framework discards the cartId on navigation, the cart starts fresh each time.
