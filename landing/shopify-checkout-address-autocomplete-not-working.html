Shopify Checkout — Address Autocomplete

Shopify Checkout Address Autocomplete Not Working

Shopify checkout uses Google Maps Places API for address autocomplete. When suggestions stop appearing, fill incorrectly, or fail silently on mobile — there are five distinct root causes. This guide covers each one with exact fix paths.

5 root causes 6 silent failure modes 3 deep dives Updated June 2026
Key insight: Shopify does not build or maintain its own address autocomplete — it relies entirely on Google Maps Places API. When autocomplete breaks, the fault is almost always one of: a misconfigured or quota-exhausted API key, a checkout extension overriding the address form, browser-level API blocking, or the customer's country not being supported by the Google Places API. Shopify's native checkout (without extensions) cannot "fix" autocomplete on its own if the underlying Maps API call fails.
Architecture

How Shopify Checkout Address Autocomplete Works

Understanding what Shopify actually does under the hood is the fastest path to diagnosing autocomplete failures. The system has four layers, each of which can fail independently.

Layer 1
Google Maps Places API
Shopify calls the Places Autocomplete API when a customer types in the address field. Google returns a list of candidate addresses. Shopify renders them as a dropdown. If the API call fails or returns no results, no dropdown appears.
Layer 2
API Key (Shopify or Merchant)
By default, Shopify uses its own platform-level Google Maps API key. Some apps and themes allow merchants to configure their own key. If a merchant-configured key is invalid, expired, or quota-exhausted, autocomplete fails for that store only.
Layer 3
Address Field Mapping
When a customer selects a suggestion, Shopify maps the Google Places "address components" (street number, route, locality, postal code, country) to specific checkout form fields. If field IDs are wrong (due to extension modification), mapping breaks and only some fields populate.
Layer 4
Browser / Network
Content blockers (uBlock Origin, Ghostery, Brave Shield), corporate firewalls, or regional restrictions (China, Iran) can block the Maps API request entirely before it reaches Google. The autocomplete script loads but the API call is silently dropped.

Deep Dive A

Google Maps Places API — Quota, Keys, and How to Test

Most merchant-side autocomplete failures originate from API key configuration. Here is how to diagnose and resolve them.

Deep Dive — API Key Diagnosis

Shopify Default Key vs. Merchant-Configured Key

By default, Shopify uses a platform-wide Google Maps API key shared across all Shopify stores. Merchants do not have visibility into or control over this key's quota. If Shopify's platform-level key experiences quota issues, all stores are affected simultaneously — this is extremely rare and would be reported on Shopify's status page.

The more common scenario: a merchant (or their developer/app) configured a custom Google Maps API key in Shopify's theme settings or through an app like "Address Validator," "Store Locator," or a checkout customization app. This custom key overrides Shopify's default. If this custom key has: (a) daily quota exhausted, (b) API restrictions that don't include "Places API" or "Maps JavaScript API," (c) HTTP referrer restrictions that don't include the checkout domain, or (d) been invalidated in the Google Cloud Console — autocomplete fails only for that store.

How to identify if a custom key is configured: Shopify Admin → Online Store → Themes → Edit Code → search for "google_maps_api_key", "maps_api_key", or "MAPS_API_KEY" in theme files. Also check Apps → look for any address or mapping-related app → check the app's settings page for an API key field.

Fastest fix: Remove the custom API key entirely (clear the field, save). Shopify reverts to its platform default. Test checkout autocomplete immediately. If autocomplete returns, your custom key was the problem — fix the key configuration in Google Cloud Console or leave Shopify's default key in place.

How to check Google Cloud Console for a custom key: console.cloud.google.com → APIs & Services → Credentials → click the API key → check "API restrictions" (must include Places API) → check "Application restrictions" (HTTP referrers must include *.myshopify.com/* and your custom checkout domain). A wildcard (*) referrer restriction is acceptable for testing but not recommended for production (security risk).

Deep Dive — Quota Monitoring

Understanding Google Maps Places API Quota

Google Maps Places Autocomplete API has a free tier of 0 requests (Places API switched to a fully paid model in 2018 with a $200/month credit). Each autocomplete session costs approximately $0.017 per session (a session = all keystrokes + one place detail lookup). For a store with 1,000 daily checkout visitors and an average of 80% entering an address with autocomplete: ~800 sessions/day × $0.017 = ~$13.60/day = ~$408/month. The $200 Google credit covers about half — merchants should budget ~$200+/month for autocomplete on any meaningful volume.

If your store suddenly stops showing autocomplete mid-day, check Google Cloud Console → APIs & Services → Quotas → Maps API Requests. If you see a sharp drop to zero that correlates with when autocomplete stopped, you've hit your billing limit or daily quota cap. Increase your billing limit or remove the per-day spend cap in Google Cloud Console billing settings.

Per-Country Address Autocomplete Availability

Google Maps Places API coverage varies significantly by country. For international stores, this directly affects which customers can use autocomplete and which must enter their address manually.

Country / Region Autocomplete Support Notes for Shopify Merchants
United States, Canada, UK, Australia Full Address-level precision, postal code + city auto-populate reliably
Germany, France, Netherlands, Belgium Full German addresses: house number comes after street name in German format; autocomplete handles this correctly
Japan Partial Japanese addresses use prefecture → city → ward → block → building hierarchy; autocomplete covers major cities well, rural areas less reliably. Kanji vs. romaji entry affects results.
Brazil Partial Major cities (São Paulo, Rio, Brasília) have good coverage. CEP (postal code) lookup is reliable. Smaller municipalities or new developments may not have structured data.
India Partial Urban addresses in major cities work well. PIN codes are supported but city/state auto-populate is inconsistent for smaller towns and new pin zones.
Mexico Partial Colonia (neighborhood) is a required part of Mexican addresses; autocomplete may not populate it consistently. Manual entry often needed.
China Blocked Google services are blocked in mainland China. Autocomplete will never work for customers in China. Manual entry required. Consider a China-specific address app if significant volume.
Iran, North Korea, Cuba, Sudan, Syria Restricted US export controls restrict Google services. Autocomplete unavailable. Manual entry only.
Southeast Asia (Indonesia, Vietnam, Philippines) Limited Major city coverage is good. Provincial/rural addresses may have incomplete structured data. Postal code lookup reliability varies by country.
Sub-Saharan Africa Limited Major capitals covered. Many rural areas lack street-level data in Google Maps. Informal settlement addresses may not appear at all.
Deep Dive C

Checkout Extension Conflicts — Diagnosis and Fix

Checkout extensions (apps that modify the Shopify checkout form) are the most common cause of address autocomplete breaking. Here is a systematic diagnosis process.

Deep Dive — Extension Conflict Diagnosis

How Extensions Break Autocomplete

Shopify's address autocomplete script targets checkout form fields by their HTML ID attributes (e.g., #checkout_shipping_address_address1). When a checkout extension replaces or wraps these fields with its own components, the IDs can change (e.g., #ext-address-field-1). The autocomplete script no longer finds the fields it expects, so it either stops initializing or fires events on elements that don't exist — the dropdown appears but selecting a suggestion does nothing.

Diagnostic test (2 minutes): Open your store's checkout in a private/incognito window on desktop Chrome. Go through to the shipping address step. Type 3–4 characters of a known address in the address field. If the Google Maps dropdown appears and works correctly — and you know it doesn't work for some customers — the issue is browser-specific (Cause 3: content blocker). If the dropdown doesn't appear even in incognito clean Chrome, continue to extension diagnosis.

Extension isolation: Admin → Apps → identify apps with "checkout," "upsell," "post-purchase," "rebuy," "bold," "zipify," or "pagefly" in the name. Temporarily uninstall or disable the extension from the checkout (in the app's settings, not in Apps). Retest autocomplete. Repeat until you find the conflicting app. Contact that app's developer with the specific conflict: "Your extension replaces the native Shopify checkout address field IDs, breaking Google Maps autocomplete. Please implement your field overrides in a way that preserves the native field IDs or re-attaches the autocomplete listener after DOM modification."

Shopify Plus — Checkout Editor: If you've customized your checkout layout in the Checkout Editor (Admin → Settings → Checkout → Customize), a layout change that moves or duplicates the shipping address section can break the autocomplete listener's DOM targeting. Open the Checkout Editor → Sections → Address → verify the shipping address block is using the default Shopify implementation, not a custom component. Reverting to the default address block fixes this in most cases.

Checkout UI Extensions: If you've deployed a custom Checkout UI Extension that modifies the shipping address step, the extension's component tree may shadow the native address fields. In your extension code: do not replace the <ShippingAddressField /> component with a custom text input — instead, add your UI before or after the native field using the extension's supported slot positions.

Fix Checklist

6-Step Address Autocomplete Fix Checklist

Run through these steps in order. Most stores find the fix at step 1 or 2.

  • 1
    Test in clean incognito Chrome — establish baseline Open checkout in a private Chrome window with no extensions. Type 3–4 characters in the address field. If autocomplete appears and works → customer-side blocker (step 3). If it doesn't work → store-side issue (continue to step 2).
  • 2
    Check for a custom Google Maps API key in your store Admin → Online Store → Themes → Edit Code → search "google_maps_api_key". Also check any address-related apps for custom key settings. If a custom key exists, remove it and test. Shopify reverts to its platform-level key — if autocomplete returns, your custom key was the issue.
  • 3
    Identify and isolate checkout extension conflicts Admin → Apps → temporarily disable apps that touch checkout (upsells, address validators, checkout customizers). Retest after each disable. When autocomplete returns, you've found the conflicting app — contact their support with the specific DOM conflict.
  • 4
    Verify custom domain referrer restriction in Google Cloud Console If you use a custom checkout domain: console.cloud.google.com → APIs & Services → Credentials → your API key → Application restrictions → HTTP referrers. Add your custom checkout domain (e.g., checkout.yourdomain.com/*) to the allowed list. Without this, the Maps API will reject requests from your custom domain.
  • 5
    Test on mobile specifically — check for native keyboard conflict Open checkout on an iPhone (Safari) and Android (Chrome). On iOS: if the keyboard autocomplete bar shows address suggestions above the keyboard instead of Google's dropdown, this is native keyboard conflict — expected behavior, not a bug. On Android: if Google Autofill overlay appears instead of Maps suggestions, disable Google Autofill for the checkout URL in Chrome settings temporarily to test.
  • 6
    Communicate country-specific limitations to affected customers For customers in China, Iran, or countries with limited Google Maps coverage: add a help text note at the address step ("Address suggestions may not be available in your country — please enter your address manually"). This can be done via Checkout Editor (Plus) or a Checkout UI Extension. It reduces support tickets and abandonment from customers confused about why autocomplete doesn't appear.
FAQ

Address Autocomplete — Frequently Asked Questions

Why did Shopify checkout address autocomplete suddenly stop working?
The most common sudden-stop causes: (1) a checkout extension was installed or updated that conflicts with the autocomplete script, (2) a custom Google Maps API key was added or modified and has a configuration error. Test in clean incognito Chrome first. If it works there, look at browser extensions. If it doesn't, check your store's checkout apps and API key settings.
Why do address suggestions appear but selecting one doesn't fill all the fields?
Partial autofill — the most common silent autocomplete failure. Causes: (1) the Google Places result for that address is missing structured components (no postal code data, city unclear); (2) a checkout extension changed address field IDs so Shopify's mapping script can't find them. Test with a well-known urban address (e.g., "123 Main St, New York") — if that populates correctly but your customer's address doesn't, it's a data gap issue. If even the test address partially fails, it's an extension conflict.
Why does address autocomplete work on desktop but not on mobile?
Mobile platforms have their own address autocomplete systems that can conflict with Google Maps suggestions. On iOS Safari, native keyboard address suggestions and content blockers can prevent the Google dropdown from appearing. On Android Chrome, Google Autofill sometimes intercepts address fields. Test in Chrome on Android first (fewest conflicts). If mobile-only: this is a known cross-platform UX limitation — Shopify doesn't have a server-side fix for native keyboard override.
Why is the postal code lookup returning the wrong city?
Google Maps returns one canonical city for each postal code. If that postal code spans multiple communities or was reassigned in a municipal consolidation, Google's data may show a different city than what your customer considers their address. This is a Google Maps data accuracy issue, not a Shopify bug. Add a note in checkout: "Please verify the auto-populated city and state before placing your order."
Can I add my own Google Maps API key to improve autocomplete in Shopify?
Shopify does not provide an official merchant-facing setting to configure a custom Google Maps API key for checkout autocomplete. Some third-party apps offer this, but using a custom key introduces dependency on your own key's quota, billing, and configuration — and often breaks autocomplete when the key is misconfigured. In most cases, Shopify's platform-level key performs better for checkout specifically. Only use a custom key if you have a specific, confirmed need (e.g., using a custom domain with strict referrer restrictions).
Why does address autocomplete show results from the wrong country?
Shopify passes a country bias parameter to the Google Places API based on the customer's detected market or the store's primary market. If Shopify's market detection is incorrect (e.g., a German customer is placed in the US market), autocomplete suggestions will be biased toward the wrong country. Fix: Admin → Settings → Markets → verify the correct market is configured for the relevant countries, including correct geographic definitions. Customers in the correctly configured market should then see country-appropriate autocomplete suggestions.
