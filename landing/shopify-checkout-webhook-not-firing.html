Root Causes

Five Reasons Shopify Webhooks Stop Firing

1. Endpoint Timeout — The 5-Second Rule

Shopify requires your webhook endpoint to return an HTTP 2xx response within 5 seconds of receiving the POST request. If your endpoint takes longer — even 5.1 seconds — Shopify marks the delivery as failed and schedules a retry.

This is the most common cause of intermittent missing webhooks. It happens when:

  • Your endpoint performs synchronous database writes before responding
  • Your endpoint calls an external API (shipping, ERP, email) before responding
  • Your server is under load and response time spikes beyond 5 seconds
  • Cold start latency (serverless functions) adds initial delay

Fix: Decouple response from processing. Return HTTP 200 immediately upon receiving the webhook, then process the payload asynchronously in a background job or queue.

// Node.js example — return 200 immediately
app.post('/webhooks/orders-create', express.raw({type: 'application/json'}), (req, res) => {
  // 1. Verify HMAC first (fast — no I/O)
  if (!verifyShopifyHmac(req)) {
    return res.status(401).send('Unauthorized');
  }
  // 2. Return 200 BEFORE any async work
  res.status(200).send('OK');
  // 3. Process asynchronously
  processOrderWebhook(req.body).catch(console.error);
});

2. Subscription Auto-Disabled After 19 Failures

If your endpoint returns non-2xx responses or times out on 19 consecutive delivery attempts, Shopify automatically deletes the webhook subscription with no notification. After deletion, zero webhooks arrive. There is no alert, no email, no dashboard warning.

The retry schedule uses exponential backoff — retries spread across approximately 48 hours before the subscription is deleted. This means an endpoint outage over a weekend can silently kill your webhooks with no one noticing until orders are missing from your system.

How to check:

# Shopify Admin API — list all webhook subscriptions
GET /admin/api/2024-04/webhooks.json

# If your subscription is missing from the response, it was deleted.
# Re-register it:
POST /admin/api/2024-04/webhooks.json
{
  "webhook": {
    "topic": "orders/create",
    "address": "https://your-endpoint.com/webhooks/orders-create",
    "format": "json",
    "api_version": "2024-04"
  }
}
Prevention: Set up uptime monitoring on your webhook endpoint URL. If your endpoint goes down, you want to know before 19 retries exhaust and Shopify deletes the subscription. CheckPulse monitors your checkout and can alert you to endpoint failures.

3. API Version Sunset

Shopify deprecates API versions quarterly and eventually sunsets them. A webhook subscription created against a sunset API version stops delivering — Shopify stops processing delivery for webhooks on unsupported versions.

Shopify's API version support window: each version is supported for approximately 12 months after release, with quarterly releases (2024-01, 2024-04, 2024-07, 2024-10). Check the Shopify API versioning page for current support status.

Fix:

  • List your webhook subscriptions and note the api_version field for each
  • Delete subscriptions on sunset or deprecated versions
  • Re-register subscriptions specifying a current API version (e.g., 2024-04)
  • Include API version upgrade in your quarterly maintenance checklist

4. Wrong Topic — orders/create vs checkouts/create

Shopify's webhook topics map to specific events, and the most common confusion is between order events and checkout events. orders/create fires only when an order is confirmed — payment accepted, free order placed, or draft order converted. It does NOT fire for abandoned carts.

TopicWhen It FiresUse Case
orders/create Order confirmed (payment accepted or free/draft) Fulfillment, inventory, ERP sync, confirmation email
orders/paid Payment captured on an existing order When you need confirmed payment, not just order creation
orders/updated Any field on an existing order changes Address edits, line item changes, fulfillment updates
checkouts/create Customer initiates a checkout session Abandoned cart capture, checkout analytics
checkouts/update Checkout modified (email entered, address, shipping selected) Progressive profile building, real-time cart recovery
checkouts/delete Checkout expires (abandoned after ~3 hours) Abandoned cart cleanup, analytics

Note: The checkouts/* topics require the read_checkouts access scope. If your app does not have this scope, subscribing to checkout topics will return a 422 error.

Also note: test orders placed from the Shopify Admin do not trigger orders/create webhooks by default in some configurations. Use the Shopify CLI or Partner Dashboard webhook trigger to test without placing real orders.

5. HMAC Verification Rejecting Valid Payloads

Shopify sends a X-Shopify-Hmac-SHA256 header with every webhook — an HMAC-SHA256 signature of the raw request body, base64-encoded. If your verification logic rejects valid Shopify webhooks, your endpoint returns 401 or 500, Shopify marks it as failed, and retries begin.

The three most common implementation mistakes:

  • Parsing JSON before verifying: HMAC is computed over the raw bytes. If you call JSON.parse(body) and then re-serialize before hashing, the byte-level representation may differ.
  • Wrong secret: Public apps use the client secret from the Partner Dashboard. Custom apps use the webhook secret configured in Admin → Settings → Notifications. These are different values.
  • Non-constant-time comparison: Using string equality (===) instead of a constant-time function exposes a timing attack surface and can also produce false mismatches due to encoding differences.
// Node.js — correct HMAC verification
const crypto = require('crypto');

function verifyShopifyHmac(req) {
  const hmacHeader = req.headers['x-shopify-hmac-sha256'];
  const secret = process.env.SHOPIFY_WEBHOOK_SECRET;
  // req.body must be the RAW Buffer — not parsed JSON
  const digest = crypto
    .createHmac('sha256', secret)
    .update(req.body) // raw Buffer
    .digest('base64');
  // constant-time comparison
  return crypto.timingSafeEqual(
    Buffer.from(digest),
    Buffer.from(hmacHeader)
  );
}

If you rotated your app's client secret, existing webhook subscriptions still use the old secret. Delete and re-register all webhook subscriptions after a secret rotation.

Silent Failures

Seven Ways Shopify Webhooks Fail Without Warning

1

Subscription deleted after 19 failures

No email, no dashboard alert. Your subscription simply stops existing. Orders continue coming in; your system stops receiving them.

2

Endpoint returns 200 but silently drops the payload

HMAC verification fails, your code returns 200 anyway (to suppress retry), but never processes the order. Shopify considers it delivered. Your system has a hole.

3

Test order does not trigger orders/create

Shopify test orders skip some webhook delivery paths in development store mode. You test, see nothing, assume webhooks are broken — but they work fine in production.

4

Order stuck in payment pending — orders/create never fires

orders/create fires on order confirmation. If a payment is in pending state (bank redirect, BNPL authorization delay), the webhook does not fire until the payment resolves.

5

API version sunset — silent delivery stop

Shopify stops delivering on sunset API versions. No error. Webhooks that worked for months simply stop after the version's end-of-life date.

6

WAF or rate limiter blocking Shopify IP ranges

Cloudflare, AWS WAF, or a custom rate limiter blocks Shopify's delivery IPs. Webhooks arrive in bursts, the limiter triggers, deliveries fail, retries exhaust, subscription is disabled.

7

Delivery to staging URL instead of production

Webhook registered pointing to a staging or ngrok URL that no longer exists. Deliveries fail silently. Production processes nothing. The subscription URL is easy to overlook during a deployment.

Deep Dives

Three Topics Worth Understanding Fully

A. Webhook Delivery Mechanics — Timeout, Retry, and Auto-Disable

Understanding exactly how Shopify delivers webhooks helps you build a system that never loses an event.

The 5-second timeout

Shopify initiates a POST request to your endpoint and waits a maximum of 5 seconds for any HTTP response. If no response arrives in 5 seconds — or if the response is non-2xx — the delivery is marked failed. There is no body inspection; only the status code matters. Return 200 and nothing else as quickly as possible.

Retry schedule

Shopify uses exponential backoff across 19 retry attempts spanning approximately 48 hours:

AttemptApprox. delay after previousTime from first failure
1 (initial)—0
2~1 min~1 min
3~5 min~6 min
4~10 min~16 min
5~30 min~46 min
6-10~1h each~5h
11-19~4-6h each~48h

After attempt 19, Shopify deletes the subscription. No further deliveries occur. You must re-register.

Checking delivery health

Query your webhooks via Admin API and verify the subscription list matches what you expect. For partner apps, the Shopify CLI provides a trigger command:

# Trigger a test webhook delivery (Shopify CLI)
shopify webhook trigger --topic=orders/create --delivery-url=https://your-endpoint.com/webhooks/orders-create

# List all registered webhook subscriptions
curl -X GET \
  "https://your-store.myshopify.com/admin/api/2024-04/webhooks.json" \
  -H "X-Shopify-Access-Token: your-token"

Idempotency

Because Shopify retries on failed deliveries, your endpoint may receive the same webhook multiple times. Always implement idempotent processing: check if you have already handled the order ID before writing to your database or calling external APIs. Use the X-Shopify-Webhook-Id header as a deduplication key.

  • Subscription orders (Recharge, Bold): Subscription apps typically generate orders through the Shopify API — orders/create fires for each renewal order they create. However, the renewal logic itself lives in the app, not in Shopify native webhooks.
  • Draft orders: Draft order lifecycle has its own topics: draft_orders/create, draft_orders/update, draft_orders/delete.
  • GDPR webhooks: Three GDPR webhooks are mandatory for public apps: customers/data_request, customers/redact, shop/redact. These must be registered and respond with 200 or Shopify can revoke app access.

C. HMAC Verification — Correct Implementation and Common Mistakes

HMAC verification is how you confirm a webhook actually came from Shopify and was not tampered with in transit. Getting it wrong breaks all webhook processing.

How Shopify computes the HMAC

  1. Take the raw request body bytes (exactly as received — no parsing, no re-serialization)
  2. Compute HMAC-SHA256 using your webhook secret as the key
  3. Base64-encode the result
  4. Send as the X-Shopify-Hmac-SHA256 header

Your verification must mirror this exactly.

Correct implementation (Node.js)

const crypto = require('crypto');

// Middleware: parse body as raw buffer
app.use('/webhooks', express.raw({ type: 'application/json' }));

function verifyWebhook(req, res, next) {
  const hmacHeader = req.headers['x-shopify-hmac-sha256'];
  if (!hmacHeader) return res.status(401).send('Missing signature');

  const secret = process.env.SHOPIFY_WEBHOOK_SECRET;
  const digest = crypto
    .createHmac('sha256', secret)
    .update(req.body) // Buffer — NOT JSON.parse(req.body)
    .digest('base64');

  // Use timingSafeEqual to prevent timing attacks
  const safe = crypto.timingSafeEqual(
    Buffer.from(digest, 'utf8'),
    Buffer.from(hmacHeader, 'utf8')
  );

  if (!safe) return res.status(401).send('Invalid signature');
  next();
}

Correct implementation (Python)

import hmac
import hashlib
import base64

def verify_webhook(data: bytes, hmac_header: str, secret: str) -> bool:
    # data must be the raw request body bytes
    digest = hmac.new(
        secret.encode('utf-8'),
        data,
        hashlib.sha256
    ).digest()
    computed = base64.b64encode(digest).decode('utf-8')
    # constant-time comparison
    return hmac.compare_digest(computed, hmac_header)

Common mistakes

Wrong: Parse JSON first

// BAD — JSON re-serialization changes bytes
const body = JSON.parse(req.body);
const raw = JSON.stringify(body);
const digest = hmac.update(raw);

Correct: Raw buffer

// GOOD — raw bytes before any parsing
const digest = hmac
  .update(req.body) // Buffer
  .digest('base64');

After rotating your secret

When you rotate your app's client secret in the Partner Dashboard, Shopify does not automatically update existing webhook subscriptions. You must:

  1. Get the list of all webhook subscriptions (GET /admin/api/2024-04/webhooks.json)
  2. Delete each subscription (DELETE /admin/api/2024-04/webhooks/[id].json)
  3. Re-register each subscription — they will now use the new secret
Diagnostic Checklist

7-Step Webhook Debugging Process

1

Verify the subscription still exists

Query GET /admin/api/2024-04/webhooks.json and confirm your subscriptions are present. If missing → they were auto-disabled. Re-register and fix the endpoint.

2

Check the API version on each subscription

Note the api_version field. Cross-reference against Shopify's current supported versions. Delete and re-register subscriptions on deprecated versions.

3

Send a test webhook to your endpoint

Use shopify webhook trigger --topic=orders/create --delivery-url=your-url and check if your endpoint receives and processes it correctly. Check endpoint logs for errors or timeouts.

4

Measure your endpoint response time

Log the time from request receipt to HTTP response. If it exceeds 3 seconds regularly, you are at risk of exceeding the 5-second limit under load. Move processing to a background queue.

5

Verify HMAC with a known-good payload

Use Shopify CLI's triggered payload to confirm your HMAC verification passes. If it fails: check you are reading raw body bytes, using the correct secret, and doing base64 comparison.

6

Confirm you are subscribed to the right topic

Map what event you are trying to catch (abandoned cart? confirmed order? payment?) to the correct Shopify topic. See the topic reference table above.

7

Check your WAF and rate limiter for Shopify IPs

If you have a WAF (Cloudflare, AWS, Nginx rate limit), confirm Shopify's delivery IP ranges are allowlisted. A WAF blocking Shopify IPs causes intermittent failures that look like random webhook drops.
