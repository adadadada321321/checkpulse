Shopify One-Page Checkout Not Working

Layout migration conflicts, extension rendering failures, payment section stuck loading — why one-page checkout breaks and how to diagnose each failure type.

Last updated June 2026  ·  Applies to: All plans (Checkout Extensibility: Plus only)  ·  CheckPulse
The core problem Shopify replaced the traditional 3-page checkout flow with a single-page layout in 2023. Apps, scripts, and themes built for the 3-page flow made assumptions about page URLs, DOM structure, and load order that the one-page layout violates. On Shopify Plus, one-page checkout is required for Checkout Extensibility — the replacement for the now-deprecated checkout.liquid — meaning Plus merchants who migrate lose all prior customizations until they rebuild them as UI extensions. The result: broken upsell widgets, payment sections that stall on "Loading," app extensions that render on desktop but disappear on mobile, and confusing errors with no clear owner.
Architecture

Shopify's Four Checkout Layers

Understanding which layer owns your checkout behavior is the first step in diagnosing failures.

Layer 1

3-Page Checkout (Legacy)

Contact → Shipping → Payment as three separate page loads. URLs: /checkout/contact, /checkout/shipping, /checkout/payment. Now replaced for all stores by one-page checkout. Apps targeting these URLs or their DOM no longer work.

Layer 2

One-Page Checkout (2023+)

All three steps on a single React page. Steps collapse and expand sequentially. A single URL (/checkout) for the whole flow. Default for all new stores since August 2023, rolled out to existing stores through 2024. No page navigations — section transitions are client-side.

Layer 3

Checkout Extensibility (Plus)

The official customization API for one-page checkout. Replaces checkout.liquid. Includes: Checkout UI Extensions (React components in designated slots), Checkout Branding API (visual style), Shopify Functions (discount, shipping, payment logic). Requires Plus plan and Checkout Extensibility enabled in partner dashboard.

Layer 4

Accelerated Checkout

Shop Pay, Apple Pay, Google Pay, and PayPal Express — these bypass the checkout page entirely for repeat customers. They have their own UI surfaces separate from one-page checkout. Conflicts with one-page layout arise when express buttons appear in unexpected positions or when their completion flow redirects incorrectly.

checkout.liquid is deprecated — this is the most common source of confusion

Shopify Plus merchants who customized checkout via checkout.liquid (a theme file) had that file removed when they enabled Checkout Extensibility. All custom CSS, JavaScript, upsell widgets, address restrictions, and form field modifications in checkout.liquid stopped working. The replacement requires rebuilding each customization as a Checkout UI Extension or using the Branding API — a completely different development model. checkout.liquid was deprecated in August 2024 for new stores and all Checkout Extensibility-enabled Plus stores.

Root Causes

Six Root Causes of One-Page Checkout Failures

01

App targets 3-page DOM or page URLs

Apps that used JavaScript to manipulate checkout by waiting for /checkout/shipping or /checkout/payment page loads, or that queried DOM selectors like #checkout-shipping-method or .step--shipping-method, fail silently on one-page checkout. The one-page layout renders checkout as a React single-page application. The 3-page DOM structure doesn't exist. Scripts that depended on Shopify.Checkout.step or listened for page navigation events get no trigger. The app appears to do nothing, with no visible error to the merchant or customer.

02

checkout.liquid removed — customizations lost

On Shopify Plus, checkout.liquid was the template file controlling checkout appearance and behavior. Merchants added custom CSS, JavaScript upsell widgets, address restriction logic, trust badges, custom form fields, and analytics scripts directly to this file. When the store migrated to Checkout Extensibility (required for newer Plus features), Shopify stopped rendering checkout.liquid. Everything in it stopped working — visually and functionally — without a clear notification. The checkout reverts to Shopify's default visual style until the Branding API and UI Extensions are configured.

03

Payment section stalled by upstream section error

One-page checkout renders sections sequentially: contact → shipping → payment. The payment section only becomes interactive after contact and shipping sections are resolved. If an upstream section has a validation error — no shipping rates for the address, a checkout UI extension throwing a JavaScript error, an address autocomplete API timing out — the payment section stays in a loading or disabled state indefinitely. On the 3-page checkout, each page was independent. On one-page checkout, a failure in section two prevents customers from reaching payment even if the payment system itself is working perfectly.

04

Checkout UI extension fails to render

Checkout UI Extensions are React components deployed via Shopify's CLI and served from Shopify's CDN. They fail to render for several reasons: (1) the extension is deployed to a development store but not the production store; (2) the extension has a runtime error that Shopify's sandbox catches and suppresses (no visible error to the customer — the extension slot just appears empty); (3) the extension is targeting a render target that doesn't exist in the current checkout configuration (e.g., checkout:dynamic:render on a store that hasn't enabled dynamic extensions); (4) the extension's network requests to fetch data timeout; (5) the extension version deployed doesn't match what's activated in the admin. Extensions fail silently — no error message appears in checkout, just an empty extension slot.

05

Theme-level scripts blocked from checkout

Scripts added to theme.liquid (the main theme template file) do not execute in checkout. Checkout is served from a different subdomain (checkout.shopify.com or the store's custom checkout domain) and does not load the theme. Merchants who add analytics tags, chat widgets, conversion tracking pixels, or custom scripts to theme.liquid expecting them to fire in checkout get no execution. This was also true on the 3-page checkout — it's not specific to one-page. But merchants often discover the issue when they switch to one-page checkout because the migration draws attention to checkout behavior. To add scripts to checkout, use the Additional Scripts field in Shopify admin → Settings → Checkout → Order status page scripts (executes on the order confirmation page only, not during checkout), or use a Checkout UI Extension with the headless render target for tracking, or use Shopify's Web Pixel API for analytics.

06

Mobile layout breaks extension or order summary

One-page checkout uses a two-column layout on desktop (form on left, order summary on right) and a single-column layout on mobile (order summary collapses to a header accordion). Checkout UI extensions that assume a minimum width or that are designed for the desktop order summary column can break on mobile — appearing cut off at the column boundary, failing to expand correctly, or not rendering at all when the order summary accordion is collapsed. Additionally, accelerated payment buttons (Shop Pay, Apple Pay) appear at the top of the checkout on mobile and below the form on desktop — an inconsistency that confuses merchants testing on different devices and can cause misalignment with custom extension UI that tries to appear near those buttons.

Silent Failures

Seven Ways One-Page Checkout Fails Without a Clear Error

Extension slot appears but is empty

The extension mounted but its render returned null or threw an error in the sandbox. Shopify shows the slot with no content. No error message visible to customers or merchants in the checkout UI.

Payment section frozen at "Calculating..."

Shipping rate API returned an error or no rates. One-page checkout can't advance to payment without a valid shipping selection. Customers see an indefinite loading state in the payment section.

Upsell widget disappeared after migration

checkout.liquid upsell (in-checkout cross-sell widget) stopped rendering after enabling Checkout Extensibility. The widget's code still exists in checkout.liquid but the file is no longer served. Nothing breaks visually — the widget just doesn't appear.

Custom checkout CSS not applying

Checkout.liquid CSS overrides (custom fonts, brand colors, button styles) dropped after Extensibility migration. Checkout reverts to Shopify defaults. Branding API controls these now but must be configured separately — it doesn't inherit from checkout.liquid.

Extension works in preview, fails in live

The Shopify CLI development build works with --dev flag but the deployed extension fails. Common cause: API keys or metafield definitions used by the extension exist in the dev store but weren't migrated to the production store.

Analytics pixel not firing at checkout steps

theme.liquid Google Tag Manager or Meta Pixel fires on all storefront pages but not checkout. Checkout is a separate surface. Merchants see attribution gaps — purchases tracked but checkout step events (begin_checkout, add_shipping_info, add_payment_info) missing.

Address autocomplete loads then breaks

Address autocomplete (Google Places, powered by Shopify) works on desktop but times out on mobile connections. On one-page checkout, an address autocomplete timeout can delay the shipping section, which in turn delays the payment section loading.

Diagnosis

How to Diagnose One-Page Checkout Failures

Open browser DevTools and check the Console tab. Navigate to checkout in an incognito window. Open DevTools (F12 or Cmd+Option+I) and go to Console. JavaScript errors from checkout UI extensions, failed network requests, and Shopify checkout lifecycle errors appear here. Look for errors referencing extension IDs, checkout-web, or app-specific script names. These errors won't appear in any Shopify admin dashboard — only in the browser console.

Identify whether the failure is pre-migration or post-migration. Check your Shopify admin → Settings → Checkout → Checkout customization. If "Checkout Extensibility" is shown as enabled (Plus) or if you see the Checkout editor rather than checkout.liquid, the migration has occurred. Any customizations from checkout.liquid are gone. If Checkout Extensibility is not yet enabled, you may still be on the legacy checkout system and experiencing a different category of bug.

Test in a completely clean session with no apps interfering. Go to Shopify admin → Apps → and temporarily disable all third-party apps that have checkout permissions. Test checkout end-to-end. If the issue resolves, re-enable apps one at a time to identify the conflicting app. Apps with "checkout" in their permissions (checkout UI extensions, CartTransform Functions, Discount Functions) are the most likely culprits.

Check the payment section stall: is shipping resolving? Fill in contact info and address. Watch whether the shipping section resolves (shows shipping options) before the payment section loads. If shipping never shows options, the issue is in shipping (carrier API, shipping zones) — not payment. If shipping shows but payment doesn't load, check the DevTools Network tab for failing requests to checkout.shopify.com or pay.shopify.com.

Test on mobile Safari specifically. Open checkout on an iPhone in Safari. One-page checkout layout shifts to single-column. Extension render targets behave differently. Accelerated payment buttons appear at the top. If the issue only reproduces on mobile, the cause is likely a checkout UI extension that doesn't handle the narrower viewport, or an app script that doesn't detect the mobile layout.

Check if the extension is deployed to the right store. In Shopify Partners → your app → Extensions, check which stores have the extension deployed and which version is active. A common mistake: the extension is deployed to a development store and tested there, but pushed to production under a different version that has a bug, or the extension was never deployed to the production store at all. Use shopify app deploy from the CLI and confirm the production store's admin shows the extension under Extensions → Checkout.

Reproduce the checkout.liquid gap by checking the branding. On Plus with Checkout Extensibility, go to admin → Online Store → Themes → Customize → Checkout. If the checkout looks like Shopify's default (no custom fonts, default colors, standard button style), your checkout.liquid CSS is no longer applying. Configure the Checkout Branding API or use the visual editor to restore brand colors, fonts, and button style. This is a separate step from rebuilding function-based customizations.

Deep Dives

Migration Path, Extension Architecture, and Tracking Setup

A — checkout.liquid to Checkout Extensibility Migration Map
What was in checkout.liquid → What replaces it

Custom CSS / brand colors / fonts: Checkout Branding API (admin → Online Store → Themes → Customize → Checkout, or via Admin API checkoutBrandingUpsert mutation). Controls: primary button color, font family, corner radius, form field style, header logo.

Custom JavaScript (analytics, chat, tracking): Shopify Web Pixel API (for analytics/tracking), Checkout UI Extension with headless target (for scripts that need checkout context), or App Bridge. Scripts in theme.liquid don't run in checkout.

In-checkout upsell widgets: Checkout UI Extension targeting purchase.checkout.block.render or purchase.checkout.cart-line-item.render-after. Must be a React component using Shopify's extension UI components.

Address restriction / validation: CartTransform Function (modifies cart before checkout), or a Checkout UI Extension with form validation logic.

Custom form fields: Checkout UI Extension targeting purchase.checkout.contact.render-after using the TextField component. Data stored as order metafields via cartMetafieldsSet or checkoutMetafieldsSet.

B — Checkout UI Extension Render Targets

Where extensions can appear in one-page checkout

Static extension points (rendered at a fixed position)

purchase.checkout.contact.render-after — below the email/contact section
purchase.checkout.shipping-option-list.render-after — below shipping options
purchase.checkout.payment-method-list.render-after — below payment methods
purchase.checkout.reductions.render-after — below discount code field
purchase.checkout.cart-line-item.render-after — below each cart line item
purchase.checkout.actions.render-before — above the "Complete order" button
purchase.thank-you.block.render — on the order confirmation page

Dynamic extension points (Plus only — Checkout Extensibility)

purchase.checkout.block.render — drag-and-drop blocks in the checkout editor. These can be repositioned by the merchant in the visual editor. Requires Checkout Extensibility enabled. The merchant sees a block component in the editor and can drag it between sections.

Common mistake: targeting non-existent extension points

Extension targets must be declared in the extension's shopify.extension.toml file. If the target doesn't match what's in the config, the extension silently doesn't mount. Verify the target string exactly — even a typo (e.g., purchase.checkout.contact.after instead of purchase.checkout.contact.render-after) results in the extension never appearing with no error.

C — Analytics and Tracking on One-Page Checkout

How to fire checkout step events on the new layout

Web Pixels (the correct approach)

Shopify's Web Pixel API is the official way to run analytics code inside checkout. Web Pixels run in a sandboxed iframe and receive checkout lifecycle events: checkout_started, checkout_contact_info_submitted, checkout_shipping_info_submitted, checkout_completed. Create a Web Pixel in your app's config (shopify.app.toml) and subscribe to these events. The pixel runs in the sandbox — it cannot access the parent DOM but can make fetch requests to analytics endpoints. Google Analytics 4, Meta Pixel, TikTok Pixel, and most major analytics platforms now have official Shopify app integrations that use Web Pixels instead of script injection.

Order status page scripts (legacy — order confirmation only)

Shopify admin → Settings → Checkout → Order status page → Additional scripts. This field accepts JavaScript that runs on the order confirmation page (after purchase), not during checkout. The Shopify.checkout JavaScript object is available here with order details. This is not the same as checkout tracking — it fires after the purchase is complete, not at the begin_checkout or add_payment_info steps. Use it only for post-purchase confirmation tracking.

The most common post-migration surprise

Merchants enabling Checkout Extensibility for the first time (to use a new Plus feature or UI extension) discover that their checkout suddenly looks plain and their in-checkout upsell widget is gone. Both are caused by checkout.liquid no longer rendering. The fix requires two separate steps: configure the Branding API to restore visual style, and rebuild each checkout.liquid feature as a proper UI Extension. Neither step happens automatically during the migration — Shopify does not port checkout.liquid content to the new system.
