Shopify Fawry Not Working — Reference Code Not Generating, Order Stuck in Pending & IPN Callback Failures

Updated June 2026 · 7 failure modes · Egypt merchants · Direct Fawry integration (not via Paymob)

Fawry is Egypt's largest offline payment network, with over 300,000 agent locations including Fawry machines, ATMs, supermarkets (Carrefour, Spinneys, BIM), and bank branches. Unlike card or wallet payments, Fawry works through a unique reference-code flow: the merchant generates a code via API, the customer pays it at any outlet within a validity window, and Fawry notifies the merchant via an IPN callback. Each step in this chain is a potential failure point — and when any one fails, the result looks different to the merchant and customer. This guide covers the 7 failure modes merchants encounter when integrating Fawry directly with Shopify.

How Fawry's Direct Integration Differs From Fawry via Paymob

There are two ways to accept Fawry payments on Shopify — and they fail differently. The first is through Paymob: you install the Paymob Shopify app, configure a Fawry integration ID in the Paymob dashboard, and Paymob's API handles all Fawry calls on your behalf. The second is direct Fawry integration: you call Fawry's own API (atfawry.com) using your Fawry Business merchant code and security key, with no intermediary. This guide focuses on direct Fawry integration. If you are using Paymob and experiencing Fawry issues, see the Paymob not working guide instead — the root causes are completely different (Paymob: missing integration ID; direct Fawry: API auth / IPN callback).

Fawry has no official Shopify app. Unlike Tabby, Paymob, or valU — which publish official apps in the Shopify App Store — Fawry's e-commerce API integration requires either a third-party Shopify plugin or custom development. This means the quality of the integration (specifically: signature generation, error handling, and IPN callback processing) varies significantly between plugins and custom implementations. Most Fawry checkout failures trace back to implementation-level issues in the plugin or custom code, not to Fawry's API itself.

Fawry processes in EGP only. Fawry's charge API requires the transaction amount in Egyptian Pounds. If your Shopify store operates in a different currency, your integration must convert the order total to EGP before calling the Fawry API. Many integration failures in multi-currency stores come from this step being missing or using a stale exchange rate. Fawry does not accept USD, EUR, or any non-EGP currency in the charge request.

Fawry's payment flow is asynchronous. Unlike a card payment (which completes or fails synchronously at checkout), a Fawry payment is a two-stage process: (1) reference code generation at checkout time, (2) customer payment at an outlet + Fawry IPN callback to the merchant hours or days later. This means Fawry orders are intentionally in "Pending" status until the outlet payment occurs. The failure mode merchants most often confuse with a bug is simply an unpaid reference code — the integration worked correctly, the customer just hasn't paid yet.

7 Failure Modes Covered

  1. Reference code not generating (API auth failure)
  2. Test environment in production (staging URL in live store)
  3. IPN callback not configured (order never updates from pending)
  4. IPN callback URL mismatch (Fawry cannot reach Shopify)
  5. Currency not EGP (amount rejected or wrong value charged)
  6. Reference code expired (customer missed the payment window)
  7. Fawry Business account not activated for e-commerce API
1

Reference Code Not Generating — API Authentication Failure

Merchant Configuration

The most common Fawry failure: the customer reaches the Fawry payment step at checkout, the page appears to process, but no Fawry reference code is displayed. The customer sees a generic "something went wrong" message or is returned to the checkout without a reference number.

Fawry's charge API uses signature-based authentication. Every API request must include a SHA-256 hash computed from the merchant code, security key, customer profile ID, customer name, merchant reference number, payment expiry, and order amounts. If any of these inputs are wrong — or if the signature computation itself has a bug — the Fawry API returns a 403 or an authentication error, and no reference code is generated.

The three most common causes: (1) the merchant code or security key was copied incorrectly from the Fawry dashboard — Fawry merchant codes and security keys are long alphanumeric strings, and a single character difference causes signature mismatch; (2) the plugin is using outdated Fawry API documentation and generating the signature in a deprecated format; (3) the security key was regenerated in the Fawry dashboard (for security reasons) but the plugin was not updated with the new key.

Silent failure pattern: Fawry API auth failures often produce no visible error at checkout. The plugin makes the API call, receives a 403, and either shows a blank reference section or falls back to "COD" without alerting the merchant. Check your Shopify error logs or enable plugin debug mode to see the raw Fawry API response.

How to Fix — Reference Code Not Generating

  1. Log into your Fawry merchant portal at merchant.fawry.com → go to Profile → API Credentials → copy the Merchant Code and Security Key exactly (no leading or trailing spaces).
  2. Update your plugin or custom integration with the current merchant code and security key. Save and test with a small order amount.
  3. Verify the signature algorithm matches Fawry's current API docs — Fawry uses SHA-256 on a concatenated string of specific fields in a specific order. The order of fields in the hash input matters; swapping any two fields produces a wrong signature that Fawry rejects.
  4. Enable debug logging in your Fawry plugin to capture the raw API request and response. A 403 response confirms auth failure; a 200 with an error code in the body indicates a different issue (e.g., account not activated for e-commerce).
  5. Test in Fawry's staging environment (staging.atfawry.com) with test credentials first — if reference codes generate in staging but not in production, the issue is almost certainly credentials (wrong merchant code / security key for the live environment).
2

Test Environment in Production — Staging URL in Live Store

Merchant Configuration

Fawry has two separate environments: a staging/test environment at staging.atfawry.com and a production environment at atfawry.com. During integration testing, merchants (or developers) configure the staging URL. If this URL is not updated to the production URL before the store goes live, the integration appears to work — reference codes are generated, the checkout completes — but the reference codes are sandbox-only and cannot be paid at any real Fawry outlet.

This failure is particularly damaging because it is not immediately apparent to the merchant. Orders are placed, reference codes are generated and sent to customers, but customers report that the Fawry code "doesn't work at the machine." The customer has attempted to pay but the outlet's system returns "invalid reference" because the code was generated against the test environment.

This failure also affects the IPN callback: Fawry's staging environment will never send real payment confirmations to production stores, so all orders placed during this period remain in "Pending" indefinitely.

How to identify this failure: Check the Fawry reference codes generated on your store. Staging reference codes often have a distinct format or prefix compared to production codes. Also check your plugin's API endpoint setting — if it contains "staging" in the URL, you are in test mode on a live store.

How to Fix — Test Environment in Production

  1. Open your Fawry plugin settings in the Shopify admin → find the API endpoint URL field.
  2. Confirm the URL is the production endpoint: https://atfawry.fawrystaging.com/ECommerceWeb/Fawry/payments/charge is staging — replace with the live Fawry production URL from your Fawry merchant documentation.
  3. Update API credentials to production credentials — staging credentials only work on the staging environment; production credentials only work on the production environment. You need separate credential sets for each.
  4. Cancel and re-take all orders placed while in test mode — reference codes generated against staging cannot be redeemed. Contact affected customers and advise them to checkout again.
  5. Verify the IPN callback URL is also pointing to your live store's domain, not a localhost or staging domain from the development phase.
3

IPN Callback Not Configured — Orders Never Update From Pending

Merchant Configuration

When a customer pays a Fawry reference code at any Fawry outlet, Fawry's system sends a payment confirmation to your store via an IPN (Instant Payment Notification) callback — an HTTP POST to a URL you configure in your Fawry merchant dashboard. If this callback URL is not configured, Fawry has nowhere to send the confirmation, and all Fawry orders remain in "Pending payment" status in Shopify indefinitely — even after customers have successfully paid.

This failure is invisible to customers (they pay successfully, receive a receipt from the Fawry outlet) but highly disruptive for merchants: orders are paid but appear unpaid, Shopify doesn't fulfill them automatically, and merchants may cancel paid orders thinking they are abandoned. Without the IPN callback, there is no automated way to connect a Fawry outlet payment to a Shopify order.

Many merchants discover this failure weeks after launch, when they manually check Fawry's dashboard and see successful payments that never updated in Shopify. The gap between Fawry's records (paid) and Shopify's records (pending) is the IPN callback.

Where to configure the IPN callback: Log into your Fawry merchant dashboard → Integration Settings → Merchant IPN URL → enter the callback URL provided by your Fawry Shopify plugin (typically something like: https://yourstore.myshopify.com/apps/fawry/callback). Save and test with a staging payment to confirm callbacks are being received.

How to Fix — IPN Callback Not Configured

  1. Find your Fawry IPN callback URL in your plugin documentation or plugin settings — it is the URL Fawry should POST payment confirmations to.
  2. Log into the Fawry merchant portal → Integration → IPN/Callback Settings → enter the callback URL and save.
  3. Test the callback using Fawry's staging environment — make a test payment and verify the order updates in Shopify from pending to paid.
  4. For historical pending orders: log into your Fawry dashboard to see which reference codes have been paid → manually update those Shopify orders to paid → fulfill them to customers.
4

IPN Callback URL Mismatch — Fawry Cannot Reach Shopify

Merchant + Technical

Even when an IPN callback URL is configured in the Fawry dashboard, it can fail to deliver payment confirmations if the URL does not exactly match what your Shopify store expects. Fawry's IPN system sends HTTP POST requests — if the URL is wrong, Fawry's POST goes to a dead endpoint and Shopify never receives the payment confirmation.

Common IPN URL mismatch patterns: (1) the store domain changed (custom domain added or changed) but the Fawry dashboard still has the old .myshopify.com URL; (2) the callback URL uses HTTP instead of HTTPS — Shopify stores require HTTPS and may reject unencrypted callbacks; (3) a trailing slash difference (fawrystaging.com/callback vs. fawrystaging.com/callback/ — some web servers treat these differently); (4) the Fawry plugin was updated and the callback URL endpoint path changed, but the Fawry dashboard was not updated; (5) the store was migrated to a new subdomain or new platform and the old callback URL is no longer valid.

This failure produces the same symptom as "IPN not configured" — orders stay in pending — but the root cause is different: callbacks are being sent by Fawry but not received by Shopify because the URL is wrong. The distinction matters for diagnosis: check Fawry's dashboard for callback delivery logs — if you see "delivery failed" or HTTP error responses, the URL is the problem.

How to Fix — IPN Callback URL Mismatch

  1. Get the exact current callback URL from your Fawry Shopify plugin settings page — copy it character-by-character, including protocol (https://) and any trailing slash.
  2. Update the Fawry merchant dashboard with the exact URL and save.
  3. Check Fawry's callback delivery log in the Fawry dashboard — look for recent delivery attempts and their HTTP response codes. A 404 means the URL path is wrong; a connection error means the domain is wrong; a 200 means delivery is working.
  4. Test with a staging payment to confirm end-to-end: place a test order → generate a Fawry reference → simulate payment in Fawry staging → verify the Shopify order updates to paid.
  5. Set up a backup polling mechanism in your plugin if available — some Fawry plugins include a cron job that queries Fawry's API for pending order statuses and reconciles them without relying solely on IPN callbacks.
5

Currency Not EGP — Amount Rejected or Wrong Value Charged

Merchant Configuration

Fawry's charge API accepts transaction amounts in Egyptian Pounds (EGP) only. If your Shopify store is configured to display prices in USD, EUR, GBP, or any other currency, your Fawry integration must convert the order total to EGP before calling the Fawry API. If this conversion is missing, two things can happen: (1) the Fawry API rejects the request with a currency error, producing no reference code; or (2) the API accepts the amount but interprets it as EGP, resulting in a reference code for the wrong value — a $100 USD order becomes a 100 EGP reference code, which is about $2 USD, causing serious undercharging.

Multi-currency Shopify stores that add Fawry as an additional payment option without updating their Fawry plugin's currency handling are particularly vulnerable to silent undercharging. The reference code generates successfully, the customer pays it, the Fawry IPN callback fires, the Shopify order marks as paid — everything looks correct — but the merchant received 30x less than the order value.

This failure is especially risky for merchants who do not regularly reconcile their Fawry dashboard payouts against their Shopify order values. It can go undetected for weeks while the merchant ships orders and receives only a fraction of the correct payment.

Undercharging risk: If your store sells in USD and Fawry is generating EGP reference codes without conversion, every Fawry order has been undercharged by approximately 50x. Check your last 30 Fawry payments in the Fawry dashboard and compare the EGP amounts against the expected USD-to-EGP converted values. If there is a systematic mismatch, the conversion is missing.

How to Fix — Currency Not EGP

  1. Check your Fawry plugin settings for a currency conversion option — most modern Fawry plugins have a toggle to convert from the store currency to EGP using a configured rate or a live exchange rate feed.
  2. Enable EGP conversion and set a conservative static rate (e.g., $1 USD = 50 EGP) if a live rate is not available — a static rate protects against the API rejection even if not perfectly precise.
  3. Verify the conversion is working by placing a test order in your non-EGP currency and checking the reference code amount in the Fawry dashboard — the EGP amount should match the order total multiplied by the exchange rate.
  4. Review historical Fawry payments for undercharging — if found, contact the affected customers or absorb the loss and fix the integration to prevent future undercharging.

Reference Code Expired — Customer Missed the Payment Window

Customer Experience

Fawry reference codes are not permanent. Every code has a validity window set at the time of generation — after this window closes, the code is invalidated at all Fawry outlets. When a customer attempts to pay an expired reference code, the Fawry outlet terminal returns "invalid reference" or "code expired." The customer may then contact the merchant believing the payment system is broken, when in fact the code simply expired.

The validity window is set in the Fawry charge API request's paymentExpiry parameter as a Unix timestamp. Merchants can configure this from as little as a few hours to as long as 30 days, depending on their Fawry account type and the transaction category. The default for e-commerce transactions in most plugins is 24 hours — appropriate for fast-moving orders but too short for customers who browse, place an order, and plan to pay later.

Note that an expired reference code does not automatically cancel the Shopify order — the order remains in "Pending payment" status indefinitely unless the merchant has set up automatic order cancellation for pending payments. Merchants should configure Shopify's "Payment pending" automatic cancellation to match or slightly exceed the Fawry reference code validity window.

Customer communication matters here: Include the reference code expiry time in the order confirmation email. "Your Fawry reference code [XXXXXXXXXX] is valid until [time/date]. Pay at any Fawry outlet, ATM, Carrefour, or BIM before this deadline." Customers who know the deadline pay on time. Customers who don't may wait and be surprised when the code expires.

How to Handle — Expired Reference Codes

  1. For the customer: Explain the code expired (not a system failure) and ask them to return to checkout to generate a new reference code. The original order should be cancelled first to release inventory.
  2. Extend the validity window in your plugin settings — for most Egyptian e-commerce, 48–72 hours is a better default than 24 hours. Verify your Fawry account allows extended validity for your transaction type.
  3. Add expiry time to order confirmation emails — include the exact deadline (not just "24 hours") so customers can plan their payment trip to a Fawry outlet.
  4. Configure Shopify automatic order cancellation under Settings → Checkout → "Automatically cancel orders not paid within" → set this to match your reference code validity plus a buffer (e.g., validity is 48h → auto-cancel at 72h, giving customers time if IPN is slightly delayed).

Fawry Business Account Not Activated for E-Commerce API

Account Status

Fawry has multiple products for businesses: a bill payment service, a point-of-sale system, and an e-commerce payment API. A business that registered with Fawry for bill payment or POS services does not automatically get e-commerce API access. If the merchant registered with Fawry but did not specifically request e-commerce API activation, calls to the Fawry charge API will fail with an "account not authorized for this service" error, even if the merchant code and security key are correct.

Additionally, Fawry requires business registration documents for e-commerce API activation — typically: Egyptian commercial registration, tax card, and bank account details. If any of these are missing or under review, the e-commerce API access remains in a pending state. The Fawry merchant portal shows the account as "active" for some services but not for the e-commerce charge API specifically.

Merchants who receive their Fawry merchant code and immediately try to integrate the API without confirming e-commerce API activation experience this failure — the credentials are valid but the API access is not yet granted. This is analogous to Paymob's KYB pending state, but at the Fawry account level rather than the gateway level.

How to Fix — Fawry Business Account Not Activated

  1. Log into your Fawry merchant portal and check your account's enabled services — look for "E-Commerce API," "Online Payments," or "Fawry for Business" activation status.
  2. Contact your Fawry account manager or Fawry merchant support and request e-commerce API activation. Provide your merchant code, commercial registration number, and tax card number.
  3. Submit any missing documentation — Fawry typically requires Egyptian commercial registration (سجل تجاري) and tax card (بطاقة ضريبية) to activate the e-commerce API for business accounts.
  4. While waiting for activation: use Fawry's staging environment to complete your integration testing so you are ready to go live as soon as the production account is activated.
  5. Ask Fawry support for test credentials that work on the staging environment while your production account is being reviewed — this prevents the activation delay from blocking your integration work.

Fawry Shopify Integration Verification Checklist

Run through this before going live

API credentials: Merchant code and security key match exactly what is shown in your Fawry merchant portal → Profile → API Credentials. No extra spaces or characters.

Production endpoint: Plugin API URL is pointing to the Fawry production endpoint (atfawry.com), not the staging environment (staging.atfawry.com or fawrystaging.com).

IPN callback configured: Fawry merchant dashboard → Integration Settings → IPN/Callback URL is set to your Shopify store's Fawry callback endpoint on the live domain with HTTPS.

Frequently Asked

Common Questions About Fawry on Shopify

Can I use Fawry direct integration alongside Paymob on the same Shopify store?

Yes — but it is rarely the right choice. If you are already using Paymob as your primary gateway, activating the Fawry integration ID within Paymob is simpler and keeps all payment methods under one dashboard. Adding a separate direct Fawry integration on the same store creates duplicate payment options at checkout (Fawry via Paymob and Fawry direct), which confuses customers. The main reason to choose direct Fawry integration over Fawry via Paymob is if you want direct settlement from Fawry (bypassing Paymob's settlement cycle) or if you specifically need a feature of Fawry's direct API that Paymob's Fawry integration does not expose.

How long does it take for a Fawry payment to reflect in Shopify after the customer pays at an outlet?

When the IPN callback is configured correctly, Fawry order status in Shopify updates within minutes of the customer paying at a Fawry outlet, ATM, supermarket, or bank. Fawry's IPN system is real-time — as soon as Fawry's network registers the payment, it sends the callback to your configured URL. In practice, the delay from physical payment to Shopify order update is typically under 5 minutes. If orders are taking longer to update or are not updating at all, the IPN callback is the issue — check the Fawry dashboard for delivery failure logs and verify the IPN URL is correct and reachable.

Does Fawry support refunds via the Shopify admin?

Fawry refunds via Shopify's admin refund flow depend on whether your Fawry plugin supports the Fawry refund API. Fawry has a refund endpoint in its API, but not all Shopify plugins expose it — some require merchants to initiate refunds manually through the Fawry merchant portal. If your plugin does support Fawry refunds via Shopify admin, issuing a refund in Shopify triggers a refund request to Fawry's API, which then reverses the payment back to the customer's Fawry account or source. The refund timeline varies: refunds to mobile wallets via Fawry are typically fast (same-day), while refunds to the original payment source for Fawry-outlet-paid orders may take 2–5 business days depending on how the refund is routed.
