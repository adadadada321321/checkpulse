Shopify Notification Email Customization Not Working — 8 Causes Explained

Logo not showing in order confirmation? Liquid error turning your email to plain text? Shopify notification emails going to spam? Klaviyo taking over and breaking defaults? Here are all 8 root causes for Shopify transactional notification email failures — for merchants and developers.

Updated June 2026 8 failure modes Liquid Templates · Brand Settings · SPF/DKIM · Klaviyo Override · Shopify Email vs Notifications · Deliverability

Shopify's notification email system has a structural complexity that trips up most merchants: notification templates (transactional) and Shopify Email (marketing) are completely separate systems, brand settings do not auto-apply to email templates, and a single Liquid syntax error in any template collapses the entire email to unformatted plain text. Adding a third-party email app like Klaviyo or Omnisend introduces a second separate override layer that persists even after uninstallation.

The result: a merchant changes their logo, configures brand colors, and expects their order confirmation to look great — but customers receive a plain white email with no logo, or worse, no email at all. Eight distinct failure modes produce these symptoms with completely different root causes and fixes.

Two Separate Email Systems in Every Shopify Store
Notification emails (transactional): Order confirmation, shipping confirmation, refund, cancellation. Edited at Admin → Settings → Notifications. Liquid-based templates. Fire automatically on order events. These are what this guide covers.

Shopify Email (marketing): Newsletters, promotions, abandoned cart campaigns. Managed in the Shopify Email app or a third-party app like Klaviyo. Completely separate codebase. Changes here have zero effect on notification templates.

Confirm which system is broken before spending time in the wrong place.

Failure Mode 1: Liquid Syntax Error Breaks the Entire Template — Falls Back to Plain-Text Email

The most severe notification template failure: customers receive a plain-text email with no layout, no logo, no colors, and just raw order data as unformatted text. This is Shopify's fallback behavior when a Liquid rendering error is detected in the notification template.

Why a single Liquid error wipes all formatting

Shopify notification templates are rendered by the Liquid templating engine server-side. If the Liquid parser encounters an error — an unclosed tag, an invalid filter, a misspelled variable — the entire HTML template is abandoned and Shopify falls back to a minimal plain-text representation of the order data. The plain-text fallback is not a separate template you can edit; it is auto-generated from the order object.

This Is Silent — No Error Email to Merchant
When a Liquid error triggers the plain-text fallback, Shopify sends the degraded email to the customer without notifying the merchant. You will not receive an alert. The only ways to detect it: a customer reports the email looked broken, you notice it during a test order review, or you check the template preview and see a Liquid error in the preview panel.

Most common Liquid errors in notification templates

Error Pattern Example Correct Syntax
Wrong variable path {{ order.customer.email }} {{ customer.email }}
Unclosed if block {% if line_item.sku %} ... (no endif) Add {% endif %} to close every {% if %}
Invalid filter {{ subtotal_price | money_with_tax }} {{ subtotal_price | money }}
Line item property typo {{ line_item.properties["_custom"] }} {{ line_item.properties["custom"] }} (no underscore prefix in emails)
For loop not closed {% for item in line_items %} without {% endfor %} Every {% for %} needs {% endfor %}

How to find and fix the Liquid error

  1. Go to Admin → Settings → Notifications → select the affected notification (e.g., Order confirmation)
  2. Click "Edit" to open the template editor
  3. Click "Preview" — if a Liquid error exists, it renders inline in the preview with the line number and error description
  4. Fix the Liquid syntax error on the indicated line
  5. Preview again to confirm no more errors
  6. Save and test by placing a test order or using "Send test notification"
Reset to Default if You Can't Find the Error
If the Liquid error is hard to locate, scroll to the bottom of the notification editor and click "Revert to default" to restore Shopify's original template. You will lose customizations but regain a working email immediately, then re-apply your changes one section at a time.

Failure Mode 2: Brand Logo and Colors Don't Auto-Apply to Notification Templates

The second most common merchant complaint: "I updated my logo in Settings → Brand but my order confirmation email still shows the old logo (or no logo at all)." Shopify's brand settings and notification templates are not linked — the brand settings are a storage vault, not an auto-injection system.

How Shopify brand settings actually work

Uploading a logo at Admin → Settings → Brand stores the image in Shopify's CDN and makes it available as a Liquid variable. But the notification templates do not read from brand settings automatically. Each template is a standalone HTML/Liquid file. If the template contains a hardcoded src="https://old-logo-url.com/logo.png", it will always display that URL regardless of what is in brand settings.

To use the brand logo in notification templates, the template must explicitly reference the Liquid variable:

{% if shop.email_logo_url %}
  <img src="{{ shop.email_logo_url }}"
       alt="{{ shop.name }}"
       width="{{ shop.email_logo_width }}"
       style="max-width:100%;">
{% else %}
  <h1>{{ shop.name }}</h1>
{% endif %}

Brand-related Liquid variables for notification templates

Variable Source in Admin Description
shop.email_logo_url Settings → Notifications → (logo upload field) URL of the logo specifically uploaded for notification emails — separate from Settings → Brand logo
shop.email_logo_width Settings → Notifications → (logo width) Pixel width for the notification email logo
shop.email_accent_color Settings → Notifications → (accent color) Hex color used for buttons and accent elements in default notification templates
shop.name Settings → General → Store name Store name as text fallback when no logo
Two Separate Logo Upload Locations
Shopify has a logo in Admin → Settings → Brand (used for the storefront and Shopify Email marketing campaigns) AND a separate logo upload in Admin → Settings → Notifications → Email customization (used for transactional notification emails). These are different files. Changing one does not change the other. If your storefront logo and notification email logo look different, this is why.

Failure Mode 3: Shopify Email (Marketing) and Notification Templates Are Completely Separate Systems

Merchants regularly spend time customizing Shopify Email templates and then expect those changes to appear in order confirmation or shipping notification emails — they don't. These are two distinct systems with no shared state.

The two email systems side by side

Feature Notification Emails (Transactional) Shopify Email (Marketing)
Where to edit Admin → Settings → Notifications Shopify Email app or third-party (Klaviyo, Omnisend)
Templating language Liquid (HTML-based) Drag-and-drop builder or HTML editor in app
When they fire Automatically on order/fulfillment/refund events Manually sent or scheduled/automated campaigns
Affected by theme changes Never — completely separate from storefront theme Never — app manages its own templates
Examples Order confirmation, shipping, refund, cancellation Newsletters, abandoned cart, welcome series, promotions
Sending infrastructure Shopify's transactional email servers Shopify Email app servers (or Klaviyo/Omnisend servers)

If you installed Shopify Email to redesign your "order confirmation" and customized a template there, you may have been editing a marketing template — not the transactional notification. The actual order confirmation email is still the unmodified template at Settings → Notifications → Order confirmation.

Abandoned cart email confusion

Shopify's built-in abandoned checkout notification (Admin → Settings → Notifications → Abandoned checkout) fires 10 hours after a cart is abandoned. If you have Klaviyo or Omnisend installed and configured with an abandoned cart flow, both systems may fire — the customer gets two emails. To avoid this, disable Shopify's native abandoned checkout notification when using a third-party email marketing app's abandoned cart flow.

Failure Mode 4: Notification Type Silently Disabled by App Install or Manual Toggle

Shopify has over 25 individual notification types, each with its own on/off toggle. These can be disabled silently during app installs, store migrations, or accidental admin changes — and there is no dashboard showing which are currently disabled without clicking into each one.

Notification types with the highest disable-rate

Notification Type Common Reason for Accidental Disable Impact
Order confirmation Email marketing app installation (app takes over and disables Shopify default) Critical Customers don't receive order receipt
Shipping confirmation Fulfillment app (ShipBob, ShipStation) disables to send its own branded email Critical Customers don't get tracking information
Refund notification Returns management app install High Customers unaware their refund was processed
Abandoned checkout Klaviyo/Omnisend abandoned cart flow configured without disabling Shopify default High Double emails, or notification off but app flow also missing
Customer account welcome CRM app installation Medium New account registration silent

How to audit all notification statuses

  1. Go to Admin → Settings → Notifications
  2. Scroll through the full list of notification types — look for any labeled "Disabled"
  3. Click each critical notification (Order confirmation, Shipping confirmation, Refund) to verify it is enabled and the template has no errors
  4. Place a test order using a Shopify test payment gateway to trigger real notification sends — verify the emails arrive
Use "Send test notification" Carefully
The "Send test notification" button in Admin → Settings → Notifications sends a preview using sample data. It does not use the same delivery path as a real order. A test notification can succeed even when real order notifications are disabled or broken. Always verify with a real test order to be certain.

Failure Mode 5: Third-Party Email Apps Override Notifications — Disabling the App Doesn't Restore Defaults

Email marketing apps — Klaviyo, Omnisend, Drip, OrderlyEmails, AfterShip — work by registering Shopify webhooks for order events. When an order is created, the webhook fires and the app sends its own branded email instead of (or in addition to) Shopify's default notification. The critical issue: when you uninstall the app, the webhook is removed but Shopify's default notifications are not automatically re-enabled.

What each app-type does to your notification flow

App Type What It Does What Happens on Uninstall
Klaviyo, Omnisend (marketing platforms) Registers order webhooks, sends own emails; may or may not disable Shopify defaults Webhooks removed, but if Shopify defaults were disabled during install they stay disabled
OrderlyEmails, Spently (template designers) Replaces Shopify notification template HTML with custom-designed templates via the app Custom HTML may remain in the template even after uninstall — app doesn't revert template content
ShipStation, ShipBob (fulfillment apps) Disables Shopify's shipping confirmation notification and sends its own Shopify shipping confirmation remains disabled — must manually re-enable
Returns management apps (Loop, Happy Returns) Adds return portal links to order notification templates or takes over return emails Template modifications may persist — check template HTML for orphaned app code

Recovery steps after uninstalling an email override app

  1. Go to Admin → Settings → Notifications
  2. Check each critical notification for "Disabled" status — manually enable any that were turned off
  3. Open each enabled template and review the HTML — look for app-specific code, broken links to app CDN assets, or references to app variables that no longer resolve
  4. For any template with orphaned app code: click "Revert to default" to restore Shopify's clean template
  5. Re-apply your brand customizations (logo, colors, footer links) to the reverted templates
  6. Send a test order to verify delivery
Klaviyo-Specific: Check Flow Active Status
If you uninstall Klaviyo but your Klaviyo account still exists with active flows, and if Shopify still has a Klaviyo webhook registered (check Admin → Settings → Notifications → Webhooks), Klaviyo may still be attempting to send emails for your orders — unsuccessfully, since the store is disconnected. This results in zero emails going out. After uninstalling Klaviyo, verify at Admin → Settings → Notifications that no Klaviyo webhooks remain registered, and that Shopify's default order confirmation is enabled.

Failure Mode 6: SPF/DKIM Misconfiguration Routes Notification Emails to Spam or Promotions

The symptom: customers report not receiving order confirmation or shipping emails. The emails are being sent by Shopify — but they are landing in the spam folder, the Gmail Promotions tab, or being outright rejected by the receiving mail server. The cause is email authentication failure at the DNS level.

How Shopify sends notification emails

By default, Shopify sends notification emails from noreply@shopify.com (or a subdomain on Shopify's infrastructure) on behalf of your store. These have Shopify's SPF and DKIM already configured — they generally land in inbox without DNS setup from the merchant.

If you changed the notification sender address to a custom email (e.g., orders@yourbrand.com at Admin → Settings → Notifications → Sender email), Shopify sends from that address — but your domain's DNS must now authorize Shopify as a sender. Without this authorization, receiving mail servers see the email as spoofed and filter it.

Required DNS records for custom sender email

Record Type What to Add Why It Matters
SPF (TXT record) Add include:shops.shopify.com to your domain's SPF record Authorizes Shopify's mail servers to send email from your domain
DKIM (CNAME records) Add Shopify's two DKIM CNAME records (found in Admin → Settings → Notifications → Sender email section) Cryptographically signs outgoing emails to prove authenticity
DMARC (TXT record) Start with v=DMARC1; p=none; rua=mailto:dmarc@yourdomain.com Specifies handling policy for failed authentication; p=reject without SPF/DKIM aligned will reject all your notification emails
Never Use Gmail/Yahoo as Notification Sender Address
If your Shopify notification sender email is yourstore@gmail.com or yourstore@yahoo.com, all your notification emails will either be rejected or marked as spam. Google and Yahoo enforce strict DMARC policies (p=reject) — they will not allow Shopify to send email from a Gmail or Yahoo address. Use a custom domain email address (orders@yourbrand.com) as the sender instead.

Diagnosing deliverability failures

  1. Place a test order with an email address you control — check spam folder first, then check Gmail Promotions tab
  2. If the email arrived in spam: check email headers (in Gmail: three-dot menu → Show original) — look for "SPF: FAIL" or "DKIM: FAIL" in the Authentication-Results header
  3. Use MXToolbox SPF checker (mxtoolbox.com/spf.aspx) with your sending domain to verify the SPF record includes include:shops.shopify.com
  4. Use MXToolbox DKIM checker with selector shopify1._domainkey and your domain to verify DKIM is configured
  5. Add missing records in your domain's DNS provider (Cloudflare, GoDaddy, Namecheap, etc.) — DNS propagation takes 15 minutes to 48 hours

Failure Mode 7: Storefront Theme Changes Have No Effect on Notification Email Templates

Shopify's storefront theme (Dawn, Prestige, Impulse, etc.) and notification email templates share nothing — they are completely separate systems. Theme file changes, theme customizations, and checkout.liquid modifications have zero effect on any notification email template.

Why the systems are completely separate

The storefront theme lives in your theme's file system (accessible at Online Store → Themes → Edit code). Notification email templates live at Admin → Settings → Notifications and are stored in a completely separate Shopify admin data store. The two systems do not share CSS, Liquid files, assets, or configuration. A change to templates/index.liquid in your theme will never appear in an email. A change to the Order confirmation email template will never change anything on your storefront.

What You Changed Affects Storefront Affects Notification Emails
Theme Liquid files (templates/, sections/, snippets/) Yes Never
Theme CSS / JavaScript Yes Never
checkout.liquid Checkout only Never
Admin → Settings → Notifications templates Never Yes
Admin → Settings → Notifications → Email customization (logo, colors) Never Yes, if template references the variables

If you need a consistent visual identity between your storefront and notification emails, you must manually replicate the relevant design choices (colors, fonts, logo) into the notification templates using Liquid variables and inline CSS. Notification emails do not support external stylesheets — all CSS must be inline or in a <style> block within the template, and many email clients strip <style> blocks entirely. Use inline styles for maximum compatibility.

Failure Mode 8: "Send Test Notification" Works But Customers Don't Receive Real Order Emails

The most confusing scenario: you click "Send test notification" in Admin, the test email arrives in your inbox immediately, you confirm the template looks correct — but real customers placing orders never receive emails. The test notification and the real order notification use different code paths.

Why test notifications and real order notifications behave differently

The "Send test notification" button generates a preview using static sample data and sends it through Shopify's immediate notification preview pipeline. It bypasses the event queue that processes real order events. Real order notifications go through Shopify's background job system: the order event fires a webhook, the webhook enqueues a background job, the job renders the template with real order data, and the email is sent through Shopify's transactional email infrastructure. If the background job system has an issue, or if the specific notification is disabled for the real event trigger (but the test uses a different trigger path), test and real can diverge.

Conditions where test succeeds but real fails

Condition Test Result Real Order Result Fix
Notification disabled via third-party app webhook override Test sends No email Check if third-party app webhook is registered; re-enable notification or configure app correctly
Liquid variable renders fine with test data but breaks with real order data Test renders Plain text fallback Identify variables that are null/missing for some real orders (e.g., missing customer.first_name for guest checkouts)
Custom sender domain failing SPF/DKIM Test sends from Shopify default Rejected or spam Configure SPF/DKIM for custom sender domain (see Failure Mode 6)
Guest checkout order — customer object has fewer fields Test uses full customer object Liquid error on guest order Wrap customer-specific Liquid in {% if customer %} guard blocks
POS order — different order origin properties Test uses online order data shape Missing properties for POS order type Test with a real POS order; guard POS-incompatible variables

How to verify real order notification delivery

  1. Use Shopify's test payment gateway (Bogus Gateway) to place a real test order with a real email address you control
  2. Check your inbox — including spam — for the order confirmation within 5 minutes
  3. If it doesn't arrive: go to Admin → Orders → select the test order → scroll to Timeline → look for "Order confirmation sent to [email]" — if this line is missing, the notification was not sent at all
  4. If the timeline shows "sent" but email hasn't arrived: deliverability issue (check spam, verify SPF/DKIM)
  5. If the timeline does not show the notification as sent: notification is disabled or a webhook is intercepting the event
Order Timeline Is Your Ground Truth
Admin → Orders → [any order] → Timeline shows exactly which notifications were sent, to which email address, and at what time. If a notification should have fired but the timeline shows nothing, the notification was not sent by Shopify. If the timeline shows it was sent but the customer didn't receive it, the issue is deliverability. These are two different problems with different fixes.
