Overview

Why Shopify Returns No Shipping Rates

Shopify calculates shipping rates dynamically at checkout based on four inputs: the delivery address, the cart's total weight, the cart's total value, and the shipping profiles assigned to the products in the cart. If any of these inputs fails to match a configured rate, Shopify returns an empty rate list instead of an error — and customers see the "no shipping rates available" state.

The most dangerous scenario is a shipping app or carrier API that was working yesterday but fails silently today: your Admin still shows rates configured, your test orders go through, but real customers in specific locations or with specific cart contents see nothing.

The eight causes below cover both configuration gaps (fixable in Admin settings) and runtime failures (that require monitoring to detect proactively).

Root Causes

8 Reasons Shopify Checkout Shows No Shipping Options

CAUSE 01 / 08
Most Common

No shipping zone configured for the customer's country or region

Shopify shipping zones define which countries and regions you ship to. If a customer's delivery address is in a country not covered by any zone, Shopify returns zero rates. This is the most common cause of selective no-shipping reports — international customers or customers in states/provinces you forgot to add see no options while domestic customers are unaffected.

Common scenario: you added a "Rest of World" zone but forgot to include a specific country. Or your General Profile has the US covered, but a new Canadian customer finds nothing.

FIX → Admin → Settings → Shipping and Delivery → Shipping Profiles → General Profile → Shipping Zones. Verify the customer's country is listed under at least one zone. Add a "Rest of World" zone with a flat rate as a catch-all if you ship globally.
CAUSE 02 / 08
Configuration Gap

Cart value below minimum threshold — no fallback rate exists

Conditional shipping rates (e.g., "Free Shipping — orders over $50") only display when the cart meets the condition. If your only configured rate is a free shipping threshold and a customer's cart is $48, Shopify returns an empty list — because the single rate's condition isn't met and there is no fallback.

This silently affects every customer whose cart value is below your free shipping minimum if you haven't added a paid fallback rate. These customers see "no shipping options" and assume your store doesn't ship to them.

FIX → Admin → Settings → Shipping and Delivery → open each shipping zone → confirm you have at least one rate with NO minimum order value (or with a minimum of $0). Add a flat "Standard Shipping — $X" rate as your unconditional fallback so all customers always see at least one option.
CAUSE 03 / 08
Configuration Gap

Product has no weight set — carrier-calculated rates fail silently

Carrier-calculated shipping rates (FedEx, UPS, USPS, Canada Post) require the total cart weight to generate a quote. If any product in the cart has weight set to zero or left blank, the total weight calculation returns 0 — and many carrier services refuse to quote for a zero-weight shipment, returning no rates instead of an error.

This is particularly hard to notice because it's product-specific: carts containing only well-configured products work fine; carts containing the one product with missing weight show nothing.

FIX → Admin → Products → open each physical product → scroll to "Shipping" → confirm "This is a physical product" is checked and a non-zero weight is set. For a bulk fix: export products to CSV (Admin → Products → Export), filter for rows where weight is blank or 0, update, and re-import.
CAUSE 04 / 08
Silent Runtime Failure

Carrier-calculated shipping API timeout or credentials expired

Shopify requests real-time rates from carrier APIs during checkout. FedEx, UPS, USPS, and Canada Post all experience periodic API outages and rate limits. When the carrier API request times out, returns a 5xx error, or the stored credentials in Shopify have expired, Shopify cannot fetch a rate — and returns nothing to the customer.

This is the hardest failure to catch because it's intermittent. During the outage window, customers see no rates and abandon. After the outage ends, rates return and everything looks normal. You have no record of the failure.

FIX (immediate) → Admin → Settings → Shipping and Delivery → Carrier accounts → check for error banners on each carrier. Re-authenticate if credentials show as expired.

FIX (resilience) → Add at least one flat-rate fallback ("Standard Shipping — $X") so customers always have a non-carrier option during API outages. Rate it competitively so customers don't feel penalized when carriers fail.
CAUSE 05 / 08
Address-Specific

P.O. Box address — carrier services don't cover it

FedEx and UPS do not deliver to P.O. Boxes. If a customer enters a P.O. Box as their shipping address and your only shipping rates are FedEx or UPS carrier-calculated, those rates disappear — because the carrier API returns no quote for that address type.

USPS and Canada Post do service P.O. Boxes. If you only have FedEx/UPS configured, P.O. Box customers are stuck with no options. This affects a small but real percentage of customers, particularly in rural areas where P.O. Box delivery is standard.

FIX → Add USPS or a flat-rate option to your shipping zones so P.O. Box customers always have at least one choice. In Admin → Settings → Shipping → add a "P.O. Box compatible — USPS" rate or a flat rate that isn't carrier-calculated. Alternatively, add a note in checkout asking customers with P.O. Boxes to select USPS.
CAUSE 06 / 08
Third-Party App

Third-party shipping app returning errors or invalid responses

Shipping apps like ShipBob, Shippit, EasyPost rate calculators, and custom carrier apps can register themselves as the rate source for products or shipping profiles. When the app's API endpoint is down, returns malformed JSON, or its credentials in Shopify have expired, Shopify receives no rates from the app and shows nothing to the customer.

The critical risk: you can see "Carrier and app rates" configured in your Admin, everything looks set up correctly from the settings view — but at runtime the app's endpoint is timing out. No error is surfaced to you. Every customer hits an empty list.

FIX → Admin → Settings → Shipping and Delivery → "Carrier and app rates" → check each app entry for error states. Temporarily disable the shipping app in shipping settings (don't uninstall) and see if native rates return. If they do, the app's rate endpoint is the problem. Contact the app's support with the error, or rotate the app's API credentials in both Shopify and the app's own dashboard.
CAUSE 07 / 08
Shipping Profile

Product assigned to a custom shipping profile with different zones

Shopify lets you create custom shipping profiles to give certain products different rates or restrictions. If a product is assigned to a custom profile that has a narrower set of shipping zones than your General Profile, carts containing that product will only show rates valid for both profiles. If there's no overlap, the customer sees nothing.

This is common when you add a new product, assign it to a custom profile for a specific reason (like oversized items), and forget that the custom profile only covers domestic shipping while the General Profile covers international — international customers with that product in cart get no rates.

FIX → Admin → Settings → Shipping and Delivery → Shipping Profiles → open each custom profile → verify the zones and rates match or are a superset of what customers need. If a product is in a custom profile unintentionally, move it back to the General Profile by editing the custom profile and clicking "Remove product."
CAUSE 08 / 08
Local Pickup Only

Only local pickup configured — no actual shipping rate

Local Pickup is not a shipping rate in Shopify — it's a separate fulfilment method that appears on the shipping step. If Local Pickup is your only fulfilment option and a customer is not near your location (or doesn't realize local pickup exists), the shipping rate list may appear empty to them depending on your Shopify plan and checkout version.

Also common: stores set up for click-and-collect only, later want to start shipping, but haven't configured any shipping rates in the General Profile — the Local Pickup option shows but no shipping rates do.

FIX → Admin → Settings → Shipping and Delivery → Shipping Profiles → General Profile → add at least one Shipping Zone with a flat rate or carrier-calculated rate. Confirm "Local pickup" is correctly labeled so customers understand it requires physical pickup (add your store address in the pickup settings so it displays to customers).

Step-by-Step Fix

5-Step Diagnosis When Shipping Rates Are Missing

1

Reproduce the exact customer scenario

Open a private/incognito browser window, add the same product(s) to cart, and enter the exact address the affected customer used. This isolates whether the issue is address-specific, product-specific, or global. If rates appear in your test, the cause is likely an edge case (P.O. Box, specific country, cart value). If no rates appear for you either, it's a broader configuration or runtime problem.

2

Verify shipping zone coverage in Admin

Admin → Settings → Shipping and Delivery → open the General Profile → check every shipping zone. Confirm the customer's country appears in at least one zone. Confirm each zone has at least one rate with no minimum order restriction (or a $0 minimum). If any zone lacks a fallback rate, add one immediately.

3

Check carrier account and shipping app status

Admin → Settings → Shipping and Delivery → scroll to "Carrier and app rates." Look for error indicators on each carrier account. If you see authentication errors or "last successful fetch" timestamps that are hours old, re-authorize the carrier. For shipping apps, check the app's own dashboard for API error logs and verify credentials haven't expired.

4

Audit product shipping data

Admin → Products → filter to "physical products" → spot-check products in the affected cart for missing weights. Any product with weight = 0 or blank is a candidate for causing carrier rate failures. Also check the "Shipping" section of each product to confirm "This is a physical product" is checked — if it's unchecked, Shopify excludes it from shipping calculations entirely.

5

Add a flat-rate fallback and test end-to-end

Regardless of the root cause identified, add a flat-rate shipping option ($X standard shipping, no conditions) to every shipping zone as a backstop. This ensures customers always have at least one option even during carrier API outages or app failures. After adding it, re-test the original scenario in incognito to confirm rates now appear.

The Monitoring Gap

Why Shipping Rate Failures Go Undetected for Hours

Shipping rate failures — especially carrier API timeouts and shipping app errors — are completely invisible to standard monitoring. Your checkout URL returns HTTP 200. Your uptime tool shows green. Your homepage, product pages, and cart all load normally. The failure happens inside the checkout session's shipping rate calculation — a backend call that standard ping-based monitors never touch.

Carrier API outages are the worst scenario: they strike suddenly, can last minutes to hours, and leave no trace. During the outage window, every customer at the shipping step sees an empty list and abandons. You find out via a revenue dip or a customer complaint — typically hours after the damage is done.

Why standard tools miss this
🔴
Ping monitor: checks if your checkout URL returns 200. It does — the page loads. The empty shipping list is an application-layer failure, invisible to TCP/HTTP checks.
🔴
Uptime robot / Pingdom: same limitation. They verify the HTTP handshake, not the checkout session's ability to compute shipping rates.
🟡
Manual testing: only catches the issue during business hours, when you happen to be watching, and only if your test address + cart triggers the failure.
🟢
End-to-end checkout simulation: the only approach that verifies the full checkout flow — including that a valid session is created and shipping is reachable — on an automated schedule.

FAQ

Shopify No Shipping Options — Frequently Asked Questions

Why does Shopify checkout show no shipping options for some customers but not others?
Address-based mismatches are the most common cause of selective failures. Your shipping zones may not cover the customer's country, the customer entered a P.O. Box address that FedEx/UPS won't quote for, or their cart value is below a minimum threshold and you have no fallback rate. Go to Admin → Settings → Shipping and add a "Rest of World" zone with a flat fallback rate to eliminate address gaps.
How do I fix "no shipping rates available" in Shopify?
Step 1: Admin → Settings → Shipping and Delivery → confirm the customer's country is in a shipping zone. Step 2: Check every zone has at least one rate with no minimum order requirement. Step 3: Verify all physical products in the cart have a non-zero weight set. Step 4: If you use carrier-calculated rates, check Carrier accounts for authentication errors. Step 5: Add a flat-rate backup in every zone so customers always have one option.
Why did shipping rates disappear after I installed a new app?
The new app likely registered itself as the rate provider for your shipping profile and is returning errors or empty responses. Temporarily disable the app in Admin → Settings → Shipping and Delivery → Carrier and app rates (click the toggle, don't uninstall). If native rates come back, the app's endpoint is broken — contact the app developer with the error details, and add a flat-rate fallback to cover the gap while you troubleshoot.
Why does carrier-calculated shipping sometimes not show at Shopify checkout?
Shopify fetches carrier rates in real-time during checkout. FedEx, UPS, USPS, and Canada Post APIs have occasional outages and rate limits. When the API call times out or returns a 5xx error, Shopify shows no rates rather than an error message. Add at least one flat-rate option per zone so customers have a fallback during carrier API outages.
Can Shopify checkout show no shipping options even if I have rates configured?
Yes. Configured rates and runtime rates are different things. A rate can be correctly configured in Admin but fail at runtime because: the carrier API is down, a shipping app is returning errors, a product has no weight set and breaks the carrier calculation, or the delivery address doesn't match any zone despite the zone existing. Always add a flat-rate fallback with no conditions as insurance against runtime failures.
How do I know if my Shopify shipping rates are broken without waiting for a customer complaint?
Standard uptime monitoring won't catch this — your checkout URL returns 200 even when the shipping step is empty. The only early-warning option is a tool that simulates the full checkout flow and verifies the shipping step is reachable.
