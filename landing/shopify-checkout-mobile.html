Diagnosis First

Is This a Mobile-Specific Problem or a General One?

Before diving into mobile fixes, confirm the issue is mobile-specific. This saves hours of debugging.

2-Minute Diagnosis Flow

01 Test checkout on desktop (any browser). If checkout also fails on desktop, the issue is not mobile-specific — see our general checkout troubleshooting guide.
02 On the same mobile device, test checkout in Safari and then in Chrome. If one works but not the other → the issue is browser-specific (likely Safari ITP or cookie policy).
03 Test in Safari Private Browsing. If checkout fails in Private but works in regular Safari → a cross-site cookie or third-party iframe is blocked by Private mode.
04 Disable all Safari Content Blockers (Settings → Safari → Extensions). If checkout works → a content blocker is intercepting a payment or checkout script.
05 Test on a different iPhone or Android device. If only one device has the issue → device-specific (saved card conflict, OS version, iOS Content Restrictions).

Root Causes

9 Mobile-Specific Checkout Problems — and Fixes

Safari · iOS
01

Safari ITP Blocking Payment Cookies

Safari's Intelligent Tracking Prevention (ITP) treats cross-site cookies as trackers and deletes or blocks them after 7 days. Some third-party payment gateways — particularly older ones or those with complex redirect flows — depend on cross-site cookies to maintain session state during the checkout-to-bank-to-checkout handoff. When ITP blocks these cookies, the payment step fails, loops, or silently errors.

Symptoms: Payment fails in Safari on iPhone; succeeds in Chrome or on desktop. Card data enters correctly but "Place Order" either does nothing or shows a generic error.

FIX: → Switch to Shopify Payments as primary gateway — it uses first-party session handling that ITP doesn't block → Ensure Shop Pay is enabled — Shop Pay handles auth natively without cross-site cookies → For alternative payment gateways: contact the gateway's support and ask if they have an ITP-compatible integration mode → As a temporary workaround: tell customers to disable "Prevent Cross-Site Tracking" in iPhone Settings → Safari → Privacy & Security — but this is a customer-side change and not a real fix
Safari · iOS
02

Apple Pay Not Appearing or Failing to Authorize

Apple Pay only renders in Safari on iOS and macOS — never in Chrome, Firefox, or any non-Safari browser on iOS. When Apple Pay fails to appear or fails after the Face ID / Touch ID step, there are several distinct causes depending on the failure point.

Symptoms: Apple Pay button missing entirely; or Apple Pay sheet appears but authorization fails after biometric; or Apple Pay appears but "Payment not completed" error.

FIX — button missing: → Admin → Settings → Payments → Shopify Payments → Manage → Apple Pay → Verify domain: every domain and subdomain must be verified separately → Custom domains added after initial Shopify Payments setup need a re-verification step → The .well-known/apple-pay-merchant-sessions path must return 200 on your domain FIX — authorization fails after Face ID: → Check if the card in the customer's Wallet is expired or locked → Check if the order total changed between Apple Pay sheet open and auth (discounts applied mid-flow can cause mismatch) → Enable "Failed Apple Pay logs" in Shopify Admin → Analytics → Reports to see decline reasons FIX — headless / custom checkout: → Shopify's built-in Apple Pay does not automatically pass through to Hydrogen or custom checkout implementations → Must implement the Apple Pay JS API (PaymentRequest API) separately in your checkout code
Safari · iOS
03

3D Secure Redirect Loop in Safari

The 3D Secure flow redirects the customer from Shopify checkout → bank authentication page → back to Shopify. This round-trip sets a return cookie on the bank's domain that Shopify uses to confirm the completed authentication. Safari's ITP often deletes or blocks this cross-site cookie, causing the checkout to treat the authentication as incomplete and redirect again — creating an endless loop.

Symptoms: Checkout redirects to bank page, authentication succeeds (customer sees "approved"), then returns to checkout — which immediately redirects to bank again. Or: payment step hangs indefinitely after bank authentication.

FIX: → Switch to Shopify Payments — it uses a native 3DS2 implementation that doesn't rely on cross-site cookies → If using a third-party gateway, ask them if they support 3DS2 (not 3DS1) — 3DS2 uses a different, ITP-compatible auth mechanism → Customer workaround: clear Safari cache (Settings → Safari → Clear History and Website Data), then retry checkout — this resets the session state → Test using Stripe's test card 4000002760003184 (3DS required) to reproduce the exact redirect loop in your dev store before going to production
iOS · All Browsers
04

Keyboard Covering Checkout Form Fields

On iOS, the virtual keyboard slides up from the bottom without resizing the viewport in older behavior — it overlaps the page content instead. This causes checkout form fields to appear "behind" the keyboard and forces the customer to scroll awkwardly, sometimes making the active field or the "Next" button inaccessible. iOS 15+ changed behavior slightly but the problem persists in certain viewport configurations.

Symptoms: Customer taps a field, keyboard appears, and the field is now hidden behind the keyboard. Page appears frozen — but is actually waiting for input. "Place Order" button inaccessible after entering payment details.

FIX (for custom checkout implementations or checkout extensibility): → Ensure the checkout page uses: <meta name="viewport" content="width=device-width, initial-scale=1.0, interactive-widget=resizes-content" /> → Add padding-bottom equal to the keyboard height using the Visual Viewport API: window.visualViewport.addEventListener('resize', () => { document.body.style.paddingBottom = window.innerHeight - window.visualViewport.height + 'px'; }) → For Shopify's standard checkout: this is handled automatically by Shopify — if you're seeing this on standard checkout, it's a theme customization or app-injected script overriding the viewport → For checkout extensibility blocks: test all custom UI blocks on iOS Safari specifically before launch
All Mobile
05

Third-Party App Scripts Causing Mobile Failures

Apps that work adequately on desktop often break entirely on mobile. Mobile JavaScript execution is 3–5x slower than desktop (CPU speed), network is 3–10x slower (bandwidth and latency), and mobile browsers enforce stricter timeouts. A chat widget that loads in 400ms on desktop may take 2000ms on a mid-range Android on 4G — and some browsers abort scripts that take too long, causing dependent payment initializations to never complete.

Symptoms: Checkout works on desktop but payment step hangs or "Place Order" never responds on mobile. Or: checkout loads but is missing expected payment methods (because the payment init script timed out).

FIX: → Go to Admin → Settings → Checkout → App embeds — disable every embed you don't need at checkout → Priority apps to remove from checkout: live chat (Gorgias, Tidio, Intercom), loyalty widgets (Smile.io, LoyaltyLion), review badges (Okendo, Judge.me) → For each app: check if "checkout-only" embed can be disabled without removing the app from storefront pages — most apps support this via their settings → Test after each removal on a real mid-range Android device on 4G (not just iPhone on WiFi) → See the full app-by-app speed audit guide: Shopify checkout app performance audit
Safari · Private Browsing
06

Checkout Broken in Safari Private Browsing

Safari Private Browsing blocks all third-party cookies and sandboxes localStorage — meaning no cross-site data can be read or written during the session. Many payment widgets and some checkout apps rely on third-party storage to maintain their session, display personalized content, or initialize their SDK. In Private mode, these initializations fail silently, often causing the payment section to render empty or unresponsive.

Symptoms: Checkout broken in Safari Private window on iPhone. No payment methods show. Or: checkout appears to load but "Place Order" does nothing. Works in normal Safari on the same device.

FIX: → This is expected behavior in Private mode for apps that use cross-site storage — not a bug → Switch payment handling to first-party methods: Shopify Payments, Shop Pay, Apple Pay — these work in Private mode → Remove third-party payment widgets that depend on cross-site localStorage initialization → If your checkout entirely breaks (not just optional features), a critical initialization script is using cross-site storage — contact your payment gateway about a first-party storage option → Customer workaround: exit Private Browsing and retry. Explain to support agents to ask "Are you in Private/Incognito mode?" as a diagnostic question
Android · Chrome
07

Google Pay Not Appearing or Failing on Android

Google Pay on Shopify requires specific configuration in both Shopify Payments and the customer's Google account. Unlike Apple Pay (which uses device hardware for biometric auth), Google Pay relies on a server-side API call to Google that can fail for several reasons.

Symptoms: Google Pay button missing on Android Chrome. Or: Google Pay sheet opens but "Pay now" fails with a generic error. Or: Google Pay only appears on some devices but not others.

FIX — button missing: → Google Pay only shows when: (1) customer is logged into Chrome with a Google account that has a saved payment method, AND (2) the domain is verified with Google Pay → Admin → Settings → Payments → Shopify Payments → Manage → Google Pay — verify domain is listed and active → Google Pay uses the browser's Payment Request API — it won't show if the customer is using Chrome in incognito mode FIX — authorization fails: → Check if the customer's saved Google Pay card is expired → Google Pay requires the order currency to match a currency supported by the customer's card — cross-currency orders fail → If the order has a custom shipping rate or unusual tax calculation, the total may mismatch the authorized amount — switch to simpler shipping/tax config FIX — missing on some devices: → Google Pay requires Android 5.0+, Chrome 61+, and the Google Pay app installed or Google Play Services up to date → Older devices running Android 4.x will never show Google Pay — no fix, offer alternative methods
All Mobile
08

Slow Checkout on Mobile — Performance Gap vs Desktop

Mobile checkout is structurally slower than desktop due to CPU speed, network conditions, and the serial nature of mobile JavaScript parsing. What merchants experience as "checkout not working" on mobile is often checkout working — but taking 8–12 seconds, causing customers to tap away before it finishes. The checkout eventually loads, but for the customer it appeared broken.

Symptoms: Customers report "checkout doesn't work on phone" but you can't reproduce on desktop. Low mobile conversion rate despite desktop working fine. No error logs — just high abandonment at checkout.

FIX: → Run a mobile-specific checkout speed test: use Chrome DevTools → toggle Device Toolbar → set to "Mid-tier mobile" throttling → go through checkout and measure each step → Target: under 3 seconds on Mid-tier mobile is acceptable; under 5 seconds is minimum viable; over 5 seconds loses most customers → Highest-impact mobile speed fixes (in order): 1. Remove chat widget from checkout (saves 800–2000ms on mobile) 2. Remove loyalty/rewards widget from checkout (saves 400–900ms) 3. Remove any analytics pixels beyond GA4 (saves 200–500ms per pixel) → Enable Shop Pay — it preloads customer data and eliminates most form-filling, cutting checkout time by 40–60% for returning customers → Full guide: Shopify checkout slow — all causes and fixes
iOS
09

iOS Content Restrictions Blocking Checkout

iOS Screen Time and Content Restrictions (parental controls) can block access to online shopping, payment pages, or specific domains. When a customer's device has these restrictions active, Shopify checkout may load but payment options won't appear, or the checkout page may redirect to a blocked-content warning. This affects children's devices and sometimes corporate iOS device management (MDM) policies.

Symptoms: Checkout appears to work but payment step is blank. "Web Content" restrictions in Screen Time prevent certain URLs. Or: checkout URL is redirected to a warning page. Only affects specific customers' devices.

FIX (customer-side): → Settings → Screen Time → Content & Privacy Restrictions → Content Restrictions → Web Content → Change from "Limit Adult Websites" to "Unrestricted Access" — this is almost always what blocks checkout → Or: add your store domain and checkout.shopify.com to the Always Allowed list in Screen Time → For corporate MDM restrictions: the customer's IT department needs to whitelist *.shopify.com, *.myshopify.com, and the payment gateway domains → These cases appear in your customer service queue as "checkout doesn't work" — adding a help article about Screen Time restrictions can deflect these tickets

FAQ

Mobile Checkout — Frequently Asked Questions

Why is Shopify checkout not working on mobile?

Mobile checkout failures have different root causes than desktop failures. The most common mobile-specific causes: iOS Safari's Intelligent Tracking Prevention (ITP) blocks cross-site cookies that some payment gateways rely on; Safari blocks third-party iframes in Private mode; mobile keyboards cover form fields in some configurations; Apple Pay requires separate domain verification; and third-party app scripts that work on desktop often time out on mobile CPUs. Start by testing in Safari vs Chrome on the same device — if Chrome works but Safari doesn't, the issue is Safari-specific (ITP or cookie policy).

Why does Apple Pay not show on my Shopify checkout?

Apple Pay only appears in Safari on iOS and macOS — never in Chrome, Firefox, or other browsers. If it's missing in Safari: check that your domain is verified in Admin → Settings → Payments → Shopify Payments → Manage → Apple Pay. Custom domains require separate verification. Also confirm the customer has a payment card set up in their Wallet app — Apple Pay won't appear if no card is configured. For headless storefronts, you must implement the Apple Pay JS API separately.

Why does Shopify checkout redirect loop on iPhone?

iPhone redirect loops at checkout are almost always Safari's ITP blocking the cross-site cookie used during 3D Secure authentication. The 3DS flow sends the customer to a bank page and returns them — but the return cookie is cross-site and ITP deletes it, making Shopify think authentication never completed. Fix: switch to Shopify Payments (which uses native 3DS2 without cross-site cookies). Customer workaround: clear Safari cache and retry.

Why is Shopify checkout slow on mobile but fast on desktop?

Mobile checkout is structurally slower: JavaScript runs 3–5x slower on mobile CPUs, and 4G networks add latency that WiFi hides. Third-party app scripts (chat widgets, loyalty apps) cause the largest mobile slowdowns — a script adding 300ms on desktop can add 800–1500ms on a mid-range Android on 4G. Removing the chat widget from checkout is the single highest-ROI mobile speed fix. See our app audit guide for per-app impact data.

Can I test mobile checkout without a real phone?

Partially. Chrome DevTools lets you simulate mobile screen size and network speed — open DevTools (F12) → click the device icon → select a mobile preset → set network throttling to "Mid-tier mobile". This gives accurate performance data. However, it cannot replicate Safari's ITP behavior, iOS keyboard behavior, or Apple/Google Pay — for those, you need real devices. Keep an iPhone (for Safari ITP testing) and a mid-range Android (for performance testing) as part of your QA workflow.
