Root Cause

Why Checkout Localization Errors Are Invisible to Merchants

Merchants test their checkout from their own location, in their own language, using their own phone number format. Everything works. But an international customer — a German buyer who expects a German-language checkout, or a Japanese customer entering their phone number in the local format — hits a silent wall.

Shopify's checkout localization has two independent systems: language (controlled by Markets and locale URLs) and field validation (controlled by country-specific rules). A third layer — third-party checkout customizations — can override either. When any part of this breaks, customers get blocked without any error appearing in Shopify Admin, and the merchant only finds out through abandoned cart emails or support tickets.

The distinct localization failure mode

Localization failures differ from general checkout errors: the checkout URL returns HTTP 200, payment gateways are fine, and your default-locale customers complete orders normally. The failure only affects customers in specific locales, using specific phone number formats, or entering non-Latin characters. Standard checkout monitoring in your home locale won't detect these failures.

Known Issues by Region

Locales With Common Checkout Validation Problems

Japan
Phone format + address field order
Japanese phone numbers use 10-11 digits in 090-XXXX-XXXX format. Address field order (prefecture before city) differs from Shopify's default. Kana/Kanji characters in name/address fields can fail some checkout customizations.
Brazil
CEP postal code + CPF tax ID
Brazilian CEP codes use a 5-3 format (01310-100). Some checkout customizations require the hyphen; others reject it. B2B customers may need a CPF/CNPJ field which isn't natively in Shopify checkout without customization.
Germany / EU
VAT number field + language toggle
German business buyers expect a Umsatzsteuer-ID (VAT number) field at B2B checkout. Without it, they can't reclaim VAT. Language toggles often don't pass the locale to Shopify checkout session, so checkout stays in English.
Arab countries
RTL layout + Arabic locale
Arabic checkout requires right-to-left (RTL) layout. Shopify's native checkout supports Arabic (ar locale) but checkout customizations built without RTL awareness break visually — form fields, buttons, and error messages display incorrectly.
Hong Kong / Macau
No postal code required
Hong Kong doesn't use postal codes. If the checkout's postal code field is required (which it may be by default depending on theme configuration), Hong Kong customers can't complete checkout without entering a fake code.
Ireland
Eircode validation
Irish Eircodes introduced in 2015 use a 7-character alphanumeric format (A65 F4E2). Some checkout validation patterns still expect the old format or reject Eircodes because they don't match a simple numeric-only postal code regex.
Diagnosis

7 Checkout Localization Failure Modes

CAUSE 1 Checkout stays in default language — locale not passed to checkout session
The customer switches language on your storefront but checkout opens in English (or your store's default language). Shopify checkout picks up the locale from the storefront session URL. If your theme's language switcher changes the display language with JavaScript but doesn't redirect to the locale URL (e.g., /fr/ for French), the Shopify session never updates — and checkout inherits the default locale.
Fix: Verify the language switcher redirects to the locale URL. In Shopify Admin → Online Store → Themes → Edit code, find the language-switcher snippet and confirm it generates a URL like https://yourstore.com/fr/ rather than simply toggling a CSS class. Also confirm in Admin → Markets → [market] → Languages that the target language is Published (not just added). Unpublished languages won't appear in checkout.
CAUSE 2 Translation app translates storefront but not checkout
Third-party translation apps (Weglot, Langify, Transcy, GTranslate) translate storefront pages but have limited or no ability to modify the Shopify checkout directly — Shopify restricts third-party access to the checkout page. Customers experience a seamlessly translated storefront that drops them into an untranslated checkout. This is one of the most reported localization complaints from international Shopify merchants.
Fix: Shopify checkout language is controlled natively through Shopify's Markets + Languages system (Admin → Markets → [market] → Languages), not through translation apps. For checkout to display in a given language: (1) the language must be added to the market in Admin, (2) the customer must reach checkout via the locale URL, and (3) you need to add checkout translations for that language via Admin → Online Store → Translations. Translation apps can handle storefront translations; Shopify Translate handles checkout translations.
CAUSE 3 Phone number validation rejecting valid international formats
Shopify checkout validates phone numbers against the shipping country's expected formats. When a customer's phone format differs from the shipping destination's expected format — common with expats, travelers, or customers with foreign SIM cards — the checkout rejects the number. A Chinese customer with a +86 number shipping to the US will have their phone rejected if the US format validation runs. A Japanese customer entering 09012345678 (without international prefix) may be rejected if the form requires the +81 format.
Fix: This is often a Shopify limitation rather than a configuration issue. For customers blocked by phone validation: instruct them to enter their number in international format (+[country code][number]). If you use a checkout customization that overrides the phone field, review its validation pattern — ensure it accepts international formats (+XX prefix) in addition to local formats. Consider making the phone field optional if phone isn't critical to your operations (Admin → Settings → Checkout → Phone number → Optional).
CAUSE 4 Postal code validation blocking customers from non-standard countries
Shopify has country-specific postal code validation. Problems appear in two scenarios: (1) Countries without postal codes (Hong Kong, Macau, parts of Latin America, some Caribbean countries) — if the postal code field is set to required, customers from these countries can't complete checkout. (2) Countries with non-numeric postal codes (Canada uses "A1A 1A1" with letters, Ireland's Eircodes use alphanumeric) — some checkout customization scripts have overly strict numeric-only validation.
Fix: For countries without postal codes: Admin → Settings → Shipping and delivery — verify these countries are in a shipping zone that doesn't require postal codes, and consider setting the postal code field to optional for these markets. For countries with alphanumeric postal codes: test checkout from the specific country using a VPN or by temporarily setting your own address to that country. If a checkout customization is the cause, locate the postal code validation script and update the regex to allow alphanumeric formats.
CAUSE 5 Non-Latin characters blocked in name or address fields
Shopify's native checkout accepts all Unicode characters (Chinese, Japanese, Korean, Arabic, Cyrillic, Hebrew, etc.) in name and address fields. When non-Latin characters are rejected, the cause is a checkout customization — a validation script that only permits ASCII/Latin characters, an address normalization app that sanitizes input to Western characters, or a fraud prevention app that flags non-Latin addresses as suspicious.
Fix: Test checkout with a non-Latin name and address without any checkout apps active. If it works without apps but fails with apps: Admin → Settings → Checkout → Checkout Extensions — identify which app is running. Disable apps one by one and test after each to find the conflicting script. For fraud prevention apps that flag non-Latin addresses: most have an allowlist or sensitivity setting; adjust the character-set policy to permit Unicode inputs.
CAUSE 6 RTL (right-to-left) language checkout breaks visually
Shopify's checkout natively supports RTL languages including Arabic (ar) and Hebrew (he). However, checkout customizations built without RTL awareness — checkout.liquid customizations on Shopify Plus, or Checkout Extensions that use absolute CSS positioning — break the RTL layout. Form fields, error messages, and buttons can overlap, appear on the wrong side, or become inaccessible when the checkout direction flips to RTL.
Fix: Test checkout in Arabic or Hebrew by setting your browser locale and using the /ar/ or /he/ locale URL. Identify which custom elements are misaligned. In checkout.liquid (Plus) or Checkout Extension CSS: use logical CSS properties (margin-inline-start instead of margin-left, padding-inline-end instead of padding-right) rather than physical directional properties. This ensures elements reflow correctly in both LTR and RTL modes. The native Shopify checkout elements don't require changes — only custom-added elements.
CAUSE 7 EU VAT / tax ID field missing for B2B buyers
European B2B buyers often need to enter a VAT registration number at checkout to ensure zero-rated (tax-free) intra-EU purchases. Shopify's standard checkout doesn't have a built-in VAT field for consumer checkout. Without this field, EU B2B buyers must pay tax upfront and then reclaim it — or they abandon checkout because the total price is wrong for business accounting. This is technically a localization issue because it only affects buyers in specific jurisdictions.
Fix: Shopify Plus stores can add a VAT number field to checkout using a Checkout Extension. Several Shopify apps also add VAT number fields to checkout for non-Plus stores (search "VAT number checkout" in the Shopify App Store). The collected VAT number should then be included in the order and on the order invoice. Verify the field validates EU VAT number formats (two-letter country code + digits/letters, e.g., DE123456789) — not just a free-text field that accepts anything.

Testing Protocol

Localization Checkout Test Checklist

Run this for every market you actively target. Use a VPN to simulate the customer's country, or use a browser locale override. Test in a clean incognito window.

  • Navigate to your store via the locale URL (e.g., /fr/ for French) — verify checkout opens in the target language
  • Add a product and proceed to checkout — verify all checkout UI elements are in the correct language (not mixed)
  • Enter a phone number in the local format for the target country — verify it's accepted
  • Enter a postal code in the format used by the target country — verify it's accepted
  • For CJK markets (Japan, China, Korea): enter name and address in local script — verify no "invalid characters" error
  • For Arabic/Hebrew markets: verify the checkout layout is RTL and form fields are usable
  • For EU B2B markets: verify a VAT number field is available if your buyers need it
  • For markets without postal codes (Hong Kong, etc.): verify checkout completes without a postal code
  • Complete a full test order in each target locale — verify confirmation email arrives in the correct language
  • After any checkout customization or translation app change: rerun all locale tests

FAQ

Checkout Localization Questions

Why is my Shopify checkout showing in the wrong language?
Shopify checkout language is set by the market the customer is in, not the browser language. The language must be added and Published in Admin → Markets → [market] → Languages. The customer must access the store via the locale URL (e.g., /fr/ for French). Translation apps can't directly modify the Shopify checkout page — use Shopify's native translation system for checkout strings.
Why does Shopify checkout reject valid international phone numbers?
Shopify validates phone numbers against the expected format for the shipping country. A phone number in one country's format shipping to a different country can fail. The most reliable fix: make the phone field optional (Admin → Settings → Checkout → Phone number → Optional) or instruct customers to enter numbers in international format (+[country code][number]). If a checkout app is overriding phone validation, the app's validation script is the source — disable apps and test to identify the culprit.
Why is Shopify checkout failing postal code validation for international customers?
Shopify uses country-specific postal code validation. Issues arise for: countries without postal codes (Hong Kong — make the field optional for that market), countries with alphanumeric codes (Canada, Ireland — ensure checkout customizations allow letters), and countries where the format requires a specific separator (Brazil's CEP hyphen). Test from the specific country and check whether the failure is Shopify native validation or a checkout customization script.
How do I fix Shopify checkout language not changing when customers switch language?
The theme's language switcher must redirect to the locale URL (/fr/, /de/, etc.) — not just toggle a display language with JavaScript. Check the switcher code in your theme. Also verify the language is Published (not just added) in Admin → Markets → [market] → Languages. If you use a translation app, enable its "checkout locale handoff" setting — most major apps (Weglot, Langify) have this option.
Why does Shopify checkout block non-Latin characters in name or address fields?
Shopify's native checkout accepts all Unicode characters. If non-Latin input is rejected, a checkout customization is the cause — either an app running ASCII-only validation or a checkout.liquid script on Shopify Plus. Disable checkout extensions one by one to identify the source. The fix is updating the validation regex in the conflicting script to accept Unicode ranges rather than only ASCII characters.
