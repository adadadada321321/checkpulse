Which Apps Slow Down Shopify Checkout — Performance Audit

Apps are the single largest merchant-controllable source of checkout latency. A single chat widget can add a full second. Stacked across 6–8 checkout-touching apps, the cumulative impact can reach 3–5 seconds — invisible to Shopify, invisible to your uptime monitor, and quietly costing you conversions every day.

Impact Reference

App Categories by Checkout Load Time Impact

Each app category below injects resources into the checkout flow in a different way. "Blocking" means the checkout UI waits for the request to complete before rendering. "Non-blocking" means it loads in parallel — still adds to total time, but less severe.

Live Chat (Gorgias, Tidio, Intercom, Zendesk) +400–1,200ms Non-blocking SDK + polling FIRST
Subscription Billing (Recharge, Skio, Bold) +300–800ms Blocking API call for eligibility FIRST

Audit Process

How to Run a Shopify Checkout App Audit

This audit isolates each app's contribution to checkout load time. Total time: 30–60 minutes. Tools needed: Chrome DevTools (free, built in). No app changes until Step 4 — measure first.

  1. 01
    Take a baseline measurement. Open Chrome in an incognito window. Add a product to cart. Open DevTools (F12) → Network tab → check "Disable cache". Click "Proceed to checkout". When the checkout page finishes loading, look at the bottom status bar: it shows "X requests, Xms, Finished". Record these numbers. This is your starting point.
  2. 02
    Identify all checkout-touching apps. Go to Shopify Admin → Settings → Apps and sales channels. Note every app that has "checkout" or "post-purchase" in its access scope. Also check: Admin → Online Store → Themes → Customize → Checkout (look for app blocks injected into checkout UI). Also check: Admin → Settings → Checkout → App embeds.
  3. 03
    Find the slowest request in DevTools. In the Network tab, click the "Time" column to sort by duration, descending. Identify any request over 300ms. Note the domain — external domains (not *.shopify.com or *.myshopify.com) are third-party app resources. The domain name usually identifies the app directly.
  4. 04
    Disable the highest-impact app first, measure again. Start with the app whose domain appeared slowest in the Network tab. Disable it temporarily (or remove its checkout block from the theme editor). Repeat Steps 01–03 with the app disabled. Record the new load time. The delta = that app's exact contribution to your checkout latency.
  5. 05
    Repeat for each checkout-touching app. Work through the list one at a time. After each app, record: (a) app name, (b) load time with app, (c) load time without. You'll have a ranked list of which apps cost the most. Re-enable apps as you go — disable only the one you're currently testing.
  6. 06
    Make the keep/cut decision. For each app, compare its revenue contribution against its load time cost. A loyalty app generating $800/month in incremental revenue that costs 600ms is worth keeping. A chat widget that's never initiated a checkout conversation and adds 900ms is not. See the decision framework below.
  7. 07
    Set your new performance baseline. After removing underperforming apps, measure checkout load time 3 times and average it. This is your new baseline. Then set up automated checkout monitoring — so you're alerted if a future app install or auto-update regresses your checkout speed before customers notice.
App-by-App Breakdown

Why Each App Category Adds Latency

Understanding why an app is slow helps you decide whether to remove it or find a faster alternative. Not all slowness is equal — a blocking API call is worse than a deferred script of the same duration.

01
HIGH IMPACT

Live Chat Widgets — Gorgias, Tidio, Intercom, Zendesk

Chat widgets load a full JavaScript SDK at checkout — typically 40–120KB of JS that initiates a WebSocket or polling connection to the chat backend. The SDK boots as soon as the checkout page loads, competing for network and CPU with the checkout's own initialization sequence. Most merchants never receive a chat message from a customer who is actively in checkout — the widget is loaded on a page where it provides near-zero value.

Audit action: Admin → Settings → Checkout → App embeds → find the chat app embed → toggle off for checkout pages only. Most chat apps support page-level exclusion without full uninstall. Impact: typically −400 to −1,000ms on checkout first load.
02
HIGH IMPACT

Subscription Billing — Recharge, Skio, Bold Subscriptions

Subscription apps inject checkout UI blocks that fetch subscription pricing, eligibility, and product configuration from the app's own API on every checkout load — even for customers who are buying one-time items. The API call is typically blocking: the checkout UI waits for the subscription app to respond before rendering the full page. When the subscription app's server is under load, this can extend to 1–2 seconds.

Audit action: Admin → Online Store → Themes → Customize → Checkout → find Recharge/Skio checkout block → remove from checkout layout. The app remains installed; only the checkout injection is removed. Re-add only if your store actively sells subscription products and the revenue justifies the latency cost. Impact: typically −300 to −800ms.
03
MEDIUM IMPACT

Loyalty & Rewards — Smile.io, LoyaltyLion, Yotpo Loyalty

Loyalty apps fetch the customer's current point balance and available rewards from their API to render a "You have X points — apply $Y discount" widget in checkout. This requires an authenticated API call per checkout session. For stores with a small loyalty program (under 5% of customers actively redeeming), the API overhead affects 100% of checkout sessions while benefiting under 5%. Consider whether the in-checkout widget is materially driving redemptions or if email/account page is sufficient.

Audit action: disable the checkout block from the theme editor. Move loyalty redemption to the cart page or account page where it does not block checkout performance. Impact: typically −200 to −700ms.
04
MEDIUM IMPACT

Upsell & Post-Purchase Apps — ReConvert, Zipify, CartHook

Upsell apps that inject into the checkout page (not just post-purchase) fetch offer data — product recommendations, bundle logic — during checkout initialization. Apps that only load at the post-purchase step have lower impact on the checkout flow itself but still add latency to the order confirmation page. If your upsell app fires during checkout (not just after payment), the offer fetch is blocking or near-blocking.

Audit action: check your upsell app's configuration for "checkout page" vs "post-purchase" placement. Disable in-checkout injection; keep post-purchase only if AOV data justifies it. Impact: typically −300 to −600ms for in-checkout placement; −100 to −300ms for post-purchase only.
05
MEDIUM IMPACT

Fraud Prevention — Signifyd, NoFraud, Kount

Fraud scoring apps make an API call during checkout to score the session for fraud risk. For legitimate customers (the vast majority), this call adds latency with no benefit to the transaction. The call is often blocking — the checkout cannot display the payment step until the fraud score returns. If your store is not in a high-fraud category (digital goods, high-value electronics), the false positive rate from fraud prevention may not justify the blanket latency cost on all sessions.

Audit action: review your fraud app's analytics — what % of orders are flagged, and what's the chargeback rate without it? If chargeback rate is under 0.3%, the ROI vs. latency trade-off may not hold. If you need fraud prevention, check if the app supports async scoring (score after order creation, not during checkout). Impact: −200 to −500ms.
06
LOWER IMPACT — but additive

Social Proof / Review Widgets

Review apps configured to show trust badges or star ratings inside checkout fetch that data from their API on each page load. Unlike category 1–5 apps, most social proof widgets are non-blocking — they load asynchronously and don't delay the checkout's critical rendering path. However, they still add to total page weight and time-to-fully-loaded, which affects perceived performance on slower connections.

Audit action: check if the widget loads synchronously or asynchronously (DevTools → Network → look for the app domain — if it appears before the DOMContentLoaded marker, it's synchronous). Async is acceptable; sync is not. Disable checkout injection if the widget is synchronous. Impact: typically −100 to −400ms.
07
LOWER IMPACT — but additive

Custom Analytics Pixels

Additional tracking pixels added via Shopify Customer Events or checkout.liquid (beyond GA4 and Meta Pixel, which are standard) each add a network request. At 100–300ms per pixel, they seem small — but three redundant pixels add up to 300–900ms. Common culprits: duplicate GA implementations, legacy pixel from a previous agency, platform-specific pixels for ad networks you no longer actively use.

Audit action: Admin → Settings → Customer events → audit every event listed. Remove any pixel for a platform you are not actively running ads on or reporting from. Ask your agency which pixels they need — duplicate pixels are common after agency transitions. Impact: −100 to −300ms per removed pixel.
Decision Framework

Keep vs. Cut: App ROI vs. Latency Cost

Not every slow app should be removed. The decision is ROI versus latency cost. Use this framework to evaluate each app you find in the audit.

Scenario App Revenue / Value Latency Cost Decision
Subscription app, 30% of revenue Critical — $15K+/mo +600ms KEEP — irreplaceable revenue driver
Loyalty app, 8% of customers redeeming Moderate — $2K/mo incremental +500ms OPTIMIZE — move to cart/account page, remove checkout injection
Chat widget, 0 checkout conversations last 90 days None at checkout +900ms REMOVE from checkout — no value, pure latency

Setting a Baseline — and Keeping It

The audit is a one-time event. Shopify apps auto-update. New apps get installed. The performance you recover today can regress silently tomorrow — and you won't know until checkout is slow again and customers are abandoning.

FAQ

Frequently Asked Questions

Which Shopify apps slow down checkout the most?

The highest-impact app categories: live chat widgets (Gorgias, Tidio, Intercom, Zendesk) typically add 400–1,200ms. Subscription billing apps (Recharge, Skio) add 300–800ms from blocking API calls. Loyalty/rewards apps add 200–700ms from customer point balance fetches. Upsell apps with in-checkout injection add 300–600ms. Fraud prevention apps add 200–500ms. With 6–8 of these running simultaneously, cumulative checkout latency easily reaches 3–5 seconds beyond baseline.

How do I find which app is slowing down my Shopify checkout?

Open Chrome DevTools (F12) → Network tab → Disable cache → add to cart → proceed to checkout. When checkout loads, sort the Network tab by "Time" (descending). Look for requests over 300ms. External domains (not *.shopify.com) are third-party apps — the domain name usually identifies the app. Then disable that app and retest checkout timing. The difference is that app's exact contribution to your checkout latency. Repeat for each app.

How do I audit checkout apps without breaking anything?

Disable one app at a time, not all at once. For checkout extension apps, remove the checkout block from the theme editor (Admin → Online Store → Themes → Customize → Checkout) without uninstalling the app. Re-enable after each test. Work during low-traffic hours and have another tab open to re-enable quickly. Never disable subscription billing apps during active sales periods — test those at 2am on a slow day.

Can I remove a chat widget from checkout only, without removing it everywhere?

Yes — most chat apps (Gorgias, Tidio, Intercom) support page-level exclusion. In the app settings, look for "excluded pages" or "page rules" and add your checkout URL. Alternatively, disable the app embed in Shopify Admin → Settings → Checkout → App embeds — this removes it from checkout specifically while keeping it on your storefront pages where it has actual value.

How much faster will my checkout be after removing slow apps?

Typical results from a thorough app audit on a store with 6–8 checkout-touching apps: 1.5–3 second improvement in checkout load time. At $50K/month in revenue, a 2-second improvement is worth approximately $7,000–$10,000/month in recovered conversion (7–10% per second × 2 seconds × $50K). The audit itself takes 30–60 minutes. Few 30-minute investments have higher per-hour ROI for an e-commerce store.

Will my uptime monitor tell me if an app slows down checkout after I fix it?

No. Standard uptime monitors (UptimeRobot, Pingdom) measure server response time — typically 100–400ms. They cannot detect checkout performance degradation because they do not execute JavaScript, initialize the React checkout UI, or fetch payment methods. Only a tool that runs a real browser checkout simulation, like CheckPulse, measures the full checkout load time that customers experience and can alert you to performance regressions from app updates or new installs.
