Shopify Checkout — Product Customization

Shopify Checkout Product Customization Not Working — Line Item Properties Lost, Engraving / Monogram Not Showing in Order

Shopify has no native product customization system. Every engraving field, monogram text box, personalization option, and custom image upload is a third-party app that writes line item properties to the cart. When those properties are never written — because the customer clicked Buy Now, because a JavaScript error killed the widget, or because a theme update deactivated the App Embed — the order arrives with no customization data and no error message.

Updated: June 2026 Affects: Infinite Options · SC Product Options · Product Customizer · Teeinblue · Hulk
The silent failure: When product customization is lost at checkout, neither the merchant nor the customer receives an error. The order completes normally. The merchant doesn't discover the problem until they open the order and find blank Properties — after they've already started production on the item.
How Shopify Product Customization Works

The Three-Layer Architecture — and Where Each Can Break

Every Shopify product customization flow has three components. A failure at any layer means the customization data is either never collected, never written, or never displayed — often with no visible error for the customer or merchant.

Layer 1
Product Page Widget
The customization app injects a widget on the product page — text fields, dropdowns, image uploads, font pickers. Failure here: App Embed disabled after theme update, JS error, widget injection selector broken by theme change, or customer bypassing the product page entirely.
Layer 2 — Most Common Failure
Cart Line Item Properties
When Add to Cart is clicked, the app writes the customization data as line item properties attached to the cart line. This step is skipped entirely when Buy Now or Shop Pay express is used — those buttons bypass the cart and open checkout immediately, before properties can be written.
Layer 3
Order Attribute Display
Line item properties carry through Shopify's checkout to the order. In Admin, they appear under the line item as a small gray Properties list. They are NOT in the default packing slip or order confirmation email unless explicitly added to those templates.

Deep Dive

The Buy Now Bypass — Why Express Checkout Kills Customization

Architecture Detail

How Buy Now Skips the Customization Step

When a customer clicks Add to Cart on a customized product, the customization app's JavaScript intercepts the form submission, collects all widget field values, packages them as a properties object, and passes them to the Shopify Cart API alongside the variant ID and quantity. This is the only moment when customization data is written to the cart.

The Buy Now button — and the Shop Pay express button — bypass this interception entirely. They call Shopify's Draft Order or Checkout Create API directly, passing only the variant ID and quantity. The customization app's form submission listener is never triggered. No properties are collected. No properties are written. The resulting cart goes straight to checkout with a bare line item.

This is not a bug in the customization app. It is an architectural limitation: the app can only intercept Add to Cart events, not the Buy Now checkout-create call. The only way to make Buy Now work with customization is to rebuild the Buy Now flow so that it opens the cart (or a custom page) first — which defeats the purpose of express checkout.

# Infinite Options buy_now_button_disable setting # In Infinite Options dashboard → Settings → Advanced: # "Disable dynamic checkout buttons" = ON # This hides Buy Now, Shop Pay, Google Pay, Apple Pay # on all product pages with active option sets

The practical recommendation: disable dynamic checkout buttons (Buy Now, Shop Pay) on all product pages where customization is required. For gift cards and non-customized products, leave them enabled. The loss in conversion rate from disabling express checkout on customized products is almost always smaller than the operational cost of fulfilling orders with missing customization data.

Architecture Detail

Checkout Extensibility and the Loss of Required Field Validation

Before Checkout Extensibility, some product customization apps (notably SC Product Options / Bold Product Options) used checkout.liquid to add a validation step at checkout. If a line item had a required property field that was empty, the checkout.liquid script would block the customer from completing the order.

Shopify's Checkout Extensibility — the post-checkout.liquid architecture — deprecated all checkout.liquid modifications. These validation scripts stopped running silently when merchants migrated their themes to the new extensibility layer. Merchants who previously relied on the checkout-level validation block discovered that customers could now complete orders with empty required fields.

As of mid-2026, very few product customization apps have released Checkout UI Extensions that replicate the required field validation inside the new checkout. The gap is real and known to app vendors. The only current workaround that does not require waiting for an app to release a Checkout UI Extension is an order-level hold via Shopify Flow:

# Shopify Flow: hold orders with missing customization Trigger: Order Created Condition: Order has line item where Properties does not contain key "Engraving Text" Action: Hold fulfillment → Tag order "missing-customization" Action: Send internal email alert with order ID

This does not prevent the purchase, but it does create an operational gate before fulfillment. The merchant can email the customer to collect the missing customization before the item goes into production.

Step-by-Step Diagnosis

Product Customization Checkout Failure — 5-Step Diagnosis

Work through these steps in order. The Buy Now bypass accounts for the majority of reports. Check it first before investigating theme or app issues.

  • 1
    Reproduce the failure path exactly Ask: did the customer use Buy Now, Shop Pay express, or Add to Cart? Buy Now and Shop Pay bypass the customization widget entirely — if the customer used either of these, the customization was never written and this is expected behavior given the current architecture. Test both paths yourself: (a) fill in customization and click Add to Cart → proceed to checkout → verify properties in order. (b) fill in customization and click Buy Now → verify properties. If (a) works and (b) fails, this is a Buy Now bypass issue, not an app bug.
  • 2
    Check App Embed status in the current active theme Go to Shopify Admin → Online Store → Themes → click Customize on your live theme → click App Embeds in the left sidebar → find your customization app and verify it is toggled ON. After any theme update or theme switch, App Embeds from the old theme are not transferred. If your App Embed is off or missing, re-enable it. Then also check the product template: Admin → Themes → Customize → navigate to a product page template → verify the customization app's App Block or section is present.
  • 3
    Inspect line item properties in the order Go to Shopify Admin → Orders → click the affected order → find the line item for the customized product. Directly below the variant name and SKU, there should be a small Properties section listing each customization field and its value. If Properties is present but empty, the widget rendered but the customer didn't fill it in (or bypassed it). If Properties is completely absent from the line item, no properties were ever written to the cart — trace back to steps 1 and 2. If Properties is present with correct values, the problem is display-only (packing slip or email template — see the template fix in Step 5).
  • 4
    Open browser DevTools console on the product page In an incognito window, navigate to a product page with customization. Open DevTools (F12) → Console tab. Look for JavaScript errors from your customization app (error messages often reference the app name, a CDN domain like hulkapps.com or product-customizer.com, or use terms like "options", "properties", "widget"). An error here means the widget is failing to load or initialize — usually because the injection selector doesn't match your current theme's product form HTML. Check your app's dashboard for a theme compatibility or widget position setting, or contact app support with your theme name and version.
  • 5
    Fix display issues in email and packing slip templates If properties ARE correctly in the order (Step 3 confirmed) but not appearing in the order confirmation email or packing slip: (a) For order confirmation email — go to Admin → Settings → Notifications → Order confirmation → Edit code → find the line items loop → add: {% for p in line_item.properties %}{% unless p.last == blank %}{{ p.first }}: {{ p.last }}<br>{% endunless %}{% endfor %} (b) For packing slips — install Order Printer Pro (free tier available), which includes line item properties in its default packing slip templates. Shopify's native packing slip template does not support properties without custom code.

FAQ

Shopify Product Customization Checkout — Frequently Asked Questions

Why does Shopify Infinite Options not work when customers use the Buy Now button?
The Buy Now button and Shop Pay express button send the customer directly to checkout without passing through the cart. Infinite Options — and most other customization apps — write the customization data (engraving text, font choice, color selection) as line item properties at the moment Add to Cart is clicked. When Buy Now is used instead, the cart is created directly from the variant ID and quantity only, without any properties attached. Infinite Options has a "Disable dynamic checkout buttons" setting in its dashboard under Settings → Advanced. Enable this for all product pages with required customization fields. This hides Buy Now, Shop Pay, Google Pay, and Apple Pay on those pages, ensuring customers go through the cart and the customization widget before reaching checkout.
Why are line item properties not showing in the Shopify order after a custom product purchase?
There are two distinct issues: properties not written versus properties not displayed. If properties were not written, the order will have no Properties section under the line item in Shopify Admin — the data was lost before checkout (most likely a Buy Now bypass, a disabled App Embed, or a JavaScript error on the product page). If properties WERE written but are not appearing in the order confirmation email or packing slip, that is a template issue. The default Shopify order confirmation email does not include line item properties — you need to edit the notification template to add a properties loop. The default Shopify packing slip also excludes properties — use Order Printer Pro or a custom packing slip template that includes the properties loop. To confirm whether properties exist: Shopify Admin → Orders → click the order → scroll to the line item → look for a small gray "Properties" label with the customization fields listed.
Why did my Shopify product customization stop working after a theme update?
Theme updates break customization apps in three ways. First, App Embeds do not transfer when you switch or update a theme — re-enable the customization app's App Embed under Admin → Online Store → Themes → Customize → App Embeds. Second, the app injects its widget by targeting a CSS selector in your theme's product form (for example, .product-form__submit or #AddToCartForm). If the theme update renamed these elements, the injection fails silently. Check your app's dashboard for a "Widget position" or "Injection selector" setting and update it. Third, some apps used checkout.liquid scripts for required field validation at checkout — those scripts stopped working when Checkout Extensibility deprecated checkout.liquid. If you relied on checkout-level validation, you need an app with a Checkout UI Extension (very few currently offer one) or a Shopify Flow hold as a workaround.
Can Shopify customers bypass required customization fields and complete checkout without providing personalization?
Yes — required field validation in most customization apps runs at Add to Cart, not at checkout. Any path that bypasses the product page widget (Buy Now, direct checkout URL, abandoned cart email, shared cart link) will skip the required field check entirely. The only reliable solution is a Checkout UI Extension that re-validates the required property at checkout — as of mid-2026, very few customization apps have released one. The practical workaround: (1) Disable Buy Now on customizable products, (2) use Shopify Flow to tag and hold orders where the expected property key is missing, and (3) email customers to collect the missing customization before fulfillment. This adds operational steps but prevents producing items without customization data.
Does Shopify product customization work with subscription apps like Recharge?
This is one of the highest-conflict pairings in the Shopify app ecosystem. Subscription apps (Recharge, Skio) and customization apps both modify the cart line item at checkout — subscriptions attach a selling plan ID, customization apps write properties. When both transform the same line item, one of three outcomes occurs: the subscription transformation strips the properties, the customization transformation drops the selling plan (silently converting a subscription to a one-time purchase), or checkout throws an error. Recharge has a line item properties pass-through feature for initial subscription orders — but renewal orders are headless API calls that bypass the product page widget entirely, so customization properties cannot be collected at renewal. Recommendation: for subscription + customization combinations, use Shopify's native variant system for any customization relevant to subscription products, and only use dynamic customization app fields for non-subscription items.
