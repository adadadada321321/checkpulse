Shopify Buy Now Button Not Working — 8 Causes Explained

Dynamic checkout buttons disappeared after changing payment gateway? Buy Now button losing line item properties and custom options? Subscription product hiding all express buttons? Apple Pay not showing? Here are all 8 root causes of Shopify buy now button failures — for merchants and developers.

Updated June 2026 8 failure modes Dynamic Checkout Buttons · Shopify Payments · Line Item Properties · Subscription Conflict · Apple Pay · Buy Button Channel · Headless

Shopify's "Buy Now" button — officially called the dynamic checkout button — shows express payment options (Shop Pay, Apple Pay, Google Pay, PayPal Express) directly on product pages, allowing customers to skip the cart and go straight to checkout. When it stops working, the failure is almost always silent: the buttons simply disappear, or orders arrive missing customization data, with no error message explaining why.

The two most common causes merchants don't know about: (1) switching from Shopify Payments to a third-party gateway silently removes all four express buttons, and (2) adding a subscription option to any product immediately hides all dynamic checkout buttons for that product. Neither generates an admin notification.

Quick Diagnostic — Which Buy Now Failure Do You Have?
Buttons disappeared suddenly? → Check if you recently changed payment gateways (Failure Mode 1) or added a subscription selling plan to the product (Failure Mode 3).

Button visible but orders missing customization? → Line item properties lost via Buy Now bypass (Failure Mode 4).

Apple Pay not showing? → Browser/device restriction or domain verification (Failure Mode 6).

Embedding buy now on external site? → This is the Buy Button channel, a completely separate product (Failure Mode 7).

Headless storefront, button does nothing on click? → Storefront API implementation missing (Failure Mode 8).

Failure Mode 1: Shopify Payments Required — Third-Party Gateway Removes All Dynamic Checkout Buttons

This is the most common cause of dynamic checkout buttons disappearing, and it's almost never accompanied by any explanation in Shopify Admin. Switch from Shopify Payments to a third-party gateway, and all four express buttons vanish from every product page in your store simultaneously.

Which gateways trigger this removal

Every third-party gateway triggers it: Stripe (when configured as a third-party provider rather than using Shopify Payments which is Stripe-powered under the hood), PayPal (when set as the primary gateway), Authorize.net, Square, Braintree, Adyen, and any other non-Shopify-Payments processor. The reason is architectural: Shop Pay, Apple Pay, Google Pay, and PayPal Express on Shopify product pages are all rendered by Shopify Payments infrastructure. They cannot function without it.

Third-party gateway (Stripe, PayPal, Authorize.net) Removed Removed Removed Removed

Shopify Payments Availability by Country
Shopify Payments is not available in all countries. Merchants in markets where Shopify Payments is unavailable cannot offer dynamic checkout buttons at all — they must use third-party gateways which excludes the express options. Check Shopify's official supported countries list for current availability.

Fix

Re-enable Shopify Payments as your primary gateway in Admin → Settings → Payments. If you have a specific reason to use a third-party processor (e.g., lower rates, existing merchant agreement, specific banking relationship), note that in some regions you can run Shopify Payments alongside additional payment methods — this preserves the express buttons while still accepting cards through your preferred processor.

Failure Mode 2: Theme Toggle "Show Dynamic Checkout Button" Is Disabled

Every Shopify theme has a setting that controls whether dynamic checkout buttons render on product pages. It's easy to accidentally toggle off during theme customization, and it's sometimes disabled by default in newer themes that prioritize a single Add to Cart call-to-action.

How to find and re-enable it

  1. Go to Online Store → Themes → Customize
  2. Navigate to a product page (click "Products" in the page selector)
  3. In the left sidebar, look for the "Product information" section or the "Buy buttons" / "Product form" section
  4. Look for a toggle labeled "Show dynamic checkout button," "Enable dynamic checkout button," or "Show payment buttons"
  5. Enable it → Save
This Setting Is Theme-Specific and Resets on Theme Change
When you switch to a new theme or duplicate your theme, this toggle returns to the new theme's default state. Always verify it after any theme change. Some themes (particularly minimal or conversion-optimized themes) disable it by default because some merchants prefer a single, focused Add to Cart button without multiple express options competing for attention.

Developer note

In custom themes, the dynamic checkout button is rendered by the {% form 'product' %} block combined with {{ form | payment_button }} in the product form template. If your custom theme omits {{ form | payment_button }} or places it outside the product form context, the buttons won't render regardless of any theme settings.

Failure Mode 3: Subscription / Selling Plan Product Hides All Dynamic Checkout Buttons Simultaneously

This is one of the most surprising behaviors for merchants who add subscription options to their products. The moment a selling plan is attached to a product — whether via native Shopify subscriptions, Recharge, Bold Subscriptions, Seal Subscriptions, PayWhirl, or any other subscription app — all four dynamic checkout buttons disappear from that product page simultaneously.

Why all four disappear at once

Subscription purchases require the customer to explicitly acknowledge and agree to recurring billing terms. Express checkout flows (Shop Pay one-tap, Apple Pay, Google Pay, PayPal Express) present a condensed payment experience with no mechanism to display recurring billing agreements or selling plan terms. Shopify's checkout system cannot obtain the required subscription consent within an express flow, so it removes all express options for any product that has a selling plan configured — even if the customer is trying to select the one-time purchase option.

This Affects the Entire Product, Including One-Time Purchase Options
If your product offers both "Subscribe & Save" and "One-time purchase" options, the presence of the selling plan configuration hides all dynamic checkout buttons — even for customers who select the one-time purchase option. There is no Shopify platform mechanism to show express buttons only when the one-time option is selected. Customers must always use the standard Add to Cart → checkout flow for products with selling plans.

What merchants typically experience

A merchant installs a subscription app or configures native Shopify subscriptions on a product. Within minutes, they notice the Shop Pay, Apple Pay, and Google Pay buttons are gone from that product page. The Add to Cart button works normally. Other products without selling plans still show the express buttons. The subscription app didn't "break" the buy now buttons — Shopify intentionally hides them as a side effect of the selling plan configuration.

Fix

There is no workaround that restores express buttons on subscription-capable products. The accepted UX approach is to communicate this clearly to customers — consider adding a brief note near the product options explaining that subscription customers complete checkout via the cart. Some merchants use A/B testing to evaluate whether removing the express buttons (by offering subscriptions) net increases or decreases revenue, since the subscription lifetime value may more than compensate for any express checkout conversion reduction.

Failure Mode 4: Buy Now Bypasses Product Form — Line Item Properties and Custom Options Lost

This is the most operationally damaging failure mode because it doesn't produce any visible error — it silently strips customization data from orders. A customer fills out a personalization field, selects a custom option, or enters an engraving request on the product page, then clicks Buy Now. The order arrives at the merchant's fulfillment team with no trace of any customization.

Why the Buy Now path skips form data

The Add to Cart button submits the product page HTML form (<form action="/cart/add">). This form includes all input fields: the variant selector, quantity, and any additional properties from <input name="properties[Field Name]" /> fields. When the form submits, all fields — including custom properties — are sent to Shopify's cart API, which stores them as line item properties on the cart line item.

The Buy Now / dynamic checkout button does not submit this form. It creates a checkout session directly using the variant ID, quantity, and nothing else. The product form's property inputs are never read.

Data Source Via Add to Cart Via Buy Now / Dynamic Checkout Button
Selected variant Transmitted Transmitted
Quantity Transmitted Transmitted
Line item properties (properties[Name]) Transmitted Lost — form not submitted
Infinite Options / SC Product Options custom fields Transmitted Lost
Gift message / engraving / personalization fields Transmitted Lost
Required acknowledgment checkboxes Checked before cart add Never shown or validated
Selling plan / subscription option selection Transmitted (and shows express buttons) N/A — express buttons hidden when selling plans exist

Fix options

Option 1 (simplest): Disable dynamic checkout buttons on products that require customization. In Online Store → Themes → Customize → Product page, turn off the dynamic checkout button setting. Or, in a custom theme, conditionally hide the {{ form | payment_button }} output for products with required properties using a Liquid check.

Option 2: Use a custom "Buy Now" button that reads form data. Build a JavaScript button that collects all property field values from the page before creating the cart item via the Cart API, then redirects to checkout with the cart's checkoutUrl. This gives you a "Buy Now" flow that respects line item properties.

// Custom Buy Now that preserves line item properties
document.getElementById('custom-buy-now').addEventListener('click', async () => {
  const form = document.querySelector('form[action="/cart/add"]');
  const formData = new FormData(form);

  // Validate required fields before proceeding
  const properties = {};
  form.querySelectorAll('[name^="properties"]').forEach(input => {
    const key = input.name.match(/properties\[(.+)\]/)?.[1];
    if (key) properties[key] = input.value;
  });

  const response = await fetch('/cart/add.js', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      id: formData.get('id'),
      quantity: parseInt(formData.get('quantity')) || 1,
      properties
    })
  });

  if (response.ok) {
    window.location.href = '/checkout';
  }
});

Failure Mode 5: Buy Now Bypasses Cart Page — Discounts, Upsells, Order Notes All Skipped

Beyond losing line item properties, the Buy Now path skips your entire cart page. This has significant revenue and operational implications that many merchants don't fully account for when evaluating where to place the express button.

What gets skipped on the cart page

  • Discount code field — Most Shopify themes place the discount code input on the cart page. Customers arriving via Buy Now reach checkout directly, where a separate discount field may or may not be present depending on theme and checkout configuration.
  • Order notes — The order note / special instructions textarea is on the cart page. Buy Now customers have no opportunity to add it. (This is covered in detail in the Order Notes Not Saving guide.)
  • Cart upsell and cross-sell apps — Apps that show frequently-bought-together recommendations, cart drawer upsells, volume discount prompts, and free gift threshold notifications all operate on the cart page. Buy Now users skip all of them.
  • Gift wrapping apps — Most gift wrap apps inject their widget on the cart page. Buy Now bypasses it. (See Gift Wrapping Not Working.)
  • Loyalty points redemption — Loyalty apps (Smile.io, LoyaltyLion, Yotpo) typically allow customers to redeem points on the cart page. Buy Now users go directly to checkout, which may or may not support redemption natively.
  • Cart-level custom fields and attributes — Any custom fields targeting cart.attributes via the cart page are bypassed.
Express Checkout AOV Impact
Studies consistently show that express checkout (Buy Now) users have lower average order values than cart-flow users, partly because upsell and cross-sell opportunities on the cart page are bypassed. Consider whether placing the Buy Now button prominently is the right call for your store's specific upsell strategy and product mix.

Failure Mode 6: Apple Pay Appears Only in Safari on Apple Devices — Five Conditions Required

Apple Pay is the most browser-and-device-restricted of Shopify's express payment options. Merchants frequently test on the wrong browser, conclude it's broken, and spend time troubleshooting what is actually working correctly in the one environment Apple supports.

The five conditions that must all be true simultaneously

  1. Safari browser — Apple Pay is only available in Safari. Chrome, Firefox, Edge, Brave, and Opera never show it — even on a Mac running macOS. Opening Safari and testing is the first diagnostic step.
  2. Apple device — macOS, iOS, or iPadOS only. Windows machines running Safari via third-party tools, Android devices, and Linux systems never show Apple Pay on Shopify product pages.
  3. Card in Apple Wallet — The device must have at least one payment card configured in Apple Wallet (or saved in Safari's autofill from Apple Pay). A Mac without Touch ID/Face ID must be paired with an Apple Watch or iPhone that has a card set up.
  4. Shopify Payments active — As covered in Failure Mode 1, Shopify Payments must be the active gateway. If suspended or replaced by a third-party gateway, Apple Pay is removed.
  5. Domain verification — Shopify automatically handles Apple Pay domain verification for *.myshopify.com domains. For custom storefront domains, Shopify handles verification at checkout.shopify.com (the checkout domain). However, for storefronts using a custom checkout domain or headless setup, the domain verification file (/.well-known/apple-developer-merchantid-domain-association) may need to be explicitly served. Mixed HTTP/HTTPS content also blocks Apple Pay silently.
Fastest Diagnostic for Apple Pay
Open your store's product page in Safari on an iPhone with a card in Apple Wallet. If Apple Pay shows, it's working — the issue is just browser/device. If it doesn't show in Safari on iPhone with a card, check Shopify Payments status in Admin → Settings → Payments. If Shopify Payments shows as Active and Apple Pay still doesn't appear, check Admin → Settings → Payments → Apple Pay section for any domain verification errors.

Country availability

Apple Pay via Shopify is available in the same countries where Shopify Payments is available. Merchants in unsupported countries cannot offer Apple Pay through Shopify regardless of device or browser configuration.

Failure Mode 7: Shopify Buy Button Channel Is a Completely Different Product

A significant portion of "Shopify buy now button not working" searches come from merchants who want to embed a purchasable product button on an external website — a blog, a landing page built on another platform, a marketing microsite, or a Squarespace/Wix/Webflow page. This is the Shopify Buy Button channel, and it's entirely separate from the product page dynamic checkout buttons.

What the Buy Button channel is

The Buy Button is a Shopify sales channel (enabled via Admin → Sales channels → Buy Button) that generates embed code for specific products or collections. The embed code renders a customizable product card with Add to Cart and/or Buy Now functionality on any external website that accepts third-party JavaScript.

Common failures specific to the Buy Button channel

Failure Cause Fix
Button not loading on external site Content Security Policy (CSP) on the hosting platform blocks sdks.shopifycdn.com Add sdks.shopifycdn.com and *.shopify.com to the platform's CSP script-src allowlist
Button renders but cart doesn't open JavaScript conflict with the hosting platform's JS (event listener collision, Promise polyfill conflict) Load Buy Button script in isolation; check browser console for JS errors
Checkout opens but closes immediately Pop-up blocker treating the checkout modal as a popup Configure moneyFormat and ensure the checkout URL opens in the same tab rather than a modal where needed
Button shows old product price or sold-out variant Embed code is cached; Buy Button embed doesn't auto-update on product changes Regenerate embed code in Admin → Buy Button channel → select product → copy new embed
Ad blockers hide the button for visitors Buy Button JS loads from sdks.shopifycdn.com which some blocklists flag Server-side render the product data and use the Storefront API instead of the Buy Button embed for critical external placements
Buy Button vs Product Page Dynamic Checkout Buttons — Key Difference
Product page dynamic checkout buttons — appear on your Shopify storefront's product pages automatically when Shopify Payments is active. Rendered by Shopify's theme engine. No configuration beyond the theme toggle and Shopify Payments setup.

Buy Button channel — requires explicit setup in Admin → Sales channels → Buy Button. Generates JavaScript embed code. Used on external sites outside your Shopify storefront. Completely separate product with its own configuration, customization options, and failure modes.

Failure Mode 8: Headless Storefront — Buy Now Requires Explicit Storefront API Implementation

In a custom or headless storefront (Shopify Hydrogen, Next.js Commerce, custom React/Vue/Svelte), there are no automatic dynamic checkout buttons. The Shopify theme engine that renders {{ form | payment_button }} doesn't exist. Every express checkout button must be implemented from scratch using the Storefront API and the appropriate payment provider SDKs.

What a complete headless Buy Now implementation requires

Express Option What You Must Implement Common Mistake
Basic "Buy Now" (direct checkout) Call cartCreate mutation with variant ID, get cart.checkoutUrl, redirect Using /cart/add.js endpoint (works for cart but doesn't return a checkout URL automatically)
Shop Pay button Call shopPayPaymentRequestSessionCreate mutation to get session token, then render the Shop Pay button component with the token Rendering the Shop Pay button UI without calling shopPayPaymentRequestSessionCreate — button appears but nothing happens on click
Apple Pay (web) Implement Apple Pay JS API: create an ApplePaySession, handle validatemerchant event with a server-side merchant session request to Apple, handle paymentauthorized event Rendering an Apple Pay button image without the Apple Pay JS session — button appears but clicking throws an error
Google Pay Implement Google Pay's PaymentRequest API or load the Google Pay JavaScript library, configure isReadyToPay, handle payment authorization Displaying a Google Pay button without calling isReadyToPay() first — button shows for customers who don't have Google Pay set up

The most common headless Buy Now mistake

Importing a Shop Pay or Apple Pay button component from a UI library and rendering it on the product page — without wiring the payment session creation API call behind it. The button renders visually (it may even have hover states and look completely functional) but clicking it either does nothing, throws a console error, or shows a generic payment failure. The session token that connects the button to an actual payment request was never created.

// Hydrogen / headless: correct Shop Pay Buy Now implementation
import { useCart } from '@shopify/hydrogen';

async function handleShopPayBuyNow(variantId) {
  // Step 1: Create a cart with the item
  const cart = await cartCreate({
    lines: [{ merchandiseId: variantId, quantity: 1 }]
  });

  // Step 2: Create a Shop Pay payment session (REQUIRED before rendering button)
  const { shopPayPaymentRequestSession } = await storefront.mutate(
    SHOP_PAY_PAYMENT_REQUEST_SESSION_CREATE,
    {
      variables: {
        variantLines: [{ merchandiseId: variantId, quantity: 1 }],
        payment: { presentmentCurrencyCode: 'USD' }
      }
    }
  );

  // Step 3: Use the token to render/activate the Shop Pay button
  // The token from shopPayPaymentRequestSession.token connects the
  // rendered button to the actual payment request
  return shopPayPaymentRequestSession.token;
}
Hydrogen Handles This For You
If you're using Shopify Hydrogen (the official React-based headless framework), the ShopPayButton component in @shopify/hydrogen handles the session creation internally. Prefer the official component over building from scratch. For custom frameworks not using Hydrogen, the Storefront API shopPayPaymentRequestSessionCreate mutation documentation provides the implementation reference.
