Shopify Thank You Page Not Working — Order Status Page Blank, Post-Purchase Extensions Missing, additional_scripts Deprecated, Guest Links Expired

Shopify's "thank you page" (the order status page at /thank_you) cannot be customized via the theme editor on non-Plus plans, all content injected via checkout.liquid and additional_scripts stopped executing after June 2025, post-purchase upsell extensions silently skip after Shop Pay, and guest customer order links expire. Here are all 8 failure modes.

📅 Updated June 2026 ⏱ 10 min read 💡 Applies to: All Shopify plans
The Thank You Page Cannot Be Edited in the Theme Editor (Non-Plus)
The most common misconception: merchants look for a "thank_you" or "order-status" template in their theme editor to customize the post-checkout page. This template does not exist in the online store theme editor on non-Plus plans. The order status page is a Shopify-hosted page controlled entirely by Shopify. Only logo, colors, and a few settings in Admin → Settings → Checkout can be changed.

Shopify's Post-Checkout Flow — What Actually Happens

Understanding the sequence helps diagnose which component is failing:

Step Page / System Who Controls It Customizable?
1. Payment submitted Shopify checkout (hosted) Shopify Plus only via Checkout UI Extensions
2. Post-purchase offer Post-purchase extension slot Shopify (app fills it) Yes — via post-purchase extension apps
3. Order status page /thank_you (order status) Shopify Non-Plus: logo/colors only
4. Confirmation email Shopify email system Shopify (merchant templates) Yes — via Admin → Settings → Notifications

The "thank you page" experience is actually three distinct systems: the post-purchase extension slot (step 2), the order status page (step 3), and the confirmation email (step 4). Problems in any of these three often get reported as "thank you page not working."

Failure Mode 1 — The Page Cannot Be Edited Without Plus (The #1 Misconception)

On standard Shopify plans, the order status page at /thank_you is a fully Shopify-controlled page. There is no Liquid template, no theme editor section, and no drag-and-drop block for it. Merchants who see customized thank you pages on other stores are either on Shopify Plus (with Checkout UI Extensions) or using a post-purchase app that inserts an offer page before the order status page.

What non-Plus merchants CAN change

  • Logo: Admin → Settings → Checkout → Logo (also applies to checkout pages)
  • Background and accent colors: Admin → Settings → Checkout → Customize
  • Custom message at top of order status page: Admin → Settings → Checkout → Order processing → "Additional content" text field (plain text, no HTML)
  • Contact information: Shown automatically from Admin → Settings → General

What requires Plus

  • Custom HTML or rich content blocks on the order status page → Checkout UI Extension
  • Custom layout changes → Checkout UI Extension
  • Native post-purchase offer in the Shopify UI slot → Checkout Extensibility
Non-Plus Alternative: Post-Purchase App
Apps like ReConvert, CartHook, and Zipify One Click Upsell inject a branded offer page between payment confirmation and the order status page, using Shopify's post-purchase extension API. These apps work on non-Plus plans and can display custom content, cross-sells, upsells, and loyalty prompts. The customer sees the offer before reaching the standard order status page.

Failure Mode 2 — checkout.liquid and additional_scripts Deprecated June 30, 2025

Shopify deprecated two mechanisms that merchants used to inject custom code into the checkout and order status page:

  • checkout.liquid — A Liquid template file (previously available to some merchants) that allowed custom HTML to be rendered in the checkout flow. Deprecated and no longer executed.
  • Additional scripts (Admin → Settings → Checkout → Order processing) — A text field where merchants could paste custom JavaScript or HTML snippets to run on the checkout confirmation page. Deprecated and no longer executed after June 30, 2025.

Both mechanisms silently stopped working. The order status page still loads and shows the customer's order details, but the injected code does not run. There is no error message shown to the merchant or customer.

What was commonly injected via these paths (and is now broken)

What Was Injected Impact of Deprecation Replacement Path
Google Analytics 4 purchase event (gtag.js) GA4 conversion tracking stopped recording purchases Google & YouTube app → Web Pixel extension
Meta Pixel Purchase event (fbq('track', 'Purchase')) Meta purchase events missing, ROAS measurement broken Meta Pixel app → Web Pixel extension
Survey widgets (Gorgias, Typeform) Post-purchase survey not displaying Checkout UI Extension (Plus) or survey app using post-purchase extension slot
Cross-sell or upsell scripts Custom offer content no longer shown Post-purchase extension app (ReConvert, CartHook, Zipify)

Affiliate tracking pixels Affiliate commissions not tracked Web Pixels API or server-side tracking via Order webhooks
Revenue Attribution Data Has Been Lost Since June 2025
Merchants who haven't migrated their tracking will have had zero purchase-event data in Google Analytics and Meta since June 30, 2025. This means all paid ad campaign ROAS calculations since that date are based on zero conversion data. If your Meta or Google Ads campaigns started spending more without clear results around mid-2025, broken conversion tracking from this deprecation is a probable cause.
  1. Go to Admin → Settings → Checkout → Order processing — if you see any code in "Additional scripts," it is no longer executing. Note what it was doing.
  2. For Google Analytics: install the "Google & YouTube" app from the Shopify App Store → connect your GA4 property → the app creates a Web Pixel extension that fires purchase events
  3. For Meta Pixel: install the "Facebook & Instagram" (Meta) app → connect your Meta Pixel → the app creates a Web Pixel extension
  4. For other tracking (TikTok, Pinterest, etc.): install the official app from each platform or configure a Customer Data Platform (Elevar, Triple Whale) that uses Web Pixels API
  5. Verify the migration is working: Admin → Settings → Customer events → check that active web pixel extensions are listed and healthy

Failure Mode 3 — Post-Purchase Extension Not Showing (ReConvert, Zipify, CartHook)

Post-purchase UI extension apps inject an offer page between checkout completion and the order status page. Merchants who've installed these apps and configured their offers frequently report the offer not showing for some or all customers.

Extension is deployed but not activated in checkout customizations

The most common cause: the app is installed and offers are configured, but the extension has not been activated in Shopify's checkout customization settings.

  1. Go to Admin → Settings → Checkout → Customize checkout
  2. In the checkout editor, look for "Post-purchase page" in the left sidebar or top navigation
  3. Verify the app's extension is shown and toggled on — it must be explicitly enabled per checkout profile
  4. Save and test with a real or test transaction (not a simulated checkout — the post-purchase slot only fires after actual payment submission)
Post-Purchase Slot Does Not Fire on Simulated / Draft Order Checkouts
Testing by placing a free ($0) test order or simulating a checkout without real payment submission will not trigger the post-purchase extension slot. You must complete an actual paid transaction (use a test card in a development store, or Shopify's Bogus Gateway) to verify the extension renders.

Offer eligibility rules excluding the order

Most post-purchase apps have configurable eligibility rules: minimum order value, specific products or collections, customer tags, first-time-buyer-only, etc. If the test order doesn't meet these conditions, no offer will show. Check the app's offer configuration for active exclusion rules.

Extension returned an error

If the post-purchase extension app's server is slow or returns an error when Shopify calls it to determine whether to show an offer, Shopify silently skips the extension and sends the customer directly to the order status page. This manifests as the extension working sometimes but not others. Check the app's dashboard for error logs or failed webhook deliveries.

Failure Mode 4 — Post-Purchase Extension Skipped After Shop Pay / Express Checkout

When customers check out via Shop Pay, Apple Pay, Google Pay, or another accelerated checkout flow, the post-purchase extension slot may not trigger or may behave differently from standard checkout.

Shop Pay's post-payment flow

After a Shop Pay purchase is completed, the customer sees Shop Pay's own confirmation screen ("Your order is confirmed"). Shop Pay then redirects to your store's order status page. The redirect happens after Shop Pay's confirmation is shown, meaning:

  • The Shopify post-purchase extension slot may render between the Shop Pay confirmation and the order status page, but the timing depends on the specific Shop Pay flow version
  • Some merchants report their post-purchase app offers showing after Shop Pay on desktop but not mobile, or vice versa
  • Shop Pay Installments has a different post-payment confirmation flow that may bypass the extension slot entirely

Apple Pay and Google Pay

Express checkout via Apple Pay or Google Pay (when initiated from product pages or cart via the payment request API) may handle the post-payment redirect differently. The key test: complete a purchase using each express payment method specifically, not just standard checkout. Many merchants test with a credit card and assume all payment methods behave identically.

Test Post-Purchase Apps With Every Active Payment Method
Create separate test transactions using: (1) standard credit card entry, (2) Shop Pay, (3) Apple Pay (on Safari/iOS), (4) Google Pay (on Chrome). Verify the post-purchase offer appears for each. If it appears for standard card but not Shop Pay, the issue is the express checkout flow, and you should contact the post-purchase app's support with this specific finding.

Failure Mode 5 — Guest Customer Order Status Link Expired

Guest customers who check out without creating an account cannot log in to view their order. Instead, Shopify embeds a unique, time-limited authenticated URL in their confirmation email:

yourstore.myshopify.com/orders/{order-id}/authenticate?key={unique-key}

This link provides one-time access to the order status page without a login. The link is time-limited and expires after a set period. After expiry:

  • Clicking the link shows "This page is no longer available" or redirects to the store homepage
  • The guest has no other self-service way to view their order status, tracking, or return eligibility
  • The guest cannot access their order history through the customer account portal because they never created an account

Merchant impact

Customers who received the order status link in their confirmation email, bookmarked it, and return weeks later to check on a return or warranty claim find the link broken. This generates support tickets and frustration that could have been prevented.

Solutions for non-Plus stores

  1. Enable "New customer accounts" (Admin → Settings → Customer accounts → New customer accounts) — this allows guests to verify their email and access order history without setting a password
  2. Update the order confirmation email template (Admin → Settings → Notifications → Order confirmed) to include a prominent "Create an account" CTA so guests convert to accounts before the link expires
  3. Configure your customer support workflow to look up orders by email address for guests who contact you after the link expires
  4. For guests who have multiple orders, note that each order has a separate unique link — there is no single guest "order history" URL
New Customer Accounts Changed How Guest Order Access Works
With Shopify's "New customer accounts" (the default for stores created after 2024), guests can enter their email on the login page and receive a 6-digit code to access their order history. This is more durable than the expiring link. If your store still uses "Classic customer accounts," the expiring link is the only self-service option for guests.

Failure Mode 6 — Order Confirmation Email Not Sent vs. Thank You Page Not Loading

Merchants frequently conflate two separate issues:

  • The thank you page didn't load — The customer is stuck at the payment step or sees an error; no order status page is shown
  • The order confirmation email didn't arrive — The thank you page showed correctly, but the email never arrived in the customer's inbox

The thank you page is served by Shopify as soon as payment is authorized. The confirmation email is queued separately by Shopify's email system and sent after the order is created. If the thank you page shows but no email arrives, the issue is in email delivery, not the page itself.

Common email non-delivery reasons

  • Email in spam / junk folder (especially common for Gmail, Apple Mail with aggressive filtering)
  • Customer's email provider soft-blocked Shopify's sending domain — check if the customer's domain has strict DMARC policies
  • Customer entered their email incorrectly at checkout (common with autocomplete filling the wrong address)
  • Shopify's notification for "Order confirmed" is disabled — Admin → Settings → Notifications → Order confirmed → verify it's enabled and the email template is not blank
  • Store has a custom sender domain configured that's missing SPF/DKIM records — email delivered to spam or rejected
  1. Find the order in Admin → Orders → check that the order exists (if the thank you page showed, the order is there)
  2. In the order detail, check the Timeline section for "Order confirmation email sent" — if it shows "Sent" but the customer didn't receive it, the issue is inbox delivery, not Shopify's sending
  3. Use "Resend order confirmation" from the order detail page to retry the email
  4. Ask the customer to check spam and search for "shopify" in their email client
  5. If this is happening for many customers, check Admin → Settings → Notifications → your sending domain → verify SPF and DKIM are correctly configured

Failure Mode 7 — Custom Redirect From Thank You Page Blocked

Some merchants attempt to redirect customers from the order status page to a custom landing page, loyalty sign-up, or review collection page using JavaScript. These redirects are blocked or overridden by Shopify.

Why redirects don't work

Shopify's order status page is hosted on Shopify's checkout domain (checkout.shopify.com or your custom checkout domain), not on your store's primary domain. The page uses a strict Content Security Policy that prevents unauthorized external script execution. Additionally, Shopify's checkout scripts monitor for navigation attempts and can override them to prevent customers from being redirected away from order confirmation before they've seen it.

Legitimate alternatives to redirects

  • Custom message in order confirmation email: Add a CTA button in the order confirmation email template that links to your loyalty page, review request form, or upsell collection
  • Post-purchase extension (for upsells): Use a post-purchase app to show your offer between payment and the order status page — no redirect needed
  • Thank you page Checkout UI Extension (Plus only): Add custom content blocks to the order status page itself using Checkout UI Extensions
  • Shopify Flow + email automation: Trigger an automated email 30-60 minutes after order placement asking for a review or redirecting to loyalty enrollment
The Confirmation Email Is More Effective Than a Thank You Page Redirect
A thank you page redirect competes with customers who immediately close the tab. The confirmation email arrives in the customer's inbox when they're ready to engage — often 1-2 hours later when they're back on their phone. Review requests and loyalty program CTAs in the order confirmation email consistently outperform thank you page redirects.

Failure Mode 8 — Headless Storefronts Need a Custom Order Confirmation Page

Merchants building headless storefronts (using Shopify's Storefront API with a custom React, Next.js, or Hydrogen frontend) must decide how to handle post-checkout flow:

Scenario A: Redirect to Shopify's hosted checkout

The most common headless approach: the storefront builds a cart using the Storefront API, then redirects the customer to Shopify's hosted checkout (checkout.shopify.com). After payment, Shopify redirects the customer to Shopify's standard order status page at yourstore.myshopify.com/thank_you. This works automatically and the customer gets the standard order status page. Post-purchase apps using the extension slot will fire normally in this flow.

Scenario B: Fully custom checkout

Merchants implementing a fully custom checkout (collecting payment details in their own frontend using the Storefront API's cartPaymentUpdate mutation or a custom payment gateway) do not automatically receive a Shopify-generated order status page. They must:

  • Build their own order confirmation page (e.g., /order-confirmation?orderId={id})
  • After payment completes, use the Storefront API to fetch order details for the confirmation page
  • Manually trigger conversion tracking events from their custom page using their tracking setup
  • Generate and display the order details, tracking information, and next steps themselves
Shopify's Checkout Extensibility Is Not Available for Fully Custom Checkouts
Checkout UI Extensions, post-purchase extensions, and Web Pixel extensions only work within Shopify's hosted checkout. If your storefront bypasses Shopify's checkout entirely, none of these extension mechanisms are available. You must implement equivalent functionality in your custom frontend code.

Hydrogen-specific considerations

Shopify's Hydrogen framework (built for headless storefronts) provides built-in utilities for redirecting to Shopify's hosted checkout. Hydrogen storefronts that use the default checkout redirect will get the standard order status page automatically. Custom Hydrogen implementations that attempt to build a checkout within the Hydrogen app must handle all of the above manually.
