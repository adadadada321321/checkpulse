Shopify Checkout Extensibility Migration Not Working — 8 Causes Explained

checkout.liquid was retired June 30 2025. Custom fields, CSS, tracking scripts, and app-injected UI silently disappeared. Here are all 8 root causes of Checkout Extensibility migration failures — for merchants, developers, and headless teams.

Updated June 2026 8 failure modes checkout.liquid Removed · Plus-Only Gate · Customizations Template · App Extensions · Additional Scripts · Extension Targets · CSS Migration · Headless

Shopify's Checkout Extensibility upgrade retired the checkout.liquid template file permanently on June 30, 2025. Stores that had customized checkout using this file — custom fields, CSS overrides, injected scripts, third-party app UI — found those elements silently gone after the sunset date. Checkout continued to function normally (orders still processed), but all visual and functional customizations disappeared without error or warning.

The replacement system is Checkout Extensibility: a combination of Checkout UI Extensions (Plus merchants only), the Checkout Customizations editor in the theme editor, and the Web Pixels API for tracking. Each piece has its own activation requirements, and failures in any one of them look identical from the outside — a blank or vanilla checkout with no customizations.

checkout.liquid Is Gone — No Rollback

Shopify removed checkout.liquid access permanently. You cannot re-enable it, roll back to it, or use it on any plan. If you're editing a file called checkout.liquid in your theme and expecting changes to appear at checkout, they will not. The file is ignored. The only migration path is Checkout Extensibility (Plus) or working within the structured Checkout Settings panel (all plans).

Failure Mode 1 — checkout.liquid was silently retired and customizations stopped rendering

On June 30, 2025, Shopify stopped reading checkout.liquid for stores that had been upgraded to Checkout Extensibility. The file wasn't deleted from the theme — it still appears in the code editor — but Shopify ignores it entirely. Any Liquid code, CSS, or HTML in that file stopped rendering at checkout.

Why it's hard to notice: checkout still works. Orders process normally. Customers can complete purchases. The only visible change is that customizations are gone — custom fields, brand colors applied via Liquid, injected app widgets, promotional banners, custom fonts. Many merchants didn't realize until a customer asked why checkout looked different or a custom required field disappeared.

What was lost for different store types:

  • Basic customization stores: Custom CSS (fonts, colors, button radius), promotional banners above the checkout form, custom header/footer content
  • App-heavy stores: Third-party app UI injected via the app's checkout.liquid override (warranty widgets, upsell offers, loyalty point displays, custom fields, age verification forms)
  • Developer-built stores: Custom Liquid logic, dynamic field rendering, conditional content based on cart contents or customer tags
  • Tracking: Google Analytics, Meta Pixel, TikTok Pixel, and affiliate tracking scripts placed directly in checkout.liquid
How to Confirm You're Affected

Go to your theme code editor (Online Store → Themes → Actions → Edit code). Look for checkout.liquid. If it exists and has content, you were using it — and that content is no longer rendering. The file is now dead weight. Your migration must use the tools described in this guide.

Failure Mode 2 — Checkout Extensibility is gated behind Shopify Plus

The replacement for checkout.liquid — Checkout UI Extensions — is available exclusively to Shopify Plus merchants. This affects:

  • Custom fields and UI blocks inside checkout (contact, shipping, payment steps)
  • Post-purchase UI extensions on the order status page
  • Installing third-party app checkout extensions (age verification, upsell widgets, custom fields)
  • Advanced branding customization via the Design extension
  • The Checkout Customizations editor (you can view it but not add extension blocks)

What non-Plus merchants can do:

  • Checkout Settings branding panel: Admin → Settings → Checkout → customize logo, accent color, button color, background color, and font from Shopify's limited set. This applies to all plans.
  • Cart page customization: The cart page (/cart) is a Liquid template fully editable on all plans. Use it for order notes, gift messages, and pre-checkout custom fields — but note that express checkout (Shop Pay, Apple Pay, Buy Now button) bypasses the cart page.
  • Web Pixels API: Tracking scripts (GA4, Meta Pixel) via the Web Pixels API are available to all plans through the official Google & YouTube and Meta Pixel apps.
Upgrade Path

If your store requires custom checkout fields (e.g., delivery date picker, gift message at checkout, age confirmation checkbox, loyalty point display), the only path is upgrading to Shopify Plus. Shopify Plus pricing starts at $2,500/month (or $2,300/month annual). If you need one custom field and you're on Basic/Shopify/Advanced, the cart page is your only option — with the caveat that express checkout bypasses it.

Failure Mode 3 — Checkout Customizations template not configured in theme editor

After upgrading to Checkout Extensibility, your checkout customizations are controlled via a new template in the Online Store theme editor — separate from the rest of your theme. This template is called "Checkout and thank you" and it lives under the page type dropdown in the theme customizer.

The most common mistake: merchants install a checkout app (or enable a Checkout UI Extension) and expect the extension to appear automatically. It doesn't. The extension block must be manually added to the Checkout Customizations template.

  1. Online Store → Themes → Customize
  2. In the page picker dropdown at the top (which says "Home page" by default), select "Checkout and thank you"
  3. You'll see sections for "Information", "Shipping", "Payment", "Thank you", and "Order status"
  4. To add an app extension: click "Add block" in the relevant section, then find your app's extension in the Apps tab
  5. Drag the extension block to the position you want within that checkout step
  6. Save the template
Theme Switch Resets App Blocks

When you switch to a new theme, the Checkout Customizations template resets to default. All app extension blocks that were added must be re-added manually in the new theme's customizer. Apps are not automatically re-added. This is a common source of "my checkout app stopped working after the theme update" tickets.

Additional causes of missing customizations template configuration:

  • The store was migrated from one Shopify account to another and the theme's checkout template was not transferred with the same extension configuration
  • A developer published a new theme version that didn't include the checkout customization blocks from the staging version
  • The checkout template was accidentally reset by publishing a different theme over the current one

Failure Mode 4 — Third-party apps not updated to publish a Checkout UI Extension

Apps that previously injected UI into checkout by overriding checkout.liquid must now publish a Checkout UI Extension — a React-based component that Shopify renders inside the checkout sandbox. Not all apps have done this migration. Many older apps still list "checkout customization" as a feature but haven't published an updated extension version.

Symptoms:

  • Installed a checkout app (age verification, warranty upsell, custom field, loyalty display) but nothing appears in checkout
  • The app's settings page in Admin says "active" or "enabled" but checkout looks unchanged
  • The app block doesn't appear in the Checkout Customizations editor (it would show under "Apps" if the extension was published)

How to diagnose:

  1. Go to Online Store → Themes → Customize → Checkout and thank you page
  2. Click "Add block" in the section where the app should appear
  3. Check the "Apps" tab — if your app doesn't appear here, it hasn't published a Checkout UI Extension
  4. Contact the app developer and ask if they have a "Checkout Extensibility compatible" version — get the specific app version number
Shopify's Checkout Compatibility Check

Shopify's Partner Dashboard shows a "Checkout Extensibility compatible" badge on apps that have published a Checkout UI Extension. Before installing any checkout-modifying app, check if it has this badge. Apps without it will not work in your checkout after the checkout.liquid migration.

Common apps that had delayed Checkout UI Extension releases (check the app's current version before assuming it works):

  • Age verification apps — many relied on checkout.liquid for the gating logic
  • Custom field apps (Infinite Options, SC Product Options for checkout fields)
  • Upsell apps that placed offers between checkout steps
  • Loyalty and reward point display apps
  • Warranty and protection plan widgets
  • Donation and round-up widgets placed at checkout

Failure Mode 5 — Additional Scripts field deprecated; tracking pixels stopped

The "Additional scripts" field in Admin → Settings → Checkout was the universal escape hatch for tracking. Merchants pasted Google Analytics, Meta Pixel, TikTok Pixel, affiliate tracking, chat widget initialization, and customer survey scripts directly into this field. It ran on the order status (thank-you) page and sometimes on other checkout steps depending on the script.

This field was deprecated alongside checkout.liquid and stopped executing scripts for stores on the Checkout Extensibility upgrade. Scripts in the field are still visible in the Admin UI but silently do nothing.

What broke for most merchants:

  • GA4 purchase event stopped firing — analytics showed zero conversions
  • Meta Pixel Purchase event missing — Facebook/Instagram ads lost conversion attribution
  • TikTok Pixel CompletePayment event gone
  • Affiliate and referral tracking (Impact, ShareASale, CJ) lost the conversion pixel
  • Post-purchase survey tools (Fairing, KnoCommerce) stopped loading

The replacement for tracking scripts:

  • GA4: Install the official Google & YouTube app → it creates a Web Pixel Extension that fires GA4 purchase events correctly from inside the checkout sandbox
  • Meta Pixel: Install the official Meta Pixel app → same mechanism via Web Pixel Extension
  • Custom tracking: Create a custom Web Pixel via Shopify's Partner Dashboard (requires a basic app) — Web Pixels run in the checkout sandbox and have access to standard checkout events including checkout_completed
  • Third-party pixels: If the pixel provider hasn't published a Web Pixel integration, use server-side Conversions API (CAPI / Measurement Protocol) triggered by the orders/create webhook instead
Double-Tracking After Migration

A common post-migration mistake: merchant installs the official Google & YouTube app to get GA4 working again, but leaves the old gtag.js script in their theme's theme.liquid file. Result: GA4 fires page_view events on storefront pages (from theme.liquid) and purchase events from the checkout sandbox (from the Web Pixel). These two instances share a measurement ID but fire from different contexts — GA4 may deduplicate correctly but Meta Pixel and TikTok often show inflated numbers. Audit all theme.liquid tracking scripts after the migration.

Failure Mode 6 — Extension targets don't match your expected placement

Checkout UI Extensions render at specific, predefined positions in checkout called "targets" or "extension points". Unlike checkout.liquid — where you could insert HTML anywhere in the DOM — Checkout UI Extensions must be assigned to one of Shopify's defined positions.

Available standard checkout extension targets (as of 2026):

  • purchase.checkout.contact.render-after — below the email/phone field
  • purchase.checkout.shipping-option-list.render-after — below the shipping rate list
  • purchase.checkout.delivery-address.render-before — above the shipping address form
  • purchase.checkout.payment-method-list.render-after — below payment method selection
  • purchase.checkout.actions.render-before — above the "Pay now" button
  • purchase.checkout.block.render — for app blocks dragged to custom positions in the Customizations editor
  • purchase.thank-you.block.render — on the thank-you page
  • purchase.order-status.block.render — on the order status page

Common placement failures:

  • Developer tries to place a "terms and conditions" checkbox immediately before the payment button — this requires purchase.checkout.actions.render-before which must be declared in the extension's configuration, not just the React component code
  • Extension blocks added to the wrong checkout step in the Customizations editor (e.g., dragging an "Information step" block into the "Payment step" section — it won't appear where expected)
  • Extension expects to render on both the checkout flow and the order status page but only one target is declared in the extension's shopify.extension.toml configuration
Checking Your Extension's Declared Targets

In your extension's shopify.extension.toml file, the [[extensions.targeting]] blocks define where the extension renders. If your extension doesn't appear at a particular checkout step, check that step's target is declared there. You cannot render at a target not listed in your toml config, even if you drag the block to that position in the Customizations editor.

Failure Mode 7 — CSS and branding customizations not migrating correctly

Many merchants used checkout.liquid for detailed CSS customization: custom fonts (loaded via @font-face or Google Fonts), specific button shapes, color palette overrides, spacing adjustments, and brand-specific styling. The Checkout Extensibility branding system does not accept raw CSS — it uses a structured visual editor.

What the Branding panel (Admin → Settings → Checkout → Branding) controls:

  • Logo upload and placement
  • Background color (solid) and optional background image
  • Accent color (applied to buttons and links)
  • Heading font (from Shopify's approved font list)
  • Body font (from Shopify's approved font list)
  • Button corner radius (sharp, slightly rounded, rounded, pill)
  • Field border style
  • Error message color

What the Branding panel cannot do:

  • Custom fonts not in Shopify's approved list — no @font-face declarations
  • Per-element CSS targeting (you can't style only the shipping section differently)
  • Custom animations or transitions
  • Multi-color layouts or gradient backgrounds with full control
  • Custom hover states on buttons
  • Hiding or repositioning native checkout elements

For Plus merchants who need more control: a "Design" Checkout UI Extension can inject custom styles using the useSettings() hook combined with Shopify's Checkout UI kit styling primitives. This is not raw CSS — it's a constrained styling API. Full arbitrary CSS injection is not available even on Plus, by design (PCI compliance isolates checkout from the storefront's CSS context).

Font Workaround for Non-Approved Fonts

If your brand font isn't in Shopify's approved font list, the closest workaround on Plus is a Design extension that uses BlockStack or Text components with inline style overrides for font-family. This has limitations and may not fully match the brand's typography. Most Plus merchants in this situation use the closest available alternative from the approved list or accept the constraint.

Failure Mode 8 — Headless storefronts using Shopify-hosted checkout cannot inject via checkout.liquid either

Headless Shopify storefronts (built with Hydrogen, Next.js + Storefront API, or custom frontends) typically use Shopify's hosted checkout on checkout.shopify.com (or the store's myshopify.com domain) for the actual payment step. Some headless teams assumed they could customize this hosted checkout via checkout.liquid — they could not, even before the retirement. checkout.liquid was exclusively for Plus stores using the standard Shopify checkout.

After the retirement, the headless checkout customization story changed:

  • Still using Shopify hosted checkout: Upgrade to Plus + implement Checkout UI Extensions. The hosted checkout now uses the Extensibility system regardless of whether your storefront is headless or native Liquid.
  • Cart customization before checkout: Headless storefronts control the pre-checkout experience (cart page, product page, mini-cart) entirely. Custom fields, gift messages, and upsells can be collected here and passed as cart attributes or line item properties.
  • Hydrogen/Remix custom checkout: Shopify's Hydrogen framework supports building a fully custom checkout experience using Cart API + Payment Sessions + Shopify Functions. This is a significant engineering investment — it bypasses the hosted checkout entirely. Most headless teams do not use this path unless they have very specific checkout requirements.

Common headless mistakes after the migration:

  • Assuming Checkout UI Extensions work automatically on a headless storefront — they require the hosted Shopify checkout and a Plus plan; they don't inject into custom-built checkout pages
  • Trying to add custom JS to the hosted Shopify checkout from the headless frontend — the checkout runs on a different domain in a PCI-isolated sandbox; JavaScript from the storefront domain cannot reach it
  • Using cart permalink URLs (/cart/[variant-id]:1) and expecting checkout customizations to apply — they do apply to hosted Shopify checkout via Checkout UI Extensions, but only if the store is Plus
The Practical Headless Migration Path

For most headless teams: upgrade to Plus, implement Checkout UI Extensions for checkout customization, and use the Storefront API for everything before checkout. The Cart API's cartAttributesUpdate mutation lets you pass custom data (collected in your headless frontend) into the checkout session where it appears as order attributes — no checkout.liquid required.
