Shopify Cart Drawer Not Working — 8 Causes Explained

Cart drawer not opening? Opens empty after Add to Cart? Order notes disappearing when you change quantity? Two drawers firing at once? Here are all 8 root causes of Shopify cart drawer failures — for merchants, theme developers, and headless teams.

Updated June 2026 8 failure modes Ajax Race Condition · Note Stripping · App Conflicts · Section Rendering API · Inventory Stale State · Headless

Shopify's cart drawer is a JavaScript-powered slide-out panel that is fundamentally different from the traditional /cart page. Apps and customizations that target the cart page often have no effect on the drawer — and vice versa. Most cart drawer problems fall into two categories: JavaScript timing issues (the drawer reads stale cart data because it opens before an AJAX call completes) and app conflicts (two systems both attach event listeners to the same buttons).

The most underdiagnosed issue: cart drawer quantity-change buttons that silently erase the customer's order note and custom attributes by sending an incomplete /cart/update.js request. The customer finishes typing a gift message, updates a quantity, and the note is gone — with no error, no warning, and no way to recover it after checkout.

Silent Failure Pattern

Cart drawer failures rarely show errors. A drawer that opens empty after Add to Cart, shows a spinner that never resolves, or silently strips order notes looks the same to the customer as a working drawer — until they reach checkout and find their cart is wrong. Many merchants only learn about these failures from customer support complaints.

Failure Mode 1 — Cart drawer and /cart page are different systems — apps target the wrong one

The Shopify cart exists in two distinct forms, and code written for one has no effect on the other:

  • The /cart page: A traditional full-page load at yourdomain.com/cart. Rendered server-side from the cart.liquid or main-cart-items.liquid template. Apps that inject into the cart page via Liquid sections or cart.liquid edits only affect this page.
  • The cart drawer: A JavaScript-rendered slide-out panel. Fetches cart HTML via the Section Rendering API or builds the UI from /cart.json data. Rendered entirely client-side on the same page the customer is on.

All modern Shopify themes (Dawn 2021+, Refresh, Sense, Craft, Crave, Studio, Spotlight) include a built-in cart drawer. The /cart page still exists but most customers never navigate to it — the drawer intercepts the cart icon click.

Common apps that target the wrong system:

  • Upsell and cross-sell apps that inject into cart.liquid — their blocks appear on /cart but not in the drawer
  • Discount code field apps that add a field to the /cart template — invisible in the drawer
  • Cart total apps that hook into cart page JavaScript events (cart:updated on /cart) — don't fire when the drawer updates
  • Gift message apps with a textarea in the cart.liquid template — customers using the drawer never see it
How to Check

Navigate directly to /cart in your browser and compare what you see with what appears in the cart drawer. If upsell blocks, discount fields, or custom content appear on /cart but not in the drawer, the app is targeting the wrong template. Check the app's documentation for a "drawer compatibility" setting or a separate App Embed for the drawer.

Failure Mode 2 — Race condition: cart drawer opens before add-to-cart AJAX completes

When a customer clicks "Add to Cart", two events fire almost simultaneously:

  1. An AJAX POST request to /cart/add.json to add the item to the cart
  2. The cart drawer's opening animation begins

If the drawer fetches cart state (/cart.json or a Section Rendering API call) before the add-to-cart response has resolved and updated the server-side cart, the drawer reads the cart before the item was added. The customer sees an empty drawer, or a drawer showing their previous cart without the item they just added.

This race condition is common in themes or apps that trigger the drawer open event at the same time as the add request, rather than waiting for the add response:

// BAD — opens drawer and fetches cart at the same time as the add request
fetch('/cart/add.json', { method: 'POST', body: formData })
openCartDrawer()           // fires immediately, reads stale cart
fetchCartAndRender()       // may complete before or after add

// CORRECT — waits for add response, then renders updated cart
fetch('/cart/add.json', { method: 'POST', body: formData })
  .then(response => response.json())
  .then(item => {
    return fetch('/cart.json').then(r => r.json())
  })
  .then(cart => {
    renderCartDrawer(cart)  // cart is guaranteed to include the new item
    openCartDrawer()
  })
Diagnosing the Race Condition

If the drawer shows empty once and then correctly shows the item if you close and re-open it, the race condition is the cause. The second open reads the cart after the add has already completed. Check your theme's cart.js or cart-drawer.js for the sequencing of the add request and the drawer open event.

Failure Mode 3 — Quantity update strips cart.note and cart.attributes

This is the most underdiagnosed cart drawer failure. When a customer changes quantity or removes an item using the drawer's +/- buttons, the drawer JavaScript typically calls /cart/update.js with something like:

// INCOMPLETE — Shopify interprets missing fields as empty and overwrites them
fetch('/cart/update.js', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    updates: { [variantId]: newQuantity }
  })
})

// CORRECT — preserve existing note and attributes
const currentCart = await fetch('/cart.json').then(r => r.json())
fetch('/cart/update.js', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    updates: { [variantId]: newQuantity },
    note: currentCart.note,          // preserve existing note
    attributes: currentCart.attributes  // preserve existing attributes
  })
})

When note and attributes are omitted from the request body, Shopify replaces them with empty values. The following data is silently lost:

  • Order notes typed by the customer (gift messages, special instructions)
  • Cart attributes set by apps (UTM tracking parameters, affiliate codes, loyalty identifiers)
  • Custom fields injected by cart customization apps (Infinite Options fields stored as cart attributes)
  • Note attributes used for discount tracking by some discount apps
No Warning to the Customer

The customer has no way to know their note was erased. The cart drawer shows their items and quantity correctly. Only at the order confirmation email (or on Admin → Orders) does it become apparent the note is missing. By then, the order has been placed and the gift message is lost permanently.

Failure Mode 4 — Duplicate cart drawer apps conflict with theme's built-in drawer

Dawn, Refresh, Sense, Craft, and most 2021+ Shopify themes include a fully-featured built-in cart drawer. Installing a third-party cart drawer app creates a second drawer, and both compete for the same user interactions.

What happens when two drawers are active:

  • Both open on click: Clicking the cart icon fires event listeners from both the theme and the app, opening both drawers simultaneously or causing JavaScript errors
  • Wrong drawer opens: The third-party app's drawer has higher CSS z-index or its event listener fires first, but the theme drawer also partially opens behind it
  • Cart count out of sync: The theme header's cart count badge reads from a different source than the app's counter — showing different numbers
  • Add-to-cart broken entirely: Conflicting event listeners cause event.preventDefault() calls that block the add-to-cart form submission entirely
Common Third-Party Cart Drawer Apps

Apps that install their own cart drawer: UpCart, Monster Cart Upsell, Cartx, Side Cart Upsell, Slide Cart Drawer, Ajax Cart. Each installs its own JavaScript that manages cart state and the drawer open/close. Use at most one of these, and disable the theme's built-in drawer if you do. Check the app's instructions for "how to disable the theme cart drawer" — usually a setting in theme.liquid or base.css.

Failure Mode 5 — Cart drawer shows stale inventory — customer can't proceed to checkout

Shopify's cart drawer uses optimistic UI — it shows "Added to cart" and updates the quantity immediately without waiting to verify real-time inventory availability. This means customers can add quantities to the drawer that exceed available stock.

The sequence that causes confusion:

  1. Product shows 1 unit in stock
  2. Two customers add to cart simultaneously
  3. Both cart drawers show success for 1 unit
  4. Customer A proceeds to checkout — succeeds
  5. Customer B proceeds to checkout — blocked: "This item is no longer available"
  6. Customer B must return from the checkout page to the store, remove the item from the cart drawer, and start over

Additional stale inventory scenarios:

  • Cart drawer allows +5 quantity for an item with only 2 in stock: Shopify's /cart/add.json does not validate against real-time inventory — it accepts the quantity and only validates at checkout
  • Overselling during flash sales: Many customers add items in the same second; the checkout-time inventory check is the only true gate; cart drawer shows all adds as successful
  • Inventory reserved by another customer's open checkout session: A 10-minute checkout hold exists for items in active checkout sessions, but the cart drawer has no visibility into these holds and allows adds for held inventory
Reducing Inventory Confusion

Display real-time inventory counts in the cart drawer using the quantityAvailable field from the Storefront API (headless) or the product.variants[].inventory_quantity Liquid variable (theme). For high-demand products, consider disabling the cart drawer's optimistic quantity increase beyond a safe maximum and showing "Only X left" messaging when stock is low.

Failure Mode 6 — Section Rendering API request blocked — drawer shows infinite spinner

Modern Shopify cart drawers use the Section Rendering API to fetch updated cart HTML after each change. The request looks like:

fetch('/cart?sections=cart-items,cart-header,cart-footer')
  .then(r => r.json())
  .then(sections => {
    document.querySelector('#cart-items').innerHTML = sections['cart-items']
  })

If this request is blocked or fails, the drawer shows a loading spinner that never resolves. The following scenarios cause the Section Rendering API to fail:

  • Ad blockers and privacy extensions: Tools like uBlock Origin, Privacy Badger, and Ghostery that block Shopify CDN resources (cdn.shopify.com, shopifycloud.com) can interfere with section render requests. Typically affects 10–20% of desktop users.
  • Corporate network firewalls: Enterprise networks that whitelist/blacklist traffic by domain may block Shopify's API endpoints while allowing the main storefront
  • Incorrect section name: If the section ID in the request doesn't match a section defined in the theme (e.g., renamed during theme editing), the response returns the section key but with empty HTML
  • Theme customization breaking section output: A Liquid syntax error in the section template causes the section render response to return an error rather than HTML, leaving the drawer stuck
  • App embedding conflict: An App Embed block with a JavaScript error in a cart-adjacent section can break the section render for all sections in the same request
Testing Section Rendering

Test the Section Rendering API directly: open your browser's developer tools → Network tab, then update cart quantity in the drawer. Look for a request to /cart?sections=.... If it returns a 200 response with valid JSON, the API is working and the issue is elsewhere. If it returns an error or never completes, the section rendering itself is failing — check for Liquid errors in your theme sections.

Failure Mode 7 — Translation app doesn't translate cart drawer content

Third-party translation apps (Weglot, Langify, GTranslate, Translation Lab) translate content by scanning the DOM when the page first loads and replacing text nodes with translated equivalents. Cart drawer content is rendered dynamically by JavaScript — it's not in the DOM at initial page load.

The result: product names, variant options, price labels ("Total", "Subtotal"), button text ("Checkout", "Remove"), and promotional messages in the cart drawer remain in the original store language regardless of the active locale. The rest of the page is translated but the drawer is not.

Why each translation approach fails for cart drawers:

Translation Method Works on Cart Drawer? Reason
Initial DOM scan ✗ No Drawer HTML injected after page load — not present during scan
MutationObserver (dynamic) Partial Works if the translation app's MutationObserver is configured to watch the cart drawer container element — not enabled by default in most apps
Shopify Markets locale-aware content ✓ Yes Liquid translations built into theme locale files are rendered server-side and work in section renders; product names from metafields work if translated in Translate & Adapt
Headless i18n (react-intl, next-intl) ✓ Yes All strings managed by i18n library regardless of when component renders

Fix options:

  • In Weglot: enable "Dynamic Content" translation and add the cart drawer container selector to the dynamic elements list in Weglot dashboard → Configuration → Dynamic elements
  • In Langify: enable "Translate after AJAX calls" and ensure the cart drawer container is included in the AJAX element scope
  • Long-term: migrate cart drawer text strings to Shopify's native locale file system (locales/en.default.json) so they're handled by the theme's built-in translation system instead of a third-party app

Failure Mode 8 — Headless cart drawer requires full Storefront API implementation — no automatic sync

Headless storefronts (Hydrogen, Next.js Commerce, custom React/Vue/Svelte) have no built-in cart drawer. Cart state is managed entirely in JavaScript — typically via React context, Zustand, Jotai, or a dedicated cart provider. There is no AJAX shortcut to /cart/update.js in headless storefronts.

Issue A — Double-click from non-optimistic updates

The most common headless cart drawer issue: quantity-change calls cartLinesUpdate via the Storefront API, but the local cart state isn't updated until the API response returns (~300–800ms). The customer clicks "+" and sees no visual response, then clicks again before the first request completes — sending a duplicate update and ending up with the wrong quantity.

// WRONG — waits for server response before updating UI
const handleQuantityChange = async (lineId, quantity) => {
  const { cartLinesUpdate } = await storefront.mutate(CART_LINES_UPDATE, {
    variables: { cartId, lines: [{ id: lineId, quantity }] }
  })
  setCart(cartLinesUpdate.cart)  // UI updates 300-800ms after click
}

// CORRECT — optimistic update first, then reconcile with server
const handleQuantityChange = async (lineId, quantity) => {
  setCart(prev => optimisticUpdate(prev, lineId, quantity))  // instant UI
  try {
    const { cartLinesUpdate } = await storefront.mutate(CART_LINES_UPDATE, {
      variables: { cartId, lines: [{ id: lineId, quantity }] }
    })
    setCart(cartLinesUpdate.cart)  // reconcile with server state
  } catch (err) {
    setCart(originalCart)  // roll back on error
  }
}

Issue B — Stale cart ID after session restart

Headless cart drawers store the cart ID in localStorage or a cookie. Common scenarios that invalidate the stored cart ID:

  • Customer starts checkout, checkout creates a new cart session, customer abandons — the cart ID in localStorage now points to an expired checkout cart
  • Mobile app handoff to web browser — the app uses a different cart than the web session
  • Browser clears localStorage between sessions (private browsing, iOS Safari ITP)

When cartLinesUpdate is called with a stale cart ID, the Storefront API returns a "Cart not found" error. Most headless implementations surface this as a generic error or show no response — the drawer appears broken with no explanation.

Issue C — cartBuyerIdentityUpdate needed for logged-in customer pricing

When a customer logs in mid-session while the cart drawer is open, the cart still reflects anonymous pricing. Customer-specific pricing (B2B price lists, loyalty member discounts, customer tag-based pricing) only applies after cartBuyerIdentityUpdate is called with the customer's access token. Failing to call this mutation means logged-in customers see anonymous prices in the drawer — then see their actual discounted prices at checkout, causing confusion about which price is correct.

// Call after customer authenticates
mutation cartBuyerIdentityUpdate($cartId: ID!, $buyerIdentity: CartBuyerIdentityInput!) {
  cartBuyerIdentityUpdate(cartId: $cartId, buyerIdentity: $buyerIdentity) {
    cart {
      lines(first: 10) {
        nodes {
          cost {
            amountPerQuantity { amount currencyCode }  # will now reflect customer pricing
            totalAmount { amount currencyCode }
          }
        }
      }
    }
    userErrors { field message }
  }
}

Quick Diagnosis Checklist

  • Navigate to /cart directly and compare with what's in the drawer — if they differ, an app is targeting the wrong template
  • Open browser DevTools → Network tab, add an item to cart, watch the request sequence — confirm add-to-cart AJAX resolves before the cart fetch that populates the drawer
  • Add an order note in the drawer, then change an item quantity — check /cart.json to verify the note is still present in the note field
  • Check for duplicate cart drawer JavaScript: open DevTools → Sources and search for multiple CartDrawer or cart-drawer class initializations
  • Test the drawer in a private/incognito browser window without extensions — if the drawer works there but not normally, an ad blocker or extension is interfering
  • If translation is missing in the drawer: test in Weglot/Langify admin with dynamic element scanning enabled for the cart container
  • For headless: add console logging to cartLinesUpdate calls and check for "Cart not found" errors in the browser console
