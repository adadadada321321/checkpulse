Shopify Express Checkout Not Working — Buy Now Button Bypasses Customizations, Shop Pay OTP Expires, Apple Pay Safari-Only, Subscription Products Hide All Express Buttons

The Buy Now button silently skips order notes, cart attributes, discount entry, and all line item property customizations. Any subscription product in the cart hides every express checkout button simultaneously. Apple Pay only appears in Safari on Apple devices — never in Chrome. Shop Pay's OTP expires in 10 minutes and mobile app-switching kills the session. Checkout UI Extensions don't fire in any express checkout lane.

📅 Updated June 2026 ⏱ 13 min read 💡 Applies to: All Shopify plans
Critical — Express Checkout Silently Bypasses Cart-Level Customizations
Buy Now, Shop Pay, Apple Pay, Google Pay, and PayPal Express all skip the /cart page. Order notes, cart attributes, upsell app widgets, gift message fields, discount code fields on the cart page, and any data collected via cart page forms is never shown to express checkout customers and never transmitted to Shopify. This is the #1 source of "why is order data missing?" from merchants who add customizations to their cart page.

Failure Mode 1 — Buy Now Button Silently Bypasses Cart-Page Data

Shopify's Buy Now button (also called the dynamic checkout button) appears on product pages, collection pages, and search results. When clicked, it sends a POST request directly to checkout with buyNow=true, skipping the cart page entirely. Every customization merchants add to the /cart route is silently skipped.

What gets bypassed

  • Order notes textarea: the note field on the cart template (name="note") is never shown, so special instructions are never collected
  • Cart-page discount code field: many themes show a discount code input on the cart page. Buy Now customers cannot enter discount codes until they reach the checkout itself (where the code field appears in the order summary)
  • Cart attribute inputs: any cart.attributes inputs (name="attributes[field_name]") on the cart template are bypassed — gift message fields, PO number inputs, delivery date pickers
  • Upsell and cross-sell apps: apps like Frequently Bought Together, ReConvert Cart, and CartHook that inject widget content into the cart page are never shown
  • Gift wrapping and gift message apps: almost all gift option apps present their UI on the cart page — this is entirely skipped by Buy Now
  • Custom shipping notes or delivery window selectors: any custom UI element on the cart page that collects data sent via cart attributes is bypassed

How to mitigate Buy Now data loss

  1. Move required inputs to the product page: use line item properties (name="properties[field_name]" inside the product form) for fields that must be collected before checkout. These are captured when the product is added to cart via the standard Add to Cart flow — but see Failure Mode 7 for why this still fails with Buy Now.
  2. Use a pre-checkout popup app: apps like Qstomizer, OrderDesk, or custom theme code can intercept the Buy Now button click and display a modal collecting required fields before proceeding to checkout.
  3. Checkout UI Extensions (Shopify Plus only): add custom fields directly inside the checkout flow at Admin → Settings → Checkout → Customize → add UI extension blocks. These appear for all checkout paths including express checkout — with one exception (see Failure Mode 5).
  4. Add messaging near Buy Now buttons: if full migration isn't immediate, add a notice such as "Need to add a gift message or special instructions? Use 'Add to cart' instead." This redirects customization-requiring customers to the standard path.
Disabling Buy Now Buttons Reduces Conversion
You can disable dynamic checkout buttons in Admin → Settings → Checkout → uncheck "Show dynamic checkout buttons" — but this removes all express checkout buttons store-wide and typically reduces conversion by 5–15% for mobile-first audiences. Only do this if missing customization data is causing operational problems that outweigh the conversion impact.

Failure Mode 2 — Subscription Products Hide All Express Checkout Buttons

This is the most surprising express checkout behavior: adding any single subscription product (a product with a selling plan) to the cart causes every express checkout button to disappear simultaneously — Shop Pay, Apple Pay, Google Pay, and PayPal Express all vanish from the cart page and product pages.

Why this happens

Express checkout flows are designed for fast, minimal-friction payment. Shopify's checkout must display the subscription agreement consent, recurring billing terms, and cancellation policy disclosures at the payment step — content that express checkout UIs cannot accommodate. As a safeguard, Shopify checks cart contents at button render time and suppresses all express checkout options when a selling plan is detected on any line item.

What this looks like to merchants

  • A customer adds a one-time purchase product → express buttons show normally
  • Customer then adds a subscription product → all express buttons disappear instantly
  • Customer removes the subscription product → express buttons reappear
  • Merchants who sell both subscription and one-time products report customers confused about why express payment disappeared "randomly"

What merchants can do

  • There is no configuration to override this — it is platform-level behavior by design
  • Inform customers via cart page messaging: "Subscription products require our standard checkout. Express payment isn't available when subscribing."
  • Consider separating subscription and one-time products into different purchasing paths (different pages or a dedicated subscription purchase flow)
  • On Shopify Plus with certain subscription app configurations, the checkout UI extension can display consent inline — but express buttons still may not appear depending on the specific selling plan configuration
This Also Affects Mixed Carts (One-Time + Subscription)
A cart with two products — one regular, one subscription — has all express buttons hidden. Customers who are trying to use Shop Pay for the regular product in their cart cannot do so if a subscription item is also present. Some merchants resolve this by setting a store policy that subscription items must be purchased separately.

Failure Mode 3 — Shop Pay OTP Expires During Mobile Session

Shop Pay uses a 6-digit one-time passcode (OTP) sent to the customer's phone number or email to verify identity before completing express checkout. This OTP expires in 10 minutes. The session can also be lost entirely on mobile devices due to app-switching behavior.

How the session gets lost on mobile

The most common sequence on iOS and Android:

  1. Customer taps Shop Pay button on product or cart page → browser sends OTP request
  2. SMS or email notification arrives → customer taps the notification → switches to Messages or Gmail app
  3. Customer copies the OTP → switches back to browser
  4. On low-memory devices or after extended time in another app: the browser tab has been evicted from memory → the Shop Pay session is gone → customer sees the cart or product page again, not the Shop Pay verification screen
  5. Customer pastes the OTP into a field that no longer exists → must start over

Additional Shop Pay failure modes

  • Email OTP spam folder: if the customer's email OTP lands in spam, they find it after the 10-minute window and the code is expired
  • Phone number not on file: if the customer's phone number changed since enrolling in Shop Pay, the SMS goes to the old number
  • Multiple devices: customers who start Shop Pay on their phone and try to complete it on a desktop browser encounter session mismatch errors
  • Shop Pay account locked: after repeated failed OTP attempts (typically 5), Shop Pay temporarily locks the account — customer must wait or use standard checkout

What to tell customers experiencing this

  1. When Shop Pay sends the OTP notification, do not tap the notification — instead, switch to the browser tab first, then switch to Messages/email by pressing the home button, copy the code, and switch back quickly.
  2. Alternatively, if your phone supports notification previews, read the OTP from the lock screen notification without unlocking to the Messages app.
  3. If the session is already lost, proceed with standard checkout (the "or" link below the Shop Pay button) — your cart items are preserved.
  4. Download the Shop Pay app — native app sessions are more stable than browser-based sessions and maintain OTP state better when switching apps.

Failure Mode 4 — Apple Pay Only Appears in Safari on Apple Devices

Apple Pay's rendering requirement is the most frequently misunderstood express checkout behavior. It requires four conditions simultaneously — and most merchant testing setups fail at condition one.

Condition Required? Common Testing Mistake
Browser must be Safari Required Testing in Chrome or Firefox on a Mac — Apple Pay will never appear
Device must be Apple (Mac, iPhone, iPad) Required Testing on Windows PC or Android — not supported
Device has a card in Apple Wallet / Keychain Required Test device with no cards enrolled — button doesn't render
Domain passes Apple Pay domain verification Required Custom domain without .well-known file reachable — silent failure
Shopify Payments enabled and active Required Using third-party gateway — Apple Pay unavailable regardless

How to diagnose Apple Pay not appearing

  1. Open your store in Safari on an iPhone or Mac that has a card in Apple Wallet. If Apple Pay appears here, it's working — your previous test environment was wrong.
  2. If it still doesn't appear in Safari on an Apple device with a card: navigate to yourdomain.com/.well-known/apple-developer-merchantid-domain-association — this file must return a JSON response, not a 404. If it 404s, domain verification has failed.
  3. Check that Shopify Payments is active (not "Pending" or "Suspended") at Admin → Settings → Payments.
  4. If your store uses a third-party payment gateway (Stripe, PayPal, Braintree, Authorize.net) instead of Shopify Payments: Apple Pay is not available. It is only available through Shopify Payments.
  5. For custom or headless domains: re-run domain verification at Admin → Settings → Payments → Shopify Payments → Manage → Apple Pay → Verify domain.
Mixed Content Blocks Apple Pay Silently
If your store loads any http:// (non-HTTPS) resource on the checkout or product page, the browser may block Apple Pay's payment session request as mixed content. The button may render but clicking it produces no popup. Check your browser console for mixed content warnings and ensure all resources load over HTTPS.

Failure Mode 5 — Checkout UI Extensions Don't Fire in Express Checkout Lanes (Plus)

Shopify Plus merchants who invest in Checkout UI Extensions — custom fields, age verification validators, consent checkboxes, post-purchase offers, checkout branding, loyalty point displays — discover that a large proportion of their orders bypass these customizations entirely.

Which express paths bypass Checkout UI Extensions

Express Path Checkout UI Extensions Apply? Notes
Standard "Check out" button Yes — fully Extensions render at all configured positions
Shop Pay express lane No Shop Pay runs its own hosted UI; extensions don't inject
Apple Pay / Google Pay No Payment sheet UI only — no extension injection points
PayPal Express No PayPal's hosted checkout flow; Shopify extensions absent
Buy Now → standard checkout Yes — fully Buy Now routes to Shopify checkout where extensions render
Meta Pay / Facebook Pay No Hosted external flow if enabled

Compliance implications

If you've implemented age verification as a Checkout UI Extension (a validator that prevents checkout completion without confirming age), customers using Shop Pay, Apple Pay, or Google Pay bypass this entirely. This is a compliance gap for merchants selling age-restricted products (alcohol, tobacco, adult content, certain supplements).

Workarounds for express checkout extension bypass

  • Shopify Flow post-purchase verification: trigger a Flow automation on orders/create where payment method is Shop Pay or an express method → send a verification email or tag the order for manual review
  • Third-party age verification apps with API-level enforcement: some apps can check age on the server side via webhooks rather than only in the checkout UI, catching express checkout orders
  • Pre-checkout popup gates: require age confirmation before the product page (not inside checkout) via a sitewide age gate cookie — this runs before any checkout path
  • Order tagging and fulfillment hold: for high-stakes compliance, auto-tag express checkout orders and add a fulfillment hold pending human verification

Failure Mode 6 — PayPal Express Currency Mismatch Causes Silent Capture Failure

PayPal Express checkout creates an order in Shopify's store currency and displays that currency to the customer during the PayPal approval screen. However, PayPal captures payment in the currency of the customer's PayPal account balance (or their PayPal account's default currency). When these differ and the merchant's PayPal account cannot receive that currency, the capture fails — often silently from Shopify's perspective.

When this happens

  • Shopify Markets multi-currency stores: a customer browsing in EUR on a USD-primary store sees prices in EUR, clicks PayPal Express, PayPal shows the EUR amount, the customer's PayPal account is set to EUR, but the merchant's PayPal business account only accepts USD — capture fails
  • Region-based pricing: the displayed price is a converted price, but PayPal cannot reconcile the currency conversion with the merchant's payout currency
  • PayPal's currency conversion vs Shopify's: if PayPal auto-converts at a different rate than Shopify, the captured amount differs from the authorized amount by more than PayPal's tolerance → capture rejected

How to detect and fix PayPal Express currency issues

  1. In PayPal Business account settings → Money, Banks, and Cards → Currency Balances: ensure your account has a balance or accepted currency that matches your Shopify store's primary currency.
  2. In Shopify Admin → Settings → Payments → PayPal: review which currency PayPal is configured to accept. It should match your Shopify store's default currency, not your markets' presentation currencies.
  3. Enable "Auto-convert" in PayPal account settings if you want PayPal to handle currency conversion — but be aware that PayPal's conversion rates are less favorable than Shopify's Markets rates, so customer-displayed prices may differ from what PayPal charges.
  4. For international stores: consider removing PayPal Express from markets where currency mismatch is most common, or test PayPal checkout flows from a customer account in each target market currency.
PayPal Capture Failures May Not Generate Merchant Alerts
When a PayPal Express capture fails after an order is created in Shopify, the failure is logged in the order timeline but does not trigger an automatic merchant notification email by default. Orders can appear to be "paid via PayPal" in the order list while actually having a failed capture. Monitor Admin → Orders → filter by "Payment status: Authorized" to catch these.

Failure Mode 7 — Buy Now Button Loses Line Item Property Customizations

Line item properties are the standard Shopify mechanism for attaching per-product customization data to cart items — engraving text, color choices, personalization fields, size notes, and any other input collected on the product page. They work by having input fields inside the product form with name="properties[Field Name]" syntax, which are submitted when the Add to Cart button is clicked.

Buy Now bypasses the product form submission. When a customer clicks Buy Now, the browser POSTs directly to /cart/add with the product variant ID and quantity, but not the form contents — because the form was never submitted. All line item properties are silently absent from the resulting checkout.

This affects all product options apps

  • Infinite Options (ShopPad): option values not transmitted via Buy Now
  • SC Product Options (formerly Bold Product Options): same behavior
  • Hulk Product Options: same
  • Custom theme line item property fields: same
  • Engraving and personalization apps: engraving text lost on Buy Now

Solutions

  • Most reliable fix: hide the Buy Now button on products that require customization. In your theme customizer, some themes allow hiding dynamic checkout buttons per product type or for specific products via metafields.
  • JavaScript interception: add a JS handler that intercepts the Buy Now click, validates that required properties are filled in, and either passes the properties programmatically or blocks the click until they are.
  • Checkout UI Extension (Plus): recreate the customization fields inside checkout itself using a TextField extension, and use applyAttributeChange to store the values as order attributes or line item changes.
  • Pre-checkout modal: trigger a modal on Buy Now click that collects required inputs before proceeding, passing them as URL parameters to the checkout with a custom checkout URL builder.

Failure Mode 8 — Headless Express Checkout Is Not Automatic

On headless Hydrogen storefronts or custom storefronts built on the Storefront API, no express checkout button works automatically. Each payment method requires its own full implementation — the UI component alone does not process payments.

What each express method requires in headless

Express Method Required Implementation Most Common Mistake
Shop Pay shopPayPaymentRequestSessionCreate mutation → session token → ShopPayButton component with token Rendering ShopPayButton without creating a payment request session first — button appears but tapping it errors
Apple Pay window.ApplePaySession JS API + server-side merchant session validation endpoint + onpaymentauthorized handler calling Storefront API Missing validatemerchant server endpoint — Apple Pay sheet opens but immediately shows "Payment not completed"
Google Pay PaymentRequest API with Google Pay config + isReadyToPay check + onpaymentauthorized calling Storefront API Using wrong merchantId or skipping isReadyToPay check — button renders but returns MERCHANT_ERROR on tap
PayPal Express PayPal JS SDK with createOrder + onApprove callbacks calling Shopify Storefront API checkout mutations Connecting PayPal's order creation directly to Shopify without mapping PayPal's order ID to a Storefront API checkout — payment captured by PayPal but no Shopify order created

Hydrogen-specific notes

// Hydrogen's ShopPayButton requires a payment request session
// Do NOT just render <ShopPayButton /> without this

const session = await storefront.mutate(SHOP_PAY_PAYMENT_REQUEST_SESSION_CREATE, {
  variables: {
    paymentRequest: {
      currencyCode: cart.cost.totalAmount.currencyCode,
      lineItems: cart.lines.nodes.map(line => ({
        quantity: line.quantity,
        variantId: line.merchandise.id,
      })),
      // ... shipping, tax, discounts
    },
  },
});

// Then pass session.token to <ShopPayButton token={session.token} />
Test Each Express Method in Isolation
When debugging headless express checkout, test one method at a time: disable all others, add verbose console logging to every callback (createOrder, onApprove, validatemerchant, onpaymentauthorized), and watch the browser network tab for failed API calls. Express checkout failures often produce no visible error to the customer while producing clear error responses in the network log.
