Shopify Masary Not Working — Bill Code Not Generating, Order Stuck in Pending & Payment Issues

Updated June 2026 · 7 failure modes · Egypt merchants · Masary offline payment network · EGP only

Masary (ماساري) is one of Egypt's established offline payment networks, allowing customers to pay e-commerce orders at a wide network of physical agents, ATMs, and participating bank branches using a reference code generated at Shopify checkout. When Masary stops working on Shopify, the failure mode is what determines the fix — a reference code not generating (API credential or signature issue) is an entirely different problem from an order stuck in pending after a customer paid (callback URL failure), which is different again from a test endpoint left in production. Understanding which of the seven failure modes you are facing is the difference between a 10-minute fix and hours of incorrect debugging.

Is Masary not generating codes or leaving orders in pending? A 4-step checkout diagnostic confirms whether your Masary integration is live and correctly configured before customers hit the problem.

How Masary Works — and Why It Fails Differently From Card Payments

Masary is an offline reference code payment network, not a real-time card payment processor. The payment flow is asynchronous: your Shopify store generates a reference code for the customer at checkout → the customer takes that code to any Masary agent, ATM, or bank branch and pays in cash or via bank card → Masary's system notifies your store via a webhook/callback → your store marks the order as paid. Each step in this chain can fail independently, and the failure looks completely different to the merchant and the customer depending on where the break occurs.

Masary is operated under the EFG Finance umbrella, the financial services arm of EFG Hermes Group — one of Egypt's largest financial institutions. This institutional backing gives Masary a wide acceptance network across Egypt's banking, ATM, and agent infrastructure, but it also means that merchant onboarding, account activation, and support follow a financial institution process rather than a pure fintech one. New merchant accounts may require additional review and documentation compared to newer BNPL providers.

Masary is integrated via direct API or through payment aggregators like Paymob. Direct integration requires a Masary merchant code, a secret key for signature generation, and a correctly configured callback URL. Via Paymob, Masary is available as one of the offline payment methods within the Paymob payment suite — requiring only a Masary-enabled Paymob account, not a separate Masary merchant registration. The failure modes differ between the two integration paths.

Masary operates exclusively in Egyptian Pounds (EGP) and is available to Egyptian customers only. Shopify stores with multi-currency pricing must convert order totals to EGP before calling the Masary API. Non-Egyptian customers cannot use Masary regardless of integration status.

7 Failure Modes Covered

  1. Masary not appearing at checkout (merchant not registered or integration not configured)
  2. Bill reference code not generating (API credential or signature failure)
  3. Order stuck in pending after agent payment (callback URL not configured or wrong)
  4. Staging or test API endpoint used in live production store
  5. Currency not EGP — amount passed in foreign currency
  6. Reference code expired before customer paid
  7. Masary merchant account not activated for e-commerce payments
1

Masary not appearing at checkout — merchant not registered or integration not configured

Merchant Issue

The most common reason Masary does not appear as a payment option at Shopify checkout is that the merchant has not completed merchant registration with Masary, or has completed registration but has not fully configured the integration within Shopify. Masary e-commerce acceptance requires a separate merchant account from any existing Masary bill payment setup — being a Masary registered issuer for bill collection does not automatically grant e-commerce merchant payment acceptance.

To accept Masary payments for Shopify orders, merchants must complete an e-commerce merchant onboarding with Masary (or via a payment aggregator like Paymob that includes Masary in its offline payment suite). Direct Masary merchant registration is handled through EFG Finance's merchant acquisition process, which typically involves submitting business documentation, signing a merchant services agreement, and receiving production API credentials. Until this process is complete and approved, the payment option cannot appear at checkout.

Merchants who have completed Masary registration but installed the integration without entering their production credentials will also see Masary absent from checkout. The Masary payment plugin or custom integration requires a production merchant code and secret key entered in Shopify's payment settings. Blank or placeholder credentials fail silently — Masary will not appear at checkout without valid credentials. Similarly, the integration must be explicitly enabled/activated in Shopify payment settings after credentials are saved; installation does not automatically make the method visible to customers.

For merchants using Paymob as their aggregator: Masary offline payment through Paymob requires Masary (or the equivalent Paymob offline payment method) to be explicitly enabled in the Paymob dashboard under payment integrations. Paymob's card and wallet setup does not automatically include Masary offline — it must be toggled on separately.

How to fix

  1. Confirm merchant registration status: Contact Masary merchant support or your EFG Finance commercial contact to confirm your e-commerce merchant account is approved and active — not just applied for. Pending applications do not allow payment acceptance.
  2. Retrieve production API credentials: Log in to the Masary merchant portal and locate your production merchant code and secret key. These differ from sandbox credentials issued during developer testing. Contact Masary merchant support if you cannot locate them or if your account is newly approved.
  3. Enter credentials in Shopify: Go to Shopify admin → Settings → Payments → Third-party providers → Masary. Enter the production merchant code and secret key exactly as provided, checking for leading or trailing spaces introduced by copy-paste. Save the configuration.
  4. Enable the payment method: After saving credentials, toggle Masary to Active or Enabled in the Shopify payment settings. Installation alone does not make it visible at checkout.
  5. If using Paymob: Log in to your Paymob dashboard → Integrations → find the Masary or offline payment method section and ensure it is enabled alongside your card and other wallet integrations.
  6. Test with a real order: Use an incognito browser window, add a product to cart, proceed to checkout, and verify Masary appears as a payment option. Complete a test transaction and verify a reference code is generated.
2

Bill reference code not generating — API credential or signature failure

Merchant Issue

When Masary appears at checkout but no reference code is generated after the customer selects it, the integration is authenticating with the Masary API but the API is rejecting the request. This is almost always an API authentication or signature failure. Masary's payment generation API uses a request signature — computed from the merchant code, secret key, order amount, and other parameters — to verify that requests come from authorized merchants. If the signature computation is incorrect, the API rejects the request and no bill code is issued.

The most common signature failure causes are: wrong secret key (copied incorrectly or from the sandbox environment instead of production), wrong merchant code, incorrect parameter order in the signature hash computation (the parameters must be concatenated in the exact order Masary's API documentation specifies), or incorrect character encoding of the input string before hashing. A single difference in any of these factors produces an entirely different hash value and triggers an authentication error.

A second category of code generation failure occurs when the Masary API rejects the request for reasons unrelated to authentication: the order amount is below Masary's minimum transaction threshold, the product category is not permitted under the merchant's account type, or the merchant account has been temporarily flagged or suspended. These rejection reasons are usually communicated in the API error response body — logging the full API response (not just the HTTP status code) is essential for diagnosing which type of failure is occurring.

For merchants using an off-the-shelf Shopify plugin for Masary, the plugin may have a bug in its signature generation code that only manifests in certain character sets, order total amounts, or Shopify versions. Checking the plugin's GitHub repository or changelog for known signature issues and verifying the plugin is on its latest version is a critical first step before modifying API credentials.

How to fix

  1. Log the full API response: Enable logging for Masary API calls in your integration and capture the complete response body when code generation fails. The Masary API error response includes a specific error code and message that identifies whether the failure is an authentication error, a signature mismatch, a business rule rejection, or something else. Diagnose the specific error before making changes.
  2. Re-enter the secret key manually: Copy the secret key from the Masary merchant portal, paste it into a plain text editor to verify it contains no hidden characters, then re-enter it in your Shopify integration. Do not copy from an email or document that may have introduced smart quotes or encoding characters.
  3. Verify signature parameter order: Compare your signature generation code against Masary's API documentation (available in the Masary developer portal). The parameter concatenation order must exactly match the documentation — even a correct secret key produces the wrong signature if parameters are in the wrong order.
  4. Confirm you are using production credentials: Verify the API endpoint URL and credentials you are using are the live production ones, not the sandbox ones. Sandbox credentials are typically labeled differently in the merchant portal.
  5. Update your plugin: If using a third-party Shopify plugin, update to the latest version. Check the plugin's issue tracker for known signature or credential handling bugs.
  6. Test with Masary's API directly: Use a tool like curl or Postman to send a test payment generation request directly to the Masary API with your credentials. This isolates whether the issue is with your Shopify plugin or with the credentials/account.
Critical: Never log or expose your Masary secret key in public error logs, browser console output, or front-end JavaScript. The secret key must remain server-side only. If your integration has ever exposed the secret key publicly, contact Masary merchant support to rotate it immediately.
3

Order stuck in pending after customer paid at Masary outlet

Merchant Issue

When a Shopify order stays in "Payment Pending" status after a customer successfully paid their Masary reference code at an agent, ATM, or participating bank branch, the problem is always a failure in Masary's payment notification delivery to your Shopify store. Masary's offline payment workflow is inherently asynchronous — your store generates a code at checkout, and separately, Masary notifies your store when that code is paid. If the notification delivery fails, your store never learns that the payment occurred.

The notification is delivered via a callback URL (also called IPN — Instant Payment Notification, or webhook) that you configure in your Masary merchant portal. Masary calls this URL with payment confirmation data after the customer pays at an outlet. For Shopify to receive this notification and update the order status, the callback URL must be correctly configured in the Masary merchant portal and must be publicly reachable from Masary's servers.

Common callback URL failure causes: the URL is not configured at all in the Masary portal (the most common cause — merchants configure credentials but forget to set the callback); the URL contains the wrong domain (pointing to a development/staging domain instead of the live Shopify store URL); the URL uses HTTP instead of HTTPS (Masary's servers may reject non-HTTPS callback URLs); a trailing slash discrepancy between the configured URL and what the server expects; or the Shopify store's endpoint returns an error response when Masary calls it, causing Masary to consider the delivery failed (though Masary may retry, repeated failures leave the order stuck).

A second category of pending order failure occurs when the callback URL is correctly configured but your Shopify store cannot process the notification — for example, because the notification format changed in a Masary API update and your plugin has not been updated to parse the new format, or because a Shopify app is blocking or intercepting the callback request.

How to fix

  1. Verify the callback URL in the Masary portal: Log in to your Masary merchant portal and navigate to Integration Settings or IPN/Callback URL configuration. Confirm that a callback URL is entered and that it points to your live production Shopify domain — not a development URL, localhost, or the wrong store. The URL should use HTTPS.
  2. Test callback URL reachability: Use a tool like curl to confirm the callback endpoint is publicly reachable and returns a 200 OK response. If the URL is behind a firewall or restricted to specific IPs, Masary's servers may not be able to reach it. Contact Masary to obtain their callback server IP ranges if IP allowlisting is required.
  3. Check for trailing slash mismatches: Verify that the URL in the Masary portal exactly matches what your server expects — including whether or not there is a trailing slash. A URL configured as `example.com/masary/callback` may fail if the server only listens at `example.com/masary/callback/` (with trailing slash) or vice versa.
  4. Review Shopify webhook logs: Check your Shopify admin → Settings → Notifications or your integration's error logs for any failed incoming callback attempts. Error responses from your server cause Masary to retry, but repeated failures eventually leave the order stuck.
  5. Manually update stuck orders: For orders already stuck in pending, contact Masary merchant support with the order reference number to confirm payment status and get a payment confirmation you can use to manually mark the Shopify order as paid. This is a workaround; fix the root callback URL issue to prevent recurrence.
  6. Update your integration plugin: If the callback is reaching your server but orders are not updating, the integration may not be parsing the Masary notification payload correctly. Verify your plugin is on its latest version and check the plugin's changelog for callback format changes.
Merchant note: Pending-order failures from callback issues are invisible until a customer contacts you saying "I paid but my order is not confirmed." By then, you may have multiple stuck orders. Consider setting up an automated Shopify order alert for orders that remain in "Payment Pending" status for more than 2 hours — this surfaces the issue before it compounds across many customers.
4

Staging or test API endpoint used in live production store

Merchant Issue

Masary, like Fawry and other Egyptian offline payment networks, operates separate API environments for development/testing and live production. The test/sandbox environment uses a different API endpoint URL from the production environment. When a Shopify store that was originally connected to the Masary sandbox for testing goes live without switching to the production endpoint and production credentials, it continues using the sandbox — and the reference codes it generates are sandbox codes that cannot be paid at any real Masary outlet, ATM, or bank branch.

This failure mode is particularly deceptive because everything appears to work technically. The checkout flow completes normally, a reference code is generated, the code is displayed to the customer, no error is shown — but the customer discovers at the agent that the code is invalid. Customers typically contact the merchant reporting "the Masary agent said the code doesn't exist" rather than describing it as a payment failure, which can be difficult to trace back to the sandbox-in-production issue without examining the API endpoint configuration.

A related variation occurs when credentials are switched to production but the API endpoint URL is left pointing to the sandbox. In this case, the production credentials fail authentication against the sandbox environment (since sandbox and production credentials are separate), producing an authentication error. Conversely, using sandbox credentials against the production endpoint also fails. Both the endpoint URL and the credentials must match — both sandbox or both production.

How to fix

  1. Identify which API endpoint your integration is using: Find the Masary API URL configured in your Shopify integration (plugin settings, environment variables, or hardcoded in custom code) and compare it to the Masary API documentation. Confirm you are using the production endpoint URL, not the sandbox or staging URL. The URLs typically differ by a subdomain or path segment (for example, "sandbox.api.masary.com" vs "api.masary.com" — check your Masary documentation for the exact URLs).
  2. Switch both the endpoint and credentials together: When moving from sandbox to production, update both the API endpoint URL and the API credentials (merchant code and secret key) at the same time. Using production credentials with a sandbox URL (or sandbox credentials with a production URL) will not work.
  3. Verify with a low-value test transaction: After switching to production, complete a low-value real transaction in a real browser, take the reference code to a Masary agent or ATM, and confirm the code is accepted and the payment processes. This is the only way to definitively confirm the production switch worked end-to-end.
  4. Cancel or void any sandbox codes distributed to real customers: If real customers received sandbox reference codes, contact Masary merchant support to identify any orders with unpayable sandbox codes and refund or re-generate valid codes for those customers.
5

Currency not EGP — order amount passed in wrong currency

Merchant Issue

Masary's payment network operates exclusively in Egyptian Pounds (EGP). If your Shopify store is configured with a multi-currency setup — displaying prices in USD, EUR, GBP, or other currencies — and your Masary integration passes the order total to the Masary API in the display currency rather than converting to EGP first, one of two things happens: the API rejects the request with a currency error (if the API validates currency), or — more dangerously — the API accepts the numeric amount and treats it as EGP, generating a valid reference code for the wrong amount.

In the second scenario, a customer whose cart is $50 USD receives a Masary reference code for 50 EGP (approximately $1 USD at 2026 exchange rates). The customer pays this small amount at the agent, Masary notifies Shopify of a payment of 50 EGP, and the order may be marked as paid — but for a tiny fraction of the real order value. This is not a theoretical edge case; it has caused real revenue loss for merchants with multi-currency Shopify stores and poorly configured offline payment integrations.

The correct behavior is for the integration to convert the order total from the display currency to EGP at a current exchange rate before calling the Masary API. Payment aggregators like Paymob handle this conversion automatically. Direct Masary integrations must implement it explicitly — either by maintaining a configured exchange rate or by fetching a current rate from the Central Bank of Egypt or a trusted exchange service before each transaction.

How to fix

  1. Verify what currency amount is being sent to the Masary API: Add logging to your Masary integration to record the exact amount and currency being sent in each API request. If the amount is not in EGP, implement currency conversion before the API call.
  2. Implement EGP conversion: Add a conversion step that fetches the order total in EGP before generating the Masary reference code. Use the Central Bank of Egypt's official rate or a reliable financial data API. Hardcoded exchange rates become incorrect as rates fluctuate — prefer dynamic rates.
  3. Restrict Masary to EGP stores only: If your store serves both Egyptian and non-Egyptian customers in different currencies, configure Masary to appear only when the checkout currency is EGP. Use Shopify Markets or checkout conditions to hide Masary for non-EGP orders.
  4. Audit past orders for amount mismatches: If you suspect this issue has been occurring, compare Masary payment notification amounts (in EGP) against the expected EGP value of those orders. Orders where the Masary payment amount is significantly lower than expected were likely processed with the currency conversion missing.
Critical: Masary codes generated for the wrong amount (due to missing currency conversion) result in customers paying less than the order total. This is a revenue loss issue, not just a technical issue. Audit past Masary transactions if you have operated with multi-currency enabled and are unsure whether conversion was applied.
6

Reference code expired before customer paid

Merchant + Customer

Every Masary reference code has a validity window — a maximum time limit within which the customer must visit a Masary outlet and pay the code. After this window expires, the code becomes invalid and cannot be paid at any outlet, ATM, or bank branch regardless of the code's face value. When a customer attempts to pay an expired code, the Masary outlet or ATM will return an error such as "code not found" or "code invalid."

The validity period is configured as a parameter when the reference code is generated via the Masary API. The configurable range and default values depend on the merchant's account type and agreement with Masary. Common validity windows for e-commerce are 24 to 72 hours, though merchants can often configure shorter or longer windows within the allowed range. Setting a validity window that is too short causes unnecessary abandoned orders from customers who fully intend to pay but are delayed — 24 hours is often too short for customers who plan to pay at an agent on their way to or from work the next day. 48 to 72 hours is a more customer-friendly default for most e-commerce use cases.

A related issue is customer confusion about the expiry — customers who receive a reference code without a clear displayed expiry time do not know when they need to pay by. Adding the expiry date and time prominently in the order confirmation, the checkout success page, and any follow-up emails significantly reduces expired code failures from customers who simply did not realize the code was time-limited.

How to fix

  1. Increase the validity window: Review the validity period parameter in your Masary API code generation call. If it is set to 24 hours or less, consider increasing to 48 or 72 hours. Check your Masary merchant agreement for the maximum allowed validity period on your account type.
  2. Display expiry prominently to the customer: Show the exact expiry date and time of the Masary code on the order confirmation page and in the order confirmation email. Customers are more likely to pay before expiry when they know explicitly when the code expires. Use a format like "Pay by: Sunday 29 June 2026 at 11:59 PM" rather than just "valid for 48 hours."
  3. Send a pre-expiry reminder email: Implement an automated email 12–24 hours before the code expires reminding the customer to pay. Many customers who receive the initial code have every intention of paying but forget — a single reminder email can recover a significant percentage of pending payments.
  4. Generate a fresh code on customer request: Add functionality on the order status page for a customer to request a new reference code if their existing one has expired. This requires cancelling or voiding the expired code (check Masary API documentation for code cancellation) and generating a new one.
  5. Avoid auto-cancelling orders immediately after expiry: Give customers a short grace period after code expiry before cancelling the Shopify order — some payment notifications from Masary may be delayed, and customers who paid near the expiry window may have their payment processed after your store thinks the code is expired.
7

Masary merchant account not activated for e-commerce payments

Merchant Issue

Masary serves multiple distinct merchant use cases: bill payment collection (utilities, telecom, subscriptions), government fee payment, and e-commerce merchant payment acceptance. A Masary merchant account registered for one purpose is not automatically enabled for all other purposes. Specifically, a business that registered with Masary for bill collection or as a biller does not automatically have the e-commerce merchant payment feature — the ability to generate dynamic reference codes for variable-amount Shopify orders — enabled on their account.

This is a particularly common confusion for merchants in industries that use Masary for subscription or service billing (the customer pays a fixed monthly bill at an agent) who then try to use Masary for their Shopify store's e-commerce orders. The API calls will fail or be rejected not because of credential errors but because the e-commerce payment feature is simply not active on the account — even with perfectly correct credentials and signature computation.

Masary's API may return a specific error code for this situation, or it may return a general rejection that is easily confused with a signature or credential error. Contacting Masary merchant support with the specific API error code and requesting confirmation of which features are enabled on the account is the fastest way to diagnose this failure mode. Adding the e-commerce merchant payment feature requires contacting your EFG Finance account manager or Masary merchant support and may require a signed addendum to the merchant agreement.

How to fix

  1. Log the specific API error code and message: When code generation fails, capture the complete error response from the Masary API, including the specific error code. An error related to account feature enablement will typically be distinct from authentication errors — note the error code before contacting support.
  2. Contact Masary merchant support: Provide the specific API error code and describe the use case: that you want to accept dynamic e-commerce payments at Shopify checkout for variable order amounts. Ask specifically whether your account has the "e-commerce merchant payment" or "dynamic payment generation" feature enabled.
  3. Request feature activation from EFG Finance: If the feature is not enabled, request activation from your EFG Finance account manager or through Masary merchant support. This may involve signing a merchant services addendum and providing additional business documentation about your Shopify store.
  4. Confirm with a post-activation test: After EFG Finance or Masary support confirms the feature has been activated, retry generating a test reference code via the API. Verify the code is accepted at a Masary outlet before considering the issue resolved.
  5. Consider Paymob as a faster path: If activating the feature on a direct Masary account is taking too long and you need to accept offline payments urgently, consider integrating Masary through Paymob instead. Paymob's onboarding for its offline payment methods (which include Masary-network payments) may be faster than direct Masary e-commerce feature activation.
Merchant note: If you were using Masary for bill collection before trying to add it to Shopify, confirm with your EFG Finance account manager that your agreement has been updated to include e-commerce merchant payments. These are different commercial relationships with different settlement terms, and some Masary account setups restrict certain features to the original registered use case.

Masary Shopify Integration Diagnostic Checklist

Masary merchant account confirmed active — e-commerce merchant payment feature enabled by EFG Finance, not just bill collection access.

Production API credentials in use — merchant code and secret key copied from the Masary merchant portal's production section, not sandbox credentials.

Production API endpoint URL configured — the Masary API URL in your integration points to the live production endpoint, not sandbox or staging.

Signature generation verified — request signature is computed with the correct parameter order per Masary API documentation. No extra spaces or encoding issues in the input string.

Callback URL configured in Masary merchant portal — IPN / notification URL is entered in the portal, points to your live Shopify store domain using HTTPS, and is publicly reachable.

Currency conversion applied — all amounts sent to the Masary API are in EGP, regardless of the Shopify store's display currency.

Reference code validity window set appropriately — expiry window is 48–72 hours and expiry date/time is displayed prominently to customers at checkout and in confirmation emails.

End-to-end test completed — a real Masary reference code generated in production was successfully paid at a physical Masary outlet or ATM and the Shopify order updated to paid without manual intervention.
