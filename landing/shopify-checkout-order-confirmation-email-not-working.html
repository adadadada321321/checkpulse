Shopify Order Confirmation Email Not Working — 8 Causes Explained

Customers not receiving order confirmation emails? Notification sent but going to spam? POS orders sending nothing? Subscription renewals silent? Here are all 8 root causes of Shopify order confirmation email failures — for merchants, developers, and headless teams.

Updated June 2026 8 failure modes Notification Toggle · Spam · Klaviyo Conflict · POS Receipt · Subscription Renewals · Template Error · Headless

Shopify's order confirmation email is a completely separate system from the order status page (the thank-you page at /thank_you). The email fires through Shopify's notification engine the moment an order is created — independently from anything the customer sees on screen. A customer can land on the thank-you page successfully while receiving zero emails, and vice versa.

The most widely misunderstood fact: Shopify's notification system does not bounce or alert the merchant when a confirmation email fails. If the notification is disabled, if the customer typed the wrong email, or if a Liquid template error breaks the template — Shopify sends nothing and records nothing visible in the storefront. The only audit trail is the order Timeline in Admin.

Silent Failure Pattern

Order confirmation email failures don't trigger any alert in Shopify Admin. A customer who didn't receive their confirmation will contact support — the merchant then checks the order and sees "Order confirmation sent" in the Timeline even though the email went to spam or was never delivered. Timeline records the send attempt, not the delivery.

Failure Mode 1 — Order confirmation notification is disabled

Shopify's order confirmation is a notification toggle that can be switched off. When it's off, no confirmation email is sent for any order — no error, no warning, nothing in the storefront UI.

Where to check: Admin → Settings → Notifications → Order confirmation. The toggle must be set to "Send" (green). It defaults to on for new stores, but can be accidentally disabled during bulk notification management.

Common reasons merchants find it disabled:

  • Staff disabled it while testing ("I'll re-enable it later")
  • Shopify plan downgrade or store transfer that reset notification settings
  • Integration with a third-party ESP (Klaviyo, Mailchimp) where someone assumed the app handles it and disabled the native notification — then the app's flow broke
  • Developer disabled it during a theme migration and forgot to re-enable
How to Resend a Missed Confirmation

Admin → Orders → [order] → More actions → Resend order confirmation. This manually triggers the notification for that specific order. For bulk resends (many orders affected), you'll need a third-party app or the Admin API: POST /admin/api/2024-01/orders/{id}/send_invoice.json — note this sends the "invoice" email, not the exact same template as order confirmation; use a custom workflow for exact duplicates.

Failure Mode 2 — Email delivered to spam or Gmail Promotions tab

Shopify sends order confirmation emails from its shared sending domain (mail.shopify.com or no-reply@shopify.com). Because thousands of Shopify stores share this domain, ISPs see high email volume from it and route some stores' emails to spam — especially newer stores with no sending history.

The Gmail Promotions tab is a softer version of this: the email was delivered, but the customer didn't see it because Gmail categorized it as promotional. Most customers check Primary and Spam but not Promotions.

Fix: configure a custom sender email address for your store:

  1. Admin → Settings → Notifications → Sender email → enter your business email address (e.g., orders@yourdomain.com)
  2. Set up SPF, DKIM, and DMARC records for your domain via your DNS provider — Shopify provides the required DNS values after you enter your email
  3. Verify the domain in Shopify — the status should show "Authenticated"
  4. Send a test order to confirm emails arrive in Primary inbox, not spam
Google Workspace Note

If your business email is hosted on Google Workspace (Gmail for business), you can use that address as your Shopify sender. Emails sent from an authenticated business domain (e.g., orders@yourstore.com) consistently land in inbox rather than spam — and customers recognize your brand name instead of "Shopify".

Additional causes of spam delivery:

  • Store name contains words that trigger spam filters (words like "free", "discount", "winner")
  • Order confirmation template was customized to include promotional content, which increases spam score
  • Customer's mail server has a strict allowlist that blocks unfamiliar senders
  • Corporate email accounts (Outlook 365, corporate Exchange) often have aggressive spam filtering that blocks external retail mail

Failure Mode 3 — Customer's email address is wrong or undeliverable

Shopify sends the order confirmation to exactly the email the customer entered at the contact step of checkout. There is no bounce notification to the merchant — if the email bounces, the order Timeline still shows "Order confirmation sent" because Shopify handed it off to its mail delivery service. Delivery failure is silent.

Common email address issues:

  • Typo at checkout: Customer typed gmai.com instead of gmail.com, or transposed letters in their own address. Shopify validates basic email format (@ symbol, domain part present) but does not verify the address is real.
  • Apple "Hide My Email": iOS customers using Sign in with Apple may use a relay address like abc123@privaterelay.appleid.com. Apple forwards emails sent to this address to the customer's real inbox — but some corporate email servers and spam filters block or delay Apple relay addresses. If the customer uses a different device or signs in differently at checkout, the relay address may not match their Apple ID.
  • Disposable email addresses: Services like Mailinator, Guerrilla Mail, and Temp Mail create addresses that expire after minutes or hours. A confirmation sent to a disposable address bounces silently after expiry.
  • Plus addressing stripped: Some older email integrations strip plus addressing (e.g., user+shopify@gmail.com is a valid Gmail address that delivers to user@gmail.com, but some systems treat it as invalid).
How to Fix a Wrong Email on an Existing Order

Admin → Orders → [order] → Customer section → Edit → update the email address → Save. Then Admin → Orders → [order] → More actions → Resend order confirmation. This sends the confirmation to the corrected address.

Failure Mode 4 — Klaviyo / ESP conflict creates double-send or zero-send

Many stores use Klaviyo or another email service provider (Mailchimp, Drip, Omnisend) alongside Shopify's native notifications. This creates two common failure patterns:

Double-send (two confirmation emails)

Both Shopify's native "Order confirmation" notification AND a Klaviyo "Placed Order" flow are active. The customer receives two nearly identical emails minutes apart. Confusing to the customer, increases unsubscribe rates.

Zero-send (no confirmation email)

The merchant disabled Shopify's native notification because they believed Klaviyo would handle it — but the Klaviyo "Placed Order" flow was paused, the list was filtered incorrectly, or the Klaviyo-Shopify integration lost connection. Result: no confirmation email for any order.

How to diagnose:

  1. Check Admin → Settings → Notifications → Order confirmation → is it set to "Send"?
  2. Check Klaviyo → Flows → search "Placed Order" → is the flow Active (not Draft or Manual)?
  3. In Klaviyo → Flows → Placed Order → check "Recent activity" to see if recent orders triggered the flow
  4. Check Klaviyo → Integrations → Shopify → confirm the integration is connected and last synced recently
  5. Place a test order and watch both Shopify's notification timeline AND Klaviyo's activity feed
Pick One System, Not Both

The cleanest architecture is to disable Shopify's native Order confirmation notification and use Klaviyo's "Placed Order" flow exclusively — OR keep Shopify's native notification and don't build a Klaviyo order confirmation flow. Running both simultaneously requires careful conditional logic in Klaviyo to avoid duplicate sends (e.g., suppress the Klaviyo flow if Shopify already sent the notification).

Failure Mode 5 — POS orders use "Receipt" not "Order confirmation"

When staff complete a sale on Shopify POS, the notification sent to the customer is not the "Order confirmation" template. It is the "POS Receipt" — a completely separate template managed in a completely separate Admin location.

Where they live:

  • Online checkout orders: Admin → Settings → Notifications → Orders section → "Order confirmation"
  • POS orders: Admin → Settings → Notifications → Point of sale section → "Receipt"

Merchants who edit or disable the "Order confirmation" template expecting it to affect POS orders see no change — because POS uses a different template entirely.

Additional POS email issues:

  • If the staff member does not enter the customer's email address at the POS terminal before completing the sale, no receipt is sent — there is no email to send it to
  • The POS Receipt template supports Liquid customization but the available variables are different from the Order confirmation template — code copied between them may break silently
  • If a POS order is later fulfilled from the online store (e.g., shipped from warehouse), the fulfillment confirmation email uses the standard shipping notification template, not the POS receipt
Sending a Receipt After POS Sale

If the customer's email wasn't captured at the time of sale, a receipt can be sent from Admin → Orders → [order] → More actions → Resend order confirmation. This sends the order confirmation template (not the POS receipt template) to whatever email is on the order. To capture email reliably at POS, train staff to add the customer's account or collect email before tapping "Charge".

Failure Mode 6 — Subscription renewal orders have no confirmation email by default

When a subscription app (Recharge, Bold Subscriptions, Loop, Skio, Awtomatic, Seal Subscriptions) processes a recurring billing charge, it creates a new Shopify order. However, the order creation path used by subscription apps goes through the Admin API or the subscription app's own billing infrastructure — not through the standard Shopify checkout flow.

The result: Shopify does not automatically send the "Order confirmation" notification for subscription renewal orders. The notification engine that fires on checkout completion does not fire the same way for programmatically-created subscription orders.

What the customer experiences:

  • They received a confirmation email for their first subscription order (placed through the standard checkout)
  • For every subsequent renewal, their bank shows a charge but no email arrives
  • They call their bank to dispute the "unrecognized" charge
  • Chargeback rate for subscription merchants is disproportionately high partly for this reason

How to configure renewal confirmation emails:

  • Recharge: Admin → Recharge → Notifications → Charge success email — enable and customize the template
  • Bold Subscriptions: Bold Dashboard → Emails → Recurring order confirmation — enable separately from Bold's initial order email
  • Loop: Loop Dashboard → Notifications → Order confirmation for renewal — each app has its own notification system
  • Klaviyo integration: Configure a Klaviyo flow triggered by the subscription app's "Charge success" event (not Shopify's "Placed Order" event) for branded renewal emails
Don't Assume Shopify Handles Renewal Emails

Every subscription app has its own email notification system that must be configured separately. The Shopify native order confirmation does not reliably fire for renewal orders. If you switch subscription apps, the notification configuration from the old app does not carry over.

Failure Mode 7 — Liquid template error causes silent notification failure

Shopify allows merchants to fully customize the order confirmation email template using Liquid. Admin → Settings → Notifications → Order confirmation → Edit code. A Liquid syntax error in the template causes the entire notification to fail silently — Shopify does not send a broken template, it sends nothing.

Common template errors that break confirmation emails:

  • Unclosed Liquid tags: {% if condition %} without a matching {% endif %}
  • Invalid filter names or incorrect filter arguments: {{ amount | money_without_decimal }} when money_without_decimal isn't a valid Shopify filter
  • Reference to a variable that no longer exists (e.g., a custom property that was removed from products)
  • Broken HTML that prevents the email client from rendering correctly (email clients are much stricter than browsers)
  • External font imports (@import url() in style tags) that time out during email generation
  • Pixel tracking scripts or JavaScript added to the email template — email clients strip JavaScript; if the script references a CDN that's unreachable, it can time out template rendering

How to detect a template error:

  1. Admin → Settings → Notifications → Order confirmation → Send test notification — if no email arrives and the order confirmation is enabled, the template has an error
  2. In the notification editor, look for red error indicators or validate the Liquid manually
  3. Restore the default template using the "Revert to default" button, then re-apply your customizations incrementally, testing after each change
  4. Check Admin → Orders → any recent order → Timeline section — if "Order confirmation could not be sent" appears, that confirms a template render failure
Version Control Your Email Templates

Copy the current notification template code into a text file and save it before making any edits. Shopify's notification editor has a "Revert to default" button, but it reverts to Shopify's original default — not your previous custom version. There is no version history for notification templates.

Failure Mode 8 — Headless checkout doesn't trigger Shopify's notification

Shopify's order confirmation notification fires automatically when an order is created through the standard checkout flow. Headless storefronts using custom checkout implementations may create orders through paths that don't trigger the notification reliably.

Scenarios where the notification doesn't fire in headless:

Order created via Admin API with suppressed notifications

When creating orders via POST /admin/api/2024-01/orders.json, the send_receipt field defaults to false in some API versions. If the order creation call doesn't explicitly set "send_receipt": true, no confirmation is sent:

// Missing send_receipt causes no confirmation email
{
  "order": {
    "line_items": [...],
    "customer": { "email": "customer@example.com" }
    // send_receipt not set — defaults to false in some contexts
  }
}

// Correct — explicitly request the notification
{
  "order": {
    "line_items": [...],
    "customer": { "email": "customer@example.com" },
    "send_receipt": true,
    "send_fulfillment_receipt": true
  }
}

Custom payment flow creates an unpaid order

Some headless checkout implementations create the Shopify order in a "payment pending" state while processing the payment externally. If the order notification fires only on creation (before payment), the customer receives a confirmation for an order that may not complete. If the notification is suppressed until payment is confirmed, and the payment webhook fails, no notification is ever sent.

Webhook-triggered ESP flow with 5-second timeout

If the confirmation email is sent via a custom webhook (orders/create → custom server → Klaviyo/Mailchimp API), the Shopify webhook has a 5-second response timeout. If the custom server takes longer than 5 seconds to process the webhook and call the ESP API, Shopify marks the webhook as failed. After 19 consecutive failures, Shopify auto-disables the webhook entirely with no merchant notification.

Webhook Auto-Disable is Silent

Shopify auto-disables webhooks after 19 consecutive delivery failures. Check Admin → Settings → Notifications → Webhooks to see if any webhooks show "Disabled" status. Re-enable and fix the endpoint latency issue — the root cause is usually a slow third-party API call in the webhook handler that consistently times out past 5 seconds.

Storefront API checkout flow and email

For headless storefronts using the Storefront API (non-Hydrogen), the checkoutComplete mutation creates the order and should trigger Shopify's standard notification pipeline. Most failures here come from the customer email not being set on the checkout object before completion — if checkoutEmailUpdateV2 was never called, the order has no email and no confirmation can be sent.
