Shopify Conversion Tracking Not Working — GA4 Not Tracking, Meta Pixel Not Firing, Web Pixels API, GDPR Consent Blocking

Shopify deprecated checkout.liquid pixel injection in June 2025. GA4 and Meta Pixel purchase events go missing due to Web Pixels API migration failures, GDPR consent blocking, duplicate event deduplication errors, and Checkout Extensibility changes. Here are all 8 failure modes.

📅 Updated June 2026 ⏱ 10 min read 💡 Applies to: All Shopify plans
checkout.liquid Pixel Injection Is Dead — June 2025
Shopify deprecated checkout.liquid and the additional_scripts field on June 30, 2025. Any GA4 gtag.js, Meta Pixel base code, TikTok Pixel, or custom tracking script injected via these paths no longer executes on Shopify's hosted checkout pages. If your conversion tracking stopped working around that date, this is almost certainly why. Shopify's replacement — the Web Pixels API — requires app-based pixel extensions, not theme-level script injection.

How Shopify Conversion Tracking Works Now (And What Breaks)

Shopify's tracking architecture changed fundamentally with the Checkout Extensibility rollout. Understanding which surface each pixel fires on is essential to diagnosing gaps:

Surface Script Injection Method Works Post-June 2025? Where Purchase Event Fires
Storefront pages (PDP, collection, cart) theme.liquid / section scripts Yes No purchase event on these pages
Checkout pages (contact, shipping, payment) checkout.liquid / additional_scripts No — deprecated Only via Web Pixel extension
Order status / thank-you page additional_scripts field No — deprecated Only via Web Pixel extension
Checkout (all steps) Web Pixels API (sandboxed) Yes — required path checkout_completed event → Purchase
Checkout (Plus only) Checkout UI Extensions Yes Via UI extension + web pixel combo

The practical consequence: if your GA4 or Meta Pixel purchase event tracking relied on code injected into checkout.liquid, the additional_scripts field, or the order status page via Shopify admin settings, it stopped firing when Shopify rolled out Checkout Extensibility. The storefront pixel (PageView, AddToCart) continues to work because those pages still use theme.liquid — but the Purchase event, which is the conversion signal your ad accounts depend on, is now silent.

Failure Mode 1 — checkout.liquid and additional_scripts Deprecated

Before Checkout Extensibility, merchants tracked purchases by injecting JavaScript into:

  • checkout.liquid — the template that controlled Shopify's checkout page layout on non-Plus plans
  • Admin → Settings → Checkout → Additional scripts — a text field where you could paste pixel code, including Shopify's order attributes like {{ checkout.order_id }}
  • Liquid injection in the order status page (the confirmation page after payment)

Shopify removed all three injection points for non-Plus stores on June 30, 2025, and has deprecated them on Plus as well (checkout.liquid becomes read-only, then sunset). Any conversion tracking set up via these methods now silently fails — no purchase events reach GA4 or Meta Pixel, and there is no visible error in Shopify Admin.

How to Check If This Affected Your Store
In Google Analytics 4: go to Reports → Realtime → Event count by event name. Complete a test purchase. If you see page_view events but no purchase event, the purchase event is not firing. In Meta Events Manager: go to your Pixel → Test Events tab → complete a test purchase. If you see PageView but no Purchase event, the web pixel extension is not running. Compare your conversion data before and after July 2025 — a significant drop in reported purchases with no change in actual orders is the clearest signal.

The fix: Web Pixels API via official app integrations

To restore GA4 and Meta Pixel purchase tracking, you must use Shopify's Web Pixels API. The correct path for each platform:

  • GA4: Install or reconnect the Google & YouTube app from the Shopify App Store. The app creates a web pixel extension that fires GA4 events (including purchase) on checkout completion. Do not manually add gtag.js to theme.liquid as a replacement — this only fires on storefront pages, not checkout.
  • Meta Pixel: Install or reconnect the Meta Sales Channel or the standalone Meta Pixel app. This creates a web pixel extension that fires Purchase events on order confirmation. Verify the web pixel is active at Admin → Settings → Customer events.
  • TikTok Pixel: Use the TikTok app from the Shopify App Store (not a manual script injection).
  • Custom pixels: If you need a custom pixel not covered by an official app, use Shopify's Customer Events interface (Admin → Settings → Customer events → Add custom pixel). Custom pixels created here run in a sandboxed Web Pixel environment and have access to Shopify's standard events including checkout_completed.

Failure Mode 2 — Meta Pixel Fires PageView But Not Purchase

This is the most common symptom after the Web Pixels API migration. The Meta Pixel base code in theme.liquid fires PageView on every storefront page — so Meta Events Manager shows pixel activity. But the Purchase event only fires inside the web pixel extension running in Shopify's checkout sandbox. If the extension is missing or broken, Purchase never fires.

Diagnosing the missing Purchase event

  1. Go to Admin → Settings → Customer events. You should see your Meta Pixel listed here as a web pixel. If it's absent, the web pixel extension was never created or was deleted.
  2. Check the "Status" column. A status of "Connected" means the extension is active. "Error" or a blank status means the web pixel is not running.
  3. In Meta Events Manager, go to your Pixel → Test Events tab. Navigate through a real checkout on your store. You should see events appearing in real time: ViewContent, InitiateCheckout, AddPaymentInfo, Purchase. If Purchase never appears, the web pixel extension is not firing the checkout_completed event.
  4. Disconnect and reconnect the Meta app: Admin → Apps → Meta → Disconnect → reinstall from App Store. This regenerates the web pixel extension.
  5. After reconnection, re-run the Test Events diagnostic to confirm Purchase now fires.
Meta Pixel vs. Conversions API — Both Can Miss
The browser-side Meta Pixel fires from inside the web pixel extension sandbox. The server-side Conversions API (CAPI) fires from your server when Shopify's webhook delivers the order. If CAPI is configured via a third-party app (e.g., Elevar, Littledata, Aimerce) and that app's Shopify webhook subscription was auto-disabled (after 19 consecutive 5-second timeout failures), CAPI purchase events also stop. Check Admin → Settings → Notifications → Webhooks for any "Failed" webhook that belongs to your CAPI integration app.

Failure Mode 3 — Duplicate Purchase Events Inflating Conversions

After discovering that conversion tracking stopped, many merchants fix one path but leave the old broken path in place — resulting in duplicate events once the new path starts working. The most common duplicate scenarios:

GA4: Google & YouTube app + manual gtag.js

The Google & YouTube app creates a web pixel extension that fires purchase events to GA4. If you also have manual gtag.js code in theme.liquid or a purchase event call in the additional_scripts field (from before the deprecation), and that code somehow still fires on the order status page, you get two purchase events per order in GA4. GA4's deduplication for this case is transaction ID-based — if both events use the same transaction ID, GA4 should deduplicate them, but this relies on the event being sent correctly from both sources with matching transaction IDs, which is rarely the case in practice.

Meta Pixel: browser pixel + CAPI without deduplication

Running both the Meta browser pixel (via web pixel extension) and a server-side Conversions API integration without deduplication causes each purchase to appear twice in Meta Events Manager and Meta Ads reporting. Meta provides a deduplication mechanism using matching event_id values — the browser pixel event and the CAPI event for the same purchase must send the same event_id. Most Shopify CAPI integration apps (Elevar, Littledata) handle this automatically, but manual CAPI implementations often omit the event_id.

How to Check for Duplicate Events
In GA4: Reports → Realtime → Events. Complete one test purchase. Count how many purchase events appear. More than one means duplication. In Meta Events Manager: Pixel → Overview → scroll to "Event Deduplication". Look for a high "Duplicate Rate" on the Purchase event. A duplicate rate above 5% suggests your deduplication isn't working.

Failure Mode 4 — GDPR Consent Manager Blocking Pixels

On stores with a cookie consent manager (OneTrust, Cookiebot, Axeptio, Consentmo, or Shopify's native privacy banner), analytics and marketing pixels are categorized as non-essential. Visitors who decline consent — typically 20–40% of EU visitors — never have GA4 or Meta Pixel loaded. Their purchase events are permanently absent from your conversion data.

Shopify's Web Pixels API has built-in Customer Privacy API integration. Web pixel extensions receive a permission grant or denial before the extension loads — the extension respects consent automatically. This means:

  • Visitors who decline analytics cookies: GA4 purchase events are not sent
  • Visitors who decline marketing cookies: Meta Pixel Purchase events are not sent
  • Visitors who haven't interacted with the consent banner yet: pixels may be held until a choice is made

This is not a bug — it is GDPR compliance working as intended. But it creates a systematic gap in your reported conversion data that many merchants mistake for a tracking problem.

Recovering consent-gated conversion signal

  • Server-side Conversions API (CAPI): CAPI events sent from your server using hashed personally identifiable information (email, phone) are consent-exempt in many jurisdictions when processed under a legitimate interest or contractual necessity basis. CAPI can recover 20–35% of EU conversion signals that browser pixels miss due to consent decline.
  • GA4 server-side Measurement Protocol: Use Shopify's orders/create webhook to send a server-side GA4 purchase hit to the Measurement Protocol API. This fires regardless of browser consent and does not require cookies, but requires correct attribution (client_id) to join to the correct user session.
  • Consent mode (Google): Enable Google Consent Mode v2 via your consent manager. This allows GA4 to model conversions for consented users and use behavioral modeling to estimate conversions from non-consented users — without actual user-level data.

Failure Mode 5 — GA4 Google & YouTube App Connection Broken

Even when the Google & YouTube app is correctly installed and a web pixel extension exists, GA4 conversion tracking can fail to attribute purchases to the right campaigns due to a broken connection between Shopify and Google:

Common broken-connection scenarios

  • Store URL changed: If you migrated from a Shopify subdomain to a custom domain, or changed your custom domain, the Google & YouTube app's property association may need to be updated. Reconnect the app at Admin → Apps → Google & YouTube → Settings.
  • GA4 property changed: If you migrated from Universal Analytics (UA) to GA4 after initial setup, the app may still be pointing to a UA property ID. Check Admin → Apps → Google & YouTube → and confirm the connected property is a GA4 property (property IDs start with numbers, not "UA-").
  • Shopify plan downgrade: Some Google & YouTube app features require a specific Shopify plan. A downgrade can disconnect certain integrations without an explicit error notification.
  • Google account permission change: If the Google account used to authorize the app loses access to the GA4 property (e.g., employee left), the app's data stream connection becomes invalid. Reconnect with an account that has GA4 Edit access.
How to Verify GA4 Web Pixel Is Active
Go to Admin → Settings → Customer events. Find the Google & YouTube entry. It should show "Connected" status. Then go to your GA4 property → Admin → Data Streams → your Shopify stream → Measurement Protocol API Secrets. If no secrets exist, the stream was set up manually, not via the Google & YouTube app's web pixel — this means checkout-step events won't fire.

Failure Mode 6 — Safari ITP and Ad Blockers Removing Attribution

Even when your pixels fire correctly on every checkout step, browser-level restrictions cause significant attribution gaps that show up as lower reported ROAS than actual:

Safari Intelligent Tracking Prevention (ITP)

Safari's ITP restricts third-party cookies and limits first-party cookie lifetimes to 7 days (or 24 hours in some scenarios). This means:

  • A customer who clicked a Meta ad on Safari iOS more than 7 days ago and purchases today will not be attributed to that ad click
  • GA4 session attribution for cross-device journeys is broken in Safari — a customer who first visited on Safari mobile is assigned a different client_id than when they complete on Safari desktop
  • Any pixel relying on the _fbp, _ga, or similar cookies set by third-party CDNs faces shortened lifetimes

Safari represents approximately 20–25% of global web traffic and 40–55% of mobile traffic in many markets. ITP's impact on attribution is not a rounding error.

Ad blocker pixel suppression

uBlock Origin, AdBlock Plus, Brave Shields, and Privacy Badger block requests to known tracking domains including connect.facebook.net (Meta Pixel), www.googletagmanager.com (GTM), and analytics.google.com (GA4). Approximately 25–35% of desktop users have an active ad blocker. Even web pixel extensions running inside Shopify's sandbox face this — the pixel's outbound HTTP requests to Meta and Google servers are blocked by the browser's network filter.

Mitigations

  • Server-side event forwarding: Route GA4 hits through a first-party subdomain (e.g., metrics.yourstore.com) via Google Tag Manager Server-Side or a custom proxy. Requests from your own domain are not blocked by most ad blockers and bypass ITP's third-party restrictions.
  • Meta Conversions API: CAPI sends purchase events from your server, not from the customer's browser — ad blockers and ITP cannot intercept server-to-server HTTP requests.
  • Enhanced Conversions (Google): Send hashed email/phone data alongside click IDs (gclid) to Google's Enhanced Conversions API. Google matches these to signed-in Google accounts, recovering attribution even when browser cookies are blocked.

Failure Mode 7 — Headless / Hydrogen Checkout Has No Automatic Tracking

Headless Shopify storefronts built on Hydrogen, Next.js Commerce, or custom React frontends do not automatically receive Shopify's web pixel extensions. Shopify's Web Pixels API only runs on Shopify-hosted checkout pages — but even those require the pixels to be created via Customer Events in the admin.

The common assumption that "installing the Google & YouTube app and Meta app will handle everything" is wrong for headless: those apps create web pixel extensions for the checkout flow, but your custom storefront pages (PDP, collection, cart) are entirely outside Shopify's pixel sandbox. You must implement GA4 and Meta Pixel tracking manually on your frontend.

What headless stores need to implement manually

Event Where It Should Fire Implementation Needed
page_view / PageView Every route change Custom router hook that fires GA4 + Meta Pixel on navigation
view_item / ViewContent Product page load GA4 + Meta event call on PDP component mount
add_to_cart / AddToCart Cart mutation success Storefront API cartLinesAdd callback triggers pixel event
begin_checkout / InitiateCheckout Checkout URL creation Fire on checkoutCreate or redirect to checkout.shopify.com
purchase / Purchase Order confirmation Web Pixel extension (via Customer Events) OR server-side CAPI/Measurement Protocol from orders/paid webhook
Use @shopify/customer-privacy for Consent Management
Hydrogen and headless storefronts should use Shopify's @shopify/customer-privacy package for consent management instead of third-party CMP libraries. This package integrates with Shopify's Customer Privacy API and ensures your pixel implementations respect the same consent state that Shopify's checkout-side web pixel extensions use.

Failure Mode 8 — Order Status Page vs. Thank-You Page Confusion

Shopify has two different post-purchase pages with confusing overlap in naming and function:

Page URL Pattern When It Shows Customizable Via
Thank-you page (new) /thank-you (inside checkout) Immediately after payment confirmation (Checkout Extensibility) Checkout UI Extensions only (Plus)
Order status page (legacy) /orders/{order_id}/authenticate?key=… After checkout completes; also accessible later via "View your order" link additional_scripts field (deprecated) / Web Pixel extension
Post-purchase page Between payment and order status Only when a Checkout UI Extension targets the post-purchase slot (Plus) Checkout UI Extensions (Plus)

Before Checkout Extensibility, the order status page (accessible via the authenticated URL after checkout) was where the additional_scripts field ran — including conversion pixels. After deprecation, this page no longer runs additional_scripts scripts. The new "thank-you" page (which appears immediately after payment) is only customizable on Shopify Plus via Checkout UI Extensions.

For tracking purposes, the purchase event should fire on checkout_completed — a Shopify Customer Events standard event that fires inside the web pixel sandbox when a customer completes payment, regardless of which page they see. The page URL is irrelevant to the pixel; what matters is the web pixel extension receiving the checkout_completed event from Shopify's event bus.
