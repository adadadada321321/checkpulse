Shopify Checkout Stuck on Processing — 8 Reasons the Spinner Never Stops

Customers click "Place Order" and get an infinite loading spinner that never resolves? No error message, no confirmation — just a frozen checkout? Here are all 8 root causes for Shopify checkout hanging on processing, with exact diagnostic steps and fixes for each.

Updated June 2026 8 failure modes Gateway Timeout · 3DS Deadlock · Extension JS Error · Post-Payment Failure · Fraud Hold · Network Drop · Shopify Degradation · Browser Conflict

A Shopify checkout stuck on processing is one of the most damaging silent failures a merchant can face — not because it is common, but because of what it looks like to the buyer. No error message, no indication of what went wrong, no retry path. The buyer sees a spinning loader and waits. After 20–30 seconds most abandon. A small percentage refresh the page, potentially submitting the order twice. An even smaller percentage email support — and those are the ones you hear about.

The buyers who leave silently are the real loss. They interpret a frozen checkout as a broken store, not as a transient technical issue. They don't try again. They go to a competitor and may leave a review describing a broken checkout experience.

First: Check Whether the Order Was Actually Created
Before diagnosing why checkout is frozen, go to Admin → Orders and check if the order appears. Shopify sometimes creates the order and sends the confirmation email successfully, but the browser-side redirect to the thank-you page fails — leaving the buyer watching a spinner while the order is already confirmed. If the order exists, the issue is a front-end redirect failure, not an order processing failure. The buyer can be emailed their order confirmation and the issue is contained. If no order exists, a genuine processing failure occurred — continue with the diagnostic steps below.

Failure Mode 1: Payment Gateway Timeout

The most common cause of a frozen Shopify checkout is a payment gateway that takes too long to respond. When a buyer clicks "Place Order," Shopify sends an authorization request to the payment processor. If the gateway takes longer than Shopify's request timeout window — typically around 30 seconds — the checkout submission request never receives a response. The browser keeps the spinner running indefinitely because it is waiting for a reply that is not coming.

Gateway timeouts are transient but can cluster: a payment processor experiencing elevated load (common during flash sales or BFCM) may respond slowly for all merchants simultaneously. A timeout during a single-buyer checkout may be a one-off; multiple buyers reporting the same freeze in a short window almost always indicates gateway-side latency.

How to identify a gateway timeout

  • Timing: The freeze happens exactly at the "Processing payment" stage, not before the "Place Order" click. The buyer has entered card details and submitted — the spinner appears and never resolves.
  • No order in Admin: Because the gateway never responded, Shopify did not create an order. Go to Admin → Orders — no order exists for the buyer's session.
  • No authorization in Payments Activity: Admin → Settings → Payments → Shopify Payments → Activity shows no authorization for the transaction. The request never made it through to create an authorization record.
  • Multiple reports at the same time: Other buyers reporting the same issue in a narrow time window strongly indicates gateway-side latency, not an individual browser issue.
  • Shopify Payments status: Check shopifystatus.com for payment processing incidents. Gateway timeouts during incidents affect all merchants simultaneously.

Failure Mode 2: 3D Secure Authentication Redirect Deadlock

3D Secure (3DS) authentication adds a verification step between the buyer clicking "Place Order" and the payment being authorized. Shopify Payments initiates a 3DS challenge — typically a redirect to the buyer's bank or a popup — and waits for the authentication result before completing the payment authorization.

If the 3DS redirect opens and the buyer closes it, or if the authentication flow silently fails without sending a result back to Shopify's callback endpoint, checkout enters a deadlock: it is waiting for an authentication response that will never arrive. The spinner runs indefinitely.

3DS deadlock scenarios

  • Buyer closes the 3DS popup: The buyer doesn't recognize the bank verification screen, assumes it's a popup ad, and closes it. Checkout continues to wait for the authentication callback. The spinner never resolves because there is no timeout surfaced to the buyer.
  • Bank app push notification not received: For banks using app-based 3DS authentication (push notification in the banking app), if the buyer's phone is off or the app notification is missed, the challenge times out silently. Shopify waits for the result; the buyer waits at the spinner.
  • Browser blocks the 3DS redirect: Corporate network firewalls, aggressive ad blockers, or privacy-focused browsers (Firefox with strict settings, Brave) sometimes block the 3DS domain or iframe. The authentication flow never starts, but checkout still enters the waiting state.
  • 3DS callback URL unreachable: The 3DS flow sends an authentication result to a callback URL on Shopify's checkout domain. In rare cases — particularly with custom checkout domain configurations — the callback fails to reach Shopify's servers. The authentication completed on the buyer's side, but Shopify never received confirmation.
  • Expired 3DS session: 3DS authentication sessions have a time limit (typically 10 minutes). If a buyer leaves the checkout open for an extended period before clicking "Place Order," the 3DS session token may have expired. The challenge opens but authentication fails immediately — with no visible error to the buyer.

How to verify and fix 3DS deadlock issues

  1. Open browser DevTools (F12) → Network tab → filter by "XHR" or "Fetch" → reproduce the freeze → look for a request to a 3DS domain or to Shopify's payment processing endpoint that is stuck in "Pending" with no response
  2. Check if the affected buyers are predominantly on specific browsers (Safari, Firefox) or networks (corporate VPN) — this pattern points to a browser or firewall blocking the 3DS flow
  3. Test 3DS checkout in Shopify's test mode using a 3DS test card — Shopify Payments provides test card numbers that simulate 3DS authentication in test mode
  4. If using a third-party gateway alongside Shopify Payments, verify that gateway supports 3DS2 — 3DS1 has known deadlock issues with certain issuers during the sunset period
  5. Review any custom JavaScript running on your checkout that might be intercepting network requests or modifying the page during the 3DS redirect phase
Add Clear Instructions for 3DS Buyers
Add a message near the checkout payment section: "After clicking Place Order, your bank may ask you to verify the transaction in your banking app or via SMS. Please do not close any popups — they are from your bank, not from our store." This reduces buyer-initiated deadlocks (closing the 3DS popup) and improves 3DS completion rates by 15–30% according to payment processor data.

Failure Mode 3: Checkout UI Extension JavaScript Error

Shopify Checkout UI Extensions allow third-party apps to run JavaScript code within the checkout flow. Post-purchase upsell apps, loyalty point displays, custom field apps, donation widgets, and subscription upsell tools commonly use checkout extensions. When one of these extensions throws an uncaught JavaScript error during the order submission phase, it can freeze the checkout UI without surfacing any error message to the buyer.

This failure mode became more common after Shopify's transition to extensible checkout in 2023–2024. Extension errors are silent: the buyer sees a spinner, the extension has crashed in the background, and nothing in the checkout UI indicates what happened.

Identifying a checkout extension error

  • Browser console error: Open DevTools (F12) → Console tab → reproduce the freeze. A checkout extension error will appear as a JavaScript error referencing a checkout extension ID, an app name, or a Shopify extension runtime error code.
  • Issue disappears in incognito: Browser extensions aren't the cause here — but if the issue disappears in an incognito window with no apps loaded, the problem may also be a browser-side conflict. If it still freezes in incognito, the Shopify checkout extension is the likely cause.
  • Issue tied to a specific app install: If the freeze started after installing a new app, that app's checkout extension is almost certainly the cause. Disable the extension in Admin → Apps → [app name] → Extension settings.
  • Issue after a Shopify update: Shopify periodically updates its checkout JavaScript APIs. Extensions built against older API versions can break when Shopify deploys a checkout update — causing freezes that appear spontaneously without any merchant action.

How to isolate and fix a checkout extension freeze

  1. Admin → Settings → Checkout → Customize → review all installed checkout extensions and note which apps have active extensions
  2. Temporarily disable each extension one at a time (using the checkout editor's visibility controls) and test checkout after each disable to isolate the culprit
  3. Once identified, contact the app developer with the console error output and the Shopify checkout update version (visible in Shopify's changelog for your store)
  4. As an immediate fix, disable the problematic extension in the checkout editor — buyers will no longer see the extension's UI but checkout will function correctly
  5. After the app developer releases a fix, re-enable the extension and verify in a test checkout before re-enabling for live traffic
Extension Errors Are Silent — Buyers Never See What Went Wrong
Unlike a payment gateway error which at least shows "Your card was declined," a checkout extension error produces no buyer-facing message. The buyer sees a frozen spinner with no way to know if they should wait, retry, or use a different browser. This silently kills conversions. If you have multiple checkout extensions installed, establish a rotation of monthly test purchases through your own checkout to catch extension errors before they affect live buyers.

Failure Mode 4: Payment Authorized But Order Creation Failed

One of the most concerning checkout freeze scenarios: the payment authorization succeeds (money is reserved on the buyer's card) but something in Shopify's post-payment processing fails — and checkout hangs at the spinner because it is waiting for the order creation confirmation that never comes.

This scenario is rare but creates a poor outcome: the buyer sees a frozen checkout and may believe their payment went through (they may see a charge pending on their bank statement) but they receive no confirmation. The merchant sees no order in Admin.

Causes of payment-authorized / order-not-created failures

  • Post-purchase webhook timeout: An app subscribed to the orders/create webhook takes too long to respond, blocking the order creation pipeline. Shopify's webhook timeout is short — if the app endpoint doesn't respond within the window, the hook fails, but this doesn't always roll back the payment authorization cleanly.
  • Inventory reservation failure: For products with multi-location inventory, the inventory reservation step during order creation can fail if there is a conflict between locations, a mismatch between available inventory and what Shopify expected at checkout time, or a third-party inventory app that rejects the reservation.
  • App integration blocking order creation: Certain ERP integrations, fulfillment apps, or third-party order management systems subscribe to Shopify's order creation pipeline. If these integrations are misconfigured or experiencing issues, they can block or delay order creation.
  • Shopify internal order creation failure: In rare cases — typically during Shopify infrastructure degradation — the order creation service fails despite a successful payment authorization. This is a Shopify-side issue logged on their status page.

What to do when payment is authorized but no order exists

  1. Admin → Settings → Payments → Shopify Payments → Activity — look for authorization records that don't have corresponding orders. Note the authorization ID and amount.
  2. Contact Shopify Support with the authorization ID and the buyer's email — Shopify can investigate whether an order creation failure occurred and may be able to manually trigger order completion.
  3. Do NOT manually create a draft order for this buyer — it will charge them again. The original authorization needs to be resolved first.
  4. Email the buyer immediately: acknowledge that their payment information was received, explain that you are investigating a technical issue, and commit to a resolution timeline. Most buyers will wait 24 hours if communicated with promptly.
  5. After resolution, review your installed apps for any that subscribe to orders/create webhooks — these are the most common cause of order creation blocks.
Payment Hold vs. Capture — Understanding the Buyer's Bank Statement
When checkout freezes after authorization, the buyer sees a "pending" or "hold" charge on their bank statement — not a completed charge. Card authorization holds are typically released within 5–7 business days if no capture occurs. However, some buyers will call their bank reporting an unauthorized charge, triggering a chargeback process. Contact affected buyers proactively to explain the hold and provide either a completed order confirmation or confirmation that the hold will release — this prevents chargebacks from buyers who don't understand authorization holds.

Failure Mode 5: Shopify Payments Fraud Hold Delaying Order Confirmation

Shopify's fraud analysis system evaluates orders for risk signals before confirming them. For most orders, this analysis is instantaneous. For orders that trigger elevated fraud scores — high value, new customer, international shipping, unusual buying pattern — the analysis takes longer, and checkout remains in a "processing" state while Shopify's risk engine completes its evaluation.

This is not a bug — it is intended behavior. But it can last 30–120 seconds for high-risk orders, which is long enough for buyers to assume something went wrong and refresh or abandon.

Signs that fraud hold is causing the freeze

  • Order eventually appears: If a buyer waits long enough (or contacts support), the order does appear in Admin — but with a fraud warning flag on it. The order was created, just delayed.
  • High-value orders specifically: If the freeze primarily affects orders above a certain value threshold, fraud hold is the likely cause — Shopify's risk engine applies more intensive analysis to high-value transactions.
  • International orders with domestic sellers: Orders where the buyer's IP, billing address, and shipping address are in different countries trigger elevated fraud signals and longer processing times.
  • New customers at high values: A first-time purchase above $500 (or whatever your typical order value is) is more likely to trigger a fraud review than a repeat customer of the same value.

How to reduce fraud hold processing time

  1. Admin → Settings → Payments → Shopify Payments → Manage → Fraud prevention — review your fraud filter thresholds; overly aggressive settings increase the frequency and duration of fraud holds
  2. Consider using a third-party fraud prevention tool (Signifyd, NoFraud, Kount) — these provide pre-authorization fraud scoring that integrates with Shopify Payments, reducing the need for Shopify's post-authorization hold
  3. Add a message on your checkout page for high-value orders: "Large orders may take an additional moment to process. Please do not refresh or click Place Order again." This prevents double-submission from buyers who assume something went wrong.
  4. Enable Shopify's "order status" page — buyers who navigate to your store with their order number can see the processing status even if the thank-you page redirect failed
  5. Review flagged orders promptly in Admin — orders held for fraud review must be approved or declined by the merchant; leaving them in a held state too long can result in authorization expiry

Failure Mode 6: Network Interruption Mid-Submission

A buyer's network connection dropping or becoming unstable between clicking "Place Order" and receiving the confirmation response produces a checkout that appears frozen. The form has been submitted — the browser sent the request — but the response never arrived. The spinner continues running because the browser is still waiting.

Mobile buyers on cellular connections are most affected by this failure mode, particularly during handoff between cell towers, in areas with poor coverage, or when switching from WiFi to cellular. Desktop buyers on flaky WiFi or via VPN experience similar issues.

Diagnosing a network interruption freeze

  • Single buyer, single incident: Network interruptions produce isolated incidents, not patterns. Multiple buyers reporting the same freeze simultaneously is not a network issue — it's a gateway or infrastructure issue.
  • Mobile browser reports: If the reports cluster on mobile Safari or Chrome Mobile, network interruption during cellular handoff is a likely factor.
  • Order may or may not exist: Depending on when the connection dropped, the order may have been created (connection dropped during the redirect to the thank-you page) or may not have been created (connection dropped before the server received the full request).
Network Drop Timing Order Created? Buyer Experience Resolution
Before request reaches Shopify No Spinner, no charge Safe to retry — no risk of double-charge
After authorization, before order creation No Spinner, hold on card Check Payments Activity; retry with caution
After order creation, before redirect Yes Spinner, charge complete Email confirmation; order exists
Add an Order Lookup to Your Thank-You Page Template
One of the most effective ways to handle network-interruption freezes is to give buyers a way to find their order without the thank-you page redirect. Add an order status lookup at yourdomain.com/orders or include a "Check your order status" link in your checkout flow that buyers can use if the page freezes. Most buyers who can verify their order exists will move on without contacting support.

Failure Mode 7: Shopify Infrastructure Degradation

Shopify's checkout and order processing services experience degradation several times per year — typically brief (15–60 minute) periods of elevated latency or partial outages affecting some or all merchants. During degradation, checkout submission requests take longer, time out, or fail entirely. The frozen spinner is the buyer-facing symptom of Shopify's infrastructure returning errors or slow responses.

This failure mode is unique in that it is entirely outside the merchant's control — there is nothing in your store configuration, apps, or checkout setup that causes it, and there is nothing you can do to fix it except wait for Shopify to resolve the underlying infrastructure issue.

Identifying a Shopify infrastructure issue

  1. Check shopifystatus.com immediately — Shopify posts incident updates there, usually within 5–10 minutes of a degradation event being detected
  2. Check @ShopifyStatus on Twitter/X for real-time incident updates during active events
  3. Compare your orders volume in Admin → Analytics → Overview against your typical hourly rate — a sudden drop in orders during a period when checkout is freezing confirms the issue is platform-wide
  4. Contact Shopify Support and open a ticket — even if the status page shows no incident, your report contributes to their detection. Include the specific time the freeze started and the error (or lack thereof) buyers are seeing
  5. During an active incident, consider posting a brief message on your social media or sending a push notification to your email list: "We are experiencing a brief technical issue — orders will resume shortly. Your saved cart will remain available." This reduces the abandonment rate and preserves buyer intent during a short outage.
Schedule Big Sales Events for Shopify's Low-Traffic Windows
Shopify's checkout infrastructure runs at highest load during BFCM, major US shopping periods, and large coordinated sales. If you are planning a flash sale or product drop, avoid launching it at the same time as other major Shopify merchants (Friday at noon EST, Monday morning, the hour after BFCM opens). Even a 30-minute offset reduces your exposure to infrastructure load spikes. Failure Mode 8: Browser or Device Conflict

Certain browsers, browser extensions, privacy settings, and device configurations conflict with Shopify's checkout submission flow. Unlike the previous failure modes — which are caused by server-side or payment-side issues — browser conflicts are client-side: the freeze happens because something in the buyer's browser environment is blocking, intercepting, or corrupting the checkout submission request.

Browser conflicts are the hardest failure mode to diagnose as a merchant because they are invisible to your server logs. You see no request from the buyer at all — or you see a malformed request that your gateway rejects. The buyer sees a frozen spinner and assumes your store is broken.

Common browser and device conflict patterns

  • Safari Intelligent Tracking Prevention (ITP): Safari's ITP partitions cookies and restricts cross-origin network requests. If Shopify's payment form or 3DS flow uses cross-origin requests that ITP treats as tracking, the request is blocked or altered. ITP behavior changes with Safari updates — a freeze that appeared after a Safari update is a strong ITP signal.
  • Ad blockers blocking payment scripts: uBlock Origin, Privacy Badger, and similar extensions sometimes block JavaScript from Shopify's payment processing domains or third-party payment gateway scripts. With the payment script blocked, the "Place Order" click fires but nothing happens — the spinner appears because the form doesn't know the payment script failed to load.
  • Corporate network firewall: Corporate networks using deep packet inspection (DPI) or web filtering sometimes block specific Shopify or payment gateway domains. Buyers placing orders from work networks may experience systematic freezes that buyers on home networks don't.
  • Outdated browser or WebView: Very old browser versions that don't support features Shopify's checkout requires (specific JavaScript APIs, TLS 1.3) may fail to complete the checkout submission. Less common today, but still relevant for buyers on older Android devices using the default browser.
  • Custom browser settings: Buyers who have disabled JavaScript for third-party origins, enabled strict content security policies, or modified their browser's fetch/XHR behavior via extensions can break Shopify's checkout without any server-side indication.

How to guide buyers through browser conflict resolution

  1. Ask the buyer to try checkout in an incognito/private browsing window — this disables most browser extensions and resets cookie partitioning
  2. Ask the buyer to try a different browser (if they're on Safari, try Chrome; if on Firefox with strict settings, try a default configuration)
  3. Ask the buyer to temporarily disable ad blockers on your checkout page — or whitelist your checkout domain in their ad blocker settings
  4. If the buyer is on a corporate network, suggest they try on a mobile connection or home network
  5. For mobile buyers experiencing the issue, ask them to try on a WiFi connection rather than cellular, and to ensure their device's browser is updated to the latest version
Test Checkout in Multiple Browser Profiles Regularly
Run a test purchase monthly across Chrome (default), Safari (default), Firefox (strict tracking protection), and Chrome with uBlock Origin enabled. This proactively surfaces browser conflicts before they affect live buyers. Pay particular attention to the checkout flow after Safari updates — ITP changes in Safari are the most common browser-side cause of checkout freezes in Shopify stores.

How to Check if Checkout Is Stuck Processing Right Now

Manual testing is not enough to reliably catch checkout freezes. By the time a buyer reports a frozen checkout, the window of impact has often lasted 15–30 minutes. Here is a systematic approach to monitoring for processing freezes:

Immediate checks

  1. Go to Admin → Analytics → Overview → look at orders per hour for today vs. the same time yesterday. A sudden drop in order volume is often the first signal of a checkout freeze affecting all buyers.
  2. Check Admin → Settings → Payments → Shopify Payments → Activity for any authorization failures or gaps in the activity stream during the suspected freeze window.
  3. Run a test order yourself using Shopify's test payment mode (Admin → Settings → Payments → enable test mode → use test card 4242424242424242). Reproduce the exact checkout flow a buyer would use — including any discount codes, shipping options, and apps that live buyers would encounter.
  4. Run the same test order in Chrome incognito, Safari private, and on mobile — if the freeze reproduces in test mode, you have a confirmed processing issue you can debug before it affects more buyers.

Proactive monitoring

For stores where checkout downtime directly costs revenue, manual checks are insufficient. Automated checkout monitoring runs a synthetic test purchase on a scheduled interval — typically every 5–10 minutes — and alerts the merchant the moment the checkout submission step exceeds a defined latency threshold or fails to complete. This approach catches processing freezes within minutes of onset, before they affect a significant number of buyers.
