Shopify Checkout Language Not Switching — 8 Causes Explained

Store translated but checkout still shows English? Weglot or Langify not translating checkout? URL prefix /fr not switching checkout language? Markets language not carrying into checkout? Here are all 8 root causes of Shopify checkout language failures — for merchants and developers.

Updated June 2026 8 failure modes Checkout Language Settings · Translation Apps · URL Prefix · Markets Language · RTL · Checkout UI Extensions · Translate & Adapt · Headless i18n

Shopify's checkout language system is one of the most misunderstood parts of international store setup. Merchants translate their storefront theme — sometimes meticulously, with every string in multiple languages — only to find checkout still displays in English. The two most common assumptions that lead to this failure: that translation apps also translate checkout, and that URL prefixes like /fr or /de automatically switch checkout language.

Neither is true. Shopify checkout runs in a completely isolated sandbox on its own domain (checkout.shopify.com). No third-party JavaScript can execute inside it. And checkout language is determined not by the URL prefix the customer was browsing, but by the Shopify Market configuration — which requires two separate Admin steps that most merchants miss one of.

The Two Separate Language Systems — Most Important Thing to Know
Storefront language — controlled by theme translations (Translate & Adapt, manual Liquid edits, third-party apps). URL prefixes (/fr, /de) work here. Third-party translation apps work here.

Checkout language — completely isolated. Controlled by Admin → Settings → Languages → checkout language file. Third-party JS cannot reach here. URL prefixes don't control this. Markets language assignment controls checkout locale.

These two systems do not share settings or translation files. Translating one does not translate the other.

Failure Mode 1: Checkout Language Setting Is Separate from Storefront Theme Translations

This is the most common cause. A merchant translates their Shopify theme — adding French, German, Japanese, or Arabic translations to product titles, descriptions, navigation, and theme strings — and then expects checkout to appear in those languages. It doesn't.

Why they're separate

Shopify's storefront theme has its own translation files (accessible via the theme editor's "Edit default theme content" or via the Translate & Adapt app). These control text in the storefront: product pages, cart, navigation, footer. But checkout has a separate set of strings — button labels ("Continue to shipping," "Complete order"), field placeholders, error messages, shipping method display names, payment section labels — that live in a completely different translation file.

What You're Translating Where It Lives Affects Checkout?
Product names, descriptions, collection titles Translate & Adapt app / metafield translations Yes — product names show in checkout in the active market language if translated via Translate & Adapt
Theme strings (nav, footer, cart page text) Theme editor → Edit default theme content No — these are storefront-only
Checkout button labels, field placeholders, error messages Admin → Settings → Languages → checkout language Yes — but only if explicitly translated in this separate file
Shipping method names Admin → Settings → Shipping → [rate name] — translatable via Translate & Adapt Yes — if translated
Payment method display names (Shopify Payments) Controlled by Shopify — not merchant-editable per language Partially — Shopify localizes some payment names automatically based on market
How to Translate Checkout Strings
Admin → Settings → Languages → Find the target language → Click the three-dot menu → "Edit default theme content" (or "Translate") → Switch to the "Checkout and system" section. This is where checkout button labels, field error messages, and UI strings are controlled. Changes here affect checkout language for that locale.

Failure Mode 2: Third-Party Translation Apps Cannot Reach Inside the Checkout Sandbox

Weglot, Langify, GTranslate, Bablic, Transcy — all of these apps work by injecting JavaScript into your storefront theme. The script intercepts DOM mutations and text nodes, substituting translations in real time as the page renders. This approach is effective on storefront pages but completely ineffective for checkout.

Why the sandbox blocks them

Shopify's checkout runs on checkout.shopify.com (or a Shopify-controlled subdomain), completely separated from your storefront domain. This isolation exists primarily for PCI compliance: the payment step must be inaccessible to third-party scripts. Shopify explicitly prevents any storefront theme JavaScript from executing inside the checkout steps. The translation app's injected script never runs, so no translation substitution ever happens.

Translation App Support Pages Often Obscure This
Many translation apps claim to "translate the entire store including checkout." What they usually mean is that product names, which are translated at the Shopify data layer via metafields (not via JavaScript), do appear translated in checkout. The checkout UI strings — button labels, error messages, field instructions — remain in the language set in Admin → Settings → Languages, which the translation app cannot change from the storefront side. Always verify by checking an actual checkout flow in a different browser language, not just the storefront pages.

What translation apps can and cannot do at checkout

Product option/variant name translation Yes Same — metafield-based, accessible to checkout
Checkout button labels No Controlled by Shopify's internal checkout language, not accessible to third-party JS
Field error messages No Same

Failure Mode 3: URL Prefix (/fr, /de) Does Not Carry Language into Checkout

Shopify Markets assigns URL prefixes to different markets and languages. A store serving French customers in France might route them through mystore.com/fr where product titles, descriptions, and theme content all display in French. When those customers click "Check out," they move to Shopify's checkout domain — and the checkout language switches back to English.

Why this happens

The URL prefix (/fr, /de, /ja) is a storefront routing mechanism. It tells Shopify which market's settings to use for the storefront experience: currency, language, pricing rules. But the checkout is hosted separately. When a customer transitions from storefront to checkout, the checkout language is determined by the market's language assignment in the checkout system — not by the URL prefix they were browsing.

For the URL prefix to also switch checkout language, both of the following must be true:

  1. The target language must be installed in Admin → Settings → Languages → Add language → Publish. "Published" means active and visible, not just added.
  2. That language must be explicitly assigned to the market in Admin → Markets → [Market] → Languages → Add. This is the step most merchants miss because it's a second location entirely separate from the Languages settings page.
The Two-Step Trap — Why One Step Isn't Enough
Installing and publishing a language in Settings → Languages makes the language available for storefront translations. It does NOT automatically assign it to any market. Markets have their own language assignment. If you only complete step 1, the French URL prefix routes customers to French product pages but checkout reverts to English because no market is configured to use French at checkout. Both steps are independently required.

Failure Mode 4: Translate & Adapt App Handles Storefront Content — Checkout Strings Are Separate

Shopify's official translation tool — Translate & Adapt — is the recommended way to translate storefront content. It manages product metafield translations, theme content translations, and metaobject translations. Merchants who use it exclusively often assume their entire store, including checkout, is translated. The checkout UI strings are not.

What Translate & Adapt does and doesn't cover

Translate & Adapt translates:

  • Product titles, descriptions, options, variant names, and tags
  • Collection titles and descriptions
  • Page and blog content
  • Navigation menu items
  • Theme content strings (headings, button labels defined in the theme)
  • Shipping rate names and descriptions
  • Metaobject fields

Translate & Adapt does not translate:

  • Checkout button labels ("Continue to shipping," "Complete order," "Pay now")
  • Checkout field placeholders and labels ("First name," "Last name," "Address")
  • Checkout error messages ("Enter a valid address," "This card was declined")
  • Checkout progress indicators ("Contact," "Shipping," "Payment")
  • Order summary section labels within checkout
  • Payment method labels within checkout (partially Shopify-controlled)
How to Find the Checkout Translation File
In Translate & Adapt: open any language → scroll to the "Checkout and system" category in the left sidebar. This section contains all translatable checkout strings. It's separate from the theme translation section. Many merchants never notice it because it's not prominently labeled. All checkout UI strings that Shopify allows merchant customization of are in this section.

Failure Mode 5: Market Language Assignment Has Two Required Steps That Merchants Often Miss One Of

This is the most common root cause of checkout showing English for international customers when the storefront correctly shows the translated language. The fix requires changes in two different locations in Shopify Admin, and the relationship between them is not obvious from the UI.

Step-by-step: correctly assigning a language to a market for checkout

  1. Install the language: Admin → Settings → Languages → Add language. Select the target language (e.g., French). Click Add. The language now appears in the list as "Unpublished."
  2. Publish the language: Click the language → set to Published. This makes it available for use across the store but does not activate it for any market yet.
  3. Assign to a market: Admin → Markets → click the target market (e.g., "France" or "European Union") → Languages section → Add language → select the published language. Save.
  4. Verify: Open a private/incognito browser, navigate to the market's URL prefix (e.g., mystore.com/fr), add a product to cart, proceed to checkout. The checkout language should now match the assigned language.
Multiple Markets and Language Overlap
If you serve multiple French-speaking markets (France, Belgium, Canada), you must assign French to each market individually. Assigning a language to the France market does not automatically assign it to Belgium or Canada. Also: if a customer visits from a country that doesn't match any market, they fall into the "International" (default) market, which typically uses the store's primary language — usually English. Customers in countries outside any configured market always see the default checkout language.

Failure Mode 6: RTL Language Checkout Layout Breaks Without Proper Locale Configuration

Arabic, Hebrew, Persian, and Urdu are right-to-left languages. When a store serves customers in RTL language markets and checkout does not have the correct RTL locale configured, the checkout layout renders left-to-right with RTL text: field labels align to the left while text flows right, button positions are mirrored incorrectly, and the progress bar displays in the wrong direction. The text is translated but the interface is unusable.

How Shopify handles RTL checkout

Shopify checkout natively supports RTL layout when the correct RTL locale is active at the checkout level — meaning the market has an RTL language (Arabic: ar, Hebrew: he, Persian: fa) properly assigned following the two-step process above. When the correct locale is active, Shopify's checkout UI automatically applies dir="rtl" to the checkout container and adjusts layout.

CSS Overrides Don't Reach Checkout
Merchants sometimes add direction: rtl; unicode-bidi: embed; CSS to their storefront theme to handle RTL Arabic or Hebrew text. This CSS applies to the storefront but has zero effect on checkout because checkout renders in an isolated domain. The only way to get RTL layout in checkout is through Shopify's own locale system — not through theme CSS.

Diagnosing RTL checkout issues

Symptom Likely Cause Fix
Text is Arabic but layout is left-to-right Arabic language installed but not published + assigned to market Complete both language install steps; assign Arabic to the relevant market
Checkout uses some Arabic words but UI elements are mixed direction Partial translation — some strings translated, RTL locale not active Ensure locale code is ar not a custom workaround; verify market assignment

Failure Mode 7: Checkout UI Extensions Hardcode English Strings and Bypass the Language System

Shopify Plus merchants using Checkout Extensibility (Checkout UI Extensions) can add custom UI components at checkout steps — banners, text blocks, validation messages, custom fields, upsell offers. If these extensions hardcode English strings or use JavaScript internationalization libraries without proper locale context, they display English regardless of the checkout language the customer is experiencing.

The correct approach for localized Checkout UI Extensions

Shopify's Checkout UI Extensions framework provides a useTranslations() hook (from @shopify/ui-extensions-react/checkout) and a locale-aware translation system. Extensions should define translation files for each supported language and use the hook to retrieve the correct string at render time.

// In your extension component
import { useTranslations } from '@shopify/ui-extensions-react/checkout';

function MyExtension() {
  const { i18n } = useTranslations();

  return (
    <Banner title={i18n.translate('banner_title')}>
      {i18n.translate('banner_body')}
    </Banner>
  );
}

// In locales/en.json
{
  "banner_title": "Free shipping on orders over $50",
  "banner_body": "Add more items to qualify."
}

// In locales/fr.json
{
  "banner_title": "Livraison gratuite pour les commandes supérieures à 50 €",
  "banner_body": "Ajoutez d'autres articles pour vous qualifier."
}
Date and Number Formatting in Extensions
Extensions that display dates, prices, or quantities must also use locale-aware formatting. JavaScript's new Date().toLocaleDateString() without a locale parameter uses the system locale of the server rendering the extension — which may differ from the customer's checkout locale. Use i18n.formatDate(), i18n.formatNumber(), and i18n.formatCurrency() from the Checkout UI Extensions framework to ensure locale-correct output for all customers.

Common internationalization mistakes in Checkout UI Extensions

  • Hardcoded English string literals in JSX — <Text>Free gift with orders over $50</Text> — always English regardless of customer locale
  • Using JavaScript Intl.NumberFormat without locale — new Intl.NumberFormat().format(price) uses server locale, not customer locale
  • Missing locale files for some languages — extension falls back to English when a translation key is missing; add fallback strings to all locale files
  • Not testing extensions with non-English market configurations — extension appears correct in merchant's English admin but fails for international customers

Failure Mode 8: Headless Storefront — @inContext(language:) Required in Every Storefront API Query, Plus Markets Configuration Required for Checkout

Headless Shopify storefronts must handle language explicitly at two independent layers: the Storefront API queries (for content displayed in the custom frontend) and the Shopify Markets configuration (for the checkout that opens after the cart). Missing either layer produces a partially translated experience.

Layer 1: Storefront API language context

Every Storefront API query returns content in the store's default language unless the @inContext directive specifies a language:

# WITHOUT language context — always returns default language
query ProductQuery($handle: String!) {
  product(handle: $handle) {
    title         # Returns "Blue T-Shirt" even on French storefront
    description   # Returns English description
  }
}

# WITH language context — returns translated content
query ProductQuery($handle: String!) @inContext(language: FR, country: FR) {
  product(handle: $handle) {
    title         # Returns "T-Shirt Bleu" if translated
    description   # Returns French description if translated
  }
}

The language value in @inContext must be an ISO 639-1 language code in uppercase: FR, DE, JA, AR, etc. The country value (ISO 3166-1 alpha-2, also uppercase) determines which Shopify Market's pricing and currency applies. Both can be set independently.

Layer 2: Markets configuration for checkout language

After a customer completes checkout initiation from a headless storefront (typically by calling checkoutCreate or cartCreate + checkout redirect via the Storefront API), they land on Shopify's hosted checkout page. The language on that checkout page is determined by the Shopify Market assigned to their country and the language configured for that market — not by anything set in the Storefront API query context.

The Headless Language Split — Most Common Confusion
A headless storefront can display perfectly in French (using @inContext(language: FR) on all queries) while checkout still shows in English (because the France market doesn't have French assigned in Admin → Markets → France → Languages). The two systems are completely independent. Fixing the Storefront API language context does nothing for checkout language. Fixing the Markets language assignment does nothing for Storefront API content. Both must be configured correctly.

Headless language implementation checklist

Component Where to Configure What It Controls
Storefront content language @inContext(language: XX) on every Storefront API query Product titles, descriptions, collection names, metafield translations
Storefront currency and pricing @inContext(country: XX) on every Storefront API query Prices, currency display, market-specific pricing rules

// Correct cart creation with language context for headless
const CART_CREATE = `
  mutation CartCreate($input: CartInput!) @inContext(language: $language, country: $country) {
    cartCreate(input: $input) {
      cart {
        id
        checkoutUrl  # This URL opens Shopify checkout — language is from Markets config

        cost {
          totalAmount { amount currencyCode }
        }
      }
    }
  }
`;

// Note: checkoutUrl language is controlled by Markets, not by @inContext here
// The @inContext here affects prices and product name translations in the cart query
// To get translated checkout, configure the market in Admin → Markets → Languages
