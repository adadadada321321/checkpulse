Shopify Checkout SSL Error — Custom Domain, Cloudflare Conflict, Mixed Content & Certificate Invalid

Updated June 2026 Shopify Basic · Shopify · Advanced · Plus 3 root causes · 8 diagnostic steps

On standard Shopify plans, checkout SSL is Shopify's responsibility — merchants don't control it. Most "SSL error at checkout" reports come from one of three sources: Cloudflare proxy mode conflicts with Shopify's checkout servers, a Shopify Plus custom checkout domain that isn't provisioned yet, or mixed-content warnings from third-party apps injecting HTTP assets into checkout pages.

How Shopify Checkout Domains Actually Work

The most important thing to understand about Shopify checkout SSL: on every plan except Plus, the checkout is not served from your domain at all. When a customer clicks "Checkout" from your cart, the browser navigates to Shopify's infrastructure — not to your store's web server.

Basic · Shopify · Advanced

checkout.shopify.com

Checkout is hosted on Shopify's domain. The URL looks like checkout.shopify.com/shop/[id]/checkouts/[token]. SSL is Shopify's certificate — you cannot customize it. SSL errors here are Shopify's issue, not yours.

Shopify Plus — Optional

checkout.yourdomain.com

Plus merchants can request a custom checkout subdomain (e.g., checkout.mystore.com). Shopify provisions the SSL certificate. Requires CNAME DNS setup and 24–48 hour provisioning. Not self-serve.

All Plans — Store Pages

yourdomain.com (store)

Your store pages (product, cart, home) are served from your custom domain. SSL for this domain is managed by Shopify automatically — a Let's Encrypt certificate is provisioned when you add your domain in Shopify admin.

Common Misconception

SSL at checkout ≠ your domain's SSL

Your store domain can have a perfect SSL certificate and your checkout can still show an SSL error — because checkout runs on a different server and domain entirely. The two SSL systems are separate.

Why this matters: If your store is on a standard Shopify plan and a customer reports "the checkout page says Not Secure," the SSL certificate for checkout.shopify.com is not something you can renew, replace, or reconfigure. The fix is almost always on the Cloudflare side (if you use it) or in a third-party app injecting HTTP content — not in Shopify admin.

Three SSL Error Types and Their Causes

Browser SSL errors at Shopify checkout fall into three categories. The error code narrows down which cause applies.

Type 1 — Certificate Mismatch (NET::ERR_CERT_COMMON_NAME_INVALID)

The browser sees a certificate that doesn't match the domain it's connecting to. At Shopify checkout this happens when: a Plus custom checkout domain is configured but the SSL certificate hasn't finished provisioning, the CNAME record points to the wrong target, or Cloudflare is proxying a checkout subdomain with its own certificate that doesn't match what Shopify's server presents.

On standard plans, this error almost never appears because the checkout domain is checkout.shopify.com — a Shopify-controlled domain with a valid wildcard certificate. If you see this error on a standard plan store, check if a browser extension or corporate network proxy is intercepting HTTPS traffic.

Type 2 — Mixed Content Warning (Shield Icon / "Not Fully Secure")

The page loads over HTTPS but resources inside it (images, scripts, stylesheets) load over plain HTTP. Browsers block or warn about this. Mixed content at checkout is almost always caused by third-party apps that inject assets with hardcoded HTTP URLs — trust badge widgets, loyalty point displays, upsell widgets, or checkout extension assets that use non-HTTPS image or script sources.

Mixed content is shown differently across browsers: Chrome shows a shield with a slash in the address bar and "Your connection to this site is not fully secure." Safari shows a grey lock instead of green. Firefox shows a broken lock. Opening DevTools → Console reveals the exact HTTP URLs being blocked.

Type 3 — Protocol / Connection Error (ERR_SSL_PROTOCOL_ERROR)

A lower-level failure where the SSL handshake itself fails — no certificate is even exchanged. At Shopify checkout, this is almost exclusively a Cloudflare configuration problem: Flexible SSL mode with misconfigured HTTPS redirect rules, a Cloudflare Page Rule that forces HTTPS but redirects to a non-HTTPS target, or a Cloudflare SSL/TLS setting that's incompatible with Shopify's server configuration.

Redirect loops and SSL errors often coexist. If Cloudflare is set to "Flexible" SSL and you've also enabled "Force HTTPS" in Shopify, you can get a redirect loop (HTTP → HTTPS → HTTP → …) that some browsers report as an SSL error rather than a redirect loop error. Check for both issues together.

Cloudflare SSL Conflicts with Shopify Checkout

Cloudflare is involved in more Shopify SSL errors than any other single cause. The combination of Cloudflare proxy + Shopify checkout creates several failure modes that are not obvious from either platform's documentation.

The Flexible SSL Trap

Cloudflare's "Flexible" SSL mode encrypts traffic between the visitor and Cloudflare, but fetches from Cloudflare to your origin (Shopify's servers) over plain HTTP. This mode is designed for origins without SSL certificates, which describes none of Shopify's servers — they all have valid certificates. Using Flexible SSL with Shopify creates a broken SSL chain that manifests as errors for some visitors, especially those on stricter browsers or corporate networks.

Fix: Cloudflare → SSL/TLS → Overview → change to "Full" or "Full (Strict)". Full (Strict) is recommended and requires the origin (Shopify) to have a valid publicly-trusted certificate, which it does.

Orange Cloud on the Checkout Subdomain (Plus)

If you're a Plus merchant with a custom checkout domain (e.g., checkout.mystore.com) and that DNS record is set to orange-cloud (proxied) in Cloudflare, two problems arise: (1) Shopify's ACME certificate provisioning challenge cannot reach the actual server to complete verification, so the SSL certificate is never issued. (2) If you have a Cloudflare origin certificate, Cloudflare presents that to visitors, but Shopify's checkout servers also try to present their certificate — the mismatch causes errors.

Fix: In Cloudflare DNS, set the checkout subdomain record to grey cloud (DNS-only). The checkout subdomain does not need Cloudflare's proxy features — Shopify handles the performance layer on its own infrastructure.

Always Use HTTPS + Redirect Misconfiguration

Cloudflare's "Always Use HTTPS" setting redirects any HTTP request to HTTPS. When combined with Shopify's own "Redirect to HTTPS" setting and Shopify's internal redirects (e.g., from your store to checkout.shopify.com), you can create redirect chains that exceed browser limits. Each redirect in the chain adds latency; if any redirect in the chain hits an HTTP URL before the final HTTPS destination, browsers may flag it.

Fix: Enable "Always Use HTTPS" in Cloudflare at the SSL/TLS → Edge Certificates level. Enable "Redirect to HTTPS" in Shopify (Settings → Domains). Disable any Cloudflare Page Rules that manually do HTTP→HTTPS redirects — let the platform settings handle it, not rules that can conflict.

Cloudflare SSL Settings Reference for Shopify

Cloudflare Setting Recommended Value Why

Shopify Plus Custom Checkout Domain Setup

Shopify Plus merchants can have checkout served from their own subdomain (e.g., checkout.yourbrand.com instead of checkout.shopify.com). This is not a self-service feature — Shopify configures it on their end, and you add the corresponding DNS record.

Setup Process

1
Request the custom checkout domain via Shopify Plus support

Contact Shopify Plus support and specify the subdomain you want (e.g., checkout.mystore.com). Shopify will configure their servers to respond to that subdomain and initiate certificate provisioning.

2
Add the CNAME record in your DNS provider

Shopify will provide the CNAME target (typically shops.myshopify.com or a similar Shopify endpoint). Add a CNAME record: checkout → [Shopify's target]. Set TTL to 300 seconds for fast propagation.

3
If using Cloudflare: set to grey cloud (DNS-only)

The checkout subdomain CNAME record must not be proxied. Set the Cloudflare proxy toggle to grey cloud (DNS-only) for this record. Shopify's provisioning bot needs direct access to the server to complete the ACME challenge.

4
Check your CAA DNS records

If you have CAA records on your domain, verify that letsencrypt.org is allowed as a certificate authority. Run dig CAA yourdomain.com. If CAA records exist but don't include Let's Encrypt, Shopify's provisioning will fail silently.

5
Wait 24–48 hours for SSL provisioning

After the CNAME propagates, Shopify's provisioning system issues a Let's Encrypt certificate. This typically completes within a few hours but can take up to 48 hours. During this window, customers will see a certificate error if they visit checkout. Do not launch publicly until provisioning is confirmed.

6
Verify the certificate

Once provisioning completes, run curl -vI https://checkout.yourdomain.com and confirm the SSL certificate is issued to checkout.yourdomain.com with a Let's Encrypt issuer. Or use an SSL checker tool to confirm the certificate chain is valid.

Do not announce your custom checkout domain before SSL provisioning is confirmed. Any customer who reaches the checkout URL before the certificate is issued will see a full-page certificate error. SSL warnings create permanent trust damage — a customer who saw "Not Secure" may not return even after the issue is fixed.

Why Provisioning Fails

  • CNAME not propagated: DNS changes can take minutes to hours. Verify propagation with dig checkout.yourdomain.com CNAME before expecting the certificate.
  • Cloudflare orange cloud: Blocks the ACME HTTP-01 challenge. Must use grey cloud (DNS-only) on the checkout subdomain.
  • CAA records blocking Let's Encrypt: Restrictive CAA records on the parent domain prevent issuance. Add 0 issue "letsencrypt.org" CAA record if needed.
  • Wrong CNAME target: Shopify's required CNAME target may differ between stores. Confirm the exact target with Shopify Plus support for your specific setup.
  • Rate limits: Let's Encrypt has rate limits per domain. If you've attempted provisioning repeatedly (after configuration errors), you may hit a rate limit. Wait before retrying.

Mixed Content Errors at Checkout

Mixed content is the most common SSL-adjacent issue on standard Shopify plans. It doesn't prevent checkout from loading, but it triggers browser security warnings that erode customer trust at the worst possible moment.

What Causes Mixed Content at Shopify Checkout

Shopify's checkout page itself loads over HTTPS — Shopify controls that. Mixed content comes from resources injected into the checkout by third parties:

  • Checkout Extensions (UI Extensions): Apps using Checkout Extensibility can inject UI elements into checkout. If the extension loads remote images or scripts over HTTP, the browser flags them as mixed content.
  • Checkout Scripts (Plus only — deprecated): Legacy checkout.liquid scripts that reference HTTP URLs for trust badges, rating widgets, or analytics pixels.
  • Buy Button embeds: If your store uses Shopify's Buy Button on an external site that loads over HTTPS, any HTTP assets in the Buy Button widget cause mixed content on the embedding page.
  • Order status page customizations: The order confirmation page (thank you page) is customizable. Scripts or images added via "Additional scripts" in Checkout settings may use HTTP URLs.

How to Find Mixed Content

1
Open Chrome DevTools at checkout

Navigate to your checkout page. Press F12 (Windows) or Cmd+Option+I (Mac) to open DevTools.

2
Check the Console tab for "Mixed Content" errors

Look for red error lines starting with "Mixed Content: The page at 'https://checkout.shopify.com/...' was loaded over HTTPS, but requested an insecure resource 'http://...'" — the URL in quotes is the HTTP resource causing the issue.

3
Check the Network tab → filter by non-secure

In the Network tab, filter by "http://" in the filter box. Any remaining requests are loading over HTTP instead of HTTPS.

4
Trace the source to the responsible app

The HTTP URL typically includes a CDN or app domain (e.g., http://assets.trustbadgeapp.com/badge.png). Contact that app's support to update their asset URLs to HTTPS, or disable the app and see if the warning clears.

Order status page scripts: Shopify admin → Settings → Checkout → scroll to "Order status page" → "Additional scripts." Any HTTP URLs in custom scripts here appear as mixed content on the thank-you page (part of the checkout flow). Replace all http:// with https:// in these scripts.

What "Force SSL" in Shopify Admin Actually Does

There's significant confusion about the "Redirect all traffic to this domain" / "Force HTTPS" toggle in Shopify admin → Settings → Domains. Merchants often try this setting when they see checkout SSL warnings, expecting it to fix the problem. It doesn't — and understanding why clarifies the entire SSL architecture.

What It Does

Enabling Force HTTPS in Shopify causes Shopify's servers to respond to any HTTP request to your store domain with a 301 redirect to the HTTPS version. A request to http://mystore.com/products/shirt is redirected to https://mystore.com/products/shirt.

This only applies to your store's main domain (product pages, collections, blog, homepage, cart page). It has no effect on the checkout domain, which is already served exclusively over HTTPS by Shopify regardless of this setting.

What It Doesn't Do

  • Does not affect checkout.shopify.com — Shopify controls that domain's HTTPS enforcement independently
  • Does not fix certificate errors if your domain's SSL certificate is not yet provisioned
  • Does not affect mixed content warnings from third-party app assets
  • Does not interact with Cloudflare's SSL settings — both can be active simultaneously without conflict, but Cloudflare's redirect rules should be removed to avoid loops
Best practice: Enable "Force HTTPS" in Shopify admin for your store domain. Enable "Always Use HTTPS" in Cloudflare (if using Cloudflare). These two settings work in parallel — Shopify handles store pages, Cloudflare handles the edge. Neither affects checkout SSL.

8-Step SSL Diagnostic for Shopify Checkout

1
Identify which domain is showing the SSL error

Look at the URL in the browser address bar when the error appears. Is it checkout.shopify.com (standard plan), checkout.yourdomain.com (Plus custom), or yourdomain.com (store pages)? The fix path depends on the domain.

2
Check if you're using Cloudflare

Log in to Cloudflare and verify the SSL/TLS mode (Overview tab). If it's "Flexible," change to "Full (Strict)." If the checkout subdomain (Plus) has orange cloud, change to grey cloud.

3
Check Shopify's SSL status for your store domain

Shopify admin → Settings → Domains → click your domain → check the SSL certificate status. If it shows "Pending" or expired, contact Shopify support — your store domain's SSL is managed by Shopify automatically but can occasionally need manual re-provisioning.

4
For Plus custom checkout domain: verify CNAME

Run dig checkout.yourdomain.com CNAME. Verify the CNAME target matches what Shopify Plus support provided. If not propagated yet, wait — don't retry provisioning during propagation.

5
Check for CAA records (Plus custom domain)

Run dig CAA yourdomain.com. If CAA records exist, confirm letsencrypt.org is in the list. If not, add 0 issue "letsencrypt.org" and wait for provisioning to retry.

6
Open browser DevTools → check Console for mixed content

Navigate through checkout to the payment step while DevTools is open. Mixed content warnings appear as red console errors listing HTTP URLs. Note which app or domain is injecting the HTTP resource.

7
Test from an incognito window on a mobile device

Browser extensions can inject HTTP assets and cause mixed content warnings that don't exist for real customers. Testing in incognito with no extensions rules out extension interference.

8
Check Shopify's status page for platform-wide SSL issues

If the error appeared suddenly without any configuration change, check Shopify's status page. Platform-wide certificate renewal issues (rare but they happen) affect all merchants simultaneously and require Shopify's engineering team to resolve.

FAQ

Why does my Shopify checkout go to checkout.shopify.com instead of my domain?
On Shopify Basic, Shopify, and Advanced plans, checkout is hosted on Shopify's domain — this is by design for PCI compliance and payment infrastructure reasons. Only Shopify Plus merchants can configure a custom checkout subdomain, and even then it requires Shopify's team to set up on their servers. Changing the checkout URL on a standard plan is not possible through any setting, app, or DNS configuration.
Will Cloudflare break my Shopify checkout SSL?
Cloudflare won't break checkout SSL on standard plans if you use Full (Strict) SSL mode — checkout.shopify.com isn't routed through Cloudflare at all on standard plans (Cloudflare only proxies your store domain). On Shopify Plus with a custom checkout subdomain, Cloudflare must be set to grey cloud (DNS-only) on the checkout record. If the checkout subdomain is orange-cloud proxied, certificate provisioning fails and customers see SSL errors.
What causes "Your connection to this site is not fully secure" at Shopify checkout?
This indicates mixed content — one or more resources on the checkout page are loading over HTTP instead of HTTPS. The checkout page itself is always HTTPS; the mixed content comes from third-party apps or scripts injecting non-secure assets. To find the specific cause: open Chrome DevTools (F12), go to Console, and look for "Mixed Content" error lines that list the HTTP URL. The domain in that URL identifies which app or script is the source.
How long does SSL provisioning take for a new Shopify domain?
When you add a new custom domain to Shopify (Settings → Domains → Connect existing domain), Shopify provisions a Let's Encrypt SSL certificate automatically. This typically completes within a few hours, but can take up to 48 hours in some cases. During this window, your store shows an SSL error for visitors who navigate to the HTTPS URL. Shopify admin → Settings → Domains shows the provisioning status. Do not redirect traffic to the new domain until SSL is confirmed as active.
Does enabling "Force SSL" in Shopify admin fix checkout SSL errors?
No. The Force SSL / Redirect to HTTPS setting in Shopify admin only affects your store domain's HTTP→HTTPS redirect for store pages. Checkout is already served exclusively over HTTPS regardless of this setting — it's on Shopify's infrastructure, not yours. If you're seeing SSL errors specifically at checkout (the checkout.shopify.com URL or your Plus custom checkout domain), the cause is elsewhere: Cloudflare configuration, an unprovisioned custom checkout domain, or mixed content from app assets.
