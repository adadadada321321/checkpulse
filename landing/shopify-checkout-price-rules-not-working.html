Shopify Price Rules Not Working — Automatic Discount Not Applying, BOGO Not Triggering, Volume Pricing Broken, Scripts Deprecated

Shopify automatic discounts silently skip the cart when any one condition is unmet, BOGO requires exact variant-level matching, discount codes disappear when any automatic discount is active, and Shopify Scripts — the Plus tool for complex tiered pricing — are being deprecated. Here are all 8 failure modes.

📅 Updated June 2026 ⏱ 10 min read 💡 Applies to: All Shopify plans
Automatic Discounts Fail Silently — No Error Is Shown to the Customer
When a Shopify automatic discount condition is not met — wrong minimum purchase, ineligible products, wrong customer segment — the discount simply doesn't appear at checkout. There is no error message. The customer doesn't know a discount was supposed to apply. This is the most common reason merchants receive support tickets asking "why didn't my sale apply?"

How Shopify Price Rules and Discounts Work (And Where They Break)

Shopify has three distinct discount mechanisms, each with different behavior, conditions, and failure modes:

Mechanism How Applied Plans Stackable?
Discount codes Customer enters code at checkout All plans One code per order (non-Plus)
Automatic discounts Applied automatically when conditions are met All plans One automatic discount per order (non-Plus)
Shopify Scripts Ruby scripts run server-side in checkout Plus only — being deprecated Fully customizable — can stack
Shopify Functions WebAssembly functions replace Scripts Plus only — replacement for Scripts Multiple Functions can run
Draft order price overrides Merchant sets line item prices manually All plans N/A — manual pricing

The critical non-Plus constraint: Shopify allows only one discount per checkout — either one discount code or one automatic discount, not both. When any automatic discount is active, the discount code input field is hidden. When a code is applied, automatic discounts that would otherwise apply do not stack on top.

Failure Mode 1 — Automatic Discount Conditions Silently Not Met

Shopify automatic discounts evaluate against several condition sets simultaneously. Every condition must be true for the discount to apply. Common silent failures:

Minimum purchase amount not reached

Shopify calculates the minimum purchase against the cart subtotal before any discounts. But there's a catch: if another discount has already been applied (e.g., a cart-level discount from a third-party app), that discount may reduce the subtotal below the threshold, causing the automatic discount to silently not apply even though the customer's original total met the requirement.

Subtotal Calculation Order Matters
Shopify evaluates the minimum purchase amount against the pre-discount subtotal for most scenarios, but the interaction with line-item discounts from apps (using the Admin API applied_discount on draft orders) can reduce the effective subtotal. If you have apps modifying line-item prices, test your automatic discount thresholds with those apps active.

Eligible products or collections condition

If your automatic discount is configured to apply only to specific products or collections, the customer's cart must contain at least one item from the eligible set. Common mistakes:

  • Product is in an eligible collection but is also tagged for exclusion in a third-party discount app — the app may be overriding Shopify's native check
  • Product was recently added to a collection but the collection update hasn't propagated to the discount evaluation cache (rare, but can cause a few minutes of incorrect behavior)
  • Discount configured to "Specific products" but specific product variants were not selected — Shopify's "specific products" picker selects variants, not just the parent product title

Customer eligibility mismatch

When the discount is set to "Specific customer segments," Shopify evaluates the logged-in customer's segment membership at the time of checkout. Guest customers (not logged in) are evaluated against the "All customers" condition. If your discount is restricted to a segment like "Customers who have placed at least 1 order" and the customer is checking out as a guest, the discount won't apply even if they have a purchase history.

  1. Go to Admin → Discounts → your automatic discount → check "Customer eligibility"
  2. If set to "Specific customer segments," verify the customer is logged in (guests cannot match segments)
  3. Check Admin → Customers → find the customer → verify they appear in the expected segment
  4. If the segment was recently updated, wait a few minutes for Shopify to sync segment membership

Failure Mode 2 — Discount Code Field Disappears When Automatic Discount Is Active

This is one of the most confusing behaviors in Shopify's checkout for non-Plus merchants. When any automatic discount applies to the cart, Shopify removes the discount code input field entirely from the checkout page. Customers who arrive with a coupon code have no way to enter it.

This affects every common sale + code combination:

  • Sitewide automatic sale (e.g., "15% off everything") + influencer code (e.g., "SARAH15") — impossible on non-Plus
  • Free shipping automatic discount + first-order coupon code — impossible on non-Plus
  • Automatic BOGO offer + loyalty redemption code — impossible on non-Plus
Customers Won't Report This — They'll Just Abandon
Customers who arrive from an influencer's link with a promise of a discount code simply see no code field. Most won't contact support — they'll quietly abandon and assume the code was fake or expired. High abandonment on influencer traffic during sale periods is often caused by this conflict.

Workarounds for non-Plus stores

  • Use a cart permalink instead of an automatic discount: Create a discount code for the influencer and share a URL like yourstore.com/discount/SARAH15?redirect=/collections/all — this pre-applies the code to the cart without requiring an automatic discount
  • Replace the automatic discount with a widely-shared code: Instead of an automatic sitewide sale, create a public discount code (e.g., "SALE15") and share it visibly on the site — this keeps the code field visible
  • Upgrade to Plus: Shopify Plus with Shopify Functions allows custom discount logic that can stack multiple discounts

Failure Mode 3 — BOGO (Buy X Get Y) Not Triggering

Shopify's Buy X Get Y discount type has specific evaluation logic that surprises merchants who expect it to work like a simple "buy one product, get another" offer.

Variant-level matching, not product-level

When you configure a BOGO with "Specific products," Shopify evaluates at the variant level. If you add "Blue T-Shirt" as the trigger product and the customer adds "Blue T-Shirt — Size L," Shopify checks whether the variant "Blue T-Shirt — Size L" is in the eligible set. If you added the parent product without selecting all variants, some variants may be excluded.

To verify: Admin → Discounts → your BOGO → click the "Customer buys" product → confirm all intended variants are shown as selected.

Minimum quantity threshold not met by eligible items only

The "Customer buys at least X" quantity count is fulfilled only by eligible items. If a customer has 3 items in their cart but only 1 is the eligible trigger product, and your BOGO requires "buys at least 2," the trigger condition is not met. Non-eligible items don't count toward the quantity threshold.

The "Y" item is removed and not re-added

When Shopify BOGO is configured to "add a product for free" (rather than "discount a product"), Shopify can automatically add the Y item to the cart when conditions are met. If the customer removes this automatically-added item, Shopify does not continuously re-add it. The discount condition is still met, but the free product is absent from the cart. The customer may complete checkout without the free item and contact support afterward.

BOGO With "Percentage Off" Is More Reliable Than "Free Product"
Instead of configuring BOGO to add a free product, configure it as "Customer gets Y products at X% off" where X is 100%. The customer must have the Y product already in their cart, but there's no silent removal behavior. This is more predictable for both the merchant and the customer.

Maximum uses per order cap reached

BOGO discounts can be configured with a maximum number of uses per order (e.g., "applies up to 3 times per checkout"). If a customer buys 6 of the trigger item expecting 3 free Y items, but the cap is set to 1, only 1 free item is granted. The cap setting is at Admin → Discounts → your BOGO → "Maximum discount uses" → "Limit to X uses per order."

Failure Mode 4 — Volume Pricing / Tiered Pricing Not Working

Shopify has no native volume or tiered pricing on any plan. There is no "buy 1-4 at $10, buy 5-9 at $9, buy 10+ at $8" feature built into Shopify's Admin. Every implementation requires a third-party app or, on Plus, Shopify Scripts (deprecated) / Shopify Functions.

Third-party volume pricing apps — failure modes

App Mechanism How It Works Common Failure Mode
Price rules + discount code App creates a Shopify discount code applied automatically at checkout Conflicts with merchant's manual discount codes (one per order limit)
Draft order pricing App intercepts checkout, creates a draft order with custom line item prices Breaks express checkout (Shop Pay, Apple Pay, Google Pay bypass draft order flow)
Storefront JavaScript price display JS modifies displayed price on product page and cart Visual only — checkout price uses Shopify's actual price, not the displayed override
Shopify Scripts (Plus) Ruby script runs in checkout, modifies line item prices server-side Being deprecated — Scripts must be migrated to Shopify Functions before deprecation deadline
Shopify Functions (Plus) WebAssembly functions apply discount logic in checkout Requires Plus; complex setup; Function must be deployed via Shopify CLI

The most dangerous failure is "storefront JavaScript price display" — the customer sees a lower price on the product page and in the cart, but Shopify's actual checkout uses the product's base price. The customer pays full price at payment, often not noticing until they check their email receipt.

JavaScript Price Modification Is Not Actual Pricing
Several volume pricing apps modify the visual display of prices using JavaScript without actually changing the Shopify price. When the customer proceeds to checkout, Shopify calculates the order total using the product's actual Shopify price, not the JS-modified display price. If a volume pricing app isn't explicitly using discount codes, draft orders, or Shopify Scripts/Functions, it may be display-only. Test by completing a test checkout and verifying the actual charged amount.

Failure Mode 5 — Usage Limit Exhausted

Shopify discount codes can be configured with two types of usage limits:

  • Total usage limit: The code can be used a maximum of N times across all customers. Once exhausted, new uses return "This discount code is invalid."
  • Per-customer usage limit: A single customer can use the code at most N times. Set to 1 for "one-time use per customer" behavior.

A common scenario: a merchant creates an influencer code with a total usage limit of 100 for a campaign. The influencer's audience exceeds 100 purchases. The 101st customer gets an "invalid" error and assumes the code was fake, escalating to the influencer who escalates to the merchant.

  1. Go to Admin → Discounts → find the discount code
  2. Check "Times used" vs "Usage limit" at the top of the discount detail page
  3. If exhausted, click "Edit discount" → increase the usage limit or remove it (unlimited)
  4. If per-customer limit is the issue, the customer can use a different email to work around it — consider whether this is the intended behavior

Failure Mode 6 — Date and Time Range Issues

Shopify discount date ranges use UTC internally, but the Admin interface displays them in the merchant's local timezone. Timezone mismatches cause discounts to go live earlier or later than intended, or expire before a campaign officially ends.

Common timing failures

  • Discount not live yet: Merchant sets start time to "midnight tonight" in Eastern time, but Shopify launches it at midnight UTC (5 hours earlier in winter) — the discount fires before the intended launch across the store
  • Discount already expired: A flash sale ends "at midnight" but the merchant's intended midnight is Pacific time while Shopify's timezone for the store is UTC — the sale ends 7-8 hours before planned
  • Scheduled sale doesn't start on time: Merchant is in UTC+8 and sets the start to 9am — Shopify interprets this as 9am UTC+8, which is 1am UTC, so the sale is already active by the time the merchant's workday begins
Always Verify Your Store's Timezone Setting
Go to Admin → Settings → General → Store timezone. This is the timezone Shopify uses for discount schedule display. If this doesn't match your expected timezone, the displayed start/end times will be misleading. Always confirm what UTC time the discount is actually scheduled for.

Failure Mode 7 — Customer Segment Eligibility Not Synced

Shopify's customer segments (used for targeted discounts) are computed periodically, not in real time. When a customer becomes eligible for a segment (e.g., they just made their second purchase, qualifying for a "returning customers" segment), there may be a delay before that eligibility is reflected in discount evaluation.

Practical consequences:

  • A "loyalty reward" automatic discount for customers who have placed 3+ orders may not fire on a customer's 3rd order if the segment hasn't updated since their 3rd order was placed
  • A segment-based welcome discount for customers in a specific geographic region may not fire if the customer's location was added to their profile too recently
  • Customers recently imported via a bulk CSV upload may not be reflected in segment counts for several hours

Workaround: for time-sensitive segment-based discounts, use discount codes distributed manually (via email) rather than automatic discounts dependent on segment sync timing.

Failure Mode 8 — Shopify Scripts Deprecated, Complex Discount Logic Breaking

Shopify Scripts is a Plus-only feature that allows merchants to run Ruby code inside Shopify's checkout to customize pricing, shipping, and payment logic. Scripts are the mechanism behind complex tiered pricing, custom BOGO logic, bundle discounts, and percentage-off-after-threshold rules on Shopify Plus.

Shopify is deprecating Scripts in favor of Shopify Functions — a WebAssembly-based system with higher performance limits. The deprecation timeline means any Plus store still relying on Scripts for checkout discounting needs to migrate.

Signs your Scripts are breaking

  • Tiered pricing that worked for years suddenly applies the wrong rate or doesn't apply at all
  • Complex BOGO logic with custom eligibility rules stops triggering correctly
  • Bundle discount that required custom logic no longer fires at checkout
  • Checkout times out intermittently, with errors in the Shopify Scripts execution log

What you need to do

  1. Go to Admin → Settings → Checkout → Scripts (Plus only) — review which scripts are active and what logic they contain
  2. Audit each script's function: is it a line item discount, shipping rate filter, or payment method filter?
  3. Contact your Shopify Plus merchant success manager or a Shopify Functions developer to scope the migration
  4. Shopify Functions are deployed via the Shopify CLI as app extensions — they require a Partner account and an app to host the Function
  5. Test the migrated Function in a development store before deploying to production
Shopify Functions Require Plus — Third-Party Discount Apps Are Not Equivalent
Shopify Scripts replacement via Shopify Functions is a Plus-only capability. If you're a non-Plus merchant trying to replicate Scripts-level discount logic, third-party apps are the only path. Be aware that app-based discount logic has the limitations described in Failure Mode 4 — particularly around express checkout bypass and the one-discount-per-order limit.
