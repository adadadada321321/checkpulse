Shopify Headless Checkout Error
Hydrogen & Storefront API Fixes

Updated June 2026 · 9 min read · By CheckPulse · Standard checkout not working? · App conflict? · Redirect loop?

Headless Shopify storefronts — built on Hydrogen, Remix, Next.js Commerce, or a custom Storefront API integration — introduce a distinct class of checkout failures. The standard Shopify Admin gives you no visibility into them. HTTP uptime monitors stay green. Shopify's status page is clean. And your checkout is silently broken.

This guide covers the six most common causes of headless checkout errors and the exact diagnostic and fix steps for each.

Key difference from standard checkout

In a standard Shopify store, checkout errors often surface as visible error messages in Shopify Admin or in the customer-facing checkout UI. In a headless storefront, the Storefront API returns HTTP 200 even when the mutation fails — errors live inside the GraphQL response body under the userErrors field, which your frontend code may not be logging or displaying.

ROOT CAUSESSix reasons headless checkout breaks

01 Storefront API token invalid or missing checkout scope
Symptom: all checkout API calls return 401 Unauthorized or "Access denied"

Your headless storefront authenticates Storefront API requests with a public access token. If this token is revoked, regenerated, or missing the checkout permission scope, every cartCreate, checkoutCreate, or checkoutLineItemsAdd call silently fails.

Diagnosis: open the browser Network tab, attempt to add to cart and proceed to checkout. Look for any Storefront API request returning 401 or a JSON body containing "message": "Access denied".

Fix: Shopify Admin → Settings → Apps and sales channels → Develop apps → your app → API credentials. Verify the Storefront API token has: unauthenticated_write_checkouts, unauthenticated_read_checkouts, unauthenticated_read_product_listings, unauthenticated_read_product_inventory. If scopes are missing, add them and regenerate the token. Update your environment variable and redeploy.
02 Checkout domain mismatch (Hydrogen PUBLIC_CHECKOUT_DOMAIN)
Symptom: checkout redirects to a blank page, 404, or redirect loop

Hydrogen generates the checkout URL using the Cart API's checkoutUrl field. This URL uses the domain set in PUBLIC_CHECKOUT_DOMAIN. If this doesn't match the primary domain in Shopify Admin → Settings → Domains, the redirect goes to the wrong host.

Diagnosis: in your browser's Network tab, find the checkout redirect and inspect the Location header. Does the domain match what's in Shopify Admin → Settings → Domains?

Fix: in your Hydrogen .env: 
PUBLIC_STORE_DOMAIN=yourstore.myshopify.com
PUBLIC_CHECKOUT_DOMAIN=www.yourcustomdomain.com
PUBLIC_CHECKOUT_DOMAIN must exactly match the primary domain in Shopify Admin — including or excluding www. If you're on Shopify Plus with a custom checkout subdomain (checkout.yourdomain.com), use that as PUBLIC_CHECKOUT_DOMAIN.
03 GraphQL mutation userErrors not surfaced (silent cart failure)
Symptom: cart appears to work but checkout never opens, or checkout page is empty

The Storefront API returns HTTP 200 even when cartCreate or checkoutCreate fails — the error lives inside the response JSON under userErrors. If your frontend code doesn't check and surface this field, the failure is completely invisible.

Diagnosis: in the Network tab, find the Storefront API GraphQL request (POST to yourstorename.myshopify.com/api/2024-10/graphql.json). Inspect the response body. Does the cartCreate or checkoutCreate object contain a non-empty userErrors array?

Common userErrors codes:
  • INVALID — invalid field value (malformed address, bad variant ID)
  • BLANK — required field missing
  • CART_DOES_NOT_MEET_DISCOUNT_REQUIREMENTS_NOTICE — discount not applicable
  • MERCHANDISE_NOT_APPLICABLE — variant archived or not in current publication
Fix: check and log userErrors in every mutation. If userErrors.length > 0, do not redirect to checkout — display the error to the customer or log it server-side for debugging.
04 CORS blocking Storefront API requests
Symptom: browser console shows "CORS policy" error; API call fails with network error

Shopify's Storefront API supports CORS for browser-side requests, but only from domains added as "Headless channels" in Shopify Admin. If your headless domain is not registered, the browser blocks the preflight request.

Diagnosis: browser console shows: Access to fetch at 'https://yourstore.myshopify.com/api/...' from origin 'https://yourheadlessdomain.com' has been blocked by CORS policy.

Fix: Shopify Admin → Settings → Apps and sales channels → Add apps → Headless → Add your storefront domain(s). Each domain (including staging and preview URLs) must be added separately. Note: Hydrogen's server-side Storefront API calls are not affected by CORS — only browser-side fetch calls (typically used in Remix loaders running client-side or in client components).
05 Payment method not enabled for headless / Storefront API
Symptom: no payment methods shown at checkout, or specific methods (Shop Pay, PayPal) missing

Most payment methods work at the hosted Shopify checkout regardless of whether the storefront is headless — the checkout page itself is served by Shopify. However, specific configurations can cause payment methods to disappear for headless storefronts.

Common causes:
  • Checkout domain mismatch (see cause 02): if the customer never reaches the actual Shopify hosted checkout, no payment methods show. This is the most frequent reason.
  • PayPal Express not enabled for Storefront API: go to the PayPal app in Shopify Admin → check for a "Headless" or "Buy Button" toggle.
  • Shopify Payments suspended: check Admin → Settings → Payments for any banner — see full guide.
  • Test mode active: if your store is in test mode but using real payment methods, or vice versa, payment methods may be hidden.
06 Wrong or missing environment variables (Hydrogen / Next.js)
Symptom: checkout works locally but breaks in production (or vice versa)

Headless storefronts depend on environment variables for API tokens, store domains, and checkout domains. A missing or mismatched variable in your production deployment is one of the most common causes of "works on my machine, broken in prod" checkout failures.

Required Hydrogen variables: 
PUBLIC_STORE_DOMAIN=yourstore.myshopify.com
PUBLIC_STOREFRONT_API_TOKEN=your_token_here
PUBLIC_CHECKOUT_DOMAIN=www.yourcustomdomain.com
PRIVATE_STOREFRONT_API_TOKEN=your_private_token  # server-side only
Diagnosis: in your Cloudflare Pages / Vercel / Oxygen deployment dashboard, compare the environment variables against your local .env. A blank value is equivalent to the variable not being set — the app won't throw an error at startup, it will just make broken API calls silently.

Fix: add all required variables to your deployment platform's environment configuration. For Cloudflare Pages: Settings → Environment variables. For Vercel: Settings → Environment Variables. Redeploy after updating.
